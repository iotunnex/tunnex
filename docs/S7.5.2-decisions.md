## Grounding (what this attaches to — all verified, file:line in the seam report)
- **Write target = `group_members`** (S7.1, migration 0018). Composite FK `(org_id, user_id) →
  memberships(org_id, user_id) ON DELETE CASCADE` (0018:42): a synced member is **auto-purged

- **Push is FREE if we go through `policy.Service.mutate`** (service.go:521): `AddGroupMember` /

- **Compiler is UNCHANGED:** group → users → *their owned devices' /32s* (compiler.go:131). It

  status-freeze + `cliauth.SweepUser` + `revoker.DeleteAllForUser` (sessions) + `PushOrgNodes`

  story"* (sso/service.go:172) — this IS that story's attach point. **No IdP `groups` claim/scope
  is requested today** (only `openid email profile`); reading groups needs a new scope+claim OR a
  Directory/Graph API call — **neither exists yet.**
- **Zero SCIM anything** — greenfield.
- **Guards auto-arm:** nil-port + `//go:build enterprise` edition gate (→ 403 `edition_required`);
  `authorize(ctx, org, PermPolicyManage)` (owner+admin, the group-management grain); querylint
  forces `org_id` on every new query by default. Reuse all three; add NO new permission (group
  sync IS policy management).

## The load-bearing choice — MECHANISM (this sizes the whole story)
"IdP groups as policy subjects" can arrive three ways; they are NOT equivalent on the standing
law (*a stale group silently granting access is the failure mode*):

| Mechanism | Staleness bound | Providers | New surface |
|---|---|---|---|
| **A. Login-time `groups` claim** | UNBOUNDED (until the user re-logs-in) | Entra only (Google has no groups claim; Entra >150 groups → overage → Graph anyway) | small |
| **B. Directory POLL + reconcile** | bounded = poll interval | Entra (Graph) + Google (Directory) uniformly | a poller + app-credential + health |
| **C. SCIM push** | near-real-time (event) | Okta/Entra strong; Google weak | a full SCIM 2.0 server + IdP-side config |

**A is rejected as the primary:** login-time claim leaves a user *removed from a group* holding
that grant for their whole session lifetime — the exact "stale group silently granting" failure
the law forbids, and it can't cover Google at all. **C (SCIM) is deferred** (argued in D4).
**LEAN = B: a directory poll + reconcile.** Bounded staleness, provider-uniform, and it rides
the existing push+compiler+deprovision so the *new* code is small.

## Decide-items (founder disposition before code)

### D1 — IdP-authoritative vs merge (conflict rules, STATED not assumed) — DECIDED (disjoint, schema-enforced)
A `user_group` is EITHER `manual` OR `idp_sync` — **never both** (a per-group `origin`), enforced
at the **schema/API layer, not by convention**.
- For an **`idp_sync` group, membership is IdP-AUTHORITATIVE**: the reconcile is the sole writer;
  the API **REJECTS (409) a manual add/remove** of its members. "Sync, not mirror" = we do not
  two-way-merge; we mirror IdP→Tunnex one way and refuse local edits that the next sync would
  stomp anyway.
- A **`manual` group stays fully manual** (unchanged S7.1 behavior).
- The **mapping** (which IdP group → which `user_group`) is a deliberate MANUAL admin act (an
  admin picks the IdP group + names/creates the Tunnex group). The *membership* is what syncs.
- **BIND-TIME EDGE (stated):** binding an EXISTING manual group to sync is **refused unless the
  group is empty** (409) — v1 does NOT convert-and-reconcile a populated manual group (simpler +
  more legible: no ambiguous "were these members yours or the IdP's?" reconciliation). The admin
  empties/recreates, or maps to a fresh group.
  - **ENFORCEMENT LAYER (carry, answered):** the schema shape CHECK cannot cross tables (a
    `user_groups` CHECK can't count `group_members`), so refuse-unless-empty is **APP-LAYER — the
    slice-4 bind handler owns it, and slice 4 OWES A RED** proving a populated manual group cannot
    flip to `idp_sync` (409), alongside the 409 on manual-editing a synced group. The schema
    enforces the disjoint SHAPE (origin ⇄ idp fields); the app enforces the empty-at-bind rule.
- **Rename/delete:** deleting the Tunnex group (or un-mapping) cascades its `group_members` +
  drops its `policy_rules` (existing FKs) — clean. Deleting the group in the IdP → the reconcile
  empties its membership (0 members = 0 grants) + surfaces "source group gone" on health.

### D2 — mechanism + cadence + "sync failed" legibility — DECIDED
Poll, interval = **10 min** (reuse the S7.5.1 `RetentionSweepInterval` ticker pattern) **with ±
jitter** (randomize each org's tick so N orgs don't hammer Graph/Directory on the same instant —
a thundering-herd guard at scale). Bounded staleness = the interval. Reconcile diffs IdP membership
vs `group_members` (origin=`idp_sync`) and applies add/remove **through `policy.Service.mutate`**
(free <5s push).
- **Fail-STATIC + LOUD** (fail-closed REJECTED: an IdP outage silently revoking live access is a
  worse breach than bounded staleness). On sync failure the last-known membership PERSISTS.
- **TWO-TIER alarm** (not one): per-org sync-health trips **IMMEDIATELY on ANY poll failure**
  (`last_sync_ok=false` + `last_sync_error`), and a **distinct ESCALATED state past the ceiling**.
  **Ceiling = 3× the poll interval (= 30 min at a 10-min poll)** — defined RELATIVE to the
  mechanism, not a wall clock. Both states are on the enterprise health surface + audited, mirroring
  the S7.5.1 `retention_failed` legible-boolean pattern (never a dead-green field).
- **Honesty line (docs + UI):** *"membership reflects the directory as of the last successful sync;
  a failed sync is surfaced, never silent."*

### D3 — deprovision semantics (trigger, latency, deliberate-red) — DECIDED (include full sweep)
Two cases, both delivered by REUSING proven primitives:
- **(a) removed from a synced group** (still an org member): the next reconcile deletes their
  `group_members` row via `mutate` → compiler drops their device /32 → gateway forward chain no
  longer grants → **deny**. **Latency = ≤ poll_interval + <5s push.**
- **(b) DISABLED / removed in the IdP** (full deprovision): the poll detects the user is gone from
  the directory and calls the **existing `DeactivateMember`** (sessions cut + CLI creds swept +
  devices out of desired state + org-wide push). **Latency = ≤ poll_interval + the DeactivateMember
  sweep.** This satisfies the anchor ("a deprovisioned user gets the full S2.6/S7.2 sweep").
- **DECIDED: v1 INCLUDES case (b)** (IdP-disable → the existing `DeactivateMember`). Deferring it
  would ship a half-deprovision; the primitive exists + is wire-proven, and the poll already holds
  the directory state, so detecting disabled + calling it is cheap.
- **TWO DELIBERATE-REDS (blocking):**
  - **Red 1 — group removal:** a user in a synced group with a live grant → remove from the IdP
    group → within ≤ interval+push, prove the `group_members` row is gone, PG shows it, and the
    gateway forward chain no longer carries their /32 (**deny on the wire**).
  - **Red 2 — full disable:** a user disabled in the IdP (or a fake directory) → within ≤ interval,
    prove the **full sweep**: session cut, device out of desired state, org-wide push fired, grant
    dead on the wire. Access dies end-to-end, wire-proven — not asserted.

### D4 — SCIM in or out (with the argument) — DECIDED (out; register S7.5.2b)
**OUT of v1 — register S7.5.2b (SCIM 2.0 inbound provisioning).** (NOT S7.5.3 — that number is
already device-posture checks; avoid the collision.) Argument:
- Poll (D2/D3) already delivers group-sync + deprovision at bounded latency **without** a SCIM
  server. SCIM's win is *event-push* (sub-poll-interval latency) — a latency refinement, not a new
  capability.
- SCIM is a distinct, sizable surface: a SCIM 2.0 server (Users + Groups resources, bearer auth,
  `filter`, PATCH ops) + per-IdP admin config. Its *create-user* half **overlaps** the existing
  invite/JIT provisioning and would need its own reconciliation with those.
- **Google's SCIM support is weak** — SCIM wouldn't even cover the Google half well, whereas poll
  (Directory API) does.
- **Non-walling:** poll-v1 and SCIM-later share the SAME reconcile core (`group_members` through
  `mutate` + `DeactivateMember`). SCIM just becomes a **second INPUT** to that reconciler — v1 does
  not architecturally exclude it. **Trigger for S7.5.2b:** a customer mandating SCIM, or needing
  provisioning latency below the poll interval.

### D5 — schema seam (same tables + origin marker; NOT a parallel model)
**LEAN:** extend the existing tables; add ONE config table. (One migration.)
- `user_groups`: add `origin text NOT NULL DEFAULT 'manual' CHECK (origin IN ('manual','idp_sync'))`
  + `idp_provider text NULL` + `idp_group_id text NULL`, with `UNIQUE (org_id, idp_provider,
  idp_group_id)` (one Tunnex group per IdP group). A `user_group` row IS the mapping — **no
  separate mapping table.**
- `group_members`: add `origin text NOT NULL DEFAULT 'manual' CHECK (...)` so the reconcile only
  ever touches its own `idp_sync` rows and can never stomp a hand-added member (belt over the
  disjoint-group rule).
- **NEW `idp_sync_configs`** (per-org, per-provider): the directory app credential (client id +
  **sealed** secret / cert — reuse the `sso_configs` AES-GCM sealing pattern), `enabled`, and the
  `last_sync_*` status. `org_id → organizations ON DELETE CASCADE`; `UNIQUE (org_id, provider)`.
- **Cascade implications (S7.1 FKs, unchanged):** the `group_members→memberships` composite FK
  keeps auto-purging synced members on deprovision (reused, not rebuilt); deleting a `user_group`
  cascades its members + `policy_rules`. **Compiler untouched.** No `deleted_at` on these tables
  (policy objects hard-delete) → querylint needs org-scoping only, no soft-delete filter.

### D6 — edition-gating + auto-arming the guards
**LEAN:** a nil `idpSyncPort` interface + `idp_sync_wire_open.go` (`//go:build !enterprise`, returns
nil) → every handler `if s.idpSync == nil { return editionRequired() }`. The enterprise impl lives
behind `//go:build enterprise` in `internal/enterprise/idpsync/` (open build never links it).
- New endpoints (map/un-map an IdP group ↔ Tunnex group; set the directory credential; trigger a
  manual re-sync; read sync health) → `authorize(ctx, org, PermPolicyManage)` for writes,
  `PermPolicyView` for reads. **REUSE `PermPolicyManage` — add NO new permission** (group sync is
  policy management; avoids an rbac-policy.json regen + a new guard to arm).
- querylint auto-arms `org_id` on the new tables/queries (default-protected). The `authorize`
  401→404→403 fail-closed walk applies for free. The directory secret is sealed-at-rest (S0.3
  master key), never returned (one-time on set), mirroring `sso_configs`.

## SIZING — materially smaller than S7.5.1? (standing scope-signal check)
**Rides existing infra** (no new invention): the <5s push (via `mutate`), the compiler, the
deprovision sweep (`DeactivateMember`), secret sealing (`sso_configs` pattern), the legible-failure
health pattern (`retention_failed`), the interval poller (`RetentionSweepInterval` pattern), and the
edition-gate + querylint + RBAC auto-arms. **Control-plane only — no kernel, no agent-side, no
on-disk durability (the tar-pit that made S7.5.1 huge is absent).**

**NEW code:** 1 migration (3 columns + 1 table) · the **IdP directory client** (the real new
integration cost) · a reconciler · a 10-min poller · a health signal · ~4 small endpoints · the
deprovision deliberate-red.

**SCOPE SIGNAL (flagged, per the standing note):** the one place this can sprawl toward S7.5.1
scale is the **directory-client integration** — Entra Graph (`/groups/{id}/members`, an app
registration + admin-consented `GroupMember.Read.All`) and Google (Directory/Cloud Identity, a
service account + domain-wide delegation) are *two different* OAuth-app + directory-API surfaces.
**Doing BOTH providers in v1 doubles the only expensive part.**

**DECIDED: Entra-only v1, Google as a fast-follow.** RIDER — the reconciler talks to a
**`DirectoryProvider` INTERFACE from day one** (Entra is the first implementation): `ListGroupMembers`,
`ResolveUserStatus` (active/disabled/gone), etc. Google is then a **second implementation, not a
refactor** (the S8.1 transport-enum discipline — name the seam up front so the second provider
plugs in). The reconcile core (`group_members` through `mutate`, `DeactivateMember`, the health
signal) is provider-agnostic behind this interface. Both-providers-v1 + SCIM-in would re-inflate to
S7.5.1 scale — that would be the scope signal to call, not absorb.

## DISPOSITION (2026-07-15) — all six DECIDED, one story, build slice-by-slice
- **D1** disjoint manual-XOR-idp_sync, schema/API-enforced; bind-an-existing-manual-group = refuse-
  unless-empty (409); manual edit of a synced group = 409.
- **D2** poll 10 min **+ jitter**; fail-STATIC + loud; **two-tier** alarm (immediate on any failure +
  escalated past the **3× = 30 min** ceiling); honesty line to docs/UI. (fail-closed rejected.)
- **D3** includes IdP-disable → the existing `DeactivateMember` full sweep; **two deliberate-reds**
  (group-removal death + full-disable sweep, both wire-proven).
- **D4** SCIM OUT → registered **S7.5.2b** (second input to the same reconciler).
- **Providers** Entra-only v1 + Google fast-follow, behind a **`DirectoryProvider` interface from
  day one** (Google = a second impl, not a refactor).
- **D5/D6** same tables + `origin`/`idp_group_id` markers + sealed `idp_sync_configs`; compiler
  untouched; nil-port edition gate + `PermPolicyManage` (no new permission) + querylint auto-arm.
- **Numbers:** poll = 10 min (jittered); staleness ceiling = 3× poll = 30 min.

SCIM 2.0 inbound provisioning (Users + Groups, bearer, filter/PATCH) as a SECOND input to the
S7.5.2 reconcile core. Trigger: a customer mandating SCIM, or provisioning latency below the poll
interval. Google SCIM is weak → poll remains Google's path even after S7.5.2b.

## DoD (when built, per the leans above)
- An admin maps an Entra group → a Tunnex `user_group`; within ≤ poll interval its members' devices
  are granted by the existing compiler (wire-observed via the S7.5.1 flow-log allow event carrying
  the rule that points at the synced group).
- Removing a user from the IdP group kills their grant end-to-end within ≤ interval + <5s
  (deliberate-red, wire-proven deny).
- Disabling the user in the IdP triggers the full `DeactivateMember` sweep (sessions + devices +
  org-wide push) within ≤ interval (deliberate-red).
- A sync failure is LOUD on the health surface (`last_sync_ok=false` + error + stale) + audited;
  never a silent stale grant.
- A manual edit of a synced group is refused (409); the schema `origin` markers keep synced +
  manual rows disjoint.
- Open build 403s every new endpoint; enterprise gate + `PermPolicyManage` + querylint org-scoping
  hold; the directory secret is sealed + never returned.

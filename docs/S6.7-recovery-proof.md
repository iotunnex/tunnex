# S6.7 (Story B) — RECOVERY-FIRST proof runbook (run BEFORE trusting persistent arming)

This proves the escape hatch un-wedges a GENUINELY dead box, and that reboot + re-arm self-heal
recover — the recovery net that must be green before the persistent arming touches the live
connect path. Nothing here uses the app or a tunnel: it arms a persistent block-all directly
(`--wfp-arm-test`), confirms networking is DEAD, then recovers. The S6.9 guard is unchanged
(Windows full-tunnel still refused); the live backend still arms via the old dynamic session.

Run everything in an **ADMINISTRATOR** PowerShell on the Windows box. The helper is at the
install dir (from the branch installer), e.g. `C:\Program Files\Tunnex\resources\helper\`. `cd`
there first, or use the full path. (`README-RECOVERY.txt` is in that same folder.)

## 0. Baseline — networking works
```
ping -n 2 1.1.1.1        # replies
```

## 1. DELIBERATE-RED the escape hatch (THE proof that matters)
```
.\tunnex-helper.exe --wfp-arm-test      # arms a PERSISTENT block-all → networking BLOCKED
ping -n 2 1.1.1.1                        # MUST time out / fail  (box is genuinely wedged)
netsh wfp show state | findstr /i tunnex   # shows the Tunnex provider + the recovery hint
.\tunnex-helper.exe --wfp-clean          # the escape hatch
ping -n 2 1.1.1.1                        # MUST reply again → networking RESTORED
```
- **PASS = ping DEAD after `--wfp-arm-test`, then ALIVE after `--wfp-clean`.** This is the whole
  point: the hatch un-wedges a box whose networking was actually dead. (If ping still worked
  after `--wfp-arm-test`, the persistent block didn't arm — tell me; do NOT proceed.)

## 2. REBOOT-recovery (no manual command)
```
.\tunnex-helper.exe --wfp-arm-test      # wedge again
ping -n 2 1.1.1.1                        # dead
# --- REBOOT the machine now (do NOT run --wfp-clean) ---
# after it boots back up, in a new admin PowerShell:
ping -n 2 1.1.1.1                        # MUST reply → the auto-start service's CleanStale cleared it
```
- **PASS = networking is back after a plain reboot, with NO manual cleanup.** Proves "a bug means
  reboot-to-recover."

## 3. RE-ARM self-heal (no ALREADY_EXISTS strand)
```
.\tunnex-helper.exe --wfp-arm-test      # arm
.\tunnex-helper.exe --wfp-arm-test      # arm AGAIN — must succeed (cleans then re-arms), not error
ping -n 2 1.1.1.1                        # still dead (one clean block, not a broken half-state)
.\tunnex-helper.exe --wfp-clean          # recover
ping -n 2 1.1.1.1                        # alive
```
- **PASS = the second `--wfp-arm-test` succeeds** (EnableFirewall cleans stale first) and a single
  `--wfp-clean` fully recovers — no orphaned/duplicate block.

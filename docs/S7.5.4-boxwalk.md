## Leg 0 — baseline (`leg0-baseline-forward.txt`)  · PASS
Both gateways' forward chain = default-deny only (`ct established accept` + `tunnex_default_drop`), no
owner allow. dev-A probe to `10.80.9.1` → 100% loss; after routing walk-dst through the tunnel, A's
`default_drop` counter = **3 packets / 252 bytes** (the blocked probe reached the forward chain). Exclusion
real BEFORE any grant.

## Leg 1 — per-user grant on the wire + exclusion + F1 org-wide strip + re-grant  · PASS
Files: `leg1-after-grant.txt`, `leg1-after-strip.txt`.
- **Grant** owner→walk-dst → allow `10.99.0.3 → 10.80.9.0/24` on **A** AND `10.99.0.2 → 10.80.9.0/24` on
  **B**. Push latency **Δ = 0.19s** (≪5s).
- **Traffic (flood):** PERMIT dev-A→walk-dst → allow counter climbs `7,53,99,146,192`; EXCLUSION
  dev-A→`10.99.0.50` (non-granted tunnel IP) → `default_drop` climbs `10,56,102,148,194`.
- **F1 org-wide strip:** DELETE the rule → allow **vanishes from A AND B** (default-deny only),
  **Δ = 0.33s**. The org-wide push reaches the non-observation gateway B.
- **Re-grant-on-wire:** re-create the same grant → allow returns on A.

## Leg 3 — downtime lapse (stateless sweep closes the audit gap)  · PASS
File: `leg3-after-restart.txt`. Temporary grant created, API stopped **while the grant was still valid**
(sweeper is in-process → down across the expiry). Past expiry, API down: allow **lingered** on A
(bounded, safe-additive direction) + row present-but-expired + **0 audits yet**. On restart, TWO
mechanisms (see decisions §two-mechanism):
- **compiler active-rules filter** strips enforcement on the post-restart refetch — **≈3s** (fast).
- **stateless sweeper** DELETEs the row + writes the `grant_expired` audit on its next tick — **+62s**;
  `row_remaining=0`, audit then present. The stateless sweep (no window cursor) picks up the grant that
  expired DURING downtime → audit-gap closed by construction, nothing lost.

## Honesty ledger

- **`walk-dst` is dark** (no host listens) — replies never return (`ping` exits 1). The accepted
  observation is **forward-chain counter movement** (allow vs `default_drop`), i.e. the packet's VERDICT
  at the enforcement hook, not a round-trip. The forward chain is the compiled+pushed artifact in the
  kernel — watching its allow-set + counters IS the wire proof.
- **Counter resets:** a trailing ruleset listing showing `0` counters after a flood is a **fresh listing
  after a push/re-apply reset the chain**, not evidence the flood never happened — the in-flight climbing
  samples are the datum.
- **dev-M first-create `401`:** the walk script `cd`'d into `scratch/` then referenced the cookie jar by
  a repo-root-relative path → `curl` sent no cookie → 401. Retry from repo root = 201. **Client-side
  script path bug; the API behaved correctly (no 5xx).** Recorded per the "no reassuring-green" rule.
- **gw-b is ruleset-only** — dev-B has no wire; Leg-3b is proven by **ruleset observation** on B (allow
  appears then strips), not traffic through B. That is the entire intended proof for the non-observation
  gateway.
- **Source-node bucketing:** allows bucket onto the **source device's node** (`compiler.go add(d.NodeID)`,
  unchanged since S7.1). "Non-hosting gateway" (S7.3 wording) = non-hosting **of the destination**. The
  Leg-3b discriminator is "the sweep's `pushOrg` reaches every gateway hosting a **source** projection of
  the lapsed grant" — gw-b qualifies via dev-B.
- **Shared compose-DB hazard:** the Go integration suite + e2e point `TUNNEX_TEST_DATABASE_URL` at the
  compose postgres — running `make test*`/`e2e` against a live box-walk stack **wipes + repollutes the
  fixture** (hit twice mid-walk: 716 leaked orgs). The D8 fix was verified against an ISOLATED throwaway
  postgres. **Do not run gates/tests against a live walk stack.**
- **Phantom seed node `010`** (`demo-gateway`, `has_endpoint=f`): in the push set (`ListActiveNodeIDsForOrg`
  = all active nodes; over-notify is explicitly safe) but never reports → its pushes no-op and its
  projection for the owner grant is empty (its strandable device was revoked for clean attribution).
- **Pool/CIDR:** device pool = `10.99.0.0/24` (masquerade scoped to it — internet-egress NAT only,
  orthogonal to the forward-chain filter). `walk-dst 10.80.9.0/24` is a dark resource CIDR routed through
  the tunnel (spoke `allowed-ips` extended) purely so probes transit the forward chain.
- **Atomic apply:** the node applies its whole ruleset in ONE atomic `nft -f -` transaction — any
  re-apply swaps old→new atomically, no enforcement gap. Load-bearing (see decisions §atomic-apply).
- **v3 = safe-by-convention:** `ProtocolVersion` bumped 1→2→3 with no consumer fail-closed gate; safe ONLY
  because every bump is observability-additive + hash-blind + safe-ignore (each unit-pinned). The gate is
  a BLOCKING S8.1 item (PLAN.md, decisions §D5-reaffirmed). A convention holding is not a guarantee.

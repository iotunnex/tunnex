# S3.7 Gateway NAT + forwarding (full-tunnel egress) — decisions (commit-one, decision-first)

Makes `full_tunnel` actually reach the internet: the node-agent enables IP forwarding +
source-NAT on the gateway so client traffic egresses via the gateway host. Today the config
connects but egress dies at the gateway (split-tunnel only works). **Pulled forward** (user
request, 2026-07-10) to make full-vs-split a real, testable customer choice — the parked
trigger was "EPIC 6 close OR beta"; resuming with this decision review, then build.

**Server/gateway-side only** — `apps/node/` (the agent) + a small API change (device-create
refusal + a `nodes` capability). No client/helper change: the S6.4 client already maps the
`gateway_no_egress` error and the full-tunnel toggle already exists.

This ADOPTS the PLAN "S3.7 paper decision" (5 points) and RESOLVES its 3 open sub-questions.

---

## Adopted from the paper (unchanged)

1. **Privilege posture — no new privileges; DETECT the host capability.** No `privileged: true`.
   The agent has `CAP_NET_ADMIN` in its own netns; `ip_forward` is a per-netns sysctl it can
   write, but source-NAT needs the HOST kernel's `nf_nat`/`nf_conntrack` + masquerade — a cap
   can't grant that. So the agent **PROBES** whether egress NAT is achievable and **degrades
   gracefully** on a locked-down host (reports the capability as false; never crashes).
2. **nftables, native (netlink) — not iptables.** The agent manages its OWN nft table
   (`tunnex`), atomically replaceable, scoped to the org pool source CIDR → egress iface. No
   shelling to `wg-quick` PostUp; no blanket rules. (Node image has neither nft nor iptables →
   the Go netlink library avoids adding a binary; see decision below.)
3. **Per-gateway capability + `full_tunnel` REFUSE (not warn).** The agent reports an egress
   capability up the existing report channel; a device create with `full_tunnel=true` against a
   gateway that lacks it is **REFUSED server-side** (typed `gateway_no_egress`) — a silently
   blackholing full tunnel is worse than a clear refusal. Split-tunnel always allowed.
4. **Desired-state reconcile + full-sweep teardown.** NAT rules are data-plane state →
   re-asserted on the S3.1 reconcile interval (heals a flushed table), never assumed. Revocation
   is a full sweep: gateway revoked (or its last full-tunnel peer gone) → the NAT table is torn
   down, no dangling masquerade.
5. **Egress e2e is the proof obligation.** A compose "internet" target reachable by the gateway
   but NOT directly by the client; a full-tunnel client reaches it ONLY through the tunnel (real
   WG + real NAT). Deliberate-red: flush the masquerade → egress fails (proves the rule carries
   it, not a leak). Negative: full-tunnel create against a no-capability gateway → `gateway_no_egress`.

---

## Open sub-questions — RESOLVED

- **(a) Capability storage → JSONB `capabilities` on `nodes`, not a boolean.** Gateway
  capabilities WILL grow (IPv6 egress, on-gateway DNS, site-to-site routing — EPIC 8). A
  `capabilities jsonb` map (`{"egress_nat": true}`) is one migration, forward-compatible; a bare
  boolean column would need a new migration per future cap. Read as `capabilities->>'egress_nat'`.
- **(b) Probe cadence → EVERY reconcile.** Host state can change under us (a kernel module
  unloaded, `ip_forward` reset, an ops change). The agent re-probes each reconcile and reports
  the current value; the API's refusal reads the LAST-reported value off the `nodes` row. (Cheap
  — the probe is a dry-run nft add + a sysctl read.)
- **(c) Typed error + edition → core/edition-neutral, `gateway_no_egress`.** Full-tunnel is core
  (like the S3.5 allocator), NOT an enterprise feature — no edition gate. The refusal is the
  typed `gateway_no_egress` the S6.4 client already maps in `friendly()`. The client also reads
  the gateway's `egress_nat` capability to disable/explain the full-tunnel toggle up front (so a
  user isn't refused only at create time).

## Additional decisions (this story)

- **nftables via the Go netlink library** (`github.com/google/nftables`) — consistent with the
  agent already speaking netlink for `wgctrl`; no `nft`/`iptables` binary added to the node image.
  A dedicated `tunnex` inet table with: a `postrouting` masquerade chain (`ip saddr <pool> oif
  <egress> masquerade`) + a `forward` chain (`wg0→egress accept`, `egress→wg0 ct state
  established,related accept`, **`wg0→wg0 accept`**).
- **Device-to-device forwarding is productized here too** (`wg0→wg0 accept`). The manual
  `iptables -A FORWARD -i wg0 -o wg0` we ran live (see `docs/gateway-device-to-device.md`) becomes
  part of the reconciled ruleset — spoke-to-spoke stops being manual/ephemeral. (Policy over who
  may reach whom stays EPIC 7.)
- **`ip_forward` owned by the agent.** The agent sets `net.ipv4.ip_forward=1` (part of the probed
  setup) → **DoD: remove the `sysctls: net.ipv4.ip_forward=1` crutch from `docker-compose.yml`**
  so the agent's own setup is the single source (the paper's "fold the POC line in").
- **IPv6 scope:** full-tunnel `AllowedIPs` includes `::/0`. v1 does **IPv4 masquerade** for
  `egress_nat`; **IPv6 masquerade (NAT66) is best-effort** — added when the host has an IPv6
  default route + NAT66 support, else IPv6 egress is unavailable and that traffic is dropped by
  the client kill-switch (**no leak** — the S6.3 kill-switch already blocks it). `egress_nat` gates
  on IPv4 working (the customer-relevant case); IPv6-egress as a first-class capability is a
  follow-up. Documented as a known limitation, not a silent gap.
- **Probe definition:** `egress_nat=true` iff (i) `ip_forward` is writable/set AND (ii) the
  `tunnex` nft table + masquerade rule can be added AND (iii) `nf_conntrack` is available (a
  masquerade without conntrack won't track return flows). A dry-run add that fails → `false`,
  logged with the reason (diagnosable on a locked host).

---

## DoD
- Agent probes egress capability every reconcile; reports `egress_nat` on the `nodes.capabilities` JSONB.
- Agent reconciles the `tunnex` nft table (masquerade + forward incl. wg0↔wg0) as desired state;
  full-sweep teardown on gateway/last-full-tunnel-peer revocation.
- API refuses `full_tunnel=true` against a no-egress gateway with typed `gateway_no_egress`
  (edition-neutral); client reads the capability to gate the toggle up front.
- **DELETE `scripts/poc-gateway-nat.sh`** + remove the `net.ipv4.ip_forward=1` compose sysctl.
- Proofs: egress e2e (full-tunnel client reaches a gateway-only target through the tunnel);
  deliberate-red (flush masquerade → egress dies); reconcile-heals (flush the table → re-asserted);
  negative (`gateway_no_egress` on a no-cap gateway).

## Non-goals (explicit)
- No per-user/per-resource egress policy (who may egress where) — that's EPIC 7 Zero Trust.
- No on-gateway DNS resolver, no IPv6-egress as a first-class capability (best-effort only), no
  site-to-site routing (EPIC 8). No client/helper change.

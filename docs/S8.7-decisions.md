## Decide-items (D1–D6 ruled; D4 mechanism the one open item, this paper's verify)

**D1 — CIDR source, org-validated (CONFIRMED).** `src_kind='cidr'` (dst symmetry, resource-refs where they
exist). The CIDR is **org-validated at rule creation** — it must be meaningful in the org's address world
(inside an approved site subnet, a declared routed range, or the device pool). An out-of-world CIDR compiles
to a rule matching nothing — a silent no-op wearing a rule's face (the reassuring-rule class). **WARN, not
refuse (RATIFIED):** the pre-staging case (an admin writing rules for a range about to be declared — the
order RouteLAN encourages) makes refusal the wrong grain. A rendered warning (`cidr_outside_org_ranges` on
the rule ROW, not just at creation) keeps the reassuring-rule class visible for the rule's whole life. **The
honesty condition:** the warning derives from CURRENT org ranges at READ time — a rule out-of-world at
creation and in-world after the range lands SHEDS its warning without edits. Red proves both directions
(warning appears, warning clears). The `saddr` emission REUSES the site-src path (one-emitter discipline —
"mirrors proven machinery" made structural), never a parallel CIDR branch.

**D2 — no version bump (CONFIRMED).** The `LocalSubnets`/`DNSForwards` precedent holds: an old agent missing
a narrower allow is missing a PERMISSION — fail-closed-safe. Both faces papered: a CIDR-source grant is an
Allow entry an old agent simply doesn't apply (denies more, never less) → no `RequiredVersion` bump.

**D3 — modal source dropdown (CONFIRMED).** The Access modal's source dropdown gains Resource/CIDR; the
free-text fallback is right (resources are a convenience, not a gate). The D1 validation refusal/warning
renders verbatim per the typed-message convention (the warn-badge on the rule row).

**D4 — scoped conntrack flush on expiry AND deletion, never blanket (CONFIRMED). MECHANISM VERIFY (this
paper, before build):**
- **The reserved seam's real shape:** `flowlog.VerdictTerminated` + `accesslog.DecisionTerminated` + the
  `RuleID` binding on `nodepolicy.AllowEntry` (`flowlog/event.go:20-27`, observability-only, EXCLUDED from
  `CanonicalHash`). A READY CONTRACT — the "terminated" verdict + the revoked grant's RuleID. The kill code
  itself is UNBUILT (grep: no `conntrack -D`, no CT-netlink call anywhere). The agent's enforcement is
  **shell-out** (`egress_linux.go`: `exec.CommandContext(ctx, "nft"...)`, `"ip"...`), NOT direct netlink.
- **Presence-in-image (the deployment truth):** the zero-touch gateway image `deploy/docker/node.Dockerfile:17`
  installs `ca-certificates wireguard-tools iproute2 nftables` — **NO `conntrack-tools`.** A shelled
  `conntrack -D` would SILENTLY NO-OP on every real gateway (the REPORTED≢STORED packaging-tier trap the D4
  ruling names). The shell-out pattern is safe ONLY for binaries the image actually has (nft/ip are present);
  conntrack is not.
- **THE PICK — netlink-via-Go (RECOMMENDED; HELD for founder disposition, per "answered before build"):**
  the packaging verify is the decider. Two paths:
  - *(a) shelled `conntrack -D`* — consistent with the nft/ip pattern, BUT requires ADDING `conntrack-tools`
    to `node.Dockerfile` (a RUNTIME binary dep the compiler cannot verify — a future image rebuild dropping
    it silently disarms the security fix; drift with no guard).
  - *(b) netlink-via-Go* — the flush lives IN the agent binary (compile-verified, no image change), a
    filtered CT-netlink dump+delete scoped to the removed rule's tuple (src/dst/proto). Reuses the
    CAP_NET_ADMIN the agent already holds (nft/wg/egress). Adds ONE Go dep (a CT-netlink lib on
    `mdlayher/netlink`, already an indirect dep). Typed errors, not CLI-output parsing.
  - **Recommendation: (b).** The consistency argument for (a) fails at the packaging tier — shell-out works
    for present binaries, and conntrack is provably absent; (b) removes the unverifiable runtime dependency
    and binds the security fix to the compiled artifact. **HELD for the founder's mechanism ruling before
    the flush is built.** Failure mode (either path): a flush error is LOGGED + surfaced (the rule is still
    removed; the lingering flows are the pre-existing behavior — degraded-not-broken, NEVER silent).

**D5 — one function, two triggers (CONFIRMED).** The scoped flush rides the SAME code path for the
grant-expiry sweep AND manual rule deletion (the sweep gains the flush step; the delete handler calls the
same function). The founder's exact scenario is the walk leg: ping running → grant expires → ping stops
within the sweep interval.

**D6 — device-revocation EXEMPT (CONFIRMED, verified).** Device revocation = PEER REMOVAL
(`devices/service.go:496`) — a crypto-layer stop (the tunnel dies), EFFECTIVE, needing NO conntrack semantics.
The vulnerable class is grant-expiry / rule-deletion where the peer STAYS (device-grant removed but device
connected; site-grant removed but sites still peered). One line, proven not assumed.

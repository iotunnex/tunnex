## Leg 1a — Graph credential probe (proves the read path + reveals the join keys) — PASS

Proves `entra.go`'s live path with these creds: client_credentials token mint, group-members read,
`accountEnabled`→status input, and the `mail=null → userPrincipalName` fallback.

Prep finding (caught by probing FIRST): `GroupMember.Read.All` alone enumerates member ids but
nulls their user attributes — reading `mail`/`userPrincipalName`/`accountEnabled` needs
**`User.Read.All` (Application)** too. Added + admin-consented; then a FRESH token (app-only tokens
bake granted roles at mint time — the stale token kept returning null until re-minted).

```
token length: 2007        # client_credentials mint OK
$ curl .../groups/$GROUP/members/microsoft.graph.user?$select=id,mail,userPrincipalName,accountEnabled
[
  { "id":"4e53e105-f28c-4ff0-81ac-af4dd56a27e0", "mail":null,
    "userPrincipalName":"testuser@iotunnexoutlook.onmicrosoft.com", "accountEnabled":true },
  { "id":"a33bec5e-54ba-46c2-a6e8-47a97fd22bf1", "mail":null,
    "userPrincipalName":"linkuser@iotunnexoutlook.onmicrosoft.com", "accountEnabled":true }
]
```

Join keys (lowercased mail||UPN): `testuser@iotunnexoutlook.onmicrosoft.com`,
`linkuser@iotunnexoutlook.onmicrosoft.com` — both active.

REVIEW LEDGER: `statusOf(nil)` treats a null `accountEnabled` as active — a fail-OPEN if Graph ever
returns it null (e.g. the missing-permission case above). Worth a defensive guard so a permission
misconfig can't silently skip deprovisioning. (Deferred to the story-end review.)

## Leg 4 — two-tier health (degraded → escalated → recover) — PASS

PUT a wrong secret → trigger → the real Graph `401` surfaces; `last_sync_at` stays FROZEN at the


D2 proven: immediate `degraded` on ANY failure (clock frozen at last success), `escalated` past the
3×poll = 30-min ceiling, clean recovery to `ok`. Membership untouched throughout (fail-static).

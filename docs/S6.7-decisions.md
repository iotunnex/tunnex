# S6.7 (Story B) ‚Äî Windows kill-switch PERSISTENCE ‚Äî decisions (decision-first)

The final Windows full-tunnel piece. Today the WFP block is armed on the upstream
`wireguard/windows` firewall, which opens a **`cFWPM_SESSION_FLAG_DYNAMIC`** session
(blocker.go:34) ‚Äî the Base Filtering Engine (BFE) auto-deletes every filter/sublayer/provider
when that session's engine handle closes, and **process death closes it**. So on crash /
`taskkill /F` the block is GONE and traffic reverts to cleartext (pcap-proven leak). macOS pf
is kernel-resident + proven; Windows is not. This makes the WFP block **survive process
death** ‚Äî the property that lets the S6.9 guard lift.

**This is the "death = enforcement" model that stranded macOS twice before it was right.** On
Windows a wrong cleanup path blocks ALL networking with no easy undo, and it can't be fixed
from macOS. So the ordering below is deliberate: **recovery net FIRST, persistent arming
second. A bug must mean "reboot to recover," never "bricked box."**

---

## Scope ‚Äî minimal fork, reuse the proven filter set

The upstream arming (registerBaseObjects ‚Üí permitWireGuardService ‚Üí blockDNS ‚Üí permitLoopback
‚Üí permitTunInterface ‚Üí permitDHCP/Ndp ‚Üí blockAll) WORKS and is live-proven (Story A). We change
**only three things**; the filter set stays byte-identical.

- **VENDOR** `golang.zx2c4.com/wireguard/windows/tunnel/firewall` (v1.0.1, MIT) into
  `apps/helper/internal/wfp/` ‚Äî an exact copy (license header preserved), so the diff vs
  upstream is auditable and pinned. `backend_windows.go` calls `wfp.EnableFirewall` /
  `wfp.DisableFirewall` instead of the upstream package. Rationale: the session flag + the
  GUIDs are created *inside* `createWfpSession`/`registerBaseObjects`, and the delete path
  needs unexported internals ‚Äî none reachable without owning the copy. Reusing the copy (vs a
  rewrite) is exactly Pawan's constraint: change session-lifetime, GUID, and cleanup; nothing
  else. **Any change beyond those three needs an argument in review.**

**Change 1 ‚Äî non-dynamic session.** Drop `cFWPM_SESSION_FLAG_DYNAMIC` from `createWfpSession`
so BFE keeps the objects after the engine handle (and the process) closes. Arm = open engine ‚Üí
begin txn ‚Üí add provider+sublayer+filters ‚Üí commit ‚Üí close. Objects persist in BFE keyed by
our GUIDs, independent of the process. (No held global session needed; `EnableFirewall`'s
"already enabled" check becomes "does our sublayer already exist," or a re-arm that first
cleans ‚Äî see CleanStale.)

**Change 2 ‚Äî FIXED provider + sublayer GUID.** `registerBaseObjects` currently calls
`windows.GenerateGUID()` twice (RANDOM per arm) ‚Äî which is *why* a persisted block can't be
found after a crash (the GUID died with the process). Replace both with our own **fixed GUID
constants** (generated once, committed). These GUIDs are the durable key CleanStale enumerates
by ‚Äî the whole recovery story hinges on them being constant.

**Change 3 ‚Äî explicit enumerate-and-delete cleanup.** `DisableFirewall` currently just
`fwpmEngineClose0` (relies on DYNAMIC auto-delete ‚Äî a no-op for a persistent session). Rewrite
it to: open engine ‚Üí begin txn ‚Üí **enumerate filters under our fixed provider/sublayer and
delete each** ‚Üí delete the sublayer by key ‚Üí delete the provider by key ‚Üí commit. This needs
FWPM syscall wrappers the upstream never added (it never deleted): `FwpmFilterCreateEnumHandle0`,
`FwpmFilterEnum0`, `FwpmFilterDestroyEnumHandle0`, `FwpmFilterDeleteById0`,
`FwpmSubLayerDeleteByKey0`, `FwpmProviderDeleteByKey0`. Idempotent (missing objects = success).
**`CleanStale` == `DisableFirewall`** ‚Äî the same enumerate-and-delete, run at startup before
re-arming; it now actually finds + removes a prior crashed session's persisted objects.

## Condition 1 ‚Äî RECOVERY NET FIRST (build + prove before the live arming)

The persistent block can wedge the whole box. Three recovery layers, built + PROVEN before the
arming is wired into the live backend:

1. **Startup self-heal (primary).** The helper service is auto-start (RunAtLoad); on every
   start it runs `CleanStale` BEFORE serving ‚Äî so any stuck block from a prior crash is removed
   at boot. **A bug ‚Üí reboot ‚Üí CleanStale clears it ‚Üí networking back.**
2. **Standalone force-clean subcommand (the escape hatch you type).** A new
   `tunnex-helper.exe --wfp-clean` runs the SAME enumerate-and-delete against our fixed GUID,
   independent of the service/tunnel. Run from an ADMIN prompt even if the service is dead:
   ```
   "C:\Program Files\Tunnex\tunnex-helper.exe" --wfp-clean
   ```
   (path per the NSIS install dir.) This is CleanStale exposed as a CLI ‚Äî same proven code path.
3. **Inspect:** `netsh wfp show state` dumps all WFP objects (find provider "WireGuard" / our
   sublayer) to confirm what's present. (netsh has no per-filter delete ‚Äî #2 is the remover.)

**Proof order (recovery legs GREEN before the no-leak gate):** arm a persistent block on a test
box, then verify EACH recovers networking: (a) `--wfp-clean` removes it; (b) reboot removes it
(service-start CleanStale); (c) a fresh arm after an un-clean prior arm self-heals (re-arm runs
CleanStale first ‚Äî no ALREADY_EXISTS, no duplicate). Only once all three recover do we trust the
arming. **The doc's escape hatch is #2 ‚Äî that is exactly what you type if the box wedges.**

**Escape hatch DISCOVERABILITY (addition ‚Äî the macOS-stranding lesson: the recovery existed but
was unreachable while locked out).** A blocked box can't reach docs or GitHub, so the literal
`--wfp-clean` command must be present ON the machine and in every locked-out surface:
  - **App UI:** the failed/unsafe tunnel state prints the exact command string (not a doc link).
  - **On disk:** the installer drops `README-RECOVERY.txt` in the install dir containing the exact
    command + one-line explanation (survives with the app even offline).
  - **Windows event log:** the helper writes an Application event-log entry WHEN IT ARMS the
    persistent block, stating the block is active and how to clear it (`--wfp-clean`) ‚Äî readable via
    Event Viewer even with networking dead.
  - **Install docs:** the recovery command is in the Windows install/troubleshooting doc.

**DELIBERATE-RED the escape hatch ‚Äî the proof that matters, BEFORE the first live arm.** NOT
"CleanStale ran without error." Actually WEDGE the box: arm a persistent block, CONFIRM egress is
dead (ping/curl to a public IP times out), THEN run `--wfp-clean`, CONFIRM egress RETURNS. This
proves the hatch un-wedges a genuinely dead box, not just that the code runs. It must pass before
the persistent arming is wired into the live connect path ‚Äî a persistent arm we can't prove we can
undo is a brick risk, not a feature.

Build sequencing that honors "recovery before arming": land the vendored package + the
enumerate-delete cleanup + `--wfp-clean` and prove the recovery legs against a manually/test-armed
persistent block, THEN flip `backend_windows.go` to arm via the persistent path. The escape hatch
ships in the SAME change, so there is never a persistent-arm build without an out.

## Condition 2 ‚Äî the `kill -9` pcap is the gate (Windows twin of the macOS proof)

One proof, and it is the gate. On a real Windows box, full tunnel up (via the Story A dev
bypass), then:
- `taskkill /F` the helper mid-full-tunnel ‚Üí **pcap on the physical NIC shows ZERO cleartext
  egress** while the WFP filters hold (the block survived process death) ‚Üí
- recovery restores networking (reboot / `--wfp-clean`).
- Plus the reboot-recovery leg: hard-reboot mid-full-tunnel ‚Üí after boot the service starts,
  CleanStale clears the stale block, host has normal connectivity (NOT permanently blocked).

**Passing this pcap (no cleartext + clean recovery) is what lifts the S6.9 guard AND removes
BOTH dev bypass flags ‚Äî in ONE atomic change** (delete the backend guard + the server win32
refusal + `TUNNEX_DANGEROUS_WINDOWS_FULLTUNNEL` + `TUNNEX_ALLOW_WINDOWS_FULLTUNNEL` + the
compose passthrough together; `coupling_test` enforces flag-and-guard removal is atomic).
**Nothing lifts on a partial pass** ‚Äî no-leak WITHOUT proven recovery is a bricked box, not a
ship.

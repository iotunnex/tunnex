# S8.2 — Route propagation (site-to-site traffic on the wire) · commit-one, decision-first

**The payoff story of EPIC 8.** S8.1 built the model + the fail-closed version gate + the disjointness
validator. S8.2 is where **traffic actually flows between sites** — Mumbai pings Bangalore, under Zero
Trust, on self-hosted hardware. Nothing here ships until the paper is ruled.

**Provenance.** Items 1–5 are the five ledgered carries pinned in PLAN.md's S8.1 pointer + S8.1's
decisions doc (D3 boundary red, D9 MSS clamp, the interlock refuse-half, the validator-input-filtering
law, the LAN-source/link-routing propagated deltas at `docs/S8.1-decisions.md` §"Propagated deltas"
line 380). Items 6–10 are the story's own questions. Substrate re-verified against the live tree this
sitting (agent + CP), cited inline. This paper is the record; a future session re-enters from it.

**Scope (S8.2).** Propagation + enforcement of site-to-site traffic + the MSS clamp. UI = S8.3, DNS =
S8.4. All dispositions below are **PROPOSED — held for founder ruling.**

**Substrate (verified this sitting — the ground the decisions stand on):**
- `DesiredState` (`apps/api/internal/nodes/service.go:59`) already carries a **`Peers[]`** section
  (topology) SEPARATE from **`Policy*`** (`policyspec.Compiled`). Site-link peers ride `Peers[]`; grants
  ride `Policy`. This split is load-bearing below.
- **Kernel routes: NOT PRESENT** — the agent programs no routes today
  (`apps/node/internal/reconcile/`); reachability is pure WG cryptokey-routing over the pool CIDR (the
  wg0 address route). S8.2 ADDS route-programming.
- Peers rendered via `wg syncconf` (`wgctrl_linux.go:276 ApplyPeers` / `:250 buildSyncConf`) — full-set
  idempotent resync; `[Peer]` = PublicKey, AllowedIPs, Endpoint.
- Forward chain (`egress_linux.go:271`) `policy drop` + `ct established,related accept` (:273) +
  compiled allows. Enforcement is the forward chain; **routes/peers are plumbing**.
- Masquerade (`egress_linux.go:262`): `ip saddr <pool-CIDR> oifname != "wg0" masquerade` — internet
  egress ONLY; wg0→wg0 (site-to-site) is NOT NATed already.
- Source match (`egress_linux.go:370 allowMatch`): `netip.ParseAddr(e.SrcIP)` — a bare HOST; a CIDR
  source is skipped. Dst already uses `ParsePrefix` (:374, S8.1).

## Item 1 — D2 classification of S8.2's new fields + THE VERSION QUESTION · PROPOSED (fork surfaced)

S8.2 introduces two categories of new wire content. Each walks the D2 enforcement-vs-observability
checklist (S7.5.1 A-1); the answers decide the version bump.

**(1a) LAN-source CIDR — the genuinely-new ENFORCEMENT change.** Real site-to-site traffic has a LAN
host as its SOURCE (`10.1.0.5`), not a Tunnex device /32. The grant is `site-A-LAN → site-B-subnet`, so
the source is a **CIDR** (`10.1.0.0/24`). This rides the EXISTING `AllowEntry.SrcIP` string (no new
field — Option-A-style, symmetric to S8.1's `DstCIDR`), BUT the agent's `allowMatch` src parse MUST
widen `ParseAddr → ParsePrefix`. `SrcIP` is IN `CanonicalHash` (`hash.go:16`), so a CIDR value changes
the hash for that grant.

| field | kind | in CanonicalHash? |
|-------|------|-------------------|
| `AllowEntry.SrcIP` now accepts a **CIDR** (site-LAN source), not only a /32 | **enforcement-significant** (widens the source match host→prefix) | YES (`src_ip` already in the projection; the value drives the hash) |
| rule-level `src_kind='site'` + `src_site_id` | **CP-side schema only** — resolved at compile to source CIDRs; NEVER on the wire | N/A (mirrors S8.1 `dst_kind='site'`) |

**(1b) Site-link peers + routes — PLUMBING (reachability, not permission).** The site-link WG peer
(remote gateway) + its AllowedIPs (remote subnets) + the kernel routes the agent programs. These live in
`DesiredState.Peers[]` + a new route surface — **OUTSIDE `policyspec.Compiled` and its hash** — exactly
where device peers already live.

| field | kind | in CanonicalHash? |
|-------|------|-------------------|
| site-link `Peer{PublicKey, AllowedIPs: remote-subnets, Endpoint}` | **plumbing** (which packets the tunnel carries; the forward chain still gates) | NO (topology section, like device peers — never in Compiled) |
| kernel routes for remote site subnets (new agent capability) | **plumbing** (reachability; grant is the enforcement boundary) | NO |
| MSS clamp rule (Item 2) | **plumbing** (node-local nft, masquerade class) | NO |

**Why plumbing is safe outside the gate:** an old agent that ignores a site-link peer field programs the
peer but NOT the route (no route-code) → the remote subnet is **unreachable, never insecure**
(fail-closed by absence). This is *why* Item 3's routed-but-dropped red is load-bearing: the route is not
the permission.

**THE VERSION FORK (argue, don't inherit — held for ruling):**

- **Option A — BUMP `ProtocolVersion 4→5` (RECOMMENDED).** The LAN-source widening is enforcement-
  significant; bumping fires the S8.1 D1 gate so an old (max=4) gateway **refuses loudly** (deny-all +
  `unsupported_policy_version`) rather than **silently SKIPPING** the LAN-source grants while reporting
  healthy. Silent-under-enforce-while-green is the EXACT reassuring-green hazard the whole S7.4b/D1 arc
  exists to kill — and S8.2's LAN-source IS the "first enforcement-significant field" D1 was built at
  S8.1 to meet. Item 4 (the refuse-half proof) then fires on this real event. Twin goldens move; the
  old-hex→new-hex delta is attributable to `version` alone (Option-A: no new hashed field beyond the
  CIDR values).
  - **KNOWN + ACCEPTED blast radius (name it, per D1's A-4 discharge):** `ProtocolVersion` is a
    compile-time constant served fleet-wide, so bumping deny-alls EVERY max=4 gateway in the org —
    **including pure-device gateways with no site involvement** — until upgraded. This is the D1 contract
    (a gateway that can't render the artifact can't be half-trusted), already accepted at S8.1's A-4
    discharge. Mitigation = **upgrade-agents-FIRST** deployment order, owed in the S8 upgrade-docs story
    (D1 already registered it). It is LOUD, not silent.
- **Option B — no bump, ride v4 (REJECTED as offered).** LAN-source rides `SrcIP` on v4; old agents
  silently SKIP LAN-source grants (fail-closed, device traffic untouched, smaller blast radius). Cost:
  an un-upgraded gateway silently under-enforces office-to-office while green — reintroduces exactly the
  reassuring-green class, for the population most likely to exist mid-rollout. Rejected for the same
  reason Option B was rejected at S8.1 (D2 refuses silent-mis-enforcement); recorded so the trade is
  findable.

**RULED (founder, 2026-07-17): Option A — bump `ProtocolVersion 4→5`.** Reasoning on the record at full
weight: this is the exact event D1 was built for. LAN-source CIDR is enforcement-significant; an old
agent that silently skips CIDR sources shows green while office-to-office blackholes — the reassuring-
green class as fleet-wide wire behavior. The gate makes the failure LOUD and diagnosable
(`unsupported_policy_version`, remedy: upgrade) instead of silent. Option B rejected as proposed.

**Blast radius accepted WITH CONDITIONS (ruled):**
- **(a) Named operational limitation (ESCALATED, Slice-2 — crosses the edition boundary)** — recorded
  here AND owed in the S11.4 upgrade docs: *"upgrade all gateways BEFORE the org becomes MULTI-SITE; the
  health surface shows exactly which gateways block it (`unsupported_policy_version`)."*
  - **The escalation the original Item-1 ruling did not contemplate:** the ruling reasoned about
    *enforcing* orgs and site-to-site *adopters*. But Routes ride `Compiled` and derive v5, so an
    **OPEN-edition, Zero-Trust-OFF org** with NO enforcement anywhere, upon adding a second site, gets a
    v5 artifact on its gateways — an un-upgraded gateway goes **deny-all, killing even the plain off-mode
    mesh it was serving**. State it plainly: the deny-all-until-upgraded blast radius applies to
    **open-edition ZT-off orgs too**, not only enforcing adopters — the trigger is "became multi-site,"
    not "turned on enforcement."
  - **Mitigating fact (VERIFIED in code, service.go siteLinkGraph):** routes derive ONLY from OTHER
    sites' approved subnets (`gSite == mySite` is skipped), and a site→site grant needs a src AND a dst
    site. So a **SINGLE-SITE org's artifact stays v4, byte-identical, untouched** — and single-site is
    the overwhelmingly common open-edition shape. The exposure is therefore bounded to **multi-site
    adopters (≥2 sites with approved subnets), upgrade-agents-first, all editions** — defensible + honest.
    (Route-attachment keys on ≥2 sites structurally, not on the first site; no ≥2-site gate needed.)
  - **(c) S8.3 scope (noted now so the UI inherits the honesty):** the add-second-site flow MUST surface
    a *"gateways below v5 will deny-all until upgraded"* warning BEFORE the action, listing the blocking
    gateways from the health surface. Registered for S8.3.
- **(b) Sequencing mercy — VERIFIED + build the conditional emit.** Current shape is
  **UNCONDITIONAL**: `Version: policyspec.ProtocolVersion` is stamped at every emit site (compiler.go:120,
  :292; service.go:304), so a naive constant-bump to 5 makes EVERY org's artifact v5 at deploy → every
  max=4 gateway refuses (device gateways included). **Ruling: emit v5 ONLY when the artifact actually
  needs it.** This is CHEAP — a compiler-side conditional `requiredVersion(Compiled) int` that returns 5
  iff the artifact contains a v5-requiring feature (a **CIDR-source `AllowEntry`** — the LAN-source grant
  — or a **`Route`** entry, Item 6), else 4. No artifact-path fork; pure function of the already-computed
  `Compiled`; deterministic (same input → same version). `policyspec.ProtocolVersion` stays the **ceiling
  (5)** = max shape the CP can emit; the EMITTED version is content-derived. Effect: a no-site org's
  artifact stays **byte-identical + hash-unchanged** (no re-converge, no refusal); only **orgs adopting
  site-to-site** flip to v5 and deny-all their un-upgraded gateways. Blast radius shrinks "every org at
  upgrade" → "orgs adopting site-to-site." Semantics SHARPEN to the honest contract — **Version = minimum
  agent version required to render THIS artifact** — which is more correct than version-as-constant.
  (Build note: the content-derived version applies at BOTH compiler emit sites + the service.go:304
  device-less fallback; `TestProtocolVersionConstantsAgree` guards the ceiling; goldens pin a no-site
  fixture at v4/unchanged-hash + a site fixture at v5 — both exercised.)

## LAW (Slice-1, ruled 2026-07-17) — `RequiredVersion` is a content-derivation ENFORCEMENT SEAM

Content-derived versioning (D1b) moves the "what shape is this artifact" decision into
`policyspec.RequiredVersion(c)`. That function is now **load-bearing**: it is precisely where a future
enforcement-significant field gets added WITHOUT updating the derivation — the artifact would then carry
new enforcement content at an OLD version number, and old gated agents would **silently accept** it (the
exact silent-mis-enforcement hazard the D1 gate exists to prevent, reintroduced one level up).

- **LAW:** every enforcement-significant content addition to `Compiled`/`AllowEntry` MUST update
  `RequiredVersion` in the SAME change (bump the ceiling AND add the content trigger). A field that
  changes the wire's enforcement shape but leaves `RequiredVersion` untouched is a silent-accept bug.
- **D2 checklist gains a row:** for every new field — *"`RequiredVersion` updated? (y/n)"*. An
  enforcement field answering "n" fails the checklist.
- **Guard-shaped test (Slice 2 is the first customer):** a `Compiled` carrying a `Routes[]` section must
  derive `RequiredVersion >= 5`. The test RED-FAILS if Slice 2 adds `Routes[]` to the enforcement shape
  without teaching `RequiredVersion` about it. (Routes are D2-classified plumbing/out-of-hash, but they
  still require a v5 agent to RENDER — so an old agent must refuse, not ignore; the version trigger is
  what makes that happen. Slice 2 applies this law explicitly.)

## Item 2 — MSS clamp (D9, binding) · PROPOSED

Double encapsulation (client-WG riding a site-WG link) drops effective MTU; PMTUD fails silently inside
tunnels → "large transfers freeze, pings work." **Decision:** the agent adds a TCP **MSS clamp** on the
site-link forward path — an nft rule in the existing `tunnex` table forward chain:
`tcp flags syn tcp option maxseg size set rt mtu` (clamp-to-PMTU), or a fixed value derived from
`sites.link_mtu` (reserved column, migration 0032) when set.

- **D2 classification (Item 1 requires it): node-local PLUMBING, OUTSIDE the hash** — the masquerade
  class (`egress_linux.go` renders it into the ruleset; it is not in `Compiled` JSON, so no version bump
  from the clamp, twin goldens untouched). Classified explicitly per D9(d), not assumed.
- **Scope:** applies to the site-link forward path only (not device egress). Rendered from the agent's
  view of the link MTU.
- **Walk owes (D9c):** a large transfer across a double-encapsulated path, **wire-proven** (a leg in the
  S8.2 box-walk — the classic "ping works, scp hangs" reproduced then fixed by the clamp).

## Item 3 — Routed-but-dropped deliberate red (D3 boundary — the epic's load-bearing proof) · PROPOSED

**Decision (named at S8.1 D3, BUILT + wire-proven here):** enforcing + zero grants → a propagated site
subnet is **ROUTED** (peer AllowedIPs + kernel route present, packets reach the forward chain) but
**DROPPED** at the forward chain (no matching allow → `policy drop`, the deny counter increments).
Routing ≠ permission, on the wire.

- **Red:** two sites, advertised + approved + route programmed, `enforcing`, NO grant → traffic
  blackholes at the forward `policy drop` (deny counter > 0), NOT at the route. Add the grant → flows.
- This is the security spine of site-to-site: the route (plumbing) never implies reachability (grant).
  S10.3 (cluster gateway) presumes this seam — load-bearing beyond EPIC 8.

## Item 5 — Validator-comparison-set law applied to route acceptance · PROPOSED

The law (S8.1 #1): a validator's comparison set is passed WHOLE; exclusions are a bypass until proven
structural. **Applied to S8.2's route-acceptance:** when the CP computes a gateway's route/AllowedIPs
set from approved advertisements, any disjointness/ambiguity check passes the WHOLE relevant set — no
"exclude self", no per-site filter.

- The org-wide subnet disjointness invariant is ALREADY enforced at approval (S8.1 D5/D7, post-#1-fix:
  approved site subnets are mutually disjoint + disjoint from the pool). Route computation **inherits**
  that invariant → no two routes can ambiguously overlap. State this reliance explicitly (a route-set
  correctness proof that leans on the approval-time guarantee).
- Any NEW comparison S8.2 introduces (e.g. a gateway's own local site subnet vs a remote route it would
  install — must not route a remote subnet that collides with its own LAN) passes the whole set + gets
  its adjacency/exact-boundary red (the S8.1 CIDR-math discipline).

## Item 6 — Propagation MECHANISM (how approved subnets become routes on other gateways) · PROPOSED

**Decision:** routes are **CP-computed desired-state output, never agent-to-agent gossip** (the S7.2
ledger: one authoritative artifact, one code path). Concretely:

- The CP computes, per gateway node, the set of **remote site subnets it must route** (from approved
  advertisements + the topology graph of Item 7) and delivers them as **site-link `Peer` entries in
  `DesiredState.Peers[]`** — each peer = the remote gateway (or the hub), `AllowedIPs` = the remote
  subnets it fronts, `Endpoint` = its public address (hub) or empty (spoke dialing out).
- The agent gains a **kernel-route capability (NEW)**: for each site-subnet AllowedIP that is not the
  pool CIDR, program `ip route replace <subnet> dev wg0` (netlink), reconciled — a flushed route heals on
  the next tick (`reconcile.go:142 runOnce`). Cryptokey-routing (AllowedIPs) decides WHICH peer; the
  kernel route makes the subnet reach wg0 at all (today the kernel only knows the pool route).
- **Routes carry NO permission** (Item 3). The GRANT (may-source-reach-subnet) is the `Compiled`
  AllowEntry (Item 1a); the route is plumbing.

**Sub-decision — route surface shape · OVERRULED (founder, 2026-07-17): explicit `Routes[]` section, NOT
derived-from-AllowedIPs.** Deriving fuses two distinct meanings into one artifact position: `AllowedIPs`
is WireGuard's crypto-routing table (what the tunnel WILL carry); making it ALSO the kernel-route intent
means neither can change independently — S8.4 (DNS routes to site resolvers), S10.3 (cluster subnets),
and any future route-metric/priority would fight the fusion. Worse, an agent INFERRING "program a kernel
route" from a peer field is the side-channel/parallel-path shape the S7.2 ledger forbids ("routes are
compiled-policy OUTPUT, never a parallel path"; derivation-from-topology is a parallel path in a thrift
costume). **Ruling:**
- An **explicit `Routes[]` section** in the v5 `Compiled` (the shape already bumping for Item 1). Each
  route = `{DstCIDR, via-peer/interface}` — the agent's route-programming reads **intent, not
  inference**.
- **D2-classified (checklist RUN, per Item 1):** a `Route` is **reachability PLUMBING, not enforcement**
  — the grant (`Allow`) permits, the route only makes packets arrive at the forward chain (Item 3). So
  `Routes[]` is **OUT of `CanonicalHash`** (two artifacts differing only in routes enforce identically;
  route drift is caught by Item 9's link-health surface, not the policy-desync hash — state it). BUT it
  triggers the content-derived `requiredVersion=5` (Item 1b): a `Route` present needs a v5 agent to
  render, so an old agent must refuse rather than ignore it. In-artifact as explicit intent; out-of-hash
  as non-enforcement.
- **`Peers[]` (DesiredState) stays** the WG-tunnel/crypto-routing surface (site-link `Peer{PublicKey,
  AllowedIPs, Endpoint}`); `Routes[]` (Compiled) is the kernel-FIB intent. Two sections, two meanings,
  independently evolvable — the decoupling the fusion would have destroyed.
- **The validator-comparison-set law (Item 5) applies to whatever accepts `Routes[]` agent-side** (the
  route set is validated whole; no per-entry exclusion).

## Item 7 — The site-link itself (how gateways become WG peers) + lifecycle · PROPOSED

S8.1 reserved `link_transport` as a site COLUMN; there is **no `site_links` table and no node↔node
peering yet**. S8.2 introduces the link.

- **Topology = hub-and-spoke, NOT full mesh (D8, binding).** A publicly-reachable gateway is the HUB;
  each site gateway (often NAT'd) peers with the hub, dialing outbound (WG + persistent-keepalive for
  NAT hole-punch); the hub forwards between spokes (the wg0→wg0 FORWARD the device-to-device path
  already uses). No STUN/relay (batch-3 #3 parked).
- **Hub designation (decide — held):** a node gains a **public endpoint attribute** (reachable
  `address:port`); a gateway with a public endpoint can serve as hub. Proposed v1: the org designates
  ONE hub node (a node attribute / flag); the CP computes the peer graph as spokes→hub. Multi-hub / mesh
  is a reserved seam (ties D6 HA / D8).
- **Keys:** reuse each node's existing WG keypair (agents already report their pubkey —
  `agent_wg_key_reported`). No new key ceremony (D6 pattern: no second enrollment).
- **CRYPTO-ROUTING PACKET-WALK (VERIFIED on paper, Slice-2 — the expectation the box-walk's first
  cross-site ping tests).** WireGuard drops an INBOUND packet whose source is not in the sending peer's
  AllowedIPs, so the graph's AllowedIPs are load-bearing in BOTH directions. Trace spoke-A LAN
  (10.1.0.0/24) → hub → spoke-B LAN (10.2.0.0/24):
  1. A-host → A-gw: kernel route `10.2.0.0/24 dev wg0` (A's `Routes[]`, all-remote) → wg0.
  2. A-gw wg0 out: dst 10.2.0.0/24 ∈ A's hub-peer AllowedIPs (all-remote incl B) → encrypt to hub.
  3. hub IN from peer-A: src 10.1.0.x ∈ peer-A AllowedIPs (**A's own subnet** — the hub's per-spoke
     entry) → accept. *(This is the load-bearing hub-side asymmetry: each hub→spoke peer carries only
     that spoke's subnet, and it serves BOTH that spoke's inbound-source check AND the hub's
     outbound-dst routing to it.)*
  4. hub FORWARD (wg0→wg0, grant-gated) → kernel route `10.2.0.0/24 dev wg0` (hub's all-remote Routes).
  5. hub wg0 out: dst 10.2.0.0/24 ∈ peer-B AllowedIPs (**B's own subnet**) → encrypt to B.
  6. B-gw IN from hub-peer: src 10.1.0.x ∈ B's hub-peer AllowedIPs (all-remote **incl A's subnet**) →
     accept → forward to B-LAN.
  Return path is symmetric (ct established). If the first ping fails, THIS is the table to diff against
  the wire — most likely culprit is a missing subnet in a hub-side per-spoke entry (step 3/5) or the
  spoke's all-remote set (step 2/6).
- **Endpoint discovery:** the hub's public endpoint is a node attribute the CP knows; spokes need no
  inbound endpoint (they dial out). Records the batch-3 #3 requirement (hub must be reachable) honestly.
- **Hub is a health-surfaced SINGLE POINT OF FAILURE (CONFIRMED addition, ruled):** with one designated
  hub, the hub going down kills EVERY spoke's site traffic — Item 9's kinds MUST distinguish **"hub down"**
  (all spokes' site traffic dead) from **"one spoke's link down"** (that spoke only), because the operator
  remedies differ entirely. Multi-hub is the reserved mitigation (ties D6 HA / D8).
- **Lifecycle — full-sweep law (BINDING):** unbinding/deleting a site, or removing an approved subnet,
  **releases the link's routes + peer entries + AllowedIPs everywhere** on the next reconcile (the
  cross-cutting revocation-is-a-full-sweep invariant). A removed link leaves NO orphan route/peer. Red:
  delete a site → its subnets vanish from every other gateway's routes + peers.

## Item 8 — Return-path + NAT interaction · PROPOSED (mostly a stated confirmation — where silence hides)

- **Site-to-site is NOT masqueraded — and the existing rule already does this right.** Masquerade is
  `ip saddr <pool-CIDR> oifname != "wg0"` (`egress_linux.go:262`): it fires only for pool-sourced
  internet egress. Site-to-site is `saddr = site-LAN` (not pool) AND `oifname = wg0` (transit) → the rule
  does NOT fire. Correct: both LANs need REAL addresses for return routing; NATing site traffic would
  break it. **No masquerade change — STATE IT** (this is exactly where a "helpful" scope-widening would
  silently break return paths). **CONFIRMED addition (ruled): one deliberate red** — site-to-site
  traffic observed **un-NAT'd on the wire** (the transited packet keeps its site-LAN source, not a pool
  address). A stated confirmation of existing behavior becomes a **pinned invariant**, so a future
  masquerade change can't silently start NATing site traffic.
- **Symmetric routes required BOTH directions:** site B's gateway must route site A's subnet (and vice
  versa) for replies to find their way back — falls out of Item 6 (the CP computes each gateway's route
  set from ALL approved advertisements, both directions).
- **Return-path acceptance:** the forward chain's `ct state established,related accept` (:273) admits
  replies once the forward direction was granted — so a SINGLE directional grant (A→B) yields
  bidirectional flow for an established connection (identical to the device model). A NEW connection
  B→A needs its own grant (default-deny). State it so operators expect directionality.

## Item 9 — Failure legibility (link down vs policy denial) · PROPOSED

An operator must distinguish these states on the health surface (S7.4b advisory-kind pattern) — CONFIRMED,
with the Item-7 hub-SPOF addition making it FOUR:

- **Policy denial** (routed-but-dropped, Item 3): route + peer healthy, forward chain drops — the deny
  counter increments; remedy = add a grant.
- **Spoke link down** (routed-but-UNREACHABLE): ONE site-link peer's last-handshake is stale — that
  spoke's traffic blackholes though a grant exists; remedy = fix that spoke's tunnel/NAT.
- **Hub down** (Item 7 SPOF): the HUB's link is down → ALL spokes' site traffic dead — a distinct,
  higher-severity kind (the remedy is the hub, not any spoke); never collapse it into per-spoke link-down.
- **Version-refused** (Item 4): `unsupported_policy_version`; remedy = upgrade the agent.

**Decision:** S8.2 adds **site-link health kinds** (e.g. `site_link_down` for a spoke, `site_hub_down` for
the hub) driven by peer handshake age, surfaced through the existing `PolicyHealthForNodes` path — so
"link down", "hub down", and "policy denies" are never the same amber. Deferred-if-scoped: a full
per-link health matrix (name the substitute if bound to S8.3's UI).

## Item 10 — Edition / opt-in split carried through propagation (D11) · PROPOSED

D11 already ruled the split; S8.2 carries it:

- **CORE (open, both suites):** the site-link peering, route propagation, kernel-route programming, the
  MSS clamp, off-mode legacy-mesh reachability. Plumbing — a self-hoster on the open edition routes
  between their sites.
- **ENTERPRISE (unlocked by edition):** ZT-governance of site traffic — `enforcing`-mode grant-gating of
  site-subnet reachability (Item 3's drop). Rides the EXISTING `zero_trust_mode` switch — **no new opt-in
  flag** (unlock-then-opt-in via the existing enforcing mode).
- **Downgrade-release (S7.5.3 law, restate):** enterprise→open → site subnets become **routed but no
  longer grant-gated** (off-mode legacy mesh); no new flag, so no new downgrade-clear beyond the
  mode revert. State it so it is not silently inherited.
- **Test placement:** core (peering/routes/clamp/off-mode) in the open suite (both editions);
  ZT-governance-of-site-traffic (Item 3 drop + downgrade release) in the enterprise suite.

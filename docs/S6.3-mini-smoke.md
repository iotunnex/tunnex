# S6.3 MINI-SMOKE (macOS) — validate the kill-switch + data plane ONLY

Scope: prove the pf kill-switch + wireguard-go data plane on a real Mac BEFORE
building the production lifecycle + Windows WFP backend on top. This is NOT the
story-end smoke (which is `docs/S6.3-smoke.md`, full lifecycle/UX/uninstall) — do
NOT merge the two scopes. **Time budget: ~25–30 min** (10 build/install, 15 test).

Prereqs: a Mac; a reachable Tunnex tenant with an enrolled gateway; the `tunnex`
CLI built. Two extra Terminal tabs. You will be prompted for your **sudo password**
several times (install, launchctl, tcpdump, kill).

---

### 0. Get a WireGuard config (device create, D2 one-time)
```
tunnex login                       # if not already
tunnex device create --name mini-smoke --full     # --full = full tunnel (both AFs)
```
Expected: `Config written to <path> (0600)`. Note that path as `$CONF`.
(Full-tunnel so the kill-switch covers v4 AND v6.)

### 1. Dev-install (STEP ZERO auth check is inside it)
```
./scripts/macos-dev-install.sh
```
Prompts: sudo password (×several). Expected key lines:
- `[3/6] STEP ZERO … tunnex-helper 0.1.0 caller-auth: native`  ← **MUST be `native`**.
  If it says `stub`, STOP — the build is wrong (not CGO_ENABLED=1).
- `[6/6] daemon up? … pid = <n>` and `caller-auth: native` in the startup log.
Budget: ~3–5 min (first `go build` pulls wireguard-go).

### 2. Tunnel up + traffic
Tab A:
```
curl -s https://api.ipify.org; echo "  (pre-tunnel IP)"
sudo /usr/local/tunnex/tunnelctl up "$CONF" --full
```
Expected: a JSON `"ok": true` with `"state": "up"`, then
`>> tunnel up; holding connection …`. LEAVE THIS RUNNING (it holds the owner
connection; closing it = graceful down).
Tab B:
```
ifconfig | grep -A3 utun            # a utun with your device address, up
curl -s https://api.ipify.org; echo "  (should now be the GATEWAY egress IP)"
```
PASS: the IP changed to the gateway's egress; `tunnelctl` prints periodic
`>> status: state=up hs=<recent>s rx=… tx=…`. Budget: ~2 min.

### 3. Enforcement-not-presence (kill-switch is live while up)
Tab B — a NON-tunneled direct target must already be BLOCKED by the kill-switch:
```
# Pick a target NOT in allowed-ips… for a full tunnel everything routes, so instead
# prove enforcement by confirming the pf anchor is EVALUATED (not just present):
sudo pfctl -a tunnex -s rules            # informational: block + passes listed
sudo pfctl -s info | grep -i enabled     # Status: Enabled
```
PASS: pf Enabled; the block/pass rules are loaded in the `tunnex` anchor. (The
decisive enforcement proof is step 4's pcap after the helper dies.)
Loopback + DHCP exception spot-check:
```
ping -c1 127.0.0.1                       # loopback exempt → replies
# DHCP/NDP pass rules present:
sudo pfctl -a tunnex -s rules | grep -E 'port (67|547)|icmp6'
```
PASS: loopback pings; the DHCP (67/547) + NDP (icmp6) pass rules are listed.

### 4. KILL-SWITCH — kill -9, process gone, pcap zero cleartext (v4 AND v6)
Tab B, pick `T=1.1.1.1` and `T6=2606:4700:4700::1111`:
```
ping T > /dev/null &  PING4=$!
ping6 T6 > /dev/null & PING6=$!
sudo tcpdump -i en0 -n host $T  -w /tmp/ks4.pcap &  TD4=$!
sudo tcpdump -i en0 -n host $T6 -w /tmp/ks6.pcap &  TD6=$!
sleep 2
HPID=$(sudo launchctl print system/io.tunnex.helper | awk '/pid =/{print $3}')
date; sudo kill -9 $HPID                 # HARD kill — no handlers run
ps -p $HPID                              # EXPECT: no such process
sleep 8
sudo kill $TD4 $TD6 $PING4 $PING6 2>/dev/null
```
Inspect (the crux):
```
sudo tcpdump -r /tmp/ks4.pcap -n 'host 1.1.1.1 and not udp' | head
sudo tcpdump -r /tmp/ks6.pcap -n 'host 2606:4700:4700::1111 and not udp' | head
```
**PASS (ALL):**
- `ps -p $HPID` shows the helper is GONE — so nothing live is enforcing.
- BOTH pcaps show ZERO direct ICMP/ICMPv6 to the targets AFTER the kill timestamp
  (the pings fail; the pf anchor, kernel-resident, keeps blocking). A single
  cleartext packet to either target = FAIL.
- `tunnelctl` in Tab A prints `>> helper connection lost` (the app-side signal).
Budget: ~5 min.

### 5. Reference-counted release
The daemon `KeepAlive` restarts the helper; on a clean graceful path pf is
released. To verify the token release path without the daemon interfering:
```
sudo launchctl bootout system /Library/LaunchDaemons/io.tunnex.helper.plist
sudo pfctl -a tunnex -F all              # flush our rules
sudo pfctl -X $(sudo pfctl -s References 2>/dev/null | awk 'NR==2{print $1}') 2>/dev/null || true
curl -s -m5 -o /dev/null -w '%{http_code}\n' https://api.ipify.org
```
PASS: after flush + release, normal traffic works again (HTTP 200) — pf is not
left force-disabled/enabled by us. (`pfctl -E`/`-X` is a refcount, not a global
toggle; a reboot fully clears if in doubt.)

### 6. Cleanup
```
./scripts/macos-dev-uninstall.sh
```
PASS: the RESIDUE CHECK block prints daemon not-found, socket gone, pf rules empty,
and `curl` returns `200` (network restored). Budget: ~2 min.

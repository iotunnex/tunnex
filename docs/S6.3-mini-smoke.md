# S6.3 MINI-SMOKE (macOS) — kill-switch fail-closed + BOUNDED recovery

Scope: prove the pf kill-switch on a real Mac — (1) it BLOCKS cleartext while the
helper is dead (fail-closed), and (2) the host RECOVERS (never stranded). This is
NOT the story-end smoke (`docs/S6.3-smoke.md`). **Time: ~25 min.**

> ⚠️ **This procedure is designed to be UNSTRANDABLE.** The full-tunnel routes use the
> WG-standard split-default, so the physical default route is **never destroyed** — the
> ONLY thing that can stop traffic is the pf block, and flushing it (below) restores
> connectivity instantly. You will NOT need to reboot. An earlier version of this smoke
> stranded a host; the recovery paths it now exercises exist and are unit-tested
> (`TestSupervisorSelfHeal`, `TestSupervisorDeadMan`).

---

## 🔴 RECOVERY — READ FIRST. Keep a SECOND terminal open on this the whole time.

If at ANY point the network is blocked and you want it back immediately, run:
```sh
sudo pfctl -a tunnex -F all        # flush the kill-switch block → traffic flows at once
```
That is the whole recovery. Three independent releases exist, fastest first:
1. **This manual flush** — instant, works even mid-test.
2. **Helper restart self-heal** — starting any helper runs `CleanStale` (flushes the
   stale anchor) before it serves: `sudo launchctl kickstart -k system/io.tunnex.helper`.
3. **Dead-man** — a LIVE helper auto-releases ≤ 90 s after the owner stops heartbeating.

Because the physical default is intact, `pfctl -F` → traffic returns immediately. If it
does NOT, the problem was a route, not the block (it won't be — but that's the
distinguisher: blocked ≠ no-route).

---

## Prereqs
A Mac; a reachable tenant with an enrolled gateway; `git pull` on
`story/S6.3-tunnel-control`. Two terminal tabs (A = driver, B = tests + recovery).
Several sudo prompts. Real internet on `en0` (so recovery is observable).

## 0. Full-tunnel config (Tab A) — no hand-editing (generator now emits ::/0)
```sh
cd ~/tunnex
( cd apps/cli && GOFLAGS=-mod=readonly go build -o /tmp/tunnex ./cmd/tunnex )
/tmp/tunnex login --server http://<your-server>
/tmp/tunnex device create --name mini-smoke --full-tunnel
CONF=~/.config/tunnex/device.conf
grep AllowedIPs "$CONF"        # MUST already be: AllowedIPs = 0.0.0.0/0, ::/0
```
If AllowedIPs is missing `::/0`, your server predates the generator fix — redeploy it;
do NOT hand-edit the config.

## 1. Install the helper (native)
```sh
./scripts/macos-dev-install.sh     # STEP ZERO must print caller-auth: native
```

## 2. Run the helper WITHOUT KeepAlive (so the kill in step 4 gives a clean window)
The installed LaunchDaemon auto-restarts on kill (which self-heals in ~2 s and would cut
the fail-closed window short). For a clean measurement, stop it and run the helper in the
foreground for the test:
```sh
sudo launchctl bootout system/io.tunnex.helper 2>/dev/null || true
sudo TUNNEX_INSTALL_DIR=/usr/local/tunnex TUNNEX_HELPER_SOCKET=/var/run/tunnex/helper.sock \
  /usr/local/tunnex/tunnex-helper &        # logs to this terminal; CleanStale runs on start
```
Note your uplink BEFORE tunneling: `route -n get 1.1.1.1 | awk '/interface/{print $2}'` → usually `en0`.

## 3. Tunnel up (Tab A) — verify via the GATEWAY, not egress
```sh
sudo /usr/local/tunnex/tunnelctl up "$CONF" --full     # {"ok":true,"state":"up"}, then holds
```
Tab B — prove the tunnel reaches the gateway (works even with S3.7 parked / no egress):
```sh
ping -c3 10.99.0.1                 # the gateway's WG address — replies = tunnel is up
route -n get 1.1.1.1 | awk '/interface/{print $2}'   # now → utun (default moved onto the tunnel)
```

## 4. FAIL-CLOSED — kill the helper, prove ZERO cleartext leaks (Tab B)
```sh
UPLINK=en0            # your uplink from step 2
T=1.1.1.1; T6=2606:4700:4700::1111
ping  $T  >/dev/null &  PING4=$!
ping6 $T6 >/dev/null &  PING6=$!
sudo tcpdump -i "$UPLINK" -n host $T  -w /tmp/ks4.pcap &  TD4=$!
sudo tcpdump -i "$UPLINK" -n host $T6 -w /tmp/ks6.pcap &  TD6=$!
sleep 2
HPID=$(pgrep -f '/usr/local/tunnex/tunnex-helper')
date; sudo kill -9 "$HPID"         # HARD kill — no handlers, no KeepAlive restart
ps -p "$HPID"                       # EXPECT: no such process
sleep 6
sudo kill $TD4 $TD6 $PING4 $PING6 2>/dev/null
echo "--- v4 (must be EMPTY) ---"; sudo tcpdump -r /tmp/ks4.pcap -n 'host 1.1.1.1 and not udp' | head
echo "--- v6 (must be EMPTY) ---"; sudo tcpdump -r /tmp/ks6.pcap -n 'host 2606:4700:4700::1111 and not udp' | head
```
**PASS:** `ps -p` → *no such process* AND both pcaps EMPTY after the kill timestamp —
the helper is GONE yet the kernel-resident pf block still holds. One cleartext packet = FAIL.
(The host is intentionally blocked right now — step 5 recovers it.)

## 5. RECOVER — prove the host is NOT stranded (the property the old smoke failed)
Pick EITHER path; confirm the internet returns:
```sh
# (a) restart the helper → CleanStale self-heals the stale block:
sudo TUNNEX_INSTALL_DIR=/usr/local/tunnex TUNNEX_HELPER_SOCKET=/var/run/tunnex/helper.sock \
  /usr/local/tunnex/tunnex-helper &
sleep 1
ping -c3 1.1.1.1                    # REPLIES = host recovered (block released, physical default intact)
# --- or (b) out-of-band flush (instant): sudo pfctl -a tunnex -F all ; ping -c3 1.1.1.1
```
**PASS:** `ping 1.1.1.1` replies → the block was released and the physical default was
always present (blocked ≠ no-route, proven — the flush alone restored traffic).

## 6. Restore the daemon + cleanup
```sh
sudo pfctl -a tunnex -F all 2>/dev/null || true    # ensure no residue
sudo pkill -f '/usr/local/tunnex/tunnex-helper' 2>/dev/null || true
sudo launchctl bootstrap system /Library/LaunchDaemons/io.tunnex.helper.plist   # KeepAlive daemon back
sudo pfctl -a tunnex -s rules      # anchor should be empty
```

## Report
Paste: the `date` line, `ps -p $HPID` (gone), both `tcpdump -r` outputs (empty), and the
step-5 `ping 1.1.1.1` (replies). PASS on all → the kill-switch enforces AND recovers →
WFP unblocks. A leaked packet, a live process, or a host that does NOT recover after
`pfctl -F` → STOP and paste it; that's a design finding, not a retry.

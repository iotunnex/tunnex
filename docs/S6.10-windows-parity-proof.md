# S6.10 (Story A) — Windows full-tunnel PARITY proof runbook

Proves Windows full-tunnel carries traffic + resolves names (adapter DNS + WFP DNS-restrict),
run BEHIND the S6.9 guard via the two loud dev bypass flags. Story A is **works-not-safe**: the
WFP kill-switch does NOT survive process death until Story B (S6.7). The app shows an UNSAFE DEV
MODE banner throughout — that state is real (a hard kill can leak). The guard does NOT lift on A.

## 0. Enable the dev bypass (BOTH flags — default OFF)
- **Gateway API** (VM): set the server flag and restart api:
  ```
  TUNNEX_ALLOW_WINDOWS_FULLTUNNEL=1 docker compose up -d api
  ```
  (docker-compose passes it through; empty by default = guard on.)
- **Windows helper**: set the helper flag for the service, then restart it:
  ```
  setx TUNNEX_DANGEROUS_WINDOWS_FULLTUNNEL 1 /M
  sc stop TunnexHelper & sc start TunnexHelper
  ```
  (machine-level env; the SCM service inherits it. Adjust the service name if different.)
- Install the Windows `.exe` built from this branch (helper carries the DNS fix; app carries the
  banner). CI uploads `tunnex-windows-installer` per push.

## 1. Connect + the UNSAFE banner
- App → tick **Route all traffic (full tunnel)** → **Connect**.
- **PASS:** it connects (no `full_tunnel_unsupported`), and a red **"UNSAFE DEV MODE — the Windows
  full-tunnel kill-switch does not persist on crash…"** banner is visible. (If the banner is
  missing while full tunnel is up under the flag, that is a FAIL — the warning must be unmissable.)

## 2. Egress works (NECESSARY, not sufficient)
PowerShell on the client:
```
Resolve-DnsName example.com          # resolves (via the tunnel resolver on the wintun adapter)
curl.exe -s ifconfig.me ; echo       # = the GATEWAY's public IP (egress via gateway NAT)
```
- **PASS:** name resolves AND `ifconfig.me` returns the gateway IP (not the client's) — proves the
  adapter SetDNS + the gateway egress NAT. A browser loads. **But egress-works does NOT prove the
  DNS fix** — §3 is what proves it.

## 3. DNS-LEAK-BLOCKED — the LOAD-BEARING check (both legs required)
`blockDNS` must force ALL DNS through the tunnel resolver ONLY. Prove it with a positive AND a
negative leg — the NEGATIVE leg is the one that proves the fix:
```
# POSITIVE — the tunnel resolver answers:
Resolve-DnsName example.com -Server 1.1.1.1        # MUST SUCCEED
# NEGATIVE (load-bearing) — a NON-tunnel resolver is blocked at the client's WFP layer:
Resolve-DnsName example.com -Server 8.8.8.8        # MUST FAIL / time out
```
- **PASS = the negative leg FAILS while the positive SUCCEEDS.** `8.8.8.8` is an unambiguous
  non-tunnel public resolver; with `blockDNS` armed, the client's WFP filter drops that query on
  outbound port 53 BEFORE it can egress (even though 0.0.0.0/0 would otherwise route it into the
  tunnel). If `8.8.8.8` RESOLVES, `blockDNS` did NOT arm — that is a **FAIL** (DNS can leak to any
  resolver), regardless of §2 passing.
- Repeat the negative leg with your actual LAN resolver IP too if you want a second data point, but
  `8.8.8.8` is the definitive one.

## 4. Deliberate-red — egress rides the gateway masquerade
On the VM, flush ONLY the NAT chain:
```
docker compose exec node-agent nft flush chain ip tunnex postrouting
```
- Client: `curl.exe -s --max-time 6 ifconfig.me` → **empty/fails** (egress dies). PASS = §2 proved
  the rule carries it, not a leak. (Reconcile heals it back within one interval — S3.7 §4.)

## 5. Disconnect restores cleanly (no strand)
- App → **Disconnect**.
- **PASS:** normal internet + LAN DNS return immediately (the adapter DNS auto-vanishes with the
  wintun adapter — no restore step, no strand), and the UNSAFE banner clears.

## Result
Record PASS/FAIL per section. §1 banner + §2 egress/DNS + §3 no-DNS-leak + §4 deliberate-red + §5
clean-disconnect green = Story A's obligations met. **The guard stays** — Windows full tunnel is
still refused for real users (flags off). Story B (S6.7 — persistent WFP session + kill -9 pcap) is
what makes it safe and lifts the guard. Turn the bypass flags OFF when done:
```
setx TUNNEX_DANGEROUS_WINDOWS_FULLTUNNEL "" /M   # + restart helper
docker compose up -d api                          # without the env → guard back on
```

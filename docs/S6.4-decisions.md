# S6.4 Connection UX — decisions (commit-one, decision-first)

Scope operationalizes the PLAN S6.4 scope note. **App-side only — the privilege helper
and both kill-switch backends are UNTOUCHED** (no privileged/kernel surface; that's why
build-through-to-story-end is pre-authorized). All changes are renderer (apps/web) +
Electron main (apps/client/src/main); the helper protocol/verbs do not change.

## Already down-paid on S6.3 (do NOT rebuild)
- **Handshake-derived status** — green "Connected" only when the last handshake is fresh
  (`TunnelControl.tsx`, `HANDSHAKE_STALE_SEC`).
- **Show tunnel IP** — "Your IP: 10.99.0.x" (main caches the config address, attaches to status).

## Build in S6.4

1. **Revocation-aware teardown (the proper version of the stopgap self-heal).**
   While connected, the client PROACTIVELY detects its device being revoked/deleted and
   auto-disconnects + clears the dead config — instead of showing a stuck "Connecting…"
   until the helper's dead-man fires.
   - Mechanism: a poll in MAIN (every ~30s while up) calls the EXISTING
     `DeviceApi.deviceExists(deviceId)` (S6.3, all-orgs + fail-safe). On a definitive
     "gone" → `tunnel.down()` + `clearTunnelConfigForOrigin` + notify the renderer with a
     distinct "device revoked — reconnect to re-enroll" state.
   - Fail-safe: `deviceExists` THROWS on any read error → the poll treats it as
     inconclusive and does NOT tear down (never disconnect a working tunnel on a blip).
   - This is the CLIENT-side proactive version; the helper's dead-man remains the
     independent kernel-side safety net. No helper change.

2. **Change-server / sign-out UI (POC item 4) — no more deleting userData files.**
   A Settings surface (desktop-only) showing the CURRENT server URL + two actions:
   - **Change server** → `config.setServerUrl(url)` (already revokes+clears the old
     credential + tunnel config before switching origin — S6.2/S6.3 posture) → reload.
   - **Sign out** → existing `auth.logout()` full-sweep (tunnel down → clear config →
     revoke → clear credential). Surface both in the desktop UI; the origin-keyed config
     already anticipates this — it just needs the controls.

3. **Server picker** — the first-run server entry already exists (setup screen); S6.4 adds
   the same entry to Settings (folds into #2's Change-server). Not a separate mechanism.

4. **Split-tunnel toggle** — a UI toggle for `full_tunnel` at device-create time (today
   `DEFAULT_FULL_TUNNEL=false`, no UI). Passes the intent to `createDevice(fullTunnel)`.
   Full-tunnel egress itself is parked (S3.7) — the toggle is present but a full-tunnel
   choice surfaces the "no internet until S3.7" caveat inline (honest, not silently broken).

5. **Tray icon** — Electron `Tray` with connect/disconnect + live status + show-window.
   Main-process only; reuses `TunnelController`.

6. **Notifications** — Electron notifications on connect / disconnect / fail-closed /
   device-revoked. Main-process, from the existing `onStatus` events.

## Tests / acceptance
- Revocation-teardown: unit-test the poll decision (gone → down+clear; error → keep) with
  a fake DeviceApi + a fake tunnel controller (like the S6.3 supervisor tests).
- Change-server/sign-out: the origin-keyed clear/revoke is already tested (deviceconfig);
  add a renderer smoke that the controls invoke `config.setServerUrl` / `auth.logout`.
- Tray/notifications: main-process wiring; no display in CI (Electron never launches in CI)
  → typecheck + build coverage, live-verified at S6.5a packaging like the other UX.

## Non-goals (explicit)

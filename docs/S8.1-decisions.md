## D1 — ProtocolVersion fail-closed gate · LOCKED (blocking)

**Decision.** The agent gains a `maxSupported` protocol version. On receiving a `Compiled` artifact with
`Version > maxSupported`, it **refuses the artifact → deny-all + `policy_degraded`** rather than
applying it. Built in commit-one, BEFORE the first enforcement-significant wire field — and
sites-as-destination (D3) IS that field. Today the node applies any version, safe-ignoring unknown
JSON fields: safe **by convention only** (every bump so far observability-additive + hash-blind +
safe-ignore, each unit-pinned). This mechanizes the convention.

**Amendment (ruled).** The refuse path sets a **distinct `policy_degraded_kind`** —
`unsupported_policy_version` — following the S7.4b advisory-kind-over-the-bool pattern. An operator
MUST distinguish "agent too old for this artifact" from every other degradation, because its remedy is
**unique: upgrade the agent.** A bare `policy_degraded` bool would send the operator hunting.

**CP-side surface (answered).** A version-refusing gateway self-reports through the existing S7.4b node
policy-health path: the agent's reconcile loop sets `policy_degraded=true` +
`policy_degraded_kind=unsupported_policy_version`, stamped with the `policy_reported_at` freshness
clock. `PolicyHealthForNodes` (the one CP-side compute) surfaces it in the node/site health UI with an
"upgrade the agent" remedy hint — a first-class, honest degradation state, not a silent stale.

**Deliberate red.** Agent handed `maxSupported+1` → deny-all + `policy_degraded` +
`kind=unsupported_policy_version`; NEVER silent-accept. Retires the interim "every bump owes its own
hash-blind pin" law once this lands.

**Go-forward-only — the pre-gate residual (Slice-1 gate, ruled).** The gate protects only agents that
HAVE it. A genuinely OLD binary (v3-era, no gate code) does not deny-all on the Slice-3 sites bump — it
**silently accepts** the artifact and safe-ignores the sites field, i.e. **enforces stale policy while
reporting healthy** (the reassuring-green class, on the wire, for exactly the population most likely to
exist mid-rollout). Recorded consequences:
- **What the interlock red actually claims (Slice 3 states it):** the interlock red simulates a *gated*
  agent pinned to the OLD `maxSupported` (the go-forward guarantee). It does NOT — cannot — cover a
  pre-gate binary. Slice 3's red comment must name the simulated agent precisely so the claim isn't
  over-read.
- **The guarantee is go-forward only** — the honest sentence, said here and owed in the walk.
- **Deployment order (S11.4-class upgrade docs, not just this file):** the v3→sites transition is
  **upgrade agents FIRST, then the CP**. If the compose quickstart ships CP+agent as one unit this is
  mostly theoretical; **multi-gateway customers upgrade piecemeal** — name it in the S8 upgrade guide
  where they read it, not only here. REGISTERED as owed at the S8 upgrade-docs story.

**Reported hash under refusal (Slice-1 gate, ruled).** A refusing agent reports its **last-good (or
empty) applied hash**, NEVER the refused artifact's hash — structural: `SetPolicy` does not store the
refused artifact, so `applyAndTrack` only ever hashes the last-good/nil policy. Echoing the refused hash
would read synced-and-healthy to the desync detector, contradicting the kind. Pinned
(`TestUnsupportedVersionRefusedIsDenyAll` note-2 assertion).

## D3 — Sites/subnets as a policy DESTINATION KIND · LOCKED

**Decision.** Extend the S7.1 policy model: `dst_kind ∈ {device, site_subnet}`. Sites/subnets are a
first-class **destination kind** in the VERSIONED `policyspec.Compiled` artifact — not a side channel.
Zero-Trust governs site-to-site by construction: under `enforcing`, a site subnet is reachable ONLY via
an explicit grant; under `off`, the legacy mesh. Mode-as-compiler-input (one code path, one artifact,
mode selects what is compiled) — the S7.1 lock. **Routing ≠ permission / approved ≠ reachable.** The
version bump introduced here is what trips D1's gate (semantic change → deny-until-agent-supports).

**S8.2 boundary.** The deliberate-red — *enforcing + zero grants → a propagated site subnet is ROUTED
but DROPPED at the gateway forward chain* — is **named here, built and wire-proven in S8.2** (route
propagation is S8.2). S10.3 (cluster gateway) already presumes this seam; it is load-bearing beyond
EPIC 8.

## D4 — Site-link TRANSPORT enum · LOCKED

**Decision.** `link.transport` is a modeled enum from day one; **`wireguard` is the only implemented
value.** This RESERVES the parked IPsec-interop seam (agent-impossible endpoints: managed cloud VPN
gateways, hardware appliances) without a later migration. IPsec stays **PARKED** (trigger = a real
customer with an agent-impossible endpoint AND after EPIC 9; strongSwan-managed-by-agent, site-to-site
only, interop matrix bounded at story-open). Routing + ZT enforcement are transport-agnostic — stated
in the schema comment beside the enum.

**Promotion path (Slice-2 ruling 1).** `link_transport` lives as a **site-level column** (migration
0032) — because v1 is single-node (a site = one gateway = one uplink; the `nodes.site_id` partial
unique index enforces it), a `site_links` table now would model a cardinality that cannot exist. When
multi-link / HA unparks, the column **promotes to a per-link table** — a mechanical migration done WITH
the feature that needs it, and **the enum/CHECK semantics carry over unchanged**. **Refuse-don't-guess
(schema twin of D1's gate):** the `CHECK (link_transport IN ('wireguard'))` makes inserting any other
value (e.g. `'ipsec'`) a **loud 23514 failure**, not a silent accept — a new transport is unusable
until its migration lands. Pinned (`TestSiteTransportCheckRefusesUnknown`).

## D5 — Subnet-advertisement model (+ D7 disjointness, folded) · LOCKED

**Decision.** A site advertises the subnets it routes; advertisement requires **control-plane/admin
APPROVAL before propagation** (a compromised site gateway must not hijack routes by advertising subnets
it doesn't own). State machine: `pending → approved`. `approved ≠ reachable` — ZT grants still gate
reachability (D3). Overlap handling: **the one forbidden outcome is silent ambiguity.**

**D7 folded in as D5's bidirectional completion (BINDING).** The S7.2 ledger's overlap-refusal covered
**site-vs-site only**. It did not assert disjointness against the org's **device pool CIDR** or
**reserved ranges** — a site subnet overlapping the pool → gateways route device-/32s toward a site
link (the silent-ambiguity class via a collision the ledger never enumerated). And it is
**bidirectional**: S4.5b's **pool-resize (grow)** must equally refuse growing into an approved site
subnet, or the check is a race.

- **ONE disjointness validator**, three inputs: existing **site subnets** + the org **pool CIDR** +
  **reserved ranges**. Pure/deterministic.
- Enforced at **BOTH seams**: advertisement-approval AND pool-resize (S4.5b grow).
- **Typed errors both directions:** advertisement-into-pool → `gateway_no_egress`-class; resize-into-site
  → `illegal_resize`-class. Never a silent accept.
- **Deliberate red each way** (advertise-overlapping-pool refused; grow-into-approved-site refused).
- **Touch to existing audited behavior (flagged):** the S4.5b resize path gains a validator call →
  **inherits the existing resize tests + a new red.** No behavior change to the happy path; the new
  refusal is additive.
- Precedence / longest-prefix / manual route pinning **DEFERRED** (trigger = first real customer need).

## D6 — Site model + the site-entity argument (+ reserved seams) · LOCKED

**The argued decision (not just tables): is a site a new ENTITY that owns nodes, or a role/attribute of
the existing node?**

**Ruling: a site is an ENTITY that owns ≥1 node.** Argument:

- **Node replacement.** A site's subnets, approved advertisements, DNS-forwarding entries (S8.4), and
  identity are **site-scoped, not node-scoped.** If a site WERE its node (attribute model), replacing a
  failed gateway box would destroy the site's identity and every advertisement/grant hanging off it —
  unacceptable operationally. As an entity, replacement = rebind `site.node`; the site survives.
- **FKs.** `sites(id, org_id, name, …)`; a node references its site (`nodes.site_id` nullable — a plain
  non-site agent has none); `site_subnets(site_id, cidr, …)`; advertisements and grants reference the
  **site**, not the node.
- **Enrollment ceremony.** No new join protocol. The node still enrolls via the **existing join-token /
  agent flow** (S3/S6 pattern). The SITE is created as an org entity (admin action), then a node is
  **BOUND** to it — a site-flavored binding step, not a second enrollment mechanism.
- **HA seam (decide-or-name — NAMED + reserved).** "Can one site have two gateways?" — **v1 enforces
  single-node-per-site** (app-layer + a uniqueness constraint). The entity-owns-≥1-node shape models
  multi-node HA **by construction** when unparked; the attribute model could not (one node = one site).
  Multi-node/HA is a **reserved seam, not a v1 feature** (trigger = HA demand; ties to D10's parked
  cache + batch-3 #4 CP-HA).

**Reserved seams (built as columns/enums, not features):** `link.transport` (D4); site
**DNS-forwarding entries** (S8.4). Reserved, not populated in S8.1.

## D8 — Hub-topology model check · LOCKED (traced, not asserted)

The framing "bricked without relay" is **REJECTED** — the no-relay limitation stays named-and-deliberate
(batch-3 #3: NAT-traversal is a documented deployment requirement; STUN/relay parked). The model must
**not** assume full-mesh direct connectivity.

**Trace through the drafted schema (D6) — hub-and-spoke of existing pieces, no new mechanism:**

- Site A (NAT'd) — its node `nA` has a **link** `{src: nA, dst: nH, transport: wireguard}`, WG dialing
  **outbound** to hub node `nH`.
- Site B (NAT'd) — its node `nB` has a **link** `{src: nB, dst: nH, transport: wireguard}`, likewise
  outbound to `nH`.
- Hub `nH` (publicly reachable `tunnex-node`) — has both peers; its gateway **forwards** between the two
  WG interfaces (the same wg0→wg0 FORWARD the device-to-device path already uses).
- `site_subnets`: A's subnet + B's subnet, each **advertised → approved** (D5). ZT **grants** gate A↔B
  reachability (D3) exactly as for any destination.

The **link model is per-link (node↔node), not per-site-pair-mesh** — nothing in the drafted tables
requires an A↔B direct link. Hub topology is therefore a **supported arrangement of existing pieces**
(per-link modeling + normal WG outbound + forwarding + ZT grants), needing **no STUN, no relay fleet,
no new mechanism.** Holds by construction against the schema as drafted.

## D9 — MSS clamping · noted in S8.1, BINDING for S8.2 commit-one · LOCKED

Double encapsulation (a client-WG session riding a site-WG link) drops effective MTU; PMTUD fails
silently inside tunnels → the "large transfers freeze, pings work" class. This is data-plane behavior of
**propagated routes = S8.2's domain.** S8.1 owes only:

- **(a)** the schema **reserves a per-link MTU attribute** if the model needs one (a nullable `link.mtu`).
- **(b)** **S8.2 commit-one MUST decide** the agent's TCP **MSS clamp** on site-link forward paths (nft,
  in the existing `tunnex` table). *(recorded binding, not built here.)*
- **(c)** the **S8.2 box-walk owes a leg** — a large transfer across a double-encapsulated path,
  **wire-proven.**
- **(d)** the clamp rule is **classified under D2's checklist** — lean node-local / outside the hash
  (the masquerade-rule class), but **classify explicitly, don't assume.**

## D10 — Site-gateway cold-start · LOCKED (restatement + PARKED variant)

The S7.2 cold-start **deny-until-first-fetch** was chosen deliberately and **stands**: cached policy on
boot = a reboot-during-CP-outage **resurrects revoked grants** — for an enforcing org, a **breach**, not
an outage. A cache is that rejected alternative in an HA costume. The proposed cached-policy fallback is
**REJECTED as offered.**

BUT the review's observation is TRUE and gets named: a site gateway is a **LAN router**, so
deny-all-on-cold-start now **severs office-to-office** — a materially larger blast radius than a single
device.

- **(a) RESTATE** the cold-start limitation with the **site-sized blast radius named** (honesty-ledger
  discipline — the limitation is now bigger, say so).
- **(b) PARK a bounded-cache variant.** Trigger = design-partner / HA demand. Constraints **pre-stated
  so a future unpark cannot soften them:** signed artifact + staleness ceiling (minutes) + **NEVER
  applies under `enforcing`** (off-mode mesh restoration only) + org-level opt-in (standing convention).
- **(c) Cross-ref batch-3 #4** (control-plane HA posture, rides S10.1/S11) as the **systemic** answer —
  the cache is a bounded palliative, HA is the real fix.

Decided-in-paper, NOT built.

## D11 — Unlock-then-opt-in + edition gating · LOCKED (the edition split)

**Is site-to-site core or enterprise-gated?** The standing convention binds every new capability
(unlock-then-opt-in, default-OFF, org-level); any part that is an enforcement flag states its
downgrade-release at commit-one (the S7.5.3 law). **Argued split:**

- **Core (open, both edition suites):** the **site model** (D6), **subnet routing**, **advertisement +
  approval** (D5), the **disjointness validator** (D5/D7), the **fail-closed version gate** (D1), and
  **off-mode legacy-mesh reachability**. Rationale: this is plumbing, like the IP allocator / device
  pool — a self-hoster on the open edition can route between their sites.
- **Enterprise (unlocked by edition):** **ZT-governance of site traffic** — i.e. `enforcing`-mode
  gating of site-subnet reachability via grants. This is enterprise **by construction**, because the
  policy engine / `enforcing` mode is already enterprise-gated (`edition_required` 403). Site subnets
  ride the **existing `zero_trust_mode`** switch — **no new opt-in flag** (unlock = enterprise edition
  makes ZT-governance available; the org opts in via the existing `enforcing` mode).
- **Downgrade-release (S7.5.3 law).** The enforcement flag here is `enforcing`-mode gating of site
  traffic. On enterprise→open downgrade it **releases to off-mode legacy mesh**: site subnets become
  **routed but no longer grant-gated** (the same mode-as-compiler-input revert S7.5.3/S7.5.4/S7.5.5
  established). No new flag, so no new downgrade-clear beyond the mode revert — **state it explicitly**
  so it is not silently inherited.
- **Test placement (why the split matters):** core tests (model / routing / advertisement /
  disjointness / version-gate) live in the **open** suite (run in BOTH editions); the
  ZT-governance-of-site-traffic tests (enforcing gates a site subnet; downgrade releases it) live in the
  **enterprise** suite.

## Slice 3 — sites-as-destination: Option A (RULED 2026-07-17) + propagated deltas

**Fork ruled: Option A — version-only bump, NO new AllowEntry field.** The crux (`hash.go:30,41`):
`Version` is IN the CanonicalHash projection, so bumping `ProtocolVersion 3→4` alone changes every
gateway's hash (the twin goldens change from the version number) AND fires the S8.1 gate on any agent
at `maxSupported < 4`. A `device→site-subnet` grant compiles to a **same-shape** `AllowEntry{Src:/32,
Dst:site-CIDR}` — enforcement-identical to a resource grant — so there is no new wire field.

**Why A (deciding argument, recorded):** Option B (a `DstKind` field on the wire) FAILS the very D2
checklist it tried to satisfy — the CIDR already fully determines enforcement, so the label is either
redundant or observability-dressed-as-enforcement (an old agent has no enforcement reason to mis-handle
it until S8.2's routing). D2 exists to REFUSE muddy-classification fields; inventing one to have
something to classify inverts the checklist. Option A's honest answer — the enforcement-significant
change at Slice 3 IS the version bump declaring the sites boundary — is what the projection discipline
looks like when the answer is "no new field."

**A-4 discharge (closure, cite `hash.go:25-28`).** That comment warned Version-in-hash was safe ONLY
while the CP serves a single version fleet-wide, with the warning pointed at EPIC 8. EPIC 8 is here: the
bump makes mixed versions real (piecemeal multi-gateway upgrade). The **S8.1 D1 gate** is what makes
Version-in-hash safe under mixed versions — an old GATED agent receiving v4 **refuses loudly** (deny-all
+ `unsupported_policy_version`) rather than diverging silently. The warning fires and its answer ships
in the SAME slice. (The ungated pre-S8.1 residual stays papered, D1 go-forward-only; same-shape Option A
means even it enforces correctly, only the upgrade-order sentence is owed in the S8 upgrade docs.)

### Conditions (ruled)
1. **D2 records the NULL classification affirmatively** — an explicit "no new AllowEntry field;
   site-subnet dst reuses `DstCIDR` (already in-hash); the version bump itself is the
   enforcement-significant change" row, so a future reader sees the checklist was RUN, not skipped.
2. **Rule-level dst_kind schema integrity** — `policy_rules` gains `site` as a dst kind + `dst_site_id`

3. **Compiler resolution semantics (CP-side, clockless-compiler pattern) — edges pinned:**

   - a site with **N subnets** compiles to **one AllowEntry per subnet**;
   - **Rule-creation against a subnetless site is ALLOWED** (not refused) — the site is a valid target;
     subnets are added independently; refusing would impose an ordering dependency, and the compile
     honestly reflects current state (yields no grant until subnets exist). Stated, not silent.
   Both edges live in the enforcing-gates red's fixture family.
4. **Goldens change gets its own gate-report sentence** — both modules' fixtures bump together with
   IDENTICAL expected hex (the twin-goldens guard is EXERCISED by this slice); the old-hex→new-hex delta
   is attributable to the `version` field ALONE (a confirmation the projection picked up nothing else).

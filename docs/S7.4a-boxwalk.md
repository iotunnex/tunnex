## HAPPY PATHS (UI action → wire cross-check on G1)

### H1 — create a group + a rule → the grant appears on G1's ruleset
UI: **Access → Groups & resources** → add group "walk-src" (add the OWNER as a member is implicit? no —
group membership drives device IPs; for a wire grant you need a device whose owner is in the group).
Simplest wire-observable reuse of what's already there: make a rule whose src is a group containing a
user who HAS an active device on G1, dst a resource.
- UI: add a resource "walk-dst" cidr `9.9.9.7/32` proto any; add group "walk-src"; add the owner to
  "walk-src" (via the members control) ; **Rules → Add rule**: src "walk-src" → resource "walk-dst".
- **Wire cross-check** (owner has active device 10.99.0.x on G1):
  ```bash
  NFT nft list ruleset | grep "9.9.9.7"
  # EXPECT: ip saddr <owner-device-ip> ip daddr 9.9.9.7 ... accept   (the compiled grant)
  ```
  Grant present on the wire = the UI create reached the compiler, not just the DB.

### H2 — enable enforcing → count-confirm + post-hoc affected list
UI: **Zero Trust mode → Enable enforcing**. The confirm dialog states the REAL rule count (wired to
`listPolicyRules`). Confirm.
- **Wire cross-check:**
  ```bash
  curl -s "${AUTH[@]}" "$BASE/api/v1/organizations/$ORG/zero-trust-mode" | jq .mode   # "enforcing"
  NFT nft list ruleset | grep -c "tunnex_default_drop"                                 # >=1 (default-deny live)
  ```
- The UI then shows the server's authoritative `affected_full_tunnel_devices` (post-hoc), NOT a
  client-computed list. (If the org has no full-tunnel devices the list is absent — correct.)

### H3 — approve a pending device → active (the folded S7.3 surface)
Pre-stage a pending device: `Access → Device approval → Require approval`, then enroll one:
```bash

RESP="$(curl -s "${AUTH[@]}" -X POST "$BASE/api/v1/organizations/$ORG/devices" -d "{\"name\":\"walk-pending\",\"node_id\":\"$NODE\"}")"
echo "$RESP" | jq '{status:.device.status, ip:.device.assigned_ip}'   # status "pending"
```
UI: the device shows in **Device approval → Pending devices**. Click **Approve**.
- **Wire cross-check:**
  ```bash

  curl -s "${AUTH[@]}" "$BASE/api/v1/organizations/$ORG/devices" | jq --arg d "$DEV" '.[]|select(.id==$d)|{status,approved_by}'
  # EXPECT: status "active", approved_by = owner uuid
  NFT wg show wg0 | grep "$(echo "$RESP" | jq -r '.device.assigned_ip')"   # peer now present
  ```

## CONDITION PATHS (what the review can't prove, the box can)

### C-a5 — rule EDIT is gap-free (create-then-delete, observable on the wire)
The proof that it's create-THEN-delete (not delete-then-create) is a TRANSIENT DOUBLE grant, never a
zero-grant window. Pick the H1 rule (src "walk-src" → resource "walk-dst" `9.9.9.7`). In one shell start a
tight poll counting that source's grants; in the UI, **Edit** the rule to a DIFFERENT dst resource
(`9.9.9.6/32`, add it first), Save.
```bash
SRCIP=<owner-device-ip>   # the saddr the walk-src grant uses
# tight poll — run DURING the Save:
while true; do echo "$(date +%s.%N) $(NFT nft list ruleset | grep -c "ip saddr $SRCIP ")"; done
```
- **EXPECT** the count go `1 → 2 → 1` (both new+old briefly present), **NEVER `1 → 0 → 1`**. The momentary
  2 IS the create-before-delete ordering on the wire; a 0 would be the forbidden delete-first gap.
- After Save: the new grant (`daddr 9.9.9.6`) present, the old (`daddr 9.9.9.7`) gone.
- (If the poll is too fast to eyeball, `... | grep -c` into a file and inspect — the 2-line is the proof.)

### C-a4 — zero-rules enforcing gate (strong copy + server-is-truth)
The zero-rules COPY is unit-proven; the box confirms the gate FIRES on a real count of 0 and the server
obeys. Cleanest without disturbing existing policy: use a SCRATCH org, OR observe on this org that the
confirm shows the ACTUAL count (N>0 now — proves the count is wired live). To see the zero-path live:
```bash
# on a scratch org with no rules, UI "Enable enforcing" -> the RED "denies ALL incl. your own access" gate.
# server-is-truth: if confirmed, mode flips regardless of count:
curl -s "${AUTH[@]}" "$BASE/api/v1/organizations/$SCRATCH/zero-trust-mode" | jq .mode   # "enforcing"
```
If staging a scratch org is not worth it, the unit test `ZERO rules → strong danger gate` + the live
count-wiring (the dialog shows the real N) is sufficient — say so and move on.

### C-a6 — unresolved vs deleted — LIKELY UNSTAGEABLE LIVE, unit test suffices
The S7.1 cascade-FK Armed Guard DELETES a rule when its referenced group/resource is deleted (no dangling
grant server-side) — so a `(deleted)` referent CANNOT be produced through the API (deleting the group
cascades its rules away). The `(unresolved — refresh)` branch needs a transient list-fetch failure, also
not deterministically stageable by hand. **Both branches are render-path (not enforcement) and are
unit-tested (deleted-referent + failed-load) — take those as sufficient** (per your offer). Nothing to do
live here; noted honestly rather than faked.

# S7.3 ‚Äî Device posture (basic): decision-first (commit-one, ON PAPER, NO code)

**Story (PLAN):** "Device posture (basic) ‚Äî require known device, block untrusted."
**Thesis:** S7.3 adds an org-level **device-approval gate** and reuses the PROVEN S7.2/S3.3 machinery
almost entirely ‚Äî device status is already a peer-desired-state filter AND a compiler input, so an
un-approved device is excluded from both with near-zero new enforcement code. This mirrors the
S7.1/S7.2 pattern exactly: an org-level mode, default OFF (no silent break), that is a COMPILER/
desired-state INPUT, enterprise-gated.

---

## What "basic device posture" means HERE (scope lock)

Every peer credential in tunnex is ALREADY identity-bound: a device is minted only through its owning
user's authenticated session (S3.3), bound to a named user, revocable <5s (S3.1). So "require known
device" is already half-true ‚Äî what S7.3 adds is an **explicit APPROVAL gate**: an enterprise org can
require that a newly-enrolled device be **admin-approved before it gets any network access**. Until
approved it is a *known but untrusted* device ‚Äî blocked.

**NON-GOALS (explicitly deferred to an ADVANCED-posture story, ledgered):** real device attestation ‚Äî
OS/patch level, disk-encryption, MDM/managed-device signals, certificate-based hardware identity,
continuous re-check. "Basic" = the approval gate + known-device requirement, nothing that needs a
client-side posture agent. (S7.3 stays server-side; the desktop client already exists but S7.3 needs no
new client capability.)

---

## LOCKED decisions (mirror S7.1/S7.2, the proven spine)

1. **Org-level setting `device_approval {off, on}`, default `off`.** Off = today's behavior (devices
   auto-active), so ZERO break for existing orgs and the open build. On = new devices land `pending`.
   Enterprise-gated (`edition_required` 403), exactly like `zero_trust_mode`. The setting is a
   compiler/desired-state INPUT, not an enforcement special-case.
2. **Enforcement = reuse the desired-state peer filter, NOT a new path.** An un-approved device is not
   an ACTIVE device, so:
   - it is dropped from `ListActivePeersForNode` ‚Üí **no wg peer served ‚Üí no tunnel** (mode-independent;
     works whether Zero Trust is off or enforcing ‚Äî the strongest, simplest choke point);
   - it is absent from `ListActiveDevicesForOrg` ‚Üí **the compiler never emits its /32 grants** (as a
     source) AND never emits its /32 as a device-to-device DESTINATION in anyone else's grant.
   This is the SAME machinery that drops a revoked device or a deactivated-user's devices. No new
   enforcement code on the gateway.
3. **`<5s` convergence on approve/reject**, reusing the existing recompile+push triggers (device
   create/revoke already push; approve/reject are the same class of device-state change ‚Üí push all
   active org nodes via the S7.2 `PushOrgNodes`).
4. **Reject = `revoked` (auditable), not hard-delete** ‚Äî same terminal state as revocation, so a
   rejected device's key can't silently re-mint and the audit trail is intact.
5. **RBAC: a new `device:approve` permission**, gated to admin/owner like `policy:manage` (approving a
   device is a security-sensitive grant of network access). Approve/reject are audited
   (`device.approved` / `device.rejected`, actor + device id + keyed pubkey fingerprint, never the raw
   key ‚Äî the S4.5 keyed-proof convention).
6. **IP-reuse safety inherited from S7.2/S3.5:** a rejected (‚Üírevoked) device's /32 is stripped from
   every ruleset where it is a destination within <5s, so S3.5 IP reuse can't inherit its old grants.
   This is the F1 full-sweep, already built.

## OPEN decisions ‚Äî need your ruling (recommendations first)

### D1 ‚Äî schema: extend `devices.status`, or a separate `trust` column? **(recommend: extend status)**
- **Option A (recommended): add `pending` to the `devices.status` CHECK** (`active | revoked | pending`).
  Enrollment under `device_approval=on` creates the device `pending`; approve = `pending‚Üíactive`;
  reject = `pending‚Üírevoked`. **Every existing `status='active'` filter ‚Äî the peer query, the compiler
  input, the per-user active-device CAP, the `(node_id, public_key)` uniqueness index ‚Äî AUTOMATICALLY
  excludes `pending` with ZERO query change.** This is the "lean heavily on proven machinery" path you
  flagged.
  - *Deliberate consequence (argued CORRECT):* a `pending` device does NOT count against the per-user
    device cap and does NOT hold pubkey uniqueness until approved. An unapproved device isn't "using" a
    slot; two pending enrollments of the same key are fine until one is approved (approval is where
    uniqueness must hold ‚Äî enforce at the approve transition).
- **Option B: a separate `trust_state` column** (`trusted | pending`), orthogonal to `status`. More
  explicit, but it changes the proven peer + compiler queries (they must now also filter trust) and
  creates a two-dimensional state machine (status √ó trust) with more combinations to reason about.
- **Recommendation: A.** Maximal reuse, single state dimension, and `pending` is genuinely a lifecycle
  state of the device, not an orthogonal attribute.

### D2 ‚Äî config at enrollment: issue immediately, or withhold until approved? **(recommend: issue immediately)**
- **Recommended:** issue the `.conf` at enrollment as today; it simply won't establish a handshake
  (no peer served) until approved, then connects within <5s of approval. The peer-omission IS the gate;
  the config is not a secret that grants access on its own (the gateway decides).
- Alternative: withhold the config until approved. More friction, no added security (the gate is
  server-side regardless). Rejected unless you want the UX signal.

### D3 ‚Äî who approves? **(recommend: admin-only; owner does NOT self-approve)**
- The owning user enrolls (authenticated session, S3.3) ‚Üí `pending`. An **admin/owner with
  `device:approve`** approves. Owner self-approval would defeat the gate (the whole point is a second
  party vouches for the device). Recommend admin-only.

### D4 ‚Äî retroactive apply when an org flips `device_approval` off‚Üíon? **(recommend: grandfather existing)**
- Existing `active` devices STAY active (grandfathered) when the setting turns on; only devices enrolled
  AFTER the flip are `pending`. Flipping it on must not black-hole an org's whole fleet mid-day (the
  S7.2 cold-start lesson: a mode flip must never be a surprise outage). An admin can still revoke any
  grandfathered device manually. Alternative (mass re-approval) is a bigger hammer ‚Äî defer to advanced.

### D5 ‚Äî auto-approval heuristics (known network, prior device, etc.)? **(recommend: NONE in basic)**
- Basic = manual approval only. Auto-approval (trust-on-known-source-IP, re-enroll-of-prior-key) is
  advanced posture ‚Äî ledger it, don't build it. Keeps S7.3 small and the security model obvious.

## Migration + surface (sketch ‚Äî for the review, not yet code)

- **Migration `0020`:** `devices.status` CHECK ‚Üí add `'pending'`; `organizations` add
  `device_approval text NOT NULL DEFAULT 'off' CHECK (device_approval IN ('off','on'))`.
- **Enrollment:** device-mint path reads the org setting; creates `pending` when on, `active` when off.
- **New endpoints (RBAC `device:approve`, enterprise-gated):** list pending devices; approve (‚Üíactive,
  push org nodes); reject (‚Üírevoked, push org nodes). Both audited.
- **Setting endpoint:** get/set `device_approval` (RBAC `device:approve` or a dedicated
  `org:update`-class perm ‚Äî D-side note), audited, enterprise-gated.
- **Peer query / compiler: UNCHANGED under Option A** (the `status='active'` filter already excludes
  pending) ‚Äî this is the whole point.

---

## DoD + box-proof (on the wire, the S7.2 bar)

- `device_approval=on`: enroll a device ‚Üí it is `pending` ‚Üí **no handshake** (peer not served), timed.
- Admin **approve** ‚Üí the peer appears and the device handshakes **within <5s**, timed.
- **Reject** ‚Üí device `revoked`, stays blocked; and under `zero_trust_mode=enforcing`, a pending/rejected
  device's /32 is **absent** as both a source grant and a device-to-device destination (nft proof).
- `device_approval=off` (default): enrollment is immediately `active` (no behavior change) ‚Äî the
  no-break proof.
- Open build: no approval concept; setting is always off.
- Deliberate-reds: pending-‚âÝ-peer; approve-lands-<5s; reject-strips-/32-everywhere (IP-reuse safe);
  grandfather-on-flip (no mass outage); flip off‚Üíon‚Üíoff round-trips with no surprise black-hole.

---

## Why this is small (the S7.2 machinery it reuses, unchanged)

- Peer desired-state drop + <5s push + /32 full-sweep ‚Äî **S3.1/S3.3/S6.4/S7.2**, proven on the wire.
- Compiler device input (pending auto-excluded) ‚Äî **S7.1/S7.2**, twin-golden-pinned.
- Org-level mode setting, default-off, enterprise-gated, compiler-input pattern ‚Äî **S7.1/S7.2**.
- Audit + keyed-proof-of-secret fingerprint ‚Äî **S4.5/S4.8**.
- New code is concentrated in: migration 0020, the enrollment branch, three thin admin endpoints +
  RBAC perm + audit. No new gateway/enforcement code.

- Mass re-approval on off‚Üíon flip (D4 alternative) ‚Äî advanced.

## D1a ‚Äî device-status CONSUMER audit (every consumer, not just excluders ‚Äî no fall-throughs)

`pending` is added to `devices.status`. Each consumer classified + its deliberate handling:

| `ListActiveFullTunnelDevices` (SetMode enum + full-tunnel) | `status='active'` | EXCLUDE | none (auto) |
| policy compiler input (`ListActiveDevicesForOrg`) | active-only | EXCLUDE ‚Üí no grants, no /32 as dst | none (auto ‚Äî verify at code) |
| `CountActiveDevicesForUser` (per-user CAP) | `status='active'` | NOT counted ‚Äî a pending device isn't using a slot (D1 deliberate) | none (auto) |
| **`ListActiveDeviceAllocations`** (the SINGLE live-allocation def ‚Äî create's lowest-free AND resize's orphan check) | `status='active'` | **INCLUDE ‚Äî a pending device HOLDS a real `assigned_ip` from creation; excluding it makes the allocator see the IP as free ‚Üí reallocate ‚Üí UNIQUE(org,ip) violation on the pending row / collision on approval.** | **FIX: `status IN ('active','pending')`** |

**D1b ‚Äî reject inherits the FULL sweep incl. the pool address.** `RejectDevice` mirrors `RevokeDevice`:
`status='revoked'`, `revoked_at=now()`, **`assigned_ip=NULL`** ‚Üí the pool address the device has held since
creation is FREED for reuse, and the /32 is stripped from every ruleset where it is a destination (<5s,
the F1 sweep). **Deliberate-red:** reject a pending device ‚Üí its IP returns to the free pool (a later
`ListActiveDeviceAllocations` no longer lists it; a new device can take it).

## D2 ‚Äî property recorded honestly
Approval gates **CONNECTIVITY, not KNOWLEDGE.** The config is issued at enrollment, so a device that is
later rejected HAS seen the gateway endpoint + its would-be tunnel address (it just never got a
handshake). Acceptable for basic posture. **Advanced-posture ledger note:** attestation flows may revisit
ISSUANCE ORDERING (withhold endpoint/config until approved) ‚Äî recorded, not built here.

## D3 ‚Äî rationale FIXED (the threat is SELF-approval, not owners)
Owners OUTRANK admins; gating owners would be wrong. `device:approve` = **admin + owner** (the
`policy:manage` grain). The designed-against case is **SELF-approval** ‚Äî approving your OWN device, which
removes the second-party vouch. Decision: **admin self-approval of their own device is ALLOWED but
distinctly audited** as `device.self_approved` (vs `device.approved` for another user's device). **Four-eyes
(hard-block self-approval) is ADVANCED posture** ‚Äî ledgered. The doc NAMES self-approval as the
designed-against case so the audit distinction is intentional, not incidental.

## D4 ‚Äî grandfather WITH visibility
Flipping `device_approval` off‚Üíon **grandfathers existing `active` devices** (no mass black-hole ‚Äî the
S7.2 cold-start lesson). Visibility: (a) `SetDeviceApproval` **records + returns the grandfathered COUNT**
(the `zero_trust_mode` affected-device ENUMERATION pattern ‚Äî server obeys, response reports blast radius);
(b) schema captures grandfathered ‚âÝ human-approved via **`devices.approved_by uuid NULL`** ‚Äî **NULL =
grandfathered / auto-active-under-off; set = explicitly approved by that actor.** S7.4 renders the
distinction; the DATA exists now.

## D5 ‚Äî no auto-approval in basic (approved as recommended). Advanced-posture ledger.

## SPINE ADDITION ‚Äî the pending / rejected CLIENT experience (LOCKED)
- **Enrollment response carries `pending_approval` explicitly** (a boolean/state field) so the client
  KNOWS it is awaiting approval ‚Äî not a silent "connected-but-nothing-works".
- **Rejection is client-observable AS revocation:** the S6.4 teardown fires, config cleared ‚Äî a rejected
  device behaves exactly like a revoked one on the client (the machinery already exists).
- **Pending is a STABLE awaiting-approval state, never an error loop** ‚Äî the client shows "waiting for
  approval", polls calmly, does not thrash or surface an error.
- **VERIFY the desktop `RevocationMonitor` / `deviceExists` does not MISREAD either state** ‚Äî the S6.4
  lesson (deviceExists THROWS on empty-orgs-200 so a blip can't false-revoke): now it must also treat
  `pending` as "exists, awaiting" (NOT gone) and `rejected` as revoked (teardown). No false-revoke of a
  pending device; no missed teardown of a rejected one.

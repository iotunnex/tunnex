# S7.3 — Device posture (basic): decision-first (commit-one, ON PAPER, NO code)

**Story (PLAN):** "Device posture (basic) — require known device, block untrusted."
**Thesis:** S7.3 adds an org-level **device-approval gate** and reuses the PROVEN S7.2/S3.3 machinery
almost entirely — device status is already a peer-desired-state filter AND a compiler input, so an
un-approved device is excluded from both with near-zero new enforcement code. This mirrors the
S7.1/S7.2 pattern exactly: an org-level mode, default OFF (no silent break), that is a COMPILER/
desired-state INPUT, enterprise-gated.

---

## What "basic device posture" means HERE (scope lock)

Every peer credential in tunnex is ALREADY identity-bound: a device is minted only through its owning
user's authenticated session (S3.3), bound to a named user, revocable <5s (S3.1). So "require known
device" is already half-true — what S7.3 adds is an **explicit APPROVAL gate**: an enterprise org can
require that a newly-enrolled device be **admin-approved before it gets any network access**. Until
approved it is a *known but untrusted* device — blocked.

**NON-GOALS (explicitly deferred to an ADVANCED-posture story, ledgered):** real device attestation —
OS/patch level, disk-encryption, MDM/managed-device signals, certificate-based hardware identity,
continuous re-check. "Basic" = the approval gate + known-device requirement, nothing that needs a
client-side posture agent. (S7.3 stays server-side; the desktop client already exists but S7.3 needs no
new client capability.)

---

## LOCKED decisions (mirror S7.1/S7.2, the proven spine)

1. **Org-level setting `device_approval {off, on}`, default `off`.** Off = today's behavior (devices
   auto-active), so ZERO break for existing orgs and the open build. On = new devices land `pending`.
   Enterprise-gated (`edition_required` 403), exactly like `zero_trust_mode`. The setting is a
   compiler/desired-state INPUT, not an enforcement special-case.
2. **Enforcement = reuse the desired-state peer filter, NOT a new path.** An un-approved device is not
   an ACTIVE device, so:
   - it is dropped from `ListActivePeersForNode` → **no wg peer served → no tunnel** (mode-independent;
     works whether Zero Trust is off or enforcing — the strongest, simplest choke point);
   - it is absent from `ListActiveDevicesForOrg` → **the compiler never emits its /32 grants** (as a
     source) AND never emits its /32 as a device-to-device DESTINATION in anyone else's grant.
   This is the SAME machinery that drops a revoked device or a deactivated-user's devices. No new
   enforcement code on the gateway.
3. **`<5s` convergence on approve/reject**, reusing the existing recompile+push triggers (device
   create/revoke already push; approve/reject are the same class of device-state change → push all
   active org nodes via the S7.2 `PushOrgNodes`).
4. **Reject = `revoked` (auditable), not hard-delete** — same terminal state as revocation, so a
   rejected device's key can't silently re-mint and the audit trail is intact.
5. **RBAC: a new `device:approve` permission**, gated to admin/owner like `policy:manage` (approving a
   device is a security-sensitive grant of network access). Approve/reject are audited
   (`device.approved` / `device.rejected`, actor + device id + keyed pubkey fingerprint, never the raw
   key — the S4.5 keyed-proof convention).
6. **IP-reuse safety inherited from S7.2/S3.5:** a rejected (→revoked) device's /32 is stripped from
   every ruleset where it is a destination within <5s, so S3.5 IP reuse can't inherit its old grants.
   This is the F1 full-sweep, already built.

## OPEN decisions — need your ruling (recommendations first)

### D1 — schema: extend `devices.status`, or a separate `trust` column? **(recommend: extend status)**
- **Option A (recommended): add `pending` to the `devices.status` CHECK** (`active | revoked | pending`).
  Enrollment under `device_approval=on` creates the device `pending`; approve = `pending→active`;
  reject = `pending→revoked`. **Every existing `status='active'` filter — the peer query, the compiler
  input, the per-user active-device CAP, the `(node_id, public_key)` uniqueness index — AUTOMATICALLY
  excludes `pending` with ZERO query change.** This is the "lean heavily on proven machinery" path you
  flagged.
  - *Deliberate consequence (argued CORRECT):* a `pending` device does NOT count against the per-user
    device cap and does NOT hold pubkey uniqueness until approved. An unapproved device isn't "using" a
    slot; two pending enrollments of the same key are fine until one is approved (approval is where
    uniqueness must hold — enforce at the approve transition).
- **Option B: a separate `trust_state` column** (`trusted | pending`), orthogonal to `status`. More
  explicit, but it changes the proven peer + compiler queries (they must now also filter trust) and
  creates a two-dimensional state machine (status × trust) with more combinations to reason about.
- **Recommendation: A.** Maximal reuse, single state dimension, and `pending` is genuinely a lifecycle
  state of the device, not an orthogonal attribute.

### D2 — config at enrollment: issue immediately, or withhold until approved? **(recommend: issue immediately)**
- **Recommended:** issue the `.conf` at enrollment as today; it simply won't establish a handshake
  (no peer served) until approved, then connects within <5s of approval. The peer-omission IS the gate;
  the config is not a secret that grants access on its own (the gateway decides).
- Alternative: withhold the config until approved. More friction, no added security (the gate is
  server-side regardless). Rejected unless you want the UX signal.

### D3 — who approves? **(recommend: admin-only; owner does NOT self-approve)**
- The owning user enrolls (authenticated session, S3.3) → `pending`. An **admin/owner with
  `device:approve`** approves. Owner self-approval would defeat the gate (the whole point is a second
  party vouches for the device). Recommend admin-only.

### D4 — retroactive apply when an org flips `device_approval` off→on? **(recommend: grandfather existing)**
- Existing `active` devices STAY active (grandfathered) when the setting turns on; only devices enrolled
  AFTER the flip are `pending`. Flipping it on must not black-hole an org's whole fleet mid-day (the
  S7.2 cold-start lesson: a mode flip must never be a surprise outage). An admin can still revoke any
  grandfathered device manually. Alternative (mass re-approval) is a bigger hammer — defer to advanced.

### D5 — auto-approval heuristics (known network, prior device, etc.)? **(recommend: NONE in basic)**
- Basic = manual approval only. Auto-approval (trust-on-known-source-IP, re-enroll-of-prior-key) is
  advanced posture — ledger it, don't build it. Keeps S7.3 small and the security model obvious.

## DoD + box-proof (on the wire, the S7.2 bar)

- `device_approval=on`: enroll a device → it is `pending` → **no handshake** (peer not served), timed.
- Admin **approve** → the peer appears and the device handshakes **within <5s**, timed.
- **Reject** → device `revoked`, stays blocked; and under `zero_trust_mode=enforcing`, a pending/rejected
  device's /32 is **absent** as both a source grant and a device-to-device destination (nft proof).
- `device_approval=off` (default): enrollment is immediately `active` (no behavior change) — the
  no-break proof.
- Open build: no approval concept; setting is always off.
- Deliberate-reds: pending-≠-peer; approve-lands-<5s; reject-strips-/32-everywhere (IP-reuse safe);
  grandfather-on-flip (no mass outage); flip off→on→off round-trips with no surprise black-hole.

the F1 sweep). **Deliberate-red:** reject a pending device → its IP returns to the free pool (a later
`ListActiveDeviceAllocations` no longer lists it; a new device can take it).

## D2 — property recorded honestly
Approval gates **CONNECTIVITY, not KNOWLEDGE.** The config is issued at enrollment, so a device that is
later rejected HAS seen the gateway endpoint + its would-be tunnel address (it just never got a
handshake). Acceptable for basic posture. **Advanced-posture ledger note:** attestation flows may revisit
ISSUANCE ORDERING (withhold endpoint/config until approved) — recorded, not built here.

## D3 — rationale FIXED (the threat is SELF-approval, not owners)
Owners OUTRANK admins; gating owners would be wrong. `device:approve` = **admin + owner** (the
`policy:manage` grain). The designed-against case is **SELF-approval** — approving your OWN device, which
removes the second-party vouch. Decision: **admin self-approval of their own device is ALLOWED but
distinctly audited** as `device.self_approved` (vs `device.approved` for another user's device). **Four-eyes
(hard-block self-approval) is ADVANCED posture** — ledgered. The doc NAMES self-approval as the
designed-against case so the audit distinction is intentional, not incidental.

## D4 — grandfather WITH visibility
Flipping `device_approval` off→on **grandfathers existing `active` devices** (no mass black-hole — the
S7.2 cold-start lesson). Visibility: (a) `SetDeviceApproval` **records + returns the grandfathered COUNT**
(the `zero_trust_mode` affected-device ENUMERATION pattern — server obeys, response reports blast radius);
(b) schema captures grandfathered ≠ human-approved via **`devices.approved_by uuid NULL`** — **NULL =
grandfathered / auto-active-under-off; set = explicitly approved by that actor.** S7.4 renders the
distinction; the DATA exists now.

## D5 — no auto-approval in basic (approved as recommended). Advanced-posture ledger.

## SPINE ADDITION — the pending / rejected CLIENT experience (LOCKED)
- **Enrollment response carries `pending_approval` explicitly** (a boolean/state field) so the client
  KNOWS it is awaiting approval — not a silent "connected-but-nothing-works".
- **Rejection is client-observable AS revocation:** the S6.4 teardown fires, config cleared — a rejected
  device behaves exactly like a revoked one on the client (the machinery already exists).
- **Pending is a STABLE awaiting-approval state, never an error loop** — the client shows "waiting for
  approval", polls calmly, does not thrash or surface an error.
- **VERIFY the desktop `RevocationMonitor` / `deviceExists` does not MISREAD either state** — the S6.4
  lesson (deviceExists THROWS on empty-orgs-200 so a blip can't false-revoke): now it must also treat
  `pending` as "exists, awaiting" (NOT gone) and `rejected` as revoked (teardown). No false-revoke of a
  pending device; no missed teardown of a rejected one.

# BUILD FINDINGS (recorded as they surface)

## Story-end review folds (6 findings — the pending state slipping pre-existing invariants)
The story-end review (touched-merged-surface Create/Revoke push change reviewed CLEAN) found 6 real
issues, all folded:

- **#1 (severe) — per-user cap bypass + org-pool DoS.** The cap counted only `status='active'`, so a user
  could create unbounded PENDING devices (each reserving a pool /32); an admin approving them all exceeds
  the cap, or the pending devices exhaust the org pool for everyone. Fix: `CountDevicesForUserCap` counts
  `active+pending`; create-at-cap refuses `device_limit`. Guard: `TestPendingDevicesCountAgainstCap`.
- **#2 — node-revoke skipped pending.** `RevokeDevicesForNode` filtered `active`, so a pending device on a
  revoked node leaked its /32 forever + lingered in the queue on a dead gateway. Fix: sweep `active+pending`
  + free the IP (full sweep). Guard: `TestNodeRevokeSweepsPendingAndFreesIP`.
- **#3 — client mode-change-while-pending dropped + no user-cancel.** The pending short-circuit ran before
  the split↔full re-mint check (mode toggle silently ignored), and an owner had no way to cancel a pending
  device. Fix: reorder (mode-mismatch first) + widen `RevokeDevice` to `active+pending` so an owner CANCELS
  their own pending device (audited `device.cancelled` ≠ `device.revoked`, IP freed). Guards:
  `TestOwnerCancelPendingAuditedDistinctly` + client `mode change while pending re-mints`.
- **#4+#6 — deviceStatus false-"gone" on an org-omitting blip / DRY.** The all-orgs scan could false-"gone"
  when a transient list omitted the device's org → false teardown. Fix: persist the device's `orgId` (at
  create) and query it DIRECTLY (fetch error → throw inconclusive; absent-in-its-own-org → gone);
  `deviceExists = deviceStatus === "active"` (ONE fail-safe — the two monitors can't diverge). Guards:
  `deviceStatus queries the OWN org directly` + `deviceExists = deviceStatus === "active"`.
- **#5 — awaiting poll leaked across a server change.** `config:setServerUrl` stopped the RevocationMonitor
  but not the ApprovalMonitor. Fix: `stopApprovalMonitor()` there too; recorded as the ORIGIN-LIFECYCLE
  rule in the ApprovalMonitor discipline list.

## Scoped re-review fold (a regression the FIRST fold introduced)
The scoped re-review found the `orgId` fold (#4/#6) introduced a MIGRATION/BACKFILL gap: a config
persisted by a pre-`orgId` build (the installed base / v0.1.0) has no `orgId`, so the monitors + self-heal
built `/organizations/undefined/devices`, threw, and went permanently inconclusive — **silently disabling
client-side revocation teardown + self-heal for every upgrader** (a real security regression). Folded:

- **#1–#5 (the regression) — scan fallback + opportunistic stamping.** `deviceStatus(deviceId, orgId)`
  now: PRESENT orgId → direct-query hardening (finding #4); FALSY orgId (legacy config) → the all-orgs
  SCAN with the empty-orgs-inconclusive throw intact, so an upgrader's monitors keep working and a
  revocation IS detected (never a malformed URL / false gone). `deviceExists = deviceStatus === "active"`
  inherits both. **FORWARD LINE (decided): legacy configs opportunistically STAMP `orgId` on the next
  resolve** (`resolveTunnelConfig` resolves the device's org via the scan and stores it), migrating onto
  the direct path so the scan path retires — NOT scan-forever. Reds:
  `revocation: legacy blank-orgId config still tears down via the scan fallback` (the regression red) +
  `deviceStatus blank orgId falls back to scan` + `resolveTunnelConfig stamps orgId onto a legacy config`.
- **#6 — locking label read.** Revoke's prior-status read (audit label cancelled vs revoked) is now
  `GetDeviceForUpdate` (`FOR UPDATE`) so a concurrent Approve serializes — no permanent mislabel in the
  append-only audit log (the cancel feature made that distinction load-bearing).
- **#7 — accepted (rationale recorded in code):** the handler's pre-tx ownership read and the service's
  in-tx locked label read serve DISTINCT layers (authorization vs the atomic label); #6's fix makes the
  in-tx read necessary, not redundant — a comment prevents a future cleanup from coupling the layers back.

### LESSON (recorded): migration-sensitive changes are tested against configs that EXIST, not only the
configs the change creates. The `orgId` fold was green against its own output and RED against the
installed base — the persisted-shape change had no backfill/fallback for pre-existing stored state. Any
change that adds a REQUIRED field to a persisted structure gets a "legacy shape (field absent)" test.

## REDUCTION (the escalation fired — the scan surface is DELETED)
The scan-fallback took THREE rounds, each finding a correctness bug, and the third was FOLD-INDUCED (the
single-fetch refactor made `resolveDeviceOrg` return a non-null org for a found-but-revoked device,
violating its own `null = gone` contract → a revoked legacy device kept + armed instead of re-minted).
Intervention base-rate = one-defect-per-fold; a fourth fold bets against that record. Per the pre-set
escalation, we STOPPED folding and REDUCED: **deleted the entire scan fault class** — `scanDevice`,
`resolveDeviceOrg`, the falsy-`orgId` fallback, the stamping, the complete-information machinery, and the
#2 transitional limitation (a documented limitation for deleted code is checkpoint-drift; deletion
satisfies its retirement condition immediately and maximally).

**New model (SIMPLE):** `orgId` is REQUIRED. New configs persist it at create; a LEGACY config (no
`orgId`) is **NEVER queried** — on next connect it is dropped + best-effort revoked (origin-keyed, S6.3;
revoke failure → orphan for admin-reap, loss=recreate doctrine) + **RE-MINTED** as a fresh device that
carries `orgId`. LOUD, not a silent identity swap: `connect()` fires a one-time "device re-created" notice
(and the pending gate raises "awaiting approval" when `device_approval=on`, since the fresh device is
pending) — carried by the existing notify machinery, no new UX surface. The monitors STRUCTURALLY never
run without an `orgId` (`deviceStatus` throws on a blank one; the re-mint happens before a monitor starts).
Red: `a no-orgId (legacy) config is re-minted, never queried` (0 query calls; fresh device; old revoked).

The COST lands on the SMALLEST legacy population this product will ever have (v0.1.0 = inner-circle,
pre-distribution) at the CHEAPEST moment — nearly free today, painful post-beta.

**CONVENTION that OUTLIVES the deleted code (the [1] lesson survives):** a DESTRUCTIVE verdict requires
COMPLETE information; partial/failed information yields UNKNOWN, never a destructive "gone" (a monitor keeps
+ backs off on a blip, never tears down). The code that taught it is gone; the rule stays.

## SECOND DEMONSTRATED LESSON — the recurring query/migration-plumbing surface
The client status-query surface ran the FULL S7.2 arc in miniature: three rounds of findings → escalation →
REDUCTION (like S7.2's staleness collapse to one field). The **enforcement core stayed clean throughout**;
the **query/migration plumbing** around it kept generating findings until the surface itself was deleted,
not just hardened. HEURISTIC (stands): when a change touches a client-side status/query path, the review
explicitly asks **(a) what does it do on PARTIAL/FAILED data? and (b) what happens to configs that ALREADY
EXIST?** — this story's two regressions were each exactly one of those questions unasked. And the process
worked: **stopping WITH the one-line fix in hand — refusing the fourth fold in favor of the reduction — was
the escalation rule doing its job.**

## CONVENTION (taxonomy lesson from #1 + #2 + the allocator finding)
**A device status is audited on TWO axes: ENFORCEMENT and RESOURCE ACCOUNTING.** `pending` is EXCLUDED from
enforcement (no peer, no grants — the two-layer `status='active'` filter) but INCLUDED in resource
accounting (the per-user CAP, the org POOL/allocator, and every SWEEP — node-revoke, reject, cancel). The
D1a consumer audit classified consumers by **exclusion-correctness** and missed the **accounting**
direction (the cap was mis-called "not counted"; `RevokeDevicesForNode` wasn't audited at all). **Every
future `status` value gets audited on BOTH axes** — does it enforce? and does it hold a resource (cap slot,
pool address) that a cap/sweep must account for?


## Commit-six pin — DEMONSTRATED CATCH (a shipped bug prevented by the investigation-before-code)
The pin asked how the desktop `RevocationMonitor` classifies the NEW exists-but-pending state
BEFORE writing client code. The investigation found the clean model ("monitor never sees pending")
was NOT the current behavior: `connect()` (ipc.ts) unconditionally `tunnel.up()`s then
`monitor.start()`s for the resolved deviceId, with NO check of `pending_approval`. So a user hitting
connect on a pending device would: arm the helper on a config whose peer does not exist server-side
(a DEAD tunnel), then within ≤30s the `RevocationMonitor` probes → `deviceExists` returns false
(pending is `status !== "active"`) → **teardown + a spurious "revoked" notification.** Wrong: pending
is not revoked. Caught on paper, never shipped. Fix (commit-six): (1) a CONNECT-GATE — a pending
device does not arm the tunnel or start the monitor; (2) a SEPARATE app-level awaiting-approval poll.
The `active→pending` transition does NOT exist (state machine: pending→active, pending→revoked,
active→revoked), so once the gate exists the monitor provably never encounters pending — not a
decide-item. Rejected classifies as `deviceExists=false` = definitive gone→teardown (never the
throw→backoff blip path).


## F1-part-3 (LESSON, + a touched-merged-surface fix)
**Every device-lifecycle event that changes COMPILED MEMBERSHIP pushes ORG-WIDE.** Approve / create /
revoke / deactivate now all `PushOrgNodes`; **"own-node push" is only ever correct for an event with NO
compiled-membership effect** — i.e. reject of a *pending* device, which was never in any allow-set.

Discovered via pin 3: device `Create`/`Revoke` pushed OWN-NODE only. Two different severities:
- **`Revoke` → org-wide is a CORRECTNESS / SECURITY fix (fail-OPEN-shaped).** A revoked device's /32 can be
  a group-resolved DESTINATION in compiled allow-sets on gateways that do NOT host it. Own-node push leaves
  that stale allow rule on the OTHER gateways for up to the reconcile interval — and revoke FREES the IP
  (S3.5/D1b, re-proven), so a reallocation inside that window hands the NEW holder the old device's
  group-dst grants on the unreconciled gateways: **the address-reuse privilege leak** (F1-part-2's fail-open
  sibling, at the push layer).
- **`Create` → org-wide is the consistency fix (fails CLOSED).** A late group-dst grant is an availability
  delay, not a leak.
- Guards (multi-node F1-part-2 shape): `TestCreateAndRevokePushOrgWide` (a node NOT hosting the device is
  nudged on both create and revoke) + `TestApprovePushesOrgWideRejectOwnNode`.
- **TOUCHED-MERGED-SURFACE (flag for the story-end review):** this edits S7.2-merged push code
  (`devices.Service.Create` / `Revoke`). The reviewer should examine it deliberately — a push-scope change
  to the merged data-plane path, not new S7.3 code.

## DEMONSTRATED-RED — the widened unique index (an inventory-worthy guard lesson)
`devices_org_ip_key` was PARTIAL on `status='active'` ONLY — silently scoped NARROWER than the allocator's
in-flight definition once `pending` holds a real IP. So a pending device's held IP was unprotected by the
DB backstop: a new ACTIVE device could take it (the partial index can't see the pending row) → a **silent
duplicate** that surfaces (as a hard failure) only when the pending device is approved. Fixed: the index is
widened to `status IN ('active','pending')`, sharing ONE definition of "holds an IP" with
`ListActiveDeviceAllocations` (cross-referenced in the migration AND the query so they can't drift).
**GUARD LESSON:** a partial UNIQUE index and the allocator query must share a single definition of

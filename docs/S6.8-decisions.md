# S6.8 — Quit continuity + fast fail-closed recovery (decision-first)

Full-tunnel is now real (S3.7). Live use surfaced a UX problem in the kill-switch's
bounded fail-closed model: **quitting the app blackholes the internet for up to ~90s**.
That is the dead-man window, and it fires on EVERY app exit — including a clean quit —
because the app never tells the helper to come Down before it dies.

This story makes exit + recovery minimally disruptive WITHOUT weakening the fail-closed
guarantee (no cleartext leak on abnormal death).

## Root cause
- The desktop app has **no `before-quit` handler**. On Cmd-Q / tray Quit / window-close
  (non-macOS) it calls `app.quit()` directly. The helper's owner IPC socket drops →
  `Supervisor.OnPeerLost()` → FailClosed (tunnel torn, pf block HELD) → the dead-man
  releases only after the full `DeadManDefault` (90s). So even a deliberate quit strands
  the user's internet for ~90s.
- `OnPeerLost` (a DEFINITIVELY closed socket) reuses the SAME 90s window as a merely
  missed-heartbeat-on-an-open-socket. A closed socket means the owner is truly gone
  (crash / SIGKILL), so the conservative full window is unnecessary there.

## Decisions
1. **Clean quit → graceful Down (app side).** Add an Electron `before-quit` handler that
   `preventDefault()`s once, calls the existing `controls.disconnect()` (which sends the
   helper a graceful Down → restore routing + release pf), then quits. Bounded by a ~3s
   timeout so a hung helper can never block quit. Result: Cmd-Q / tray Quit / window-close
   restore the internet **immediately**, never touching the dead-man. This is the common
   case and the biggest win.
2. **Definitive-death gets a SHORTER dead-man (helper side).** Split the one window in two:
   - `DeadManDefault` (90s) — the tunnel is up but heartbeats stopped on a STILL-OPEN
     socket (app wedged/slow). Unchanged — we can't tell "wedged" from "briefly slow", so
     stay conservative (and monotonic-clock-safe across sleep, per S6.3 review #9).
   - `DeadManOrphan` (5s) — the owner IPC socket was DEFINITIVELY CLOSED (`OnPeerLost`).
     The app is gone; we keep only a brief grace so a transient hiccup doesn't drop the
     kill-switch (releasing too eagerly would fail OPEN — a moment of cleartext). This cuts
     the force-quit / crash blackhole from ~90s to ~5s while STILL failing closed. Not
     lower: near-0 is effectively fail-open on a crash. Clean quit (decision 1) is ~0s.
   A crash/SIGKILL can run no handler, so decision 1 can't help it — decision 2 bounds it.
3. **Tick the dead-man loop on the MIN of the two windows.** The loop ticked every
   `DeadManDefault/3` (30s); a 12s orphan window would otherwise not fire until the next
   30s tick. Tick on `min(deadMan, deadManOrphan)/3` so the short window is honored with
   fine granularity (~4s), overhead negligible (a lock + a time compare).

## Non-goals / invariants preserved
- **No fail-OPEN.** Both windows still HOLD the block until they expire — abnormal death
  never reverts to cleartext instantly. The kernel-resident kill-switch (pf anchor / WFP)
  and CleanStale recovery are untouched; `kill -9` of the HELPER still relies on the next
  start's self-heal.
- **Monotonic clock** stays `time.Now` (pauses across sleep) — a healthy tunnel is not
  released on resume before the first post-resume heartbeat.
- Windows parity: the orphan window is platform-agnostic (it's in the Supervisor, above
  the Backend), so it applies once the Windows kill-switch persistence lands (S6.7).

## DoD
- App `before-quit` → graceful Down (timeout-bounded); clean quit restores internet ~instantly.
- Supervisor: `DeadManOrphan` window used iff the owner connection was lost (`OnPeerLost`);
  `DeadManDefault` unchanged for heartbeat-silence-on-open-socket. Dead-man loop ticks on
  the min window. Env overrides: `TUNNEX_DEADMAN`, `TUNNEX_DEADMAN_ORPHAN`.
- Tests: orphaned path releases after the short window, wedged path after the full window,
  a fresh Up clears the orphan flag. App before-quit unit-covered (disconnect called + quit
  proceeds even on disconnect error/timeout).
- Proof: clean quit while full-tunnel'd → internet back within ~1s; force-quit → back within
  ~5–7s (not ~90s); fail-closed still holds during the interim (no cleartext leak).

# S6.5a Windows proof runbook (unsigned)

Full parity with the macOS `.pkg`: a real **NSIS `.exe` installer** that registers the helper
service at INSTALL time (one UAC prompt) and removes it via Add/Remove Programs. The bare box
needs **nothing pre-installed**. Needs a reachable tenant + active gateway (same as macOS).

Artifact (built on the Mac): `apps/client/release/Tunnex-Setup-0.1.0.exe` — bundles `Tunnex.exe`,
the SCM-service-capable helper + `wintun.dll`, and the install/uninstall service hooks
(`build/nsis/installer.nsh`).

Rebuild on the Mac: `cd apps/client && bash scripts/stage-helper.sh win && CSC_IDENTITY_AUTO_DISCOVERY=false npx electron-builder --win --x64 --config electron-builder.yml`.

---

## Setup / install (on the box)
1. Copy **`Tunnex-Setup-0.1.0.exe`** to the box.
2. Run it → SmartScreen "Windows protected your PC" → **More info → Run anyway** → the
   installer is **elevated (UAC)** and registers the helper service during install.
3. Confirm the service is up (PowerShell): `sc query tunnex-helper`
   - **PASS:** **STATE: 4 RUNNING** — proves the Go SCM service dispatcher works (a plain
     console binary would time out 1053).

---

## Proof A′ — Acceptance: launch → connect → reach, zero terminal
1. Launch **Tunnex** (Start menu / desktop shortcut the installer created).
2. Enter the server URL → sign in → **Connect** (split tunnel). The service is already running
   from install, so **no per-connect prompt**.
   - **PASS:** status → **Connected** with an IP + a fresh handshake; reach an org host. Zero
     terminal.

If Connect fails with a caller-auth error, grab the helper log: `Get-Content C:\Windows\Temp\tunnex-helper.log -Tail 20` — the caller image path must be inside `TUNNEX_INSTALL_DIR` (the install dir, `C:\Program Files\Tunnex`).

---

## Proof 1 — WFP live kill-switch pcap (the deferred S6.3 Windows proof)
1. In the app, tick **Route all traffic (full tunnel)**, **Connect**. (No internet egress
   yet — S3.7 — that's expected; this proof is about NO LEAK, not connectivity.)
2. Start a capture on the **physical adapter** (built-in pktmon):
   ```
   pktmon start --capture --pkt-size 0 -f C:\tunnex-leak.etl
   ```
3. Kill the helper hard: `taskkill /F /IM tunnex-helper.exe`
4. Wait ~15s (WFP filters persist — death = enforcement), then stop + convert:
   ```
   pktmon stop
   pktmon etl2pcap C:\tunnex-leak.etl -o C:\tunnex-leak.pcapng
   ```
   - **PASS (leak):** open `tunnex-leak.pcapng` in Wireshark → **zero cleartext** packets
     to non-tunnel/non-endpoint destinations while the helper was dead (WFP held the block).
5. **Recovery:** the service's restart-on-failure brings the helper back within ~5s; its
   startup self-heal releases the WFP block.
   - **PASS (recovery):** `sc query tunnex-helper` RUNNING again; normal networking returns
     (`ping 1.1.1.1` succeeds once WFP is released and you Disconnect).

## Proof 3 — Tray / notification live display
System-tray icon present; connect/disconnect/**failed** (`taskkill /F /IM tunnex-helper.exe`)/
**revoked** (revoke the device from the admin UI) track live with notifications; tray click
shows the window; Quit exits. **PASS:** all four states + notifications.

---

## Proof 2 — Residue smoke (uninstall via Add/Remove Programs)
1. **Disconnect** the tunnel first (graceful Down removes WFP filters + the wintun adapter).
2. **Settings → Apps → Tunnex → Uninstall** (runs the NSIS uninstaller → `customUnInstall` →
   `sc stop`/`sc delete`).
3. Verify clean:
   ```
   sc query tunnex-helper          # → 1060 "service does not exist"
   Get-NetAdapter | ? InterfaceDescription -match Wintun   # → nothing
   ```
   - **PASS:** service gone (1060), no Tunnex/wintun adapter, install dir removed.

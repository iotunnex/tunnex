# S6.5a Windows proof runbook (unsigned, bare box)

The bare Windows box needs **nothing installed** — the app is built on the Mac
(`win-unpacked`, x64) and transferred as a zip; a PowerShell script registers the helper
service. Needs a reachable tenant + active gateway (same as the macOS proofs).

Artifact (built on the Mac): `apps/client/release/Tunnex-win-x64.zip` (contains
`Tunnex.exe`, `resources/` incl. the SCM-service-capable helper + `wintun.dll`, and
`install-helper.ps1` / `uninstall-helper.ps1`).

Rebuild any time on the Mac: `cd apps/client && bash scripts/stage-helper.sh win && npx electron-builder --win --dir --x64 --config electron-builder.yml`, then copy `build/win/*.ps1` into `release/win-unpacked` and zip.

---

## Setup (on the box)
1. Copy `Tunnex-win-x64.zip` to the box; **extract** it (right-click → Extract All).
2. Open **PowerShell as Administrator**, `cd` into the extracted folder (where `Tunnex.exe` is).
3. Register the helper service:
   ```
   powershell -ExecutionPolicy Bypass -File .\install-helper.ps1
   ```
   - **PASS:** `sc query tunnex-helper` shows **STATE: 4 RUNNING** (proves the Go SCM
     service dispatcher works — a plain console binary would time out 1053).

---

## Proof A′ — Acceptance: launch → connect → reach, minimal terminal
1. Double-click **Tunnex.exe** → SmartScreen "Windows protected your PC" → **More info → Run anyway**.
2. Enter the server URL → sign in → **Connect** (split tunnel). The service is already
   running, so no per-connect prompt.
   - **PASS:** status → **Connected** with an IP + a fresh handshake; reach an org host.

If Connect fails with a caller-auth error, grab the helper log: `Get-Content C:\Windows\Temp\tunnex-helper.log -Tail 20` (or check Event Viewer) — the caller image path must be inside `TUNNEX_INSTALL_DIR` (the extracted app folder).

---

## Proof 1 — WFP live kill-switch pcap (the deferred S6.3 Windows proof)
1. In the app, tick **Route all traffic (full tunnel)**, **Connect**. (No internet egress
   yet — S3.7 — that's expected; this proof is about NO LEAK, not connectivity.)
2. Start a capture on the **physical adapter** (built-in pktmon):
   ```
   pktmon start --capture --pkt-size 0 -f C:\tunnex-leak.etl
   ```
3. Kill the helper hard: `taskkill /F /IM tunnex-helper.exe`
4. Wait ~15s (WFP filters persist — death = enforcement), then stop + convert:
   ```
   pktmon stop
   pktmon etl2pcap C:\tunnex-leak.etl -o C:\tunnex-leak.pcapng
   ```
   - **PASS (leak):** open `tunnex-leak.pcapng` in Wireshark → **zero cleartext** packets
     to non-tunnel/non-endpoint destinations while the helper was dead (WFP held the block).
5. **Recovery:** the service's restart-on-failure brings the helper back within ~5s; its
   startup self-heal releases the WFP block.
   - **PASS (recovery):** `sc query tunnex-helper` RUNNING again; normal networking returns
     (`ping 1.1.1.1` succeeds once WFP is released and you Disconnect).

## Proof 3 — Tray / notification live display
System-tray icon present; connect/disconnect/**failed** (`taskkill /F /IM tunnex-helper.exe`)/
**revoked** (revoke the device from the admin UI) track live with notifications; tray click
shows the window; Quit exits. **PASS:** all four states + notifications.

---

## Proof 2 — Residue smoke
1. **Disconnect** the tunnel first (graceful Down removes WFP filters + the wintun adapter).
2. Elevated PowerShell in the app folder: `powershell -ExecutionPolicy Bypass -File .\uninstall-helper.ps1`
   - **PASS:** residue check — service removed (`sc query` → 1060), wintun adapter gone.
3. Delete the extracted app folder.

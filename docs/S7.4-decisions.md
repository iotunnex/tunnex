## STEER 1 — differentiated health surface, COLLAPSED AT DESIGN (the decision, not a mid-fold)

The ledgered "which-kind-of-degraded" surface is exactly the multi-signal→multi-field shape that produced
S7.2's staleness gap-states. Design it collapsed from the start:

**D-H1 — `policy_degraded` (bool) STAYS the single source of truth.** It answers "is this gateway's
policy healthy?" and remains the only field any LOGIC (alarms, gating) keys on. The differentiation adds
NO new decision logic and does NOT feed any branch.

**D-H2 — KIND is an advisory PROJECTION, not logic.** The three kinds are derivable from the EXISTING
JSONB caps at compute time (Investigator B, verified in code):
| Kind | Raw signature (existing JSONB) | New state? |
|---|---|---|
| `apply_failing` | `policy_error` set + `policy_failing_since` set | none — projection |
| `stuck_enforcing` | `policy_error` set + `policy_failing_since` EMPTY (+ server knows `pushed=="" && applied!=""`) | none — projection |
| `silent_desync` (existence) | server term-3 only: `pushed != applied`, error/since empty | none — server already computes term-3 in isolation |

Add ONE **read-only advisory field** `policy_degraded_kind` (enum `none|apply_failing|stuck_enforcing|silent_desync`)
to the `Node` response — computed alongside `policy_degraded` from the same caps, **pure projection, zero
new stored state, no migration.** The bool is authoritative; the enum is display detail. Web renders a
differentiated badge on the `Gateways.tsx` node row.

**DURABILITY CONDITION (Q4) — the property that keeps the collapse from eroding in a later story.** The new
field is a contract change (arms codegen-drift). A test MUST assert the enum is **COMPUTED from the SAME
JSONB caps the bool derives from, and is NEVER independently settable** — the kind can never silently acquire
its own state (a stored field, a separate write path). If a future story needs per-kind state (e.g. the
deferred desync-onset, D-H3), that is an explicit new decision, not a drift. This test is what makes "kind =
projection, not logic" durable rather than momentary.

**D-H3 — the ONE genuinely-new-state item is ISOLATED and DEFERRED.** Distinguishing *transient* vs
*stuck* `silent_desync` needs a **persisted `desync-onset` timestamp** (the server compare is instantaneous
today; `service.go:466-472` explicitly notes onset would be new state, against the reduce goal). That is
the coupled-temporal-state that spawned the arc. **It is NOT built in S7.4.** `silent_desync` ships as a
single advisory kind ("pushed≠applied") without a transient/stuck split. If the split is ever wanted, it is
its own decision-first item (a new `policy_desync_since` field, stamped on term-3 onset, cleared on
reconverge) — named here, deferred. Ledger trigger: a real "stuck desync read as a transient blip" incident,
or S7.x hardening. This keeps the arc's terminal heuristic applied PRE-EMPTIVELY.

## D-A1 — mode-toggle "warn-and-confirm": NO client-side compiler reimplementation

The server gives no pre-flight preview and no lockout guard; the affected list is post-hoc + egress-only.
Two ways to warn before the flip: (i) UI-compute the blast radius from list endpoints, or (ii) generic
warning + show the server's authoritative post-hoc list after.

**Decision: (ii) — CONFIRMED (Q2).** Building a UI-side blast-radius preview means REIMPLEMENTING the
compiler's who-loses-reachability logic in the client — a second, divergent source of truth (the exact bug
class as the S7.2 staleness surface / Steer 1). So: the confirm dialog states the consequence generically,
the flip proceeds, and the server's authoritative `affected_full_tunnel_devices` is shown prominently AFTER
as the real blast radius. A genuine pre-flight preview, if wanted, is a **server-side dry-run endpoint** (new
work) — DEFERRED, NOT a client reimplementation.

**PLUS a COUNT-BASED self-lockout advisory (Q2) — a rule COUNT read, NOT a reachability/compiler check.**
The mode-enable confirm surfaces the allow-rule count from the existing `listPolicyRules` endpoint:
- N > 0 → "Enforcing denies all traffic except what your rules allow — you have **N** allow rule(s). Continue?"
- N == 0 → the STRONGER gate: "You have **NO** allow rules — enabling Enforcing denies **ALL** traffic,
  including your own access. Continue?" (surfaces the S7.1 zero-grants footgun).

This is honest server-sourced COUNT data, not a divergent second source of truth — it makes no claim about
*which* devices are affected (that stays forbidden / server-authoritative-post-hoc). The real dry-run/preview
endpoint stays deferred per D-A1.

## D-a5 — CONSEQUENTIAL: rule editing = CREATE-THEN-DELETE, gap-free (the one enforcement-semantics touch)
There is no `updatePolicyRule` (create/delete only). "Edit a rule" is implemented as **create-new-first, then
delete-old** — NEVER delete-first:
1. POST the new rule (create).
2. ONLY on create success → DELETE the old rule.
- **Failure modes are all safe:** create fails → old rule stays, edit aborts, NO access gap. Create succeeds +
  delete fails → a transient DUPLICATE grant persists (harmless + retryable), NO gap.
- **CONDITION (legibility) — the duplicate must be VISIBLE, not just harmless.** On create-succeeds +
  delete-fails the UI reports PARTIAL SUCCESS ("new rule created; the old rule could not be removed — retry")
  and the list shows BOTH rules, with retry available. A duplicate nobody knows about is how a "deleted" rule
  keeps granting access — harmless-but-invisible is unacceptable for an enforcement artifact (same
  failure-must-be-legible principle that closed S7.3's [0]). Test: create-succeeds + delete-fails → both rules
  listed + partial-failure surfaced + retry available.
- **Why gap-free:** the model is ALLOW-ONLY and grants are a UNION (S7.1 semantics), so a transient duplicate
  rule is a no-op. This is the create-first-swap reasoning from the S7.3 migration but WITHOUT its cap/atomicity
  trap — here NO resource is freed and re-claimed (grants are purely additive), so create-first has no downside.
- Delete-first would open a real access gap on an enforcing org (traffic denied mid-edit) and lose the rule on a
  failed recreate — FORBIDDEN. This ordering is stated because it is the single place S7.4a touches enforcement
  semantics; it gets an explicit ordering test.

## D-a6 — CONSEQUENTIAL: ID→name joins render "(deleted/unknown)", NEVER omit
`PolicyRule` returns only IDs; the UI joins `src_group_id`/`dst_resource_id`/`dst_group_id` against the loaded
groups/resources. The rule row is **never omitted** — a rule the server is ENFORCING must always be visible even
if its referents are broken (**the UI never hides live policy**).
- **CONDITION (deleted ≠ unresolved) — the copy must not lie about WHY.** Distinguish the two causes:
  - referent ABSENT from a **successfully-loaded** set → "(deleted)" — the referent is genuinely gone.
  - the groups/resources list **FAILED TO LOAD** → "(unresolved — refresh)" + the page surfaces the load error.
  Conflating them makes a transient fetch failure render HEALTHY policy as broken — the false-alarm class this
  project has hit three times (staleness, desync, migration). The rule stands either way; the label must be
  honest about the cause. Test BOTH branches (deleted-referent vs failed-load).
(Cascade-FK deletes dependent rules server-side per the S7.1 guard, so a genuine dangling referent should be rare —
but the UI stays honest if one appears, rather than silently hiding an enforced grant.)

## STEER 1 — differentiated health surface, COLLAPSED AT DESIGN (the decision, not a mid-fold)

The ledgered "which-kind-of-degraded" surface is exactly the multi-signal→multi-field shape that produced
S7.2's staleness gap-states. Design it collapsed from the start:

**D-H1 — `policy_degraded` (bool) STAYS the single source of truth.** It answers "is this gateway's
policy healthy?" and remains the only field any LOGIC (alarms, gating) keys on. The differentiation adds
NO new decision logic and does NOT feed any branch.

**D-H2 — KIND is an advisory PROJECTION, not logic.** The three kinds are derivable from the EXISTING
JSONB caps at compute time (Investigator B, verified in code):
| Kind | Raw signature (existing JSONB) | New state? |
|---|---|---|
| `apply_failing` | `policy_error` set + `policy_failing_since` set | none — projection |
| `stuck_enforcing` | `policy_error` set + `policy_failing_since` EMPTY (+ server knows `pushed=="" && applied!=""`) | none — projection |
| `silent_desync` (existence) | server term-3 only: `pushed != applied`, error/since empty | none — server already computes term-3 in isolation |

Add ONE **read-only advisory field** `policy_degraded_kind` (enum `none|apply_failing|stuck_enforcing|silent_desync`)
to the `Node` response — computed alongside `policy_degraded` from the same caps, **pure projection, zero
new stored state, no migration.** The bool is authoritative; the enum is display detail. Web renders a
differentiated badge on the `Gateways.tsx` node row.

**DURABILITY CONDITION (Q4) — the property that keeps the collapse from eroding in a later story.** The new
field is a contract change (arms codegen-drift). A test MUST assert the enum is **COMPUTED from the SAME
JSONB caps the bool derives from, and is NEVER independently settable** — the kind can never silently acquire
its own state (a stored field, a separate write path). If a future story needs per-kind state (e.g. the
deferred desync-onset, D-H3), that is an explicit new decision, not a drift. This test is what makes "kind =
projection, not logic" durable rather than momentary.

**D-H3 — the ONE genuinely-new-state item is ISOLATED and DEFERRED.** Distinguishing *transient* vs
*stuck* `silent_desync` needs a **persisted `desync-onset` timestamp** (the server compare is instantaneous
today; `service.go:466-472` explicitly notes onset would be new state, against the reduce goal). That is
the coupled-temporal-state that spawned the arc. **It is NOT built in S7.4.** `silent_desync` ships as a
single advisory kind ("pushed≠applied") without a transient/stuck split. If the split is ever wanted, it is
its own decision-first item (a new `policy_desync_since` field, stamped on term-3 onset, cleared on
reconverge) — named here, deferred. Ledger trigger: a real "stuck desync read as a transient blip" incident,
or S7.x hardening. This keeps the arc's terminal heuristic applied PRE-EMPTIVELY.

## D-A1 — mode-toggle "warn-and-confirm": NO client-side compiler reimplementation

The server gives no pre-flight preview and no lockout guard; the affected list is post-hoc + egress-only.
Two ways to warn before the flip: (i) UI-compute the blast radius from list endpoints, or (ii) generic
warning + show the server's authoritative post-hoc list after.

**Decision: (ii) — CONFIRMED (Q2).** Building a UI-side blast-radius preview means REIMPLEMENTING the
compiler's who-loses-reachability logic in the client — a second, divergent source of truth (the exact bug
class as the S7.2 staleness surface / Steer 1). So: the confirm dialog states the consequence generically,
the flip proceeds, and the server's authoritative `affected_full_tunnel_devices` is shown prominently AFTER
as the real blast radius. A genuine pre-flight preview, if wanted, is a **server-side dry-run endpoint** (new
work) — DEFERRED, NOT a client reimplementation.

**PLUS a COUNT-BASED self-lockout advisory (Q2) — a rule COUNT read, NOT a reachability/compiler check.**
The mode-enable confirm surfaces the allow-rule count from the existing `listPolicyRules` endpoint:
- N > 0 → "Enforcing denies all traffic except what your rules allow — you have **N** allow rule(s). Continue?"
- N == 0 → the STRONGER gate: "You have **NO** allow rules — enabling Enforcing denies **ALL** traffic,
  including your own access. Continue?" (surfaces the S7.1 zero-grants footgun).

This is honest server-sourced COUNT data, not a divergent second source of truth — it makes no claim about
*which* devices are affected (that stays forbidden / server-authoritative-post-hoc). The real dry-run/preview
endpoint stays deferred per D-A1.

## D-a5 — CONSEQUENTIAL: rule editing = CREATE-THEN-DELETE, gap-free (the one enforcement-semantics touch)
There is no `updatePolicyRule` (create/delete only). "Edit a rule" is implemented as **create-new-first, then
delete-old** — NEVER delete-first:
1. POST the new rule (create).
2. ONLY on create success → DELETE the old rule.
- **Failure modes are all safe:** create fails → old rule stays, edit aborts, NO access gap. Create succeeds +
  delete fails → a transient DUPLICATE grant persists (harmless + retryable), NO gap.
- **CONDITION (legibility) — the duplicate must be VISIBLE, not just harmless.** On create-succeeds +
  delete-fails the UI reports PARTIAL SUCCESS ("new rule created; the old rule could not be removed — retry")
  and the list shows BOTH rules, with retry available. A duplicate nobody knows about is how a "deleted" rule
  keeps granting access — harmless-but-invisible is unacceptable for an enforcement artifact (same
  failure-must-be-legible principle that closed S7.3's [0]). Test: create-succeeds + delete-fails → both rules
  listed + partial-failure surfaced + retry available.
- **Why gap-free:** the model is ALLOW-ONLY and grants are a UNION (S7.1 semantics), so a transient duplicate
  rule is a no-op. This is the create-first-swap reasoning from the S7.3 migration but WITHOUT its cap/atomicity
  trap — here NO resource is freed and re-claimed (grants are purely additive), so create-first has no downside.
- Delete-first would open a real access gap on an enforcing org (traffic denied mid-edit) and lose the rule on a
  failed recreate — FORBIDDEN. This ordering is stated because it is the single place S7.4a touches enforcement
  semantics; it gets an explicit ordering test.

## D-a6 — CONSEQUENTIAL: ID→name joins render "(deleted/unknown)", NEVER omit
`PolicyRule` returns only IDs; the UI joins `src_group_id`/`dst_resource_id`/`dst_group_id` against the loaded
groups/resources. The rule row is **never omitted** — a rule the server is ENFORCING must always be visible even

- **CONDITION (deleted ≠ unresolved) — the copy must not lie about WHY.** Distinguish the two causes:

(Cascade-FK deletes dependent rules server-side per the S7.1 guard, so a genuine dangling referent should be rare —
but the UI stays honest if one appears, rather than silently hiding an enforced grant.)

# S7.4b commit-one — differentiated health surface (PAPER ONLY; no branch/code until S7.4a merges)

Refines the Steer-1 decision with the health-surface codebase map (Investigator B). Health signals live in
`nodes.capabilities` JSONB (`policy_error`, `policy_failing_since`, `policy_hash`, `policy_version`); the
server derives `policy_degraded` (bool, 3 OR-terms) per `ListNodes`. NONE of the rich signals cross the wire
today — only the collapsed bool.

## (a) JSONB read → a BOUNDED, ENUMERATED set of states; can't-determine is DISTINCT
The server projects the caps into ONE advisory `policy_degraded_kind` enum. The enumerated states, each
mapped to its raw signature:
| state | raw signature (server, at compute time) |
|---|---|
| `healthy` | not degraded |
| `apply_failing` | `policy_error` set + `policy_failing_since` set |
| `stuck_enforcing` | `policy_error` set + `policy_failing_since` empty (+ `pushed=="" && applied!=""`) |
| `converging` | term-3 desync (`pushed != applied`), desync age < debounce window T (see (c)) |
| `silent_desync` | term-3 desync, desync age ≥ T (stuck) |
| `desync_unknown` | the compiled-hash was UNAVAILABLE (compile fault) → the server CANNOT rule out a silent desync |

**`desync_unknown` is a first-class honest state — NEVER defaulted to `healthy` and NEVER claimed as a
specific kind.** This is the loadOne legibility law applied to the health KIND: can't-determine ≠ healthy
(the can't-determine window today makes an already-desynced gateway read not-degraded — acceptable for the
BOOL because it's on last-good fail-closed policy, but the KIND must SAY "can't determine", not imply health).

## (b) The projection stays ADVISORY — `policy_degraded` (bool) remains authoritative
`policy_degraded` is the ONLY signal any logic (alarms, gating) keys on; it answers "is this gateway
healthy?" The `policy_degraded_kind` enum is display detail — it adds NO decision logic. **Durability test
(carried from S7.4 D-H2/Q4):** the enum is computed from the SAME caps the bool derives from, and is NEVER
independently settable — a codegen-drift-armed contract test asserts it, so the kind can never silently
acquire its own state in a later story. **ADVISORY-READ-ONLY GUARD (amended):** the kind is UI-read-only —
a test (or structural boundary) asserts that NOTHING in enforcement, push targeting, or compilation reads
`policy_degraded_kind` (or `policy_desync_since` beyond the render derivation). Advisory that something
quietly starts consuming stops being advisory; `policy_degraded` (the bool) remains the sole load-bearing
signal — the S7.2 collapse must not silently un-collapse.

## (c) The debounce/window as FIRST-CLASS TEMPORAL STATE (the S7.2 fault class — decided on paper)
`converging` vs `silent_desync` cannot be told apart from the instantaneous compare (`service.go:466-472`
notes the onset is not tracked — that was deliberate in S7.2). Distinguishing them needs a **persisted
desync-onset**: a new server field `policy_desync_since` (on the node row / caps JSONB), **stamped when the
server FIRST observes term-3 for a node in the report/reconcile ingest path, cleared when the hashes
reconverge OR `policy_degraded` clears.** Then `age = now − policy_desync_since`; `age < T → converging`
(transient, expected during a push — advisory-only, NOT alarming), `age ≥ T → silent_desync` (stuck,
actionable). **T ties to the push SLA** (≈2× the <5s reconcile/push target, ~10–15s).

Modeled first-class to AVOID the S7.2 fault class: the onset is a single persisted timestamp with exactly
two write points (stamp on first term-3, clear on reconverge/heal) — NOT a second derived signal that can
gap-state against the bool. `converging` never alarms (so a normal push's brief desync is not a false
alarm — the legibility failure a naive term-3 badge WOULD cause).

**DECISION — OPTION X (build the temporal state now), SIGNED OFF, amended.** Rationale recorded (do not
re-litigate): the S7.2 ledger deferred this exact state to S7.4 "as new state" — S7.4b is the story
designated to build it honestly. Option Y (defer + render every term-3 as `desync_unknown`) would render a
KNOWN fact (a hash mismatch we DID observe) as "unknown" forever — a legibility violation of the mirror
polarity to a false alarm: `desync_unknown` must mean **we could not determine**, NEVER **we determined but
won't say**. X's risk is bounded because we do what S7.2 couldn't: model the temporal state on paper first —
single timestamp, enumerated write points, fault class named in advance.

**X-1 (required) — `silent_desync` requires CURRENT reports; the render is three-valued on a stamped node.**
The timestamp proves desync ONSET; it cannot prove desync is ONGOING — only a fresh report can. So the
render is a function of `stamp × report-freshness × applied-hash`, not the stamp alone:
- reports FRESH + hash still mismatched: `age = now − policy_desync_since`; `age < T → converging`, `age ≥ T → silent_desync`.
- reports FRESH + hash matches: reconverged → **clear the stamp** → `healthy` (bool governs).
- reports STALE (agent stopped reporting): → **`desync_unknown`, stamp RETAINED.** Silence is NOT evidence
  of reconvergence — never clear on silence, never claim `silent_desync` without current proof, never
  `healthy`. Clearing requires POSITIVE evidence (a fresh report with a matching hash).

**X-2 — ONE clock.** `policy_desync_since` is stamped by the CONTROL PLANE at report-processing time, never
taken from the agent-reported time. Two clocks on one temporal signal is a skew bug awaiting its finding
number.

**X-3 — the reds are the design (named now, a temporal-state surface):** (a) false-alarm — a fresh push →
transient mismatch → NOTHING renders `silent_desync` within T (`converging` shows, never alarms); (b) stuck —
mismatch persisting past T with reports flowing → `silent_desync`; (c) reports-stop — stamped, then reports

report clears the stamp; (e) **X-4 write-ownership** — an agent report carrying a FORGED (or absent)
`policy_desync_since` leaves the CP-computed stamp UNAFFECTED (ingest never reads a reported value into the

**Persistence — `policy_desync_since` is a dedicated CP-owned COLUMN** on `nodes` (migration 0021), NOT a
`capabilities` JSONB key. **X-4 (write-ownership, the deciding hazard):** `capabilities` is AGENT-FED — the
report-ingest builds/writes that blob from the agent's `AppliedPolicy`. Putting the CP-owned, X-2-clocked
`policy_desync_since` there reopens exactly what X-2 closed:
- **Clobber** — a blob-replace on the next report erases the CP stamp → it can never accumulate past `T` →
  `silent_desync` becomes structurally UNREACHABLE.
- **Forgery** — any agent-supplied value for that key would be an agent writing the CP's independent
  observation OF that agent, defeating the signal.
A COLUMN removes the fault class by construction: CP-only writability, agent has no path to it, no
merge/namespace-strip discipline to maintain (an argued bound this surface's history says to avoid).
The T-window render reads it as a per-node PROJECTION (age = now − it, computed in Go at `ListNodes`), NOT a
query predicate — so "column only if a predicate" doesn't apply; the deciding factor is write-ownership, not
read shape. **WRITE-OWNERSHIP IS TRANSITION EVIDENCE** (extends the state table): the stamp/retain/clear
transitions are CONTROL-PLANE-WRITE-ONLY; there is NO agent-write transition — the ingest never copies a
reported value into this column.

## State × render × transition-evidence TABLE (the authoritative model — no evidence-less state)
Render is `f(policy_degraded bool × stamp × report-freshness × applied-hash)`. Every state has an explicit
evidence-IN and evidence-OUT; a state with no honest evidence rule would be a PAPER FINDING (fix before code).
The bool is the row's PRIMARY signal; the kind is the differentiated detail — **the badge is NEVER less
alarmed than the bool** (a `converging`/`desync_unknown` kind still shows the bool's degraded state, just
annotated).

| kind | render (over the bool) | evidence IN | evidence OUT |
|---|---|---|---|
| `healthy` | none / green (bool not degraded) | not degraded; hash matches; reports fresh | any degrade condition below |
| `apply_failing` | amber, "apply failing" | `policy_error` set + `policy_failing_since` set (reports fresh) | a fresh report clears `policy_error` |
| `stuck_enforcing` | amber, "enforcing a disabled policy" | `policy_error` set + `failing_since` empty (+ `pushed=="" && applied!=""`) | a fresh report clears the error / mode resync |
| `converging` | subtle, "syncing…" (NEVER a loud alarm) | term-3 mismatch; reports fresh; `age < T` | `age ≥ T` → `silent_desync`; hash match → `healthy`; reports stale → `desync_unknown` |
| `silent_desync` | RED, "silent desync — pushed policy not applied" | term-3 mismatch; reports fresh; `age ≥ T` | fresh matching report → `healthy`; reports stale → `desync_unknown` |
| `desync_unknown` | grey, "can't determine health" | compiled-hash UNAVAILABLE (can't compute `pushed`) **OR** (stamp set AND reports STALE) | a fresh report resolves to a determinate kind above |

Note the bool during `converging`: term-3 makes `policy_degraded` briefly true on every push (its existing
S7.2 conservative behavior). The kind ADDS "this degraded is a transient sync", never subtracts alarm —
`silent_desync` is only when that degraded PERSISTS past T with fresh reports (the actionable case).

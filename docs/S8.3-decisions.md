# S8.3 — site management UI (commit-one, decision-first)

**Story:** the web consuming layer for EPIC 8's site model. S8.1 (site/gateway model + advertisement/approval) and S8.2 (route propagation + site-link health kinds) shipped **API-only**; nothing in `apps/web` fetches a site today. This story is the UI that makes the proven backend operable — plus a small backend rider (the S8.2b keepalive) and the folded Access-page polish.

**Scope discipline:** UI + the keepalive rider + the polish scope. **No new data-plane behavior. DNS forwarding is S8.4.** The only backend touches permitted are (a) the S8.2b keepalive on site-link peers, (b) wiring the already-authored-but-unwired `DeleteSite`, and (c) closing the OpenAPI Node health-kind enum gap (below) — each is enabling the UI to render wire-truth, not new enforcement.

**Re-entry anchor:** main @ `8f7bd65` (pointer) / `2df19df` (S8.2 merge). Branch `story/S8.3-site-ui`. Prior art: S7.4a (consuming a proven health backend from the web), S7.3 (device-approval queue = the advertisement-queue pattern), S7.5.4 (per-user + expiry grant fields).

## Backend truth this UI consumes (surveyed, not assumed)

1. **Site API surface (S8.1):** `listSites`, `registerSite`, `listSiteSubnets`, `addSiteSubnet` (→ pending), `bindSiteNode` / `unbindSiteNode` (D6 replace), `listPendingSiteSubnets` (the queue), `approveSiteSubnet`. All gate `authorize(PermSiteManage)` FIRST (owner+admin). No `deleteSite` on the wire yet.
2. **`Site` is thin** — `{id, name, link_transport, link_mtu?, created_at}`. It does NOT embed the bound gateway, health, or subnets. The topology/site cards must ASSEMBLE from three reads: `listSites` + per-site `listSiteSubnets` + the nodes list (the binding lives on `node.site_id`; a site's gateway = the node whose `site_id` is this site). This shapes D2.
3. **Health is per-node, via the S7.4b surface** (`node.policy_degraded` bool primary + `policy_degraded_kind` advisory). There is no per-site health rollup endpoint; a site's link health = its bound gateway's kind.
4. **BACKEND-TRUTH GAP (must close, item (c) above):** the server EMITS `site_hub_down` / `site_link_down` (S8.2 box-walk Leg 7 observed them on the wire), but the **OpenAPI `Node.policy_degraded_kind` enum, the generated TS union, and `healthview.policyHealthBadge` all lack them** — they currently fall through to the `default: "degraded"` badge. The honesty floor (D2) requires the real kind, so this story extends the enum → `make generate` → adds the two badge cases. `unsupported_policy_version` IS already in the enum (the v5-warning source, CW, is ready).
5. **`subnet_not_disjoint` already carries detail** — `apierr.Conflict("subnet_not_disjoint", "this subnet overlaps the {class} range {With}; approval refused")` + a `site.subnet_approval_refused` audit with `overlap_class` + `overlaps`. The refusal UX (D3) can render the typed message verbatim; **no API enrichment owed.**
6. **`DeleteSite` is authored + `DELIBERATELY UNWIRED`** (`db/queries/sites.sql`: `-- DELIBERATELY UNWIRED until S8.3 … needs the confirm-naming-target`). D4 wires it.
7. **Access already supports user-subject + expiry** in `RuleFormModal` (src_kind=user, `expiresAt`, `ExtendGrantModal`) and `policyview` has `resolveUser` + `grantExpiry`. The polish (CP) is LIST-ROW display + a summary line, not new create-capability — confirms "polish, not redesign."

## Carried decide-items (pre-ledgered at S8.2; instantiated here)

### CW — the add-second-site v5 upgrade warning (the named-limitation's UI layer)
**Context:** S8.2's content-derived version means the moment an org becomes multi-site with a routed grant, the compiled artifact bumps to v5; any gateway below v5 REFUSES it and goes deny-all (`unsupported_policy_version`). The limitation is named in the paper; this is its operator-facing surface.

**Which action gates?** Two candidates: (a) registering the second site, (b) the second site's first approved subnet (the action that actually makes a cross-site grant compilable). **Proposed: gate at approval of the first subnet on the org's 2nd site** — registration alone advertises nothing routable (a subnetless site compiles no grant, S8.1), so warning at registration would fire on a no-op and train dismissal. Approval is the first moment a v5 artifact can actually be produced.

**RULED: blocking confirm-with-count at first-approval-on-2nd-site, and the PRE-WARN signal is MANDATED (not optional).** Before the approval commits, if any of the org's gateways report a max-supported policy version below the ceiling (5), surface a confirm — *"Approving this subnet enables cross-site routing. N gateway(s) cannot apply the new policy version and will deny all traffic until upgraded: [named gateway list]. Approve anyway?"* Confirm over passive-warning because the consequence is a hard deny-all.

**Why pre-warn, not retroactive (ruling rationale):** a retroactive warning — "we deny-alled your gateways, here's why" — is the reassuring-green class wearing an apology. The whole point of the named-limitation's UI layer is *before*, and *before* requires a wire signal for "which gateways are below v5" that exists PRIOR to the bump (the `unsupported_policy_version` kind only appears AFTER the artifact goes v5 — too late to pre-warn).

**The signal — built in Slice 1 (backend-truth's natural home):** the agent REPORTS its max-supported policy version (the report path already carries version-adjacent data — EXTEND it, do not invent a new channel); the CP STORES it on the node row; the confirm dialog LISTS which gateways are below 5 by name. This signal is permanently useful — every future version bump inherits an upgrade-readiness surface (S11.4's docs will point at it).
**Red:** the confirm shows the named sub-v5 gateway list; a fleet that is all-v5 shows a CLEAN confirm (no gateways listed).

### CK — S8.2b keepalive lands BEFORE this story's box-walk
**Proposed disposition: a PRE-SLICE (Slice 0), backend-only, merged/walked first.** Add `PersistentKeepalive` to site-link `Peer`s (both `nodes.Peer` + `reconcile.Peer`, honoring `peersEqual` — keepalive is a desired-field so it must be compared or it churns). Rationale: the topology view (D2) renders link health from real handshake freshness; without keepalive an idle site link false-stales at 180s and the view would show `site_link_down` on a healthy-but-idle link DURING its own walk. The keepalive must be live on the wire before the UI walk so the topology view isn't proving against a false-degraded substrate. Sequenced first, its own reconcile-level red (keepalive present in desired peer + `peersEqual` treats it as significant).
**RULED: CONFIRMED as specified — peer field + `peersEqual` significance + 25s (wireguard-conventional).** ADDITION: when Slice 0 lands, the S8.2b registration entry gets its DISCHARGE LINE (the keepalive item closes, trigger "before S8.3's box-walk" met).

## Story decide-items

### D1 — Page architecture (where sites live in the IA)
Real nav today: Dashboard / Devices / Access / Users / Settings / Audit (`AppShell.tsx` `NAV`). Options: (a) a new top-level **Sites** page; (b) fold into Devices/Nodes; (c) a **Network** section grouping sites (+ future topology).
**Proposed: (a) a new top-level "Sites" nav entry**, edition-gated visibility (enterprise; the site ops are `site:manage`). Rationale: sites are a first-class ZT primitive with their own lifecycle (register/advertise/approve/bind/delete) — folding into Devices (which is per-user device credentials, a different noun) would muddy both. "Network" (c) is premature with one sub-page. The Access rule-builder's existing `dst_kind='site'` / `src_kind='site'` entries point AT these sites (already resolved to site labels in `policyview`); a Sites page gives those labels a home to click through to.
**RULED: CONFIRMED — top-level "Sites" nav, edition-gated.** Sites are an org-structural concept PEER to Devices, not a subtype of nodes; and the epic-close founder walk needs a coherent place to start. **ADDITION — the bidirectional link:** the Access click-throughs (dst/src site labels → the site page) bind the two surfaces; INCLUDE THE REVERSE LINK — the site page shows *"rules referencing this site"*. That reverse link needs a rules-referencing-site COUNT, which is **the same API addition D4's cascade preview needs** — build ONE endpoint, serve both (D1 reverse link + D4 cascade count).

### D2 — Topology view scope (what v1 renders + the honesty floor)
**Proposed render (v1):** per-site cards showing — site name, its bound gateway (as a list-of-one per CH), the gateway's **real** health badge (the S7.4b kind incl. the newly-wired site-link kinds), approved subnets, and hub designation (the endpoint-bearing gateway; derivable client-side from which gateway carries an endpoint, matching `siteLinkGraphFrom`'s hub rule — or surfaced from the backend if cleaner). Links between sites rendered from real state: a link is "up" only on fresh handshake truth, "down" on `site_link_down`/`site_hub_down`.
**Honesty floor (binding): render ONLY wire-truth.** No decorative animation implying live telemetry we don't have (no fake packet-flow pulses, no synthetic "activity"). Handshake ages and health kinds are real or the element shows "unknown" (the `desync_unknown` discipline). Stitch/design concepts are LAYOUT reference only; the never-render-unproduced-data rule binds over any concept art.
**RULED: render floor CONFIRMED** — static-truth cards + explicit link-health labels, no decorative telemetry animation (the honesty floor is law; a full node-graph canvas is S-future if warranted).

**RULED: hub source OVERRULED → BACKEND-DERIVED (not client-side).** Client-side hub derivation ("endpoint-bearing gateway") would re-implement `siteLinkGraphFrom`'s election logic in TS — a SECOND COPY of a load-bearing decision, the S8.2 #1 defect class (two paths disagreeing about one truth) relocated to the display layer. The moment hub election gains a tiebreak or a designation override (multi-hub is reserved → this logic WILL change), the UI's copy silently forks. **The CP computes the hub; the API says which node it is** (a field on the node/site payload or the health output — cheapest honest seam, implementer's pick); the UI renders what it is told. One line of API, zero divergence risk — the D2 render-floor law applied to DERIVED truth, not just telemetry.

### D3 — Advertisement / approval UX
- **Pending queue:** reuse the S7.3 device-approval-queue pattern — `listPendingSiteSubnets` → a review list with per-row Approve. Surface on the Sites page (or a sub-tab).
- **Refusal display:** on `409 subnet_not_disjoint`, render the API's typed message verbatim (it names the overlap class + colliding range — item 5). **RULED: render the message; NO client-side disjointness re-computation.** This is **the comparison-set law's UI corollary — one validator, never a second copy of the check in JS** (the server is the sole disjointness authority; a JS re-check is the S8.2 #1 two-paths-disagreeing class in the display layer).
- **Unbind / replace-gateway (D6 capability):** `bindSiteNode` / `unbindSiteNode` exist API-only. **RULED: surface bind (pick an unbound gateway node) + unbind (with a confirm — unbind is a full-sweep of site-link peers/routes, proven in the S8.2 walk Leg 6).** Replace = unbind-then-bind; v1 need not be atomic (stated).
**RULED: CONFIRMED** — queue placement (Sites page/sub-tab, admin-only per D5), verbatim-refusal, bind/unbind surface.

### D4 — Site lifecycle UI (DeleteSite — wire it + the destructive grain)
`DeleteSite` is authored + deliberately unwired pending this story. **Proposed disposition:**
- **Wire the endpoint** (`deleteSite` op, `authorize(PermSiteManage)` FIRST, edition-gated same as the others).
- **Destructive grain: confirm-with-name-typed** (the S8.1 registration/S6.5 destructive precedent) — the operator types the site name to enable Delete.
- **Cascade preview:** before delete, show what cascades — the site's subnets AND any `dst_kind='site'`/`src_kind='site'` policy rules that reference it. **RULED: the preview states what the cascade DOES from REAL counts — "N rules deleted, M subnets released, gateway unbound" — never "this may affect."** The 0033/0034 cascades handle the DB side; the UI must not let a delete silently orphan a grant the operator forgot.
- **The rules-count API:** build the "rules referencing site X" count endpoint — **SHARED with D1's reverse link** (one endpoint serves both the site page's "rules referencing this site" and this cascade preview).
**RULED: CONFIRMED** — wire `deleteSite`, confirm-with-name-typed + real-count cascade preview, build the shared rules-count API.

### D5 — Edition / RBAC surfaces (what a non-holder sees)
`PermSiteManage` gates every mutation (owner+admin). **The B6 walk-class lesson: no affordance that leads to a guaranteed 403.**
**Proposed disposition:**
- **Non-enterprise org:** the Sites nav entry shows the standard `edition_required` upsell surface (the Access-page precedent at `Access.tsx:134`), not the live UI.
- **Enterprise, member role (no `site:manage`):** **read-only topology** — sites, gateways, health, subnets are VISIBLE (operational visibility is legitimate for a member), but every MUTATION affordance (register, advertise, approve, bind, unbind, delete) is ABSENT (not disabled-with-403, absent) — `canManage` gating exactly like `Access.tsx`'s `canManage` prop.
- **RULED sharpening: the member sees health but NOT the pending queue.** The queue is an ACTION surface; visible-but-inert is B6's cousin (an affordance the member can see but can't act on). Health/topology = legitimate read; the pending queue = admin-only.
- **Non-enterprise:** the edition upsell follows the established **`403 edition_required` mirror pattern** (the `Access.tsx:134` precedent).
**RULED: CONFIRMED with the queue-hidden-for-member sharpening.**

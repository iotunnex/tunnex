# Tunnex cloud POC — deploy + connect with the desktop client

End state: a **public VM** runs the tunnex server + a WireGuard gateway; your **Mac
desktop app** signs in over the internet, clicks **Connect**, and reaches the
private org network through the tunnel.

> **Reality check (split-tunnel).** This POC proves Zero-Trust access: the client
> reaches the gateway + org subnet (`10.99.0.0/24`, e.g. `ping 10.99.0.1`). It does
> NOT route your general internet through the VPN — that's gateway NAT (**S3.7**,
> parked). Full-tunnel connects but its internet egress dies at the gateway.

---

## Part A — the VM (server + gateway)

### A1. Get a VM
Any provider (EC2 / DigitalOcean / Hetzner …): **Ubuntu 22.04+, 2 vCPU / 2 GB**,
a **public IP**. Install Docker + Compose. Open the firewall / security group:
| Port | Proto | Why |
|---|---|---|
| 80 | TCP | dashboard + API |
| **51820** | **UDP** | **the tunnel** (a TCP-only rule silently blocks every tunnel) |

### A2. Clone + configure (endpoint BEFORE enrolling)
```sh
git clone https://github.com/iotunnex/tunnex && cd tunnex
cp .env.example .env
```
Edit `.env` — set the PUBLIC address every client config will dial:
```sh
TUNNEX_NODE_ENDPOINT=YOUR_PUBLIC_IP:51820   # NOT a private/compose name
APP_BASE_URL=http://YOUR_PUBLIC_IP
```
(This is baked into device configs at creation — set it before creating any device.)

### A3. Boot + create the org
```sh
docker compose up -d --build --wait
```
Browse to `http://YOUR_PUBLIC_IP` → sign up → verify email → create your org.
(Dev email lands in Mailpit on the VM; or set real SMTP in `.env`.)

### A4. Enroll the gateway
Dashboard → **Devices → Gateways → Enroll gateway** → name it → copy the line:
```sh
TUNNEX_JOIN_TOKEN=…  TUNNEX_NODE_NAME=my-gw  docker compose up -d node-agent
```
The gateway appears **active** in the dashboard once the agent redeems the token.
(Full detail + troubleshooting: `docs/deploy-public-vm.md`.)

## Part B — the Mac desktop client

### B1. Build the client (from your local repo)
```sh
cd ~/tunnex
pnpm --filter @tunnex/web build          # bundles the SPA incl. the new Connect UI
pnpm --filter @tunnex/client build
```

### B2. Install the privileged helper (interim dev-install)
The production installer (SMAppService) isn't built yet; use the dev-install (it
builds the helper CGO=1 → native caller-auth, and loads the LaunchDaemon):
```sh
./scripts/macos-dev-install.sh           # sudo prompts; MUST print caller-auth: native
```

### B3. Launch the client, point it at the VM
```sh
TUNNEX_BUNDLE_DIR="$(cd apps/web/dist && pwd)" pnpm --filter @tunnex/client start
```
On the setup screen enter `http://YOUR_PUBLIC_IP` (validates via `/healthz`), then
**Sign in with your browser** as your org user.

### B4. Connect
On the dashboard (desktop mode) the **VPN tunnel** card appears → **Connect**.
Main creates a device (D2, once), the helper brings up the utun + routes the org
subnet. The card shows **Connected** + a live handshake + rx/tx.

### B5. Verify (the POC proof)
```sh
ifconfig | grep -A2 utun          # a utun with your 10.99.0.x address
ping 10.99.0.1                     # the gateway, THROUGH the tunnel → replies
```
Dashboard shows your device **online** with a recent handshake. Disconnect from the
card restores normal routing.

### Troubleshooting
- **Connect fails "no gateway":** the gateway isn't active (A4) — check `docker compose logs node-agent` on the VM.
- **Connects but no handshake / can't ping:** `TUNNEX_NODE_ENDPOINT` wrong, or **UDP 51820 blocked** in the security group.
- **"caller-auth: stub":** the helper was built CGO-off — rerun the dev-install on the Mac.
- **Card says "helper isn't running":** `sudo launchctl print system/io.tunnex.helper` — reload via the dev-install.

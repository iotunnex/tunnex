# S6.6 — Zero-build deploy: clean-VPS acceptance runbook

The DoD proof for S6.6: on a **clean machine with only Docker installed**, one `install.sh` yields a
reachable dashboard + an enrolled gateway + a connecting client — with **zero `git clone`, zero
`--build`, zero file edits**. Run this end to end after CI has published a release tag's images.

## Preconditions

- A fresh VPS (or VM) with a **public IP or DNS name**, nothing Tunnex installed.
- **Docker Engine + the Compose v2 plugin** (`docker compose version` works). `curl`, `openssl`.
- UDP **51820** and TCP **80** reachable from where the client + gateway will connect.
- The images for the release tag exist in ghcr (CI's `publish` job ran green on the `v*` tag).
  A release must be cut first — until then, run with `TUNNEX_VERSION=latest` to track `main`.

## 1. Install (one command, two prompts)

```sh
curl -fsSL https://raw.githubusercontent.com/iotunnex/tunnex/main/deploy/install.sh | sh
```

Answer:
1. **Public address** — the DNS name / public IP this box is reachable at (e.g. `vpn.acme.com`).
   Loopback (`localhost` / `127.*` / `::1` / `0.0.0.0`) is **refused at the prompt** — it re-asks.
2. **SMTP** — `y` + host/port/user/pass/from, or `N` to skip (local sign-in still works).

**PASS:** it prints `>> Installing Tunnex <version>`, pulls images, waits healthy, and ends with the
`Tunnex <version> is running.` next-steps block. No build output scrolls by. `./tunnex/.env` exists
with a single `TUNNEX_VERSION=`, one `APP_BASE_URL=http://<addr>`, one `POSTGRES_PASSWORD=` (no
duplicate keys).

**FAIL signals:** any `docker build` step; an `.env` with a key twice; a `localhost` prompt accepted;
`could not resolve a released Tunnex version` (no release tag yet → re-run with `TUNNEX_VERSION=latest`).

## 2. Dashboard reachable + sign up

- Open `http://<addr>/`.
- Sign up → you land on **create your first organization** (S4.7 funnel).

**PASS:** dashboard loads over the public address (not localhost); org creation succeeds; the
Overview renders. If SMTP was configured, the verify email arrives with links pointing at `<addr>`
(NOT localhost — the POC item-3 bug this installer prevents at the source).

## 3. Enroll a gateway (item 6 — the complete command)

- Dashboard → **Gateways** → optionally name it → **Generate join token**.
- The modal shows the **complete runnable command** (not just a token), copy it:
  ```
  TUNNEX_JOIN_TOKEN=<token> [TUNNEX_NODE_NAME="<name>"] docker compose -f tunnex.yml up -d --force-recreate node-agent
  ```
- Run it **in `./tunnex/`** (where `tunnex.yml` lives) on the same box.

**PASS:** the `node-agent` re-creates, redeems the token, and the gateway appears in the Gateways
list with a version + a recent "last seen". No `node_name_mismatch` loop (item-6 fix: name is inline
+ shell-quoted).

## 4. A client connects

- Issue a device config from the dashboard (or the desktop client), bring the tunnel up.
- Verify a WireGuard handshake + traffic (split or full tunnel per the client).

**PASS:** handshake completes against `<addr>:51820`; the device shows live in the dashboard
(`last_handshake` recent). This closes the loop: control plane (API) + data plane (enrolled gateway)
+ client, all from prebuilt images.

## 5. Upgrade / rollback (reproducibility check)

- Edit only `./tunnex/.env`: set `TUNNEX_VERSION` to another released tag.
- `docker compose -f tunnex.yml pull && docker compose -f tunnex.yml up -d`.

**PASS:** the stack moves to the named version; setting the prior tag rolls back. Nothing but `.env`
was ever edited.

---

### Acceptance summary

| Check | Gate |
|---|---|
| No source build anywhere | no `docker build` output; images pulled from ghcr |
| No file edits | only `install.sh`-generated `.env` touched (values only) |
| Clean `.env` | no duplicate keys; single `APP_BASE_URL` at the public address |
| Loopback refused | install.sh re-prompts on `localhost`/`127.*`/`::1`/`0.0.0.0` |
| Enroll = one command | Gateways modal prints the full `--force-recreate node-agent` line |
| Loop closes | dashboard reachable → gateway enrolled → client handshake |

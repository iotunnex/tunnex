The DoD proof for S6.6: on a **clean machine with only Docker installed**, one `install.sh` yields a
reachable dashboard + an enrolled gateway + a connecting client â€” with **zero `git clone`, zero
`--build`, zero file edits**. Run this end to end after CI has published a release tag's images.

## Preconditions

- A fresh VPS (or VM) with a **public IP or DNS name**, nothing Tunnex installed.
- **Docker Engine + the Compose v2 plugin** (`docker compose version` works). `curl`, `openssl`.
- UDP **51820** and TCP **80** reachable from where the client + gateway will connect.
- The images for the release tag exist in ghcr (CI's `publish` job ran green on the `v*` tag).
  A release must be cut first â€” until then, run with `TUNNEX_VERSION=latest` to track `main`.

## 1. Install (download-verify-inspect-run â€” the default path)

```sh
url=https://raw.githubusercontent.com/iotunnex/tunnex/main/deploy
curl -fsSL "$url/install.sh" -o install.sh
curl -fsSL "$url/install.sh.sha256" -o install.sh.sha256 && sha256sum -c install.sh.sha256
less install.sh
sudo sh install.sh
```

**PASS:** `sha256sum -c` prints `install.sh: OK`. (The convenience path is
`curl -fsSL "$url/install.sh" | sh` â€” same script.)

Answer:
1. **Public address** â€” the DNS name / public IP this box is reachable at (e.g. `vpn.acme.com`).
   Loopback (`localhost` / `127.*` / `::1` / `0.0.0.0`) is **refused at the prompt** â€” it re-asks.
2. **SMTP** â€” `y` + host/port/user/pass/from, or `N` to skip (local sign-in still works).

**PASS:** it prints `>> Installing Tunnex <version>`, pulls images, waits healthy, and ends with the
`Tunnex <version> is running.` next-steps block. No build output scrolls by. `./tunnex/.env` exists
with a single `TUNNEX_VERSION=`, one `APP_BASE_URL=http://<addr>`, one `POSTGRES_PASSWORD=` (no
duplicate keys).

**FAIL signals:** any `docker build` step; an `.env` with a key twice; a `localhost` prompt accepted;
`could not resolve a released Tunnex version` (no release tag yet â†’ re-run with `TUNNEX_VERSION=latest`).

### 1a. Pipe-safe checks (safe-to-pipe-into-root-shell)

- **Non-interactive** (no terminal): `curl -fsSL "$url/install.sh" | TUNNEX_PUBLIC_ADDR=vpn.acme.com TUNNEX_SMTP=skip sh`
  â†’ installs with zero prompts. With the address env var unset and no TTY, it **dies loud** with the
  exact re-run line (never hangs).
- **Loud on bad address**: `... | TUNNEX_PUBLIC_ADDR=127.0.0.1 sh` â†’ dies immediately (not a warning).
- **Idempotent re-run**: run install a second time â†’ it prints `Reusing the existing database
  password`, the stack stays up, and the API still connects to postgres (a new password would break
  auth against the existing volume). `grep -c '^POSTGRES_PASSWORD=' tunnex/.env` = `1`, and the value
  is unchanged from the first run.
- **No half-written `.env`**: interrupt/kill mid-run before the pull â†’ no `.env.new` left behind and
  the previous `.env` (if any) is intact.

## 3. Enroll a gateway (item 6 â€” the complete command)

- Dashboard â†’ **Gateways** â†’ optionally name it â†’ **Generate join token**.
- The modal shows the **complete runnable command** (not just a token), copy it:
  ```
  TUNNEX_JOIN_TOKEN=<token> [TUNNEX_NODE_NAME="<name>"] docker compose -f tunnex.yml up -d --force-recreate node-agent
  ```
- Run it **in `./tunnex/`** (where `tunnex.yml` lives) on the same box.

**PASS:** the `node-agent` re-creates, redeems the token, and the gateway appears in the Gateways
list with a version + a recent "last seen". No `node_name_mismatch` loop (item-6 fix: name is inline
+ shell-quoted).

## 4. A client connects

- Issue a device config from the dashboard (or the desktop client), bring the tunnel up.
- Verify a WireGuard handshake + traffic (split or full tunnel per the client).

**PASS:** handshake completes against `<addr>:51820`; the device shows live in the dashboard
(`last_handshake` recent). This closes the loop: control plane (API) + data plane (enrolled gateway)
+ client, all from prebuilt images.

## 5. Upgrade / rollback (reproducibility check)

- Edit only `./tunnex/.env`: set `TUNNEX_VERSION` to another released tag.
- `docker compose -f tunnex.yml pull && docker compose -f tunnex.yml up -d`.

**PASS:** the stack moves to the named version; setting the prior tag rolls back. Nothing but `.env`
was ever edited.

## 6. Flow / access logs â€” the FULLY-JOINED leg (enterprise, S7.5.1 deferral)

The S7.5.1 box-walk proved the flow-log pipeline end-to-end with a real WG spoke (wg-quick); this
leg closes the ledgered SUBSTITUTESâ‰ SATISFIES by driving the ACTUAL product client through it on a
clean host. Run only on the ENTERPRISE image (`TUNNEX_BUILD_TAGS=enterprise` / enterprise release tag).

- Enable Zero Trust: `PUT /organizations/{org}/zero-trust-mode {"mode":"enforcing"}`; create a group
  + resource + access rule (grant); add the client's user to the group. (Flow logs require enforcing
  mode â€” a mesh/off org renders the blanket mesh and emits NONE. This is by design; state it.)
- Arm flow logging on the gateway: `TUNNEX_FLOWLOG_GROUP=100` (opt-in; OFF renders no log clause).
- Connect the real desktop/CLI client; drive one flow that MATCHES the grant + one that does not.

**PASS:** the granted flow yields an `allow` access-event carrying the grant `rule_id` +
`dst_resource_id` (via `/access-events` and the byte-verbatim `/access-events/export`); the ungranted
flow yields a `deny` event (no rule_id); the JSONL source-of-truth SURVIVES a container recreate (it
is on the `tunnex_flowlog` volume) and `/access-log/health` reads clean. Grant-level attribution
(device/user identity still NIL by design). This is the product-client end-to-end the S7.5.1 walk
deferred.

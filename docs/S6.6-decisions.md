# S6.6 ‚Äî Zero-build deploy (EPIC 6 epic-ender) ‚Äî decisions (decision-first)

**The non-negotiable principle:** a customer must NEVER clone the repo, build from source, edit a
file, or run diagnostics to get a working tunnel. The POC required building images on BOTH the
server AND the gateway VM (`git clone ‚Ä¶ && docker compose up --build`) ‚Äî that is the friction this
story kills. **Acceptance: a clean machine with nothing installed ‚Üí run one `install.sh` ‚Üí a working
tunnel. Zero source builds, zero file edits.**

Closes POC-ledger **item 1** (build-on-both-boxes), **item 6** (join-token ceremony must print the
COMPLETE runnable enroll command), **item 7** (client Node ‚â•20 engine warning).

---

## 1. Published prebuilt images ‚Üí ghcr.io (extend S6.0b CI)

Today `docker-compose.yml` builds `api / web / nginx / node-agent / migrate` from source. Publish
them instead: **`ghcr.io/iotunnex/tunnex-{api,web,nginx,node-agent,migrate}`**.

- **CI (extend `.github/workflows/ci.yml`):** a new `publish` job builds + pushes the five images.
  Uses the workflow's default `GITHUB_TOKEN` with `packages: write` (same-org GHCR, no extra
  secret). Trigger: **push to `main` ‚Üí `:latest` + `:<short-sha>`; a `v*` release tag ‚Üí `:vX.Y.Z`**.
  Red gates still block; publish runs only after they pass.
- **Arch (DECIDED): amd64 + arm64 multi-arch** via buildx ‚Äî supports ARM servers (Ampere/Graviton)
  out of the gate. Accepts the ~2x image-CI time on main pushes.
- GHCR packages are **public** (the repo is public) ‚Üí `docker pull` needs no auth on the customer box.
- The Dockerfiles are unchanged (they already build these artifacts); we only add the push.

## 2. A hosted compose file the customer PULLS, never edits

A prod compose distinct from the dev `docker-compose.yml`: **`deploy/tunnex.yml`** (also published as
a release asset + a stable raw URL). It has **NO `build:` ‚Äî only `image:` refs pinned to a RELEASE
VERSION via `${TUNNEX_VERSION}`** (e.g. `image: ghcr.io/iotunnex/tunnex-api:${TUNNEX_VERSION}`).
**REVISED (do NOT pin `:latest`):** a customer's deployment must be reproducible + revertible ‚Äî if
every box floated on `:latest`, a bad merge to main becomes a bad pull on someone's production
gateway with no version to name, pin, or roll back to. `install.sh` writes `TUNNEX_VERSION=vX.Y.Z`
(the newest RELEASE tag) into `.env`; upgrade = bump `TUNNEX_VERSION` + `pull && up -d`, rollback =
set the prior tag. `:latest` stays published as an OPT-IN for people deliberately tracking main
(`TUNNEX_VERSION=latest`). **This story cuts the first release tag (v0.1.0)** so there is a version
to pin (CI publishes `:vX.Y.Z` on `v*` tags ‚Äî ¬ß1). To the ghcr images, real prod posture (`restart: unless-stopped`, healthchecks, published :80/:8443/:51820), and
**no mailpit, no `--build`, no source context**. Postgres/redis stay as their upstream images. The
API auto-migrates on boot (S0.3), so no separate migrate step for the customer. The customer never
opens this file ‚Äî `install.sh` fetches it next to the generated `.env` and runs `docker compose -f
tunnex.yml pull && up -d`.

## 3. `install.sh` ‚Äî asks EXACTLY two things, writes a CLEAN .env

Curl-pipeable (`curl -fsSL https://get.tunnex.io/install.sh | sh` OR a release asset). It:
1. **Prompts for exactly two inputs:**
   - **Public address** ‚Äî the DNS name / public IP users + gateways reach (‚Üí `APP_BASE_URL`,
     `TUNNEX_NODE_ENDPOINT` host, the CORS/cookie posture). **REJECT loopback at the prompt (fail
     loud, re-prompt):** `localhost` / `127.0.0.1` / `::1` / `0.0.0.0` are refused ‚Äî this is a
     DEPLOY tool, and a loopback public address is POC item 3 (verify/reset/invite emails emitted
     `localhost` links unreachable on a remote box). Prevent it at the source, same class as the
     `.env` append trap ‚Äî not a warning after the fact. (Pure-local dev uses the repo's dev
     `docker-compose.yml`, never install.sh.)
   - **SMTP ‚Äî configure or SKIP** ‚Äî if configured: host/port/user/pass/from; if skipped: email
     features (verify/reset/invite) are disabled with a clear note (local-auth still works; the
     dashboard surfaces the "SMTP not configured" state rather than silently failing).
2. **GENERATES secrets + resolves the version** ‚Äî `POSTGRES_PASSWORD` (`openssl rand`), the derived
   `DATABASE_URL`, and `TUNNEX_VERSION` = the newest RELEASE tag (from the GitHub releases API;
   pinned, never `:latest` ‚Äî ¬ß2). The API's S0.3 bootstrap auto-generates the JWT/session secrets +
   master encryption key inside the container (persisted to the `tunnex_secrets` volume) ‚Äî
   install.sh does NOT handle those; it only sets the DB password + version + the two inputs.
3. **Writes `.env` as ONE clean file from a template** ‚Äî the whole file rendered with the inputs +
   generated values substituted, written atomically. **NEVER `cat >> .env`** ‚Äî the append trap bit
   us once: duplicate keys, and compose silently used the FIRST value. install.sh writes fresh (and
   backs up an existing `.env` rather than appending).
4. `docker compose -f tunnex.yml pull && docker compose -f tunnex.yml up -d`, waits for healthy,
   then **prints the NEXT STEPS ‚Äî treated as a real deliverable, not an echo** (it's the customer's
   first experience): the exact **dashboard URL** (their public address), **"create your first org"**
   (the funnel lands a fresh signup on `/create-org`, S4.7), and **where gateway enrollment lives**
   (Dashboard ‚Üí Gateways ‚Üí Generate join token, which now prints the full enroll command ‚Äî item 6).
   No diagnostics, no "if it doesn't work run X" ‚Äî a clean, confident hand-off.

## 4. Item 6 ‚Äî the join-token ceremony prints the COMPLETE runnable command

`apps/web/src/components/Gateways.tsx` currently shows only the token. Change it to render the
**full copy-paste command** the operator runs on the gateway host, e.g.:
```
TUNNEX_JOIN_TOKEN=<token> docker compose -f tunnex.yml up -d --force-recreate node-agent
```
(the `--force-recreate node-agent` is the piece the POC operator had to know by heart). Keep the
one-time-secret hygiene (shown once, never re-served ‚Äî the existing `OneTimeSecretModal`); the
command carries the token inline. So enrolling a gateway is copy ONE line, not "here's a token, now
go figure out the command."

## 5. Item 7 ‚Äî pin Node ‚â•20 (kill the engine warning deterministically)

The client build warns on Node <20 (deps require ‚â•20). Add **`"engines": { "node": ">=20" }`** to
`apps/client/package.json` (and the root workspace), plus a **`.nvmrc` = 20**, and set
`engine-strict` so the mismatch is a clear hard error at install rather than a silent warning that
produces a subtly-wrong build. CI already runs Node 20; this makes local match.

- `deploy/tunnex.yml` (image-only, no build) + published so `install.sh` can fetch it.
- `install.sh`: two prompts, generated DB secret, CLEAN atomic `.env` (no append), `pull && up -d`,
  prints next steps. README quickstart rewritten around it (no `git clone`, no `--build`).
- Gateways.tsx prints the complete `--force-recreate node-agent` enroll command (item 6).

- **Acceptance proof:** on a clean VPS with only Docker installed, `install.sh` ‚Üí reachable dashboard

## 6. Pipe-safe from day one (addition ‚Äî marketing's `curl ‚Ä¶ | sh` hero serves THIS script)

The landing page's hero is `curl -fsSL https://get.tunnex.io | sh`, serving the SAME `install.sh`
this story produces. So the script must be safe to pipe blind into a root shell. Small addition, not
a rescope ‚Äî everything in ¬ß1‚Äì¬ß5 stands.

- **Safe-to-pipe:**
  - **Idempotent** ‚Äî re-running against an existing `./tunnex` **reuses the generated
    `POSTGRES_PASSWORD`** (a fresh `openssl rand` would not match the existing postgres volume ‚Üí
    auth failure); backs up the old `.env`; `pull && up -d --wait` is inherently re-runnable.
  - **Prereqs fail LOUD + actionable** ‚Äî missing Docker / Compose v2 / curl / openssl each die with a
    one-line fix, not a stack trace.
  - **Non-TTY aware** ‚Äî under a fully-detached pipe (no `/dev/tty`, e.g. CI) it takes the two inputs
    from **env vars** (`TUNNEX_PUBLIC_ADDR`, `TUNNEX_SMTP=skip|configure` + `SMTP_*`); if the address
    is missing with no terminal, it dies with the exact env-var re-run line. Env vars also override
    the prompts on an interactive run.
  - **Never a half-written `.env`** ‚Äî write-to-`.env.new` + atomic `mv`, plus an `EXIT` trap that
    removes `.env.new` on any failure. The `.env` is never observed partial.
- **Loud on a bad public address** ‚Äî loopback (`localhost`/`127.*`/`::1`/`0.0.0.0`) or a scheme is
  refused at the SOURCE, on BOTH paths: an interactive prompt re-asks; an env-var value dies (no
  infinite loop on a fixed bad value). POC item 3 prevented, not warned-about-later.
- **SHA256 for the script itself** ‚Äî CI's `release-assets` job (on `v*` tags) attaches `install.sh`
  **and `install.sh.sha256`** to the GitHub release (same discipline as the client `SHA256SUMS`), so
  the docs offer a verify step.
- **Docs show BOTH paths, inspect-first as the default** ‚Äî `curl -o install.sh` +
  `sha256sum -c install.sh.sha256` + `less install.sh` + `sudo sh install.sh` is the recommended
  path; `curl | sh` is the convenience one-liner. Rationale: our buyers are sovereignty/security-
  conscious ‚Äî a VPN vendor offering ONLY pipe-to-root-shell is an anti-pattern to exactly that
  audience. Offer both; don't hide the inspectable path.
- **Prerequisite stated plainly** ‚Äî "any VPS with Docker + a public address." One command installs
  the software; it doesn't conjure infrastructure (prevents the laptop-with-no-public-IP failure).
- **OWNERSHIP (recorded in PLAN so it can't drift):** exactly ONE `install.sh`, in THIS repo, produced
  by S6.6. The marketing site only SERVES it (release asset / static file) ‚Äî it must NEVER fork or
  hand-maintain its own copy. `get.tunnex.io` depends on the pending domain purchase; S6.6 does not
  block on it ‚Äî the script just gets a URL later.

## Non-goals
- No `get.tunnex.io` domain/hosting of the script itself (that's the marketing-site/separate task ‚Äî
  install.sh ships as a repo file + release asset; a vanity URL is a later wrap). No image signing /
  SBOM (S6.5b-class, deferred). No Helm/k8s (EPIC 10). No auto-update of a running deployment (pull
  a new tag + `up -d` is the manual upgrade; documented).

# S6.3 Tunnel Control — Human Smoke Script (real macOS + Windows)

The S6.3 pure core is CI-covered on Linux (`make test-helper`: protocol, config
validation, path-check, fail-closed state machine, IPC framing; plus the Electron
framing tests in the client matrix). What CI **cannot** exercise — a privileged
tun device, real routing, code-signing identity, a GUI — is scripted here and
**must be run by a human on real hardware** before S6.3 merges.

Each step states its **PASS assertion**. The kill-switch step is INSTRUMENTED: it
is proven by a packet capture, not an eyeball.

## Sequencing + target machines
- **Run macOS FIRST.** The Mac is the proven dev machine (S6.1's smoke ran there),
  so it's the fastest path to a first green tunnel + kill-switch pcap.
- **macOS target:** the proven dev Mac.
- **Windows target: [USER TO IDENTIFY — physical box or an admin-rights VM].**
  Windows needs naming NOW so smoke day isn't blocked on procurement: installing
  the `wireguard-nt`/Wintun driver + the helper service both need admin. A VM is
  fine for tunnel semantics, BUT a kill-switch pcap on a VM's virtual NIC needs
  care (capture on the VM's guest NIC, and beware the host doing NAT/routing that
  masks a guest leak — capture inside the guest, on the guest's physical-facing
  adapter).

> Prereqs: a reachable Tunnex tenant + an enrolled gateway. First connect CREATES
> a device (D2 — owned by the app, config captured once into the keychain) and
> logout REVOKES it; verify that lifecycle too. Build the app + helper for the
> platform under test.

## STEP ZERO (both platforms — do this FIRST, it gates everything)
The helper's caller-auth resolver has a real "native" build and a fail-closed
"stub" build; a stub build refuses EVERY caller (safe) but must never be silently
shipped. Before anything else:
- **macOS: the helper MUST be built `CGO_ENABLED=1`** (the caller-path uses
  libproc via cgo). A CGO-off macOS build is the stub.
- Run `tunnex-helper --version`. **PASS requires `caller-auth: native`.** If it
  prints `caller-auth: stub`, STOP — the build is wrong; no other step is valid
  (the helper would refuse all callers). The same line is logged at helper startup.

## A. macOS

Terms: physical uplink is usually `en0`; the tunnel is a `utun` device.

### A1 — Install + first-run privilege grant
1. Launch Tunnex.app, sign in.
2. Trigger a tunnel connect. Expect exactly ONE macOS admin auth prompt (the
   `SMAppService` LaunchDaemon registration).
   - **PASS:** `launchctl print system/io.tunnex.helper` shows the daemon loaded;
     a second connect does NOT re-prompt (idempotent).

### A2 — Tunnel up carries traffic
1. Note your public IP: `curl -s https://api.ipify.org` (pre-tunnel).
2. Connect (full-tunnel: allowed-IPs `0.0.0.0/0, ::/0`).
3. `ifconfig` shows a `utun` with the device address; `netstat -rn` shows the
   default route via that utun.
4. `curl -s https://api.ipify.org` again.
   - **PASS:** the IP changed to the gateway's egress IP; UI shows Connected +
     a live handshake time (`tunnel.status()` `last_handshake_sec` recent).

### A3 — Graceful down restores routing
1. Click Disconnect.
   - **PASS:** the `utun` is gone; `netstat -rn` default route is back on `en0`;
     `curl api.ipify.org` shows your ORIGINAL IP; UI shows Disconnected.

### A4 — Caller authentication (interim path-check)
1. With the helper running, from a shell run a NON-Tunnex process that connects to
   the helper socket (`/var/run/tunnex/helper.sock`) and sends a `tunnel_up`
   (a 20-line Go/Python snippet using the framing in `apps/helper/ipc.go`).
   - **PASS:** the helper replies `caller_untrusted` and closes; no tunnel comes
     up. (This process's executable is outside `/Applications/Tunnex.app`.)

### A5 — KILL-SWITCH, INSTRUMENTED (the security crux)
Goal: prove that if the helper dies mid-tunnel, NO packets bound for the tunneled
destination leak in cleartext out the physical interface, and the UI says so.

1. Connect (full-tunnel). Pick a target reachable only meaningfully via the
   tunnel, e.g. the gateway's internal peer `10.99.0.1`, or a public host you will
   watch by address, say `T=1.1.1.1`.
2. In terminal 1, start a **continuous** probe to the target:
   `ping T` (leave it running).
3. In terminal 2, start a **packet capture on the PHYSICAL interface**, filtered
   to the target, writing a file:
   `sudo tcpdump -i en0 -n host T -w /tmp/killswitch.pcap`
   (While the tunnel is up you should see little/none to `T` on `en0` directly —
   it's encapsulated inside WireGuard UDP to the gateway endpoint. That's the
   baseline.)
   Before the kill, confirm the PRE-ARRANGED block exists: `sudo pfctl -a tunnex -s rules`
   lists the kill-switch rules (installed at Up, not at death).
4. Identify + hard-kill the helper with `kill -9` so NO cleanup handler can run:
   `sudo kill -9 <helper pid>` (find it via `sudo launchctl print system/io.tunnex.helper`).
   Note the wall-clock time. Then CONFIRM the process is actually gone: `ps -p <pid>`
   returns nothing. This is the crux — with the helper dead, only pre-arranged
   kernel state can be enforcing anything.
5. Let the probe run ~10 more seconds, then stop the capture (Ctrl-C) and the ping.
6. **PASS assertions (ALL required):**
   - **Helper is GONE** (step 4) — so whatever holds the line is not live code.
   - **Zero cleartext egress:** `tcpdump -r /tmp/killswitch.pcap -n 'host T and not udp'`
     shows NO direct IP packets to `T` leaving `en0` AFTER the kill timestamp. The
     pf anchor (arranged at Up, kernel-resident) keeps blocking though the helper is
     dead, so the ping fails and nothing reaches `en0` in cleartext. (Contrast — the
     fail-OPEN bug excluded: the pcap WOULD show cleartext ICMP to `T`. ZERO is the proof.)
   - **UI surfaces it loudly:** within ~1s the app shows a prominent Disconnected/Failed
     state with a reason (the app's persistent connection dropped); `tunnel.status()`
     eventually `state: "failed"`.
7. Recovery: on next launch the helper reconciles the stale pf anchor (adopt on
   reconnect, or an explicit clear); reconnect works; graceful disconnect restores
   routing AND flushes the pf anchor.

### A6 — Clean uninstall (lifecycle is a deliverable — only smoke checks this)
1. Graceful disconnect first, then quit + remove Tunnex.app (run the uninstall path).
   - **PASS (ALL):** `launchctl print system/io.tunnex.helper` = not found (no orphaned
     root LaunchDaemon); `/var/run/tunnex/helper.sock` is gone (no orphaned socket);
     `sudo pfctl -a tunnex -s rules` is EMPTY (no residual kill-switch black-holing the
     network post-uninstall); `netstat -rn` shows no leftover blackhole/utun routes.

## B. Windows

Terms: physical uplink e.g. `Ethernet`/`Wi-Fi`; the tunnel is a Wintun adapter.
Use an elevated PowerShell for capture/kill.

### B1 — Install + service
1. Install Tunnex; connect once (accepts the one-time UAC elevation that installs
   the Windows service).
   - **PASS:** `sc query TunnexHelper` = RUNNING; reconnect does not re-prompt.

### B2 — Tunnel up / down
- Up (full-tunnel): `Get-NetIPInterface` shows the Wintun adapter; public IP
  (`curl.exe https://api.ipify.org`) becomes the gateway egress. **PASS** as A2.
- Down: adapter removed, IP reverts, UI Disconnected. **PASS** as A3.

### B3 — Caller authentication (SDDL + refused-client)
The pipe SDDL is `D:P(A;;GA;;;SY)(A;;GA;;;BA)(A;;GRGW;;;AU)` — SYSTEM + local
Administrators full control; any Authenticated User may CONNECT + read/write (so
the unprivileged app can reach the pipe); Protected DACL (no inheritance). Access
≠ authorization: the caller-path check decides which PROCESS is trusted.
1. **Refused-client test:** from a non-Tunnex process (image path OUTSIDE
   `C:\Program Files\Tunnex`), connect `\\.\pipe\tunnex-helper` and send `tunnel_up`.
   - **PASS:** the pipe OPENS (SDDL allows the authenticated user to connect) but the
     helper replies `caller_untrusted` and no tunnel comes up — proving the SDDL is
     the access gate and GetNamedPipeClientProcessId is the identity gate.
2. Confirm the Tunnex app itself IS accepted (image inside the install dir).

### B4 — KILL-SWITCH, INSTRUMENTED
1. Connect (full-tunnel); pick target `T=1.1.1.1`; start `ping -t T`. Confirm the
   WFP kill-switch filters exist (arranged at Up): `netsh wfp show filters` (or the
   app's diagnostics) lists the Tunnex sublayer/filters.
2. Start a capture on the PHYSICAL adapter filtered to T:
   `pktmon start --capture --pkt-size 0 --comp <PhysAdapterId>` (or Wireshark on
   the physical NIC with display filter `ip.addr == 1.1.1.1 && !wg`), writing to a
   file. (VM note: capture on the GUEST's adapter; ensure the host isn't NAT-routing
   in a way that hides a guest leak.)
3. HARD-kill the service process so NO handler runs: `taskkill /F /IM tunnex-helper.exe`.
   Note the time; CONFIRM it's gone (`tasklist | findstr tunnex-helper` → empty). With
   the process dead, only the pre-arranged WFP filters can enforce anything.
4. Run ~10s more; stop capture.
5. **PASS (ALL):**
   - **Helper is GONE** (step 3) — the enforcer is not live code.
   - Convert/inspect the capture (`pktmon pcapng` → Wireshark): NO cleartext IP packets
     to `T` on the physical adapter AFTER the kill. The WFP filters (kernel-resident,
     persistent sublayer) keep blocking though the helper is dead; ping reports
     "General failure"/timeouts, not replies. A single cleartext packet = FAIL.
   - UI shows Failed/Disconnected loudly within ~1s; `tunnel.status()` → `failed`.
6. Recovery: next launch reconciles the stale WFP sublayer; reconnect works; graceful
   disconnect restores routing AND removes the WFP filters.

### B5 — Clean uninstall (lifecycle deliverable)
1. Graceful disconnect, then uninstall Tunnex.
   - **PASS (ALL):** `sc query TunnexHelper` = does not exist (no orphaned LocalSystem
     service); the named pipe `\\.\pipe\tunnex-helper` is gone; `netsh wfp show filters`
     shows NO residual Tunnex sublayer/filters (nothing black-holing the network
     post-uninstall); `Get-NetRoute` shows no leftover blackhole routes; the Wintun
     adapter is removed.

## Recording

For each platform capture: the run date, OS version, app+helper build hash, the
`killswitch.pcap` (attach), the exact kill timestamp, and the tcpdump/pktmon
filter output proving zero post-kill cleartext to `T`. "No leak" is a filtered
capture with zero matching packets, not an assertion. A single leaked cleartext
packet to a tunneled destination after the kill = FAIL, fix before merge.

# S6.3 Tunnel Control — Human Smoke Script (real macOS + Windows)

The S6.3 pure core is CI-covered on Linux (`make test-helper`: protocol, config
validation, path-check, fail-closed state machine, IPC framing; plus the Electron
framing tests in the client matrix). What CI **cannot** exercise — a privileged
tun device, real routing, code-signing identity, a GUI — is scripted here and
**must be run by a human on real hardware** before S6.3 merges.

Each step states its **PASS assertion**. The kill-switch step is INSTRUMENTED: it
is proven by a packet capture, not an eyeball.

> Prereqs: a reachable Tunnex tenant + an enrolled gateway (a device the desktop
> can bring up). Build the app + helper for the platform under test.

---

## A. macOS

Terms: physical uplink is usually `en0`; the tunnel is a `utun` device.

### A1 — Install + first-run privilege grant
1. Launch Tunnex.app, sign in.
2. Trigger a tunnel connect. Expect exactly ONE macOS admin auth prompt (the
   `SMAppService` LaunchDaemon registration).
   - **PASS:** `launchctl print system/io.tunnex.helper` shows the daemon loaded;
     a second connect does NOT re-prompt (idempotent).

### A2 — Tunnel up carries traffic
1. Note your public IP: `curl -s https://api.ipify.org` (pre-tunnel).
2. Connect (full-tunnel: allowed-IPs `0.0.0.0/0, ::/0`).
3. `ifconfig` shows a `utun` with the device address; `netstat -rn` shows the
   default route via that utun.
4. `curl -s https://api.ipify.org` again.
   - **PASS:** the IP changed to the gateway's egress IP; UI shows Connected +
     a live handshake time (`tunnel.status()` `last_handshake_sec` recent).

### A3 — Graceful down restores routing
1. Click Disconnect.
   - **PASS:** the `utun` is gone; `netstat -rn` default route is back on `en0`;
     `curl api.ipify.org` shows your ORIGINAL IP; UI shows Disconnected.

### A4 — Caller authentication (interim path-check)
1. With the helper running, from a shell run a NON-Tunnex process that connects to
   the helper socket (`/var/run/tunnex/helper.sock`) and sends a `tunnel_up`
   (a 20-line Go/Python snippet using the framing in `apps/helper/ipc.go`).
   - **PASS:** the helper replies `caller_untrusted` and closes; no tunnel comes
     up. (This process's executable is outside `/Applications/Tunnex.app`.)

### A5 — KILL-SWITCH, INSTRUMENTED (the security crux)
Goal: prove that if the helper dies mid-tunnel, NO packets bound for the tunneled
destination leak in cleartext out the physical interface, and the UI says so.

1. Connect (full-tunnel). Pick a target reachable only meaningfully via the
   tunnel, e.g. the gateway's internal peer `10.99.0.1`, or a public host you will
   watch by address, say `T=1.1.1.1`.
2. In terminal 1, start a **continuous** probe to the target:
   `ping T` (leave it running).
3. In terminal 2, start a **packet capture on the PHYSICAL interface**, filtered
   to the target, writing a file:
   `sudo tcpdump -i en0 -n host T -w /tmp/killswitch.pcap`
   (While the tunnel is up you should see little/none to `T` on `en0` directly —
   it's encapsulated inside WireGuard UDP to the gateway endpoint. That's the
   baseline.)
4. Identify + hard-kill the helper: `sudo launchctl kill KILL system/io.tunnex.helper`
   (or `sudo kill -9 <helper pid>`). Note the wall-clock time.
5. Let the probe run ~10 more seconds, then stop the capture (Ctrl-C) and the ping.
6. **PASS assertions (BOTH required):**
   - **Zero cleartext egress:** `tcpdump -r /tmp/killswitch.pcap -n 'host T and not udp'`
     shows NO direct IP packets to `T` leaving `en0` AFTER the kill timestamp. If
     the kill-switch held, the OS had no route for `T` except the torn-down
     utun / a blackhole, so the ping fails and nothing reaches `en0` in cleartext.
     (Contrast — the fail-OPEN bug we are excluding: the default route would fall
     back to `en0` and the pcap WOULD show cleartext ICMP to `T`. Seeing ZERO such
     packets is the proof.)
   - **UI surfaces it loudly:** within ~1s the app shows a prominent
     Disconnected/Failed state with a reason (not a silent "connected" or a blank);
     `tunnel.status()` returns `state: "failed"`.
7. Recovery: reconnect works cleanly (new handshake); disconnect restores routing.

### A6 — Clean uninstall
1. Quit + remove Tunnex.app (and run the app's uninstall/unregister path).
   - **PASS:** `launchctl print system/io.tunnex.helper` reports not found — NO
     orphaned root LaunchDaemon left behind.

## B. Windows

Terms: physical uplink e.g. `Ethernet`/`Wi-Fi`; the tunnel is a Wintun adapter.
Use an elevated PowerShell for capture/kill.

### B1 — Install + service
1. Install Tunnex; connect once (accepts the one-time UAC elevation that installs
   the Windows service).
   - **PASS:** `sc query TunnexHelper` = RUNNING; reconnect does not re-prompt.

### B2 — Tunnel up / down
- Up (full-tunnel): `Get-NetIPInterface` shows the Wintun adapter; public IP
  (`curl.exe https://api.ipify.org`) becomes the gateway egress. **PASS** as A2.
- Down: adapter removed, IP reverts, UI Disconnected. **PASS** as A3.

### B3 — Caller authentication
1. From a non-Tunnex process, connect the named pipe `\\.\pipe\tunnex-helper` and
   send `tunnel_up`.
   - **PASS:** `caller_untrusted`; no tunnel. (Image path outside
     `C:\Program Files\Tunnex`.)

### B4 — KILL-SWITCH, INSTRUMENTED
1. Connect (full-tunnel); pick target `T=1.1.1.1`; start `ping -t T`.
2. Start a capture on the PHYSICAL adapter filtered to T:
   `pktmon start --capture --pkt-size 0 --comp <PhysAdapterId>` (or Wireshark on
   the physical NIC with display filter `ip.addr == 1.1.1.1 && !wg`), writing to a
   file.
3. Hard-kill the service host: `taskkill /F /IM tunnex-helper.exe` (or
   `sc stop TunnexHelper` then kill the process). Note the time.
4. Run ~10s more; stop capture.
5. **PASS (both):**
   - Convert/inspect the capture (`pktmon pcapng` → open in Wireshark): NO
     cleartext IP packets to `T` on the physical adapter AFTER the kill (kill-switch
     held; ping reports "General failure"/timeouts, not replies).
   - UI shows Failed/Disconnected loudly within ~1s; `tunnel.status()` = `failed`.
6. Recovery + graceful disconnect restore routing.

### B5 — Clean uninstall
1. Uninstall Tunnex.
   - **PASS:** `sc query TunnexHelper` = does not exist — no orphaned LocalSystem
     service.

## Recording

For each platform capture: the run date, OS version, app+helper build hash, the
`killswitch.pcap` (attach), the exact kill timestamp, and the tcpdump/pktmon
filter output proving zero post-kill cleartext to `T`. "No leak" is a filtered
capture with zero matching packets, not an assertion. A single leaked cleartext
packet to a tunneled destination after the kill = FAIL, fix before merge.

# S3.7 gateway egress — proof runbook (live gateway)

Proves full-tunnel egress now works end-to-end + the S3.7 proof obligations. Run on the
gateway VM + a full-tunnel client (the packaged desktop client, macOS). The agent now
installs the `tunnex` nft table itself (masquerade + forward) and reports the `egress_nat`
capability — the control plane refuses full-tunnel against a gateway that lacks it.

## 0. Redeploy the gateway with S3.7
The container gains `nftables` and the agent owns `ip_forward` + the NAT now. **Rebuild** so
the old manual iptables/poc rules are gone (a fresh container netns starts clean):
```
git pull            # get S3.7 on the VM
docker compose up -d --build node-agent
```
Confirm the agent installed the table + reports the capability:
```
docker compose exec node-agent nft list table ip tunnex        # masquerade + forward chains
docker compose logs node-agent | grep -iE "egress|reconcile"     # no egress_reconcile_degraded
# on the API side, the node's capabilities should now carry egress_nat=true
```

## 1. Negative first — gateway_no_egress (control-plane refusal)
Before the agent reports egress_nat (or on a locked-down host where it's false), creating a
**full-tunnel** device is refused:
- In the desktop app, tick **Route all traffic (full tunnel)** → Connect on a gateway whose
  `egress_nat` is false → **`gateway_no_egress`** (the UI maps it: "this gateway can't route
  full-tunnel internet traffic yet"). Split-tunnel is always allowed.
- (Unit-proven too: `devices` TestFullTunnelRequiresGatewayEgress.)

## 2. Egress e2e — full-tunnel reaches the internet THROUGH the tunnel
With the S3.7 agent reporting egress_nat=true:
1. Desktop client: **Route all traffic (full tunnel)** → **Connect**.
2. From the client: `ping 1.1.1.1` and `curl -s ifconfig.me`.
   - **PASS:** ping replies **through the tunnel** (latency ≈ your gateway RTT, NOT the 4ms
     direct path) and `ifconfig.me` returns the **GATEWAY's public IP**, not the client's —
     proving traffic egresses via the gateway's NAT, not the local NIC.

## 3. Deliberate-red — the masquerade rule is load-bearing
Prove egress rides the rule, not a leak: flush ONLY the NAT chain on the gateway, egress dies.
```
docker compose exec node-agent nft flush chain ip tunnex postrouting
```
- From the full-tunnel client: `curl -s --max-time 5 ifconfig.me` → **times out / fails**
  (packets reach the gateway but aren't NAT'd out). **PASS = egress dies when the masquerade
  is removed** (so §2 proved the rule carries it, not a pre-existing leak).

## 4. Reconcile-heals — desired-state recovery
The agent re-asserts the table every reconcile interval (`TUNNEX_AGENT_EGRESS_INTERVAL`,
default 30s). After §3's flush (or a full `nft delete table ip tunnex`), wait one interval:
```
docker compose exec node-agent nft list table ip tunnex        # masquerade rule back
```
- From the client: `curl -s ifconfig.me` → **works again** (gateway IP). **PASS = a flushed
  table self-heals** — no manual re-apply, unlike the deleted poc script.

## Result
Record PASS/FAIL per section. §1 negative + §2 egress + §3 deliberate-red + §4 reconcile-heals
green closes S3.7's proof obligations. Device-to-device (spoke↔spoke) is now part of the same
reconciled ruleset (`wg0→wg0 accept`) — no more manual `iptables -A FORWARD`.

## Note — automated compose e2e (ledgered)
A fully-automated compose e2e (a client container with a full-tunnel `.conf` reaching a
gateway-only "internet" target through real WG + real NAT) is the ideal CI proof but needs a
real-WG-client test stack (the same gap as the deferred node-enrollment e2e → enterprise-e2e
stack). This live runbook is the interim proof; the automated harness lands with that stack.

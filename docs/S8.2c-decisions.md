# S8.2c — site-to-site end-to-end via UI (gateway zero-touch) — commit-one (decision-first)

> **ZERO-TOUCH GATEWAY LAW (the acceptance bar):** a gateway is brought online by pasting the ONE install command the dashboard emits — and nothing else. Sites, subnets, enforcing, the site→site grant, and a genuinely-separate host *behind* the gateway reaching a far site are ALL achieved by clicking in the dashboard — never by SSH'ing the gateway to add networking. **Any manual networking step on the gateway is a DEFECT, not a runbook line.** (`docs/laws.md`)

**Origin:** the cross-cloud demo (`walk-artifacts/cross-cloud-demo/demo-record.md`) proved AWS Sydney↔Azure West US site-to-site at **138ms** — but only after **6 manual gateway touches + 3 UI gaps**. S8.2c closes that gap. **Sequencing:** ahead of S8.4; S8.5 shrinks by the overlap.

**Box-walk (defined BEFORE a line is written):** the cross-cloud demo re-runs CLEAN under the Zero-Touch Law — fresh org, two cloud VMs, the ONLY terminal action a pasted join command; register/advertise/approve/enforce/grant/behind-host-reaches-far-site all CLICKED. That re-run IS the story's walk. Fixture: a genuinely-separate FORWARDED host behind the gateway (the fixture-fidelity TOPOLOGY-SIBLING law — S8.2's dummy-LANs-in-netns hid the forward path).

## Decide-items (proposed dispositions — HELD for founder ruling)

### D1 — behind-host reach + mesh-mode forwarding (#5, re-characterized by demo leg (b))
**(b) EXONERATED the S8.2 compiler:** under enforcing+grant it emits the correct **symmetric, iifname-agnostic** forward rule (`saddr <src-site> daddr <dst-site> accept`). The demo's behind-host (the CP) failed for **deploy-class** reasons, not a data-plane defect: the packet was dropped by **Azure SDN before the gateway** (no UDR), and **mesh mode emits no LAN→tunnel rule** at all.
**Proposed:** (a) **mesh-mode forwarding** — decide whether OFF/mesh should forward a behind-host's site→site traffic, or whether site-to-site-for-behind-hosts DELIBERATELY requires enforcing+a grant (my lean: **require enforcing+grant** — mesh is device-mesh; a routed site grant is a governance act, consistent with unlock-then-opt-in — but state it, don't leave it an accident). (b) The **red** (the fixture-sibling): a genuinely-separate behind-gateway host initiates cross-site **end-to-end**, with the cloud route (D3) in place. HELD.

### D2 — src-hint in ApplyRoutes (#4b)
The agent programs the site route with **no `src`**, so a gateway-host-originated packet mis-sources (overlay addr) AND a manual `src` fix is **clobbered every reconcile tick**.
**Proposed:** `ApplyRoutes` sets `src <the gateway's site-subnet IP>` on the proto-static site routes. **Red:** gateway-host-originated traffic sources correctly AND **survives reconcile** (today it clobbers). Note: real behind-hosts don't hit this (they source naturally); it's the gateway-host-as-source case — but the Zero-Touch Law says even that must Just Work. HELD.

### D3 — host-networking stance + cloud-fabric routing (#3)
Bridge mode traps `wg0` in the container netns (only the overlay addr; can't reach host LANs). The demo needed manual `--network host`. Separately, cloud fabric needs **Azure UDR / AWS route-table + source-dest-check** for a behind-host to reach the gateway.
**Proposed:** (a) **host networking is REQUIRED and emitted** in the install command (D4). (b) The agent **refuses LOUDLY when it can't reach what it advertises** (an approved site subnet it can't route to the host is a health-degraded/blocked state, not silent). (c) Cloud-fabric routing (UDR/route-table) is **documented as a named prerequisite** per cloud (it's outside Tunnex's reach — but the Zero-Touch Law's "click-only" applies to Tunnex surfaces; cloud-fabric setup is stated honestly as the one unavoidable cloud-console step, minimized + documented). HELD.

### D4 — the emitted install command (#1/#2)
The join-token screen emits a `docker compose -f tunnex.yml … node-agent` line (co-located assumption); a remote gateway needs a bare `docker run` with `wgctrl`, host networking, all envs, and the public endpoint — the compose line **broke twice** when pasted, and defaulted `backend=mem`.
**Proposed:** the join-token screen emits the **ONE true `docker run`** for a **remote gateway** (public CP URLs, `TUNNEX_WG_BACKEND=wgctrl`, `--network host`, `TUNNEX_NODE_ENDPOINT`, the token) — copy-paste-correct as-is. The **compose form remains** for co-located single-host installs (a toggle/two tabs on the enroll screen). HELD.

### D5 — site rules in the Access builder (GAP-2, the headline UI gap)
S8.3 built site-rule **display** (`ruleRow` resolves site labels) but **NOT create** — the Add-rule modal is `group/user → group/resource` only. The demo's site→site grant had to be a **raw DB insert** (bypassing API validation + audit — demo-only, never documented).
**Proposed:** add **`site`** to the modal's Source AND Destination type options, creating `src_kind='site'`/`dst_kind='site'` rules **through the EXISTING policy API** (disjointness/validation/audit ride the same path — no new bypass). HELD.

### D6 — org creation + add-rule-with-zero-groups (GAP-1, GAP-3)
GAP-1: an existing owner can't create a 2nd org in-session (`/create-org` is `RequireNoOrg`-gated). GAP-3: the Add-rule button stays disabled after adding a group until a page refresh (pre-existing cross-section staleness).
**Proposed:** GAP-1 — an in-session "New organization" affordance for an existing user (or an explicit decision that multi-org is signup-only). GAP-3 — lift `groups` to the parent / reload RulesSection on group-add. Both same slice as the UI work (D5). HELD.

## RULINGS (folded 2026-07-18 — all six RULED; build S8.2c before S8.4 opens)
- **D1 RULED — mesh mode emits SYMMETRIC site-forward rules (LAN↔tunnel BOTH directions) for approved subnets.** Mesh's contract is "everything open, no doors"; a mode that means no-restrictions silently not forwarding a behind-host is a broken promise, not a design choice. Enforcing keeps grant-gated ((b)-proven correct). **Red:** a behind-gateway host initiates cross-site in mesh (passes); in enforcing-without-grant it **drops at the gateway's chain with the counter incrementing** — the drop MOVES from the invisible fabric to our VISIBLE chain wherever the fabric permits.
- **D2 RULED — mechanism (b): the CP SENDS the local site subnet** (`Compiled.LocalSubnets`); the agent picks its host address inside it for the src-hint. **The one-truth law, 4th instance:** the agent guessing its own site address (egress probe / interface heuristic) is a SECOND independent derivation of a fact the CP already owns authoritatively — which subnet this gateway fronts is exactly what the admin advertised + approved in the dashboard. Heuristics agree on cloud VMs, quietly diverge on-prem (WAN≠LAN) / multi-homed — the exact shapes S8.2c exists to make safe. (b) makes the agent a CONSUMER of the one election — same as the D2-topology overrule (backend hub, no TS re-election), the S8.3 `meta.protocol_version` seam. Four instances = the pattern is the architecture.
  - **CLASSIFICATION (D2-checklist row, explicit RequiredVersion analysis):** `LocalSubnets` is **out-of-hash plumbing like `Routes`** — an old agent ignoring it loses ONLY the src-hint (gateway-host-originated sourcing), **never reachability or enforcement — degraded-not-broken → NO version bump.**
  - **No-site edge:** no local subnet in the artifact → no src-hint, **logged ONCE (not per-tick)**, the route still programs identically to today. Red.
  - **Multi-subnet site:** the agent picks the advertised subnet containing one of its host addresses; if **NONE match** (advertised a subnet the host isn't on) → no src-hint **+ the D3 health kind fires** (refuse-loudly, not a guess).
  - **Reds:** correct sourcing (plain `ping` from the gateway host, no `-I`) · **survives reconcile** (program → manually strip → next tick restores) · the no-site + no-matching-address edges.
- **D3 RULED** — host-mode is the REQUIRED + emitted shape; the agent **detects bridge-trapped wg0 / unreachable advertised subnets → refuses LOUDLY (a health kind, the B2 pattern), never a silent no-op.** Cloud-fabric steps (Azure UDR, AWS route-table + src/dst-check) are un-codeable → **GUIDED SETUP: the site/subnet UI surfaces per-cloud "your fabric needs this route" instructions** (detected-or-declared cloud, copy-paste snippets, doc links). **ZERO-TOUCH LAW BOUNDARY CLAUSE (folded into the law):** the law's boundary is the GATEWAY VM itself — zero SSH after join stands; the cloud console gets ONE named, guided visit per side.
- **D4 RULED as proposed** — the join screen emits the one true `docker run` for remote gateways (wgctrl, host networking, endpoint, all envs, token) + a bare-metal variant; compose stays for co-located. **Red:** the emitted command, pasted verbatim on a CLEAN VM, enrolls to `agent_ready` on real WG with ZERO edits.
- **D5 RULED as proposed** — `site` src/dst options in the modal, riding the EXISTING API path (validation + audit intact; the demo's DB insert is the named anti-pattern). **Include site-rule DELETION in the same pass.**
- **D6 RULED into the UI slice** — org creation for an existing owner; **Add-rule usable with ZERO groups** (site/resource rules don't need a group — the disabled state was a FALSE dependency).
- **Slice order approved:** agent → deploy surface → UI → review → the demo-re-run walk (the acceptance: Pawan re-runs cross-cloud with the only terminal action a pasted join command + one guided cloud-console step per side). **BUILDING Slice 1.**

> **STATUS: DISPOSITIONED — this is the record.** All decide-items ruled (D1–D11); dispositions folded

Branch `story/S7.5.5-mfa`. Migration will be **0031**.

## What already exists (seam map — plug in, not greenfield)

- **Local login** — `auth.Service.Authenticate(email, pw)` (`auth/service.go:127`) verifies the password
  (argon2, timing-equalized no-user + SSO-only + dummy-verify) and surfaces `account_deactivated` (403)
  only AFTER the password checks out. The HTTP `Login` handler (`auth_handlers.go:54`) then calls
  `s.sessions.Create(user.ID)` and sets the session cookie. **The MFA second step inserts BETWEEN these**:
  Authenticate succeeds → (if challenge required) NO session is minted → a challenge is returned instead.
- **SSO login** — `SsoCallback` (`sso_handlers.go:93`) calls `sessions.Create(userID)` directly after the
  IdP callback ("SSO mints a fresh session, fixation rule"). No local password, no local factor.
- **CLI auth (S5.1)** — `CliAuthorize` (`cli_auth_handlers.go`) is `requireVerifiedSessionUser`-gated: the
  browser consent leg REQUIRES an already-authenticated cookie session; the public device-code/token legs
  only complete after that in-browser authorize. So a CLI credential can only be minted downstream of a
  full browser session.
- **Password reset** — `auth.Service.ResetPassword` (`auth/service.go:245`) consumes a single-use reset
  token, sets the new hash, **sweeps every CLI credential** (`cliauth.SweepUser`, same tx) and **revokes
  every session** (`revoker.DeleteAllForUser`). It touches no TOTP today.
- **Recovery-code-class hygiene exists** — `cliauth` tokens: `sha256` hash at rest, `TokenPrefix`, atomic
  single-use consume (code consumed on first touch, replay refused, no which-check oracle), keyed
  fingerprint audit. **The recovery-code model mirrors this exactly.**
- **Sessions** — `session.Store` with `Create(userID)` + a revoker (`DeleteAllForUser`). Cookie-based;
  `requireVerifiedSessionUser` / RequireAuth gate protected routes.
- **Edition gating** — `edition_required` 403 pattern; enterprise features are UNLOCK-THEN-OPT-IN,
  default-OFF, org-level (founder law).
- **Downgrade-releases-enforcement law (S7.5.3)** — any enterprise ENFORCEMENT flag MUST define what
  clears it on enterprise→open; binds this story.
- **Flip-on precedent (S7.5.3)** — 0%-loss grandfather on enabling an enforcement gate (never black-hole
  the fleet).

## Sizing verdict

Medium. New: `user_totp` (secret, enrolled_at, confirmed) + `user_recovery_codes` (hashed, single-use) +
`org_mfa` enforce flag (per-org, default OFF, enterprise). A login-time challenge state (NOT a session).
Enroll/verify/recovery endpoints. Admin-reset (enterprise). One migration (0031). No data-plane, no
compiler, no artifact-version bump (auth-plane only).

## Decide-items

### D1 — The gating/positioning split: enrollment OPEN, org-ENFORCE enterprise  · LOCKED

**Per-user TOTP enrollment + self-service MFA is OPEN (core, all editions).** Only the
**org-level ENFORCE** (mandate MFA for all local users) + admin controls (admin-reset, enforce-audit) are
**enterprise-gated + org-opt-in (default OFF)** — unlock-then-opt-in.

**Argument (this is the positioning call the founder flagged):** a self-hosted security/VPN product that
ships **zero 2FA in the open edition** is a security-credibility problem — a weak default posture and a bad
look for the category. Individual users must be able to self-protect regardless of edition. The ENTERPRISE
value is not "2FA exists" — it's **org-wide MANDATE + fleet compliance + admin override + audit**. This
mirrors the industry norm (GitHub/GitLab: individual 2FA free; org-mandated 2FA is the admin/paid tier).
Zero 2FA in the open edition also hands every competitor comparison a free line — capability is table
stakes; **mandate + fleet compliance is the enterprise value.** Consequence: a user who enrolls TOTP (open,
self-service) is **always** challenged at their own login — their choice, retained in every edition;
org-enforce only adds "unenrolled users must enroll."

**Build consequence (record):** the MFA endpoints arm the 401-walk / RBAC guards in **BOTH** editions.
`test-editions` must prove **enrollment WORKS open-side** (the open path is exercised, a code confirms, a
challenge gates a login) — NOT merely that the enterprise enforce toggle refuses with 403. Open-side
enrollment is a shipped feature, not an unlock stub.

### D2 — Downgrade-release (bound by the S7.5.3 law)  · LOCKED

On enterprise→open downgrade: (a) the **org ENFORCE flag clears** (enforcement releases
— no user is locked out; the mesh-widen analog); (b) **already-enrolled TOTP secrets are RETAINED-but-
unenforced** (the data-retention-on-downgrade sibling), NOT purged.

**Argument:** purging TOTP secrets on a billing event would (1) destroy users' security posture over a
non-security change, (2) force mass re-enrollment on any re-upgrade. Because enrollment is OPEN (D1), a
retained enrollment KEEPS WORKING — the user still gets their own MFA at login; the org merely loses the
power to MANDATE it for others. Retain-but-unenforced is both safer and user-respecting. (Symmetric with
S7.5.3's downgrade-release + S7.5.1b retention thinking.)

### D3 — TOTP enrollment ceremony + admin reset  · LOCKED

- **Provisioning:** standard RFC-6238 TOTP (SHA1, 6 digits, 30s). Server generates the secret; returns an
  `otpauth://` URI (for QR) + the base32 manual key ONCE (one-time-secret hygiene — never re-shown).
- **Verify-before-arm:** enrollment is a two-step ceremony. The secret is stored `confirmed=false` on
  generate; it only becomes `confirmed=true` (and MFA active for the user) when the user submits a VALID
  code. An unconfirmed secret never gates login and is overwritten on re-generate. (No half-enrolled user
  can lock themselves out.)
- **Recovery codes** issued at confirm (D4).
- **Re-enroll/reset (self):** a user re-enrolls by generating a new secret + confirming — replaces the old
  (single active TOTP per user). Losing the device → recovery code → then re-enroll.
- **Admin reset (enterprise):** an admin can **DISENROLL** another user's MFA (removes their TOTP +
  recovery codes → forces re-enrollment at next login if enforce is on). Admin reset **DOES NOT** let the
  admin authenticate as the user (it only clears the factor; the password/SSO gate still stands).
  **Safeguards:** (1) first-class audit — `actor_user_id`=admin, target=user, `mfa.admin_reset` + keyed
  fingerprint; (2) new RBAC perm (D9), never reuse; (3) **in-story: the target user is NOTIFIED**
  (best-effort email via the existing mailer) — a user whose MFA was silently reset by a compromised admin
  MUST find out. A mailer call, not a subsystem; (4) **step-up-auth DEFERRED as a CLASS, not an MFA item**
  (see below).

**Admin step-up-auth class (ledgered, not in-story):** requiring an admin's OWN fresh re-auth/MFA before
invoking an account-takeover-shaped power is not MFA-specific — `admin-resets-MFA` joins `admin-revokes-CLI-
credential`, admin device-reject, and the other admin powers that act on another user's security state.
Building step-up for MFA alone would be a one-off; it lands for the WHOLE class. Trigger = a security-review
pass or S11. Until then the power is bounded by RBAC (D9) + first-class audit + target-notification +
disenroll-only-never-impersonate (D3).

### D4 — Recovery codes: issuance (join-token-class hygiene)  · LOCKED

On TOTP confirm, issue **N=10** single-use recovery codes, shown ONCE (one-time-secret
modal, like the CLI/join-token secret). Stored **sha256-hashed** (mirroring `cliauth`), **atomic single-use
consume** (a used code is burned in the same tx that authenticates — no double-spend, no which-code oracle),
keyed-fingerprint audit on consume. Regeneration replaces the whole set (invalidates all prior). A
low-remaining-count signal surfaced to the user (e.g. ≤3 left). A recovery code is accepted at the login
challenge IN PLACE OF a TOTP code.

**Argument:** recovery codes ARE credentials of the join-token class — same hygiene, no invention. Hashing +
atomic consume is the anti-replay guarantee; the existing `cliauth` consume pattern is the template.

### D5 — SSO-vs-local interplay (+ the CLI leg)  · LOCKED

Org-ENFORCE applies to **local-password logins only**. **SSO logins are EXEMPT** — the
IdP owns the second factor; we don't hold the SSO user's TOTP and double-enforcing is both impossible and
wrong (the org delegated identity to the IdP). A user with a TOTP enrollment is challenged on their LOCAL
login; their SSO login is governed by the IdP's MFA. **CLI auth needs no separate MFA** — `CliAuthorize` is
cookie-session-gated, so the CLI credential is always minted downstream of a browser session that already
passed MFA (verified against `cli_auth_handlers.go`, per the S5.1 D1-caveat-class check — confirmed, not
assumed).

**Argument:** each auth path is governed by its own factor authority. This is why the challenge sits in the
LOCAL Authenticate→session seam (D6) and nowhere near SSO/CLI.

### D6 — Where the challenge sits + the half-authenticated session shape  · LOCKED (standout decision)

The second step is a **login-time challenge** in the local `Authenticate`→`sessions.
Create` seam. The intermediate (password-passed, TOTP-pending) state is **NOT a session** — no session
cookie is minted until TOTP/recovery passes. Instead: on password success WHERE a challenge is required,
return a short-lived (≤5 min) **single-use MFA-challenge token** bound to `(user_id, purpose=mfa_login)`
(the reset/cli-code hygiene class — hashed, single-use). The client POSTs `{challenge, code}` to
**`POST /auth/mfa/verify`** → on success `sessions.Create` mints the full session cookie. Until then the
user has NO authenticated session — RequireAuth rejects everything by construction.

**Argument:** the classic MFA footgun is a "half-authenticated session" that some route forgets to
re-check. Making the pending state a NON-session (a bare challenge token that ONLY the verify endpoint
accepts) removes that entire class — there is no partial session to leak. "Challenge required" =
`org_mfa.enforce AND local-auth` OR `user has a confirmed TOTP` (self-enrolled users always challenged).

### D7 — Sweep interactions + lockout/rate-limit  · LOCKED

- **Password reset does NOT purge TOTP.** A reset proves **inbox control, not device loss** — and (verbatim
  reasoning): **purging TOTP on reset would let an email compromise bypass the second factor entirely,
  defeating MFA's whole purpose.** Reset keeps sweeping sessions + CLI creds (unchanged); TOTP + recovery
  codes survive, so an inbox-only attacker still faces the second factor. Device loss is handled by
  recovery-code → re-enroll, or admin reset (D3).
- **Deactivation:** does not purge TOTP (reactivation restores the account whole); the `account_deactivated`
  403 already blocks login before the challenge matters.
- **Lockout — per-challenge cap now, TERMINAL:** N failed attempts (N=5) **burns the challenge token** →
  the user is dropped back to the password step (where S11.3's backoff will eventually sit). The recovery-
  code surface shares the SAME cap (a recovery-code guess counts against it). The **global brute-force
  backoff** (per-account/per-IP across login + TOTP) is **LEDGERED to S11.3** alongside the CLI brute-force
  items. **Interim exposure (named, not silent):** an attacker who already holds the password can retry
  challenges serially — each retry costs a **fresh password round** — so the per-challenge cap does not
  bound TOTAL guesses; only the password gate + the 30s TOTP window + the 10^6 space do, until S11.3 adds
  account/IP backoff. Ledger-pointer: S11.3 (with the CLI brute-force surfaces).

### D8 — Enforcement flip-on for existing users (the grandfather question)  · LOCKED

On an org enabling ENFORCE with unenrolled local users: **enroll-on-next-login grace**,
NOT hard block. An unenrolled local user, on their next login, passes the password then is routed into the
**enrollment ceremony** (forced) before a full session is minted — no one is locked out; enforcement begins
at each user's next login. This is the 0%-loss flip-on analog (S7.5.3 precedent): the flip never
black-holes the fleet; it converts "logged-in-without-MFA" into "must-enroll-to-proceed."

**Argument:** a hard block would instantly lock out every unenrolled user (mass black-hole — exactly the
class S7.5.3's grandfather avoided). Enroll-on-next-login begins enforcement without a lockout event.

> **FORK (slice 2) — the enrollment-gate mechanism: Option A (gated session + default-deny middleware),
> A over B.** D8 originally said "before a full session is minted"; that wording is AMENDED — **a gated
> session IS minted; full authority arrives on confirm.** Reasoning: D6's challenge-NOT-a-session law
> targets incomplete AUTHENTICATION (password passed, second factor pending, identity not fully proven).
> The D8 grandfather case is different in kind — authentication is COMPLETE (password verified; the user
> has no second factor to pend, which is the problem being solved). What's outstanding is a **compliance
> obligation → an AUTHORIZATION state**, not an authentication one. So the user gets a normal session with
> an authorization OVERLAY (unchanged minting/cookie/CSRF/revocation), and a **default-deny middleware**
> restricts it to enrollment until a confirmed TOTP exists. Option B (an enrollment ticket, no session)
> would contort a second D6-shaped mechanism onto a non-D6 situation, adding a new credential class /
> auth path (more surface, its own hygiene + reds) to the exact endpoints we already session-gate. The
> asymmetry that makes A safe: this state can only grant enrollment-shaped powers — a gated session's
> blast radius is bounded by the default-deny allowlist, whereas a new ticket-auth path is a whole new
> credential class to get wrong.
>
> **Middleware conditions (the allowlist IS the security property):**
> - **Default-deny; allowlist exact + minimal:** `mfaEnrollStart`, `mfaEnrollConfirm`, `logout`,
>   `currentUser` (/auth/me — and me RETURNS the enrollment_required state so the UI can route). **Disenroll
>   is NOT allowlisted** (a gated user has nothing to disenroll; disenrolling while gated self-cycles).
>   `cliAuthorize` is NOT allowlisted (a gated user must not birth a long-lived credential that outlives the
>   gate).
> - **Keyed on ROUTE IDENTITY (operationId), not path-string** — survives a path rename without silently
>   widening or breaking.
> - **Typed error `mfa_enrollment_required`** (403, distinct from generic forbidden) so web/CLI/desktop render
>   "set up 2FA to continue," not a dead permissions error (legibility law applied to the gate).
> - **Self-arming red:** walk ALL OpenAPI operations (the 401-walk shape) — a gated user gets
>   `mfa_enrollment_required` on every non-allowlisted op, so a FUTURE endpoint is gated by construction, not
>   by memory. Plus: confirm flips the gate off in the SAME request (next call succeeds, no re-login); a
>   gated session CANNOT mint a CLI credential (`cliAuthorize` refused).
> - CSRF + the existing auth stack apply UNCHANGED (it's a normal session + an authorization overlay).

**Honesty line owed (paper AND the eventual UI copy — do NOT imply instant coverage):** pre-existing sessions
of unenrolled users **remain valid until natural expiry** after enforce flips on. **"Enforce ON" ≠ "all
sessions MFA-backed"** until sessions roll over. An org needing IMMEDIATE eviction uses the existing
deactivate / session-sweep tools; enforce flip-on is a forward gate, not a retroactive one. No active-session
eviction on flip-on (chosen: lower disruption).

### D9 — RBAC  · LOCKED

New permission `PermMfaManage` (never reuse — per-feature perm convention) gates the
enterprise admin surface: the org enforce toggle + admin-reset. Self-service enrollment/verify/recovery are
NOT perm-gated (any authenticated user manages their OWN MFA). Grant table is generated + drift-guarded.

### D10 — Edition gate + audit + migration + guards  · LOCKED

Enterprise surface (enforce toggle + admin-reset) returns `edition_required` 403 in the
open edition. Audits: `mfa.enrolled` / `mfa.disenrolled` (self, actor_user) · `mfa.admin_reset` (actor=admin,
target=user) · `mfa.enforce_enabled` / `mfa.enforce_disabled` (org) · `mfa.recovery_regenerated` ·
`mfa.recovery_code_used` (keyed fingerprint, never the code). Migration 0031: `user_totp`,
`user_recovery_codes`, `org_mfa`. Both-editions build/test. TOTP secret stored encrypted at rest via the
existing secret sealer (S4.5 keyed) — NOT plaintext.

### D11 — Recovery-code CONSUMPTION semantics  · LOCKED (D4 covers issuance; this covers use)

A recovery code:
- **substitutes for the TOTP code at the challenge step ONLY** — NEVER for the password. It is accepted at
  `POST /auth/mfa/verify` in place of a TOTP code (D6), on a valid MFA-challenge token that already proves
  the password passed. It is never a password bypass.
- **counts against the same per-challenge cap** (D7) — a wrong recovery-code guess burns an attempt like a
  wrong TOTP guess.
- **is consumed atomically + single-use** (D4 hygiene) — burned in the same tx that authenticates.
- **is audited DISTINCTLY** as `mfa.recovery_code_used` (keyed fingerprint, never the code) — a recovery-code
  login is a **security signal**, not a routine event (a used recovery code often means device loss or an
  incident); it is surfaced separately from a normal TOTP login.
- **low-remaining warns** (≤3 left → nudge to regenerate); **consuming the LAST code warns loudly** (the
  user is one device-loss from lockout) but **does NOT force re-enrollment** (the TOTP still works; forcing
  re-enrollment mid-login would be its own footgun).

## Build notes + process (slice sign-off, 2026-07-16 — carried, not re-cutting)

**Slice 1 (security core):**
- **In-house RFC-6238 TOTP (build-vs-dependency, decided in-slice):** implemented in ~40 lines
  (HMAC-SHA1 + dynamic truncation + base32) rather than pulling `pquerna/otp` or equivalent. Rationale:
  the primitive is small and stable, and a full TOTP library is MORE dependency surface (QR rendering,
  multi-algorithm, provisioning helpers) than the one function we need — for an auth-critical path, less
  third-party code in the trust boundary is the safer trade. **Validated against the PUBLISHED RFC 4226
  Appendix D vectors** (`TestHOTPRFC4226Vectors`) — an EXTERNAL oracle, not round-trip self-consistency
  (these 6-digit codes equal the low-6 of the RFC 6238 Appendix B 8-digit vectors). Code comparison is
  **constant-time** (`subtle.ConstantTimeCompare`). Flag for the story-end review if a wider vector set is
  wanted.
- **Server-side TOTP replay guard (in this slice):** a code — or its accepted-window neighbor — that just
  succeeded must NOT succeed twice. Track `last_used_timestep` per user; reject any timestep ≤ last-used.
  Accepted window **±1 step** (standard); stated in code + here. Skipping this is the classic RFC-6238 footgun.
- **Challenge token burned on SUCCESS too**, not only on cap exhaustion — verify consumes it atomically
  (cliauth atomic-consume pattern under a row lock). A token that survives its own success is a replay seam.
- **Deliberate reds owed:** (a) an UNCONFIRMED enrollment (`confirmed=false`) does NOT challenge at login and
  does NOT count as enrolled; (b) a burned/expired challenge token is refused; (c) a replayed TOTP code is
  refused (replay guard); (d) a consumed recovery code is refused. (Built + green, plus cross-user-binding
  and cap-then-fresh-login per the slice-1 sign-off.)

**LAW (walk-found, slice 1) — state-that-must-survive-a-failed-operation cannot share a transaction with
that operation's failure path.** The `cap-then-fresh-login` red caught it: `VerifyChallenge` returned an
`apierr` on a wrong code, and the same `withTx` **rolled the attempt-count increment back** — so a wrong
code cost NOTHING and the terminal cap never fired (a brute-force hole). A wrong code MUST cost something;
a rollback made it free. Fix (the S7.3 outcome-not-error doctrine applied to auth): the tx COMMITS the
increment/burn regardless, and the wrong-code result is an OUTCOME value mapped to an error AFTER commit;
only a genuine infra error rolls back. **This is reusable — S11.3's rate-limiting hits the identical shape
(a counter that rolls back with the refused request is no rate limit); cross-referenced from the S11.3
ledger line.** FAN-OUT FLAG: the story-end review must probe whether the same rollback-erases-state pattern
exists elsewhere in the MFA surface (recovery consumption / challenge burn on any error path) — one found
instance means the class may have siblings.

- **RFC vectors — BOTH layers pinned:** RFC 4226 App-D (HMAC-HOTP core) + RFC 6238 App-B (the TOTP
  time-step / large-counter derivation, SHA1 low-6). The 6238 layer is not covered by 4226; both run.

**Slice 2:** the enforce gate keys on **CONFIRMED** enrollment — an abandoned-mid-ceremony (`confirmed=false`)
enrollment counts as UNENROLLED for the D8 grandfather path (red — it's the seam between the two slices).
Admin-reset is disenroll-only, so it cannot strand even the org's last owner (it removes the challenge, not
the account) — stated, no special case.

**Fan-out probe (named, slice-2 sign-off):** verify the **authorize-before-edition** ordering is UNIFORM
across every new endpoint — an open-build MEMBER must get `forbidden` (authorize first), an open-build
OWNER must get `edition_required` (edition second) — consistent with the S4.5 precedent. Asserted by the
slice-2 reds for the enforce endpoints; the review confirms uniformity across all new surface.

**Slice 3:** the mfa-pending login step is an unauthenticated-adjacent surface — the `loadOne` legibility law
applies to its error states (a failed verify renders legibly, never a reassuring blank).

**Slice-3 dep (decided in-slice): `qrcode.react@4.2.0` (exact-pinned, ISC license).** Defensible under the
less-deps ethos: a QR renderer draws squares from a string — NO parse surface, NO network, view-only (unlike
the in-house TOTP call, where the trust boundary mattered). The **manual base32 key remains the fallback**
(always shown behind "can't scan?") — the QR is an enhancement, not the only enrollment path. License is
ISC (MIT-class), noted here like the wireguard-go vendor pins. Version pinned exactly (no caret) so a QR
render can't drift under us.

**Process (pre-committed):** slice-by-slice gates as usual; **the full multi-finder review runs AFTER slice 3
over the WHOLE story diff** (S7.5.4 shape — one fan-out, token headroom, uninterrupted). Box-walk legs:
downgrade-release + flip-on grandfather (above) **plus one wire execution of each new audited-mutation class**
per the armed-guard convention — `mfa.enrolled`, `mfa.recovery_code_used`, and the **admin-reset with its
notification email observed in Mailpit**.

## Story-end review folds (record — 15 findings → 10 defects; both required probes CLEAN)

The multi-finder review (17 agents) found the rollback-erases-state class has **no siblings** and the
authorize-before-edition ordering is **uniform**. It also found 10 defects; dispositions folded:

**D5 AMENDMENT (findings #1+#2, the headline — a design-level fix, not two patches).** The enrollment
gate was auth-method-BLIND: it keyed on `IsEnrollmentGated(userID)` for every principal, so D5's SSO +
CLI exemptions — established only at the LOGIN seam (Login challenges local-password only) — did NOT hold
at the per-request middleware. An SSO user or a bearer credential in an enforcing org was wrongly gated.
**Root cause: the missing primitive was auth-method provenance on the principal, not a missing `if`.**
Fix: the session/credential records its **auth method at mint** (`authctx.AuthLocalPassword | AuthSSO |
AuthBearer`) — SSO at the SSO mint seam, bearer at credential auth — and the gate applies **only** to
`local_password` principals. The exemption rides the principal's **immutable** mint-time method (session
fixation: a session's method never changes), NOT a route-class or header sniff. Legacy sessions (empty
method, minted pre-marker) are treated as non-local → exempt, which aligns with D8 (enforce governs new
logins, not live sessions; legacy sessions age out) — so **no Redis migration is needed**. The self-arming
walk gained **auth-method variants** (`TestEnrollmentGateAuthMethodExemption`): the walk that proved the
allowlist now also proves the exemption, so the gap that hid this can't re-open. **D5's "SSO EXEMPT / CLI
rides the session" is now enforced at the PRINCIPAL level, unit-pinned — the property whose unit-level
absence is exactly what the review caught.**

**Allowlist lesson (finding #5).** The gate allowlist was derived from the happy path and dead-ended the
**unverified-email + enforced** intersection (enrollment needs a verified email, but `verifyEmail` /
`resendVerification` weren't allowlisted). Fix: the allowlist admits **exactly what a gated user needs to
become UN-gated** — email verification is upstream of enrollment, same class as the enroll ops. Now
`{mfaEnrollStart, mfaEnrollConfirm, verifyEmail, resendVerification, logout, currentUser}`.

**Other folds:** #3 `StartEnrollment` refuses over a confirmed factor (409 `already_enrolled`; re-enroll =
disenroll first) — restartable-over-unconfirmed preserved. #4 the gate-state error is never swallowed at
`/me` or Login — on a resolution error we SURFACE it (loadOne server-side), never guess (omitting gates

## Walk-found defect — the enrollment-gate BRICK + tautological guard (2026-07-16/17)

**Symptom (box-walk Leg 2, grandfather):** a gated local-password member got `403 mfa_enrollment_required`
on **every** op — including the ALLOWLISTED `GET /auth/me` and `POST /auth/mfa/enroll`. Fail-closed HELD
(deny, no bypass) but denied the whole allowlist, so the grandfather path was **bricked in production shape**:
a gated user could not reach the very ops that let them become un-gated.

**True root cause — operationId casing.** `api.GetSwagger()` (the embedded spec the gate resolves through)
carries operationIds as oapi-codegen's EXPORTED (PascalCase) Go identifiers — `CurrentUser`, `MfaEnrollStart`,
… — NOT the source-yaml camelCase. `enrollmentGateAllow` was written in camelCase. Zero matches →
`gateAllows` returned false for every escape op. Fix = key the allowlist on the embedded (PascalCase) names.

**Why every test was blind — the tautology.** The self-arming test asserted
`gateAllows(req) == enrollmentGateAllow[op.OperationID]` — the **same map on both sides**. A camel/Pascal
mismatch cancels perfectly: both lookups miss, both false, equal, GREEN. That is a *self-referential guard* —
it can only detect a thing disagreeing with itself. Distinct failure class from D5's population-blindness:
there the test proved the right property against the wrong population; here it proved **no property at all**
and displayed green doing it.

**LAW (adopt verbatim; armed-guards inventory candidate at merge — the tautological-guard class):**
> A guard's test must assert ABSOLUTE expected behavior, never derive its expectation from the same artifact
> under test. Corollary: test request-shape-sensitive middleware through the REAL chain (`NewRouter`), not an
> isolated helper. Demonstrated-red: the S7.5.5 enrollment gate's self-arming test compared `gateAllows(req)`
> against `enrollmentGateAllow[op.OperationID]` (the same map both sides), so a camelCase-vs-PascalCase opID
> mismatch (source-yaml casing vs oapi-codegen's exported-identifier casing in the embedded spec) cancelled
> out and passed while the grandfather path was fully bricked in production shape.

**Correction record (the diagnosis itself was corrected mid-fix).** The first diagnosis — that `gorillamux
FindRoute` failed on runtime path-only URLs — was **wrong, and was approved-for-build** before a local
full-chain repro (`TestEnrollmentGateThroughRealRouter` driving the real `NewRouter`) reproduced the brick and
instrumentation showed `FindRoute` resolving cleanly (`err=<nil>`, opID `CurrentUser`). The path-only "fix"
for it PASSED precisely because it was irrelevant to the live defect — itself an instance of the law: a red
that does not fail against the live defect proves nothing. Surfaced against the standing disposition rather
than shipped (halt-and-surface applies to one's own approved-but-wrong fixes too).

**Fix (three parts, landed):** (1) allowlist keyed on the embedded PascalCase opIDs, fail-closed default
untouched; (2) self-arming test de-tautologized — absolute `(method, path)` semantic pins (must-pass +
must-deny incl `mfaDisenroll`/`cliAuthorize`/`login`/a plain resource op), breadth (every op resolves + no
leak outside the pins) proven SEPARATELY; (3) `TestEnrollmentGateThroughRealRouter` kept as a PERMANENT
full-chain red so the isolated-helper blind spot stays closed by construction.

**Exposure grep (re-aimed to operationId string literals):** the gate is the ONLY runtime consumer of
`Operation.OperationID`; the RBAC mirror is not operationId-keyed; the authwalk keys
`walkBodies[strings.ToLower(op.OperationID)]` — case-insensitive, **immune** by construction. Class contained
to the gate, now fixed.

# S7.5.4 ‚Äî Per-user + temporary (time-boxed) policy grants (commit-one, decision-first)

> **STATUS: PENDING DISPOSITION.** Paper before code. Extends the Zero-Trust policy model (S7.1 model /
> S7.2 enforcement / S7.5.2 IdP-group subjects) with two orthogonal capabilities: a rule whose SUBJECT
> is a single USER (not just a group), and a rule that EXPIRES at a set time (a temporary grant, window-
> extensible). My leans are marked; the sharp calls are **D2 (extend-before-lapse race)** and **D5 (the
> S8 fail-closed-on-newer-version decide-item ‚Äî decide it, don't drift it)**.

**Three inherited obligations this paper MUST discharge (founder-directed, non-optional):**
1. **The versioned-artifact bump** ‚Äî USER as a subject kind + the S8 seam discipline + the newer-Version
   fail-closed decide-item (‚Üí ¬ßD1 resolves the framing, ¬ßD4 lands the only real bump, ¬ßD5 decides the gate).
2. **Window-extensible expiry** ‚Äî extend BEFORE lapse, never delete+recreate (‚Üí ¬ßD2).
3. **The S7.5.1 deferred device/user identity stamping** ‚Äî "a per-user grant with no user in the flow log
   is half-blind" (docs/S7.5.1-decisions.md:330) ‚Äî this story is its named carrier (‚Üí ¬ßD4).

And the **S7.5.3 ledger law (binding):** any enterprise ENFORCEMENT flag must define what CLEARS it on
downgrade. Grants' downgrade-release is stated explicitly in ¬ßD3 ‚Äî even though the honest answer is
"nothing to clear," the law requires the paper to REASON it, not skip it.

## What already exists (seam map, so we plug in, not greenfield)

- **The compiled artifact is IP-ONLY.** `policyspec.Compiled` (`policyspec.go:93`, `ProtocolVersion=2`)
  carries `Allow []AllowEntry` where `AllowEntry` is `{SrcIP /32, DstCIDR, Protocol, Ports, RuleID}`
  (`policyspec.go:67`). **There are NO group/user/device KINDS on the wire** ‚Äî every subject is resolved
  to /32s inside the pure `Compile` (`enterprise/policy/compiler.go:184-224`). The only identity metadata
  on the wire is `RuleID` (v2/S7.5.1, OBSERVABILITY ‚Äî excluded from `CanonicalHash` via `projectForHash`,
  `hash.go:40`).
- **A rule's subject is a GROUP only.** `policy_rules` (`0018_zero_trust.up.sql:74`): `src_group_id NOT
  NULL ‚Üí user_groups`, `dst_kind ‚àà {resource, group}`. No per-user/per-device subject, no expiry column.
- **The compiler resolves group‚Üí/32 at `compiler.go:188`** (`userGroups[d.UserID]` ‚Äî "a device is in a
  group iff its owner is"). A per-USER subject plugs in at exactly this site; a temporary grant is a
  pre-loop filter on `s.Rules`.
- **4 recompile+push triggers, one choke point** (`enterprise/policy/service.go:538` `mutate ‚Üí pushOrg ‚Üí
  AffectedNodeIDs ‚Üí NotifyMany`): rule CRUD, membership CRUD, device change (incl. `health_blocked`),
  mode. **Expiry-on-a-timer has NO mutation hook ‚Üí it is a NEW 5th trigger (a sweeper).**
- **Flow logs are HALF-BLIND by construction.** `access_events` (`0023_access_events.up.sql`) HAS
  `src_device_id` / `src_user_id` columns but they are **always NIL** ‚Äî enrichment is grant-only
  (`accesslog/ingest.go:251`: "NO src_ip‚Üídevice lookup anywhere ‚Ä¶ a racy IP map is forbidden"). The
  honest fix is agent-STAMPED device identity (docs/S7.5.1-decisions.md:324-334, slot = S7.5.4).
- **Downgrade-release precedent:** `ReleaseAllHealthBlocks` at open-build boot (`devices/health.go:267`,
  `main.go:234-248`) ‚Äî the pattern the ledger law points at.
- **Expiry precedents are all delete/consume/recreate** (`auth_tokens`, `cli_*`); NONE extend in place.
  The reusable machinery is the SWEEPER SHAPE (`SweepStaleHealthBlocks` + `StartHealthSweeper`,
  `health.go:509`: ticker + TTL-filtered clear + per-row system audit + per-org push).
- **Version-bump discipline (S8 seam):** `policyspec.go:45` "bump on any shape change; classify
  enforcement (‚Üí hash projection) vs observability (‚Üí out)"; `TestProtocolVersionConstantsAgree` guards
  the twin constants (`nodes.ProtocolVersion` == `policyspec.ProtocolVersion`). **The agent does NOT
  fail-close on a newer/unknown `Compiled.Version` today** ‚Äî additive forward-compat is convention, and
  `A-4` (single version served fleet-wide) holds (`hash.go:26-34`). The genuine fail-closed gate is
  UNBUILT ‚Äî the S8 seam.

## Sizing verdict

Model change (a `src_kind` discriminator + an `expires_at` on `policy_rules`) + one compiler branch +
one sweeper (the 5th trigger, cloned from `SweepStaleHealthBlocks`) + one OBSERVABILITY artifact field
(`device_id`, a v3 bump exactly like v2's `rule_id`) + ingest enrichment. **S7.5.3-sized ‚Äî no new
enforcement path, no kernel contact.** The compiled artifact's ENFORCEMENT shape is UNCHANGED (still
/32‚ÜíCIDR). The one sprawl vector is the flow-log identity stamping (¬ßD4) ‚Äî but it's additive-observability,
fully precedented by `rule_id`. If the paper starts reaching for a per-user KIND on the wire, or a
grant that DENIES/narrows, that's the scope signal to CALL (see ¬ßD3 lock).

## Decide-items

### D1 ‚Äî Per-user subject (and the "USER as a subject kind" framing, resolved)

- **Model: a `src_kind` discriminator on `policy_rules`** mirroring the existing `dst_kind` pattern.
  `src_kind ‚àà {group, user}`; `src_group_id` XOR `src_user_id` (a CHECK enforces exactly one, like the
  `dst_*` columns). A per-user rule names a single org member as the source subject.
- **Compiler: one branch at `compiler.go:188`.** The `userGroups[d.UserID]` membership test gains a
  `|| (r.SrcKind == "user" && r.SrcUserID == d.UserID)` arm ‚Äî a per-user rule resolves to THAT user's
  active device /32s, skipping the group indirection. Everything downstream (dedup, allow-set keying,
  the 4 triggers) is unchanged.
- **RESOLVES the "USER as a subject kind in `policyspec.Compiled`" framing (the load-bearing
  clarification):** USER becomes a subject kind at the RULE-SPEC level (`policy_rules.src_kind`), **NOT
  on the compiled wire artifact** ‚Äî which stays IP-only (/32‚ÜíCIDR) because the per-user subject is
  resolved to the user's device /32s CP-side, identically to how a group is. **Therefore the per-user
  subject bumps NO enforcement artifact version.** The only version bump in this story is D4's
  observability `device_id`. (This is why the artifact stayed IP-only through S7.1/S7.2/S7.5.2 ‚Äî identity
  never reaches the wire as a kind; S7.5.4 keeps that invariant.)
- The membership FK cascade that protects group grants (`group_members (org_id,user_id) ‚Üí memberships ON
  DELETE CASCADE`) must have a per-user analog: a `src_user_id` rule must CASCADE-drop (or be swept) when
  its user leaves the org, or it dangles as a grant to a departed member. Sub-decision below.
- **RECOMMEND** the `src_kind` discriminator. *Open sub-decision:* the departed-user cleanup ‚Äî a direct
  `src_user_id ‚Üí memberships(org_id,user_id) ON DELETE CASCADE` FK (clean, structural, mirrors
  `group_members`) vs a service-layer sweep. Recommend the **FK cascade** (structural > procedural; it's
  the S7.1 discipline).

### D2 ‚Äî Temporary grants: expiry model + WINDOW-EXTENSIBLE (the inherited obligation)

- **Model: a nullable `expires_at timestamptz` on `policy_rules`.** NULL = permanent (every rule today);
  set = a temporary grant that lapses at `expires_at`. A rule can be per-user, temporary, both, or
  neither ‚Äî the two features are orthogonal columns, not a new entity.
- **Compiler: a pre-loop filter** `WHERE expires_at IS NULL OR expires_at > now()` on `s.Rules` (the
  correctness backstop ‚Äî an expired rule stops compiling on the NEXT recompile REGARDLESS of the
  sweeper). This is belt; the sweeper is braces.
- **The 5th recompile trigger ‚Äî a grant-expiry SWEEPER** (cloned from `SweepStaleHealthBlocks`,
  `health.go:509`): a `time.Ticker` (interval ~1 min ‚Äî expiry PROMPTNESS matters more than the 5-min
  health cadence, tunable) that, when a temporary rule crosses `expires_at`, recompiles + pushes the
  affected org so the grant's /32s leave the gateway within the sweep interval, not on the reconcile
  fallback. Per-lapse audit: system actor, cause `grant_expired` (`auditSystem`, mirroring
  `report_stale`).
- **EXTEND-BEFORE-LAPSE (the obligation, LOCKED):** extending a grant is `UPDATE policy_rules SET
  expires_at = $new WHERE id = $1 ‚Ä¶` ‚Äî bumping the window IN PLACE, never delete+recreate (which would
  churn the /32 out and back, and lose the rule's identity/audit lineage). **The sharp call ‚Äî the
  extend-vs-sweep RACE:** the sweeper may lapse a grant in the same instant an admin extends it. Guard:
  **extend REFUSES a rule whose `expires_at` has already passed** (`‚Ä¶ AND expires_at > now()` in the
  UPDATE; 0 rows ‚Üí a `409 grant_lapsed` "this grant already expired ‚Äî create a new one"). A lapsed grant
  is terminal; only a live grant's window moves. This makes extend and the sweeper serialize on the row
  without a lock: whoever commits first wins, and the loser sees the honest terminal state. (Diverges
  from every existing expiry row ‚Äî `auth_tokens`/`cli_*` are consume-once, never extended ‚Äî so this race
  is genuinely new; stating it is the point.)
- **RECOMMEND** `expires_at` on `policy_rules` + the pre-loop filter + the 1-min sweeper + extend-refuses-
  lapsed. *Open sub-decision:* the sweep interval (tighter = promoter expiry, more recompiles). Recommend
  ~1 min (a grant that says "expires at 5:00" leaving at 5:04 is a poor look; health's 5 min was fine
  because staleness is fuzzy, expiry is a promise).

### D3 ‚Äî Downgrade-release (the S7.5.3 ledger law ‚Äî MANDATORY statement)

The law: any enterprise ENFORCEMENT flag must define what clears it on downgrade. For grants the honest
answer is **"nothing to clear" ‚Äî but here is the REASONING the law requires, not a skip:**

- **Grants add NO persistent enforcement STATE on any row** (unlike `health_blocked`, a device-row flag
  that survived a downgrade and stranded the device). A grant is a `policy_rules` row that is a COMPILER
  INPUT. Enforcement lives ONLY in the compiled artifact.
- **On downgrade (policy edition OFF ‚Üí `NewPolicyEdition()` nil port), the compiled artifact vanishes and
  agents fall back to the legacy MESH (default-ALLOW within the org).** So every grant becomes moot and
  access WIDENS, never narrows ‚Äî no user is stranded off the network by a grant on downgrade. The grant
  ROWS persist harmlessly (re-compiled on re-upgrade if still unexpired; the sweeper resumes).
- **Therefore no boot-time `Release*` is needed** ‚Äî the "release" is structural: disabling the feature
  removes the artifact that carried the enforcement, and the fallback is MORE open. This is the OPPOSITE
  of health_blocked's petrification, and the paper states it so a future reader (S7.5.5) doesn't assume
  every enforcement feature needs a boot-release. *(If a downgrade could ever STRAND access, a release
  would be mandatory ‚Äî see the lock below that keeps grants additive so it can't.)*
- **LOCK ‚Äî grants are ADDITIVE-ONLY (allow grants), never DENY/narrow.** A per-user or temporary grant
  only ADDS allow entries on top of the default-deny model; there is no "deny grant" / "restrict grant."
  This (a) keeps the downgrade safe (mesh reopens, nothing stranded), (b) keeps the default-deny model
  clean (a deny is the ABSENCE of an allow, not an explicit rule), and (c) is the scope fence ‚Äî a
  temporary DENY (quarantine) is a DIFFERENT feature (would need its own downgrade-release + a device-row
  flag like health_blocked). Out of S7.5.4.

### D4 ‚Äî Flow-log identity stamping (the S7.5.1 deferred obligation ‚Äî its named carrier)

"A per-user grant with no user in the flow log is half-blind" ‚Äî you cannot audit "did user X's temporary
grant actually get used" if the event only has a src IP. This story makes USER-level events real:

- **Agent-STAMPED device identity (the honest path, NOT an `src_ip‚Üídevice` DB lookup ‚Äî that racy map is
  forbidden, `ingest.go:44`).** The compiled artifact carries the /32‚Üídevice mapping so the agent can
  stamp it: **`AllowEntry` (or a parallel map on `Compiled`) gains `SrcDeviceID` ‚Äî OBSERVABILITY, exactly
  like `RuleID`.** The agent decodes it, stamps `device_id` on the `flowlog.Event`, it rides `WireEvent`,
  and `ingest.enrich` writes it straight through to `access_events.src_device_id` ‚Äî no DB IP lookup.
- **`src_user_id` joins CP-side** from `src_device_id ‚Üí devices.user_id` at ingest (a clean FK join, not
  a wire field) ‚Äî so the wire carries ONE new observability field (device_id), and user is resolved
  server-side.
- **This is a v3 `ProtocolVersion` bump ‚Äî classified OBSERVABILITY:** `SrcDeviceID` stays OUT of
  `projectForHash` on BOTH sides (`policyspec` + `nodepolicy`), so existing pushed/applied canonical
  hashes are UNDISTURBED (no false staleness on every enforcing gateway ‚Äî the exact trap v2's `rule_id`
  navigated). Guarded by `TestProtocolVersionConstantsAgree` (twin constants move together) + the hash-
  projection reds + the nodepolicy field-order golden.
- **RECOMMEND** agent-stamped `SrcDeviceID` (v3, observability) + CP-side `src_user_id` join. *Open sub-
  decision:* stamp device_id for ALL flow events, or only per-user-grant ones? Recommend ALL (device
  identity is universally useful for audit; scoping it to per-user grants is a false economy and a branch
  in the hot path). The flow-log UI surfacing of the new identity columns is a fast-follow, not commit-one.

### D5 ‚Äî The S8 fail-closed-on-newer-version decide-item (DECIDE it, don't drift it)

The founder flagged the "newer-Version fail-closed decide-item." With v3 (device_id) landing, this is the
moment to decide ‚Äî and the decision is **defer-with-a-named-trigger, reasoned:**

- **v3 is OBSERVABILITY, so an old agent ignoring `SrcDeviceID` is CORRECT and SAFE** ‚Äî it simply doesn't
  stamp identity; enforcement is byte-identical (the field is out of the hash). So **v3 does NOT require a
  fail-closed gate** ‚Äî additive-observability forward-compat (unknown JSON fields ignored) is exactly
  right here, as it was for v2.
- **The agent still does NOT fail-close on a newer `Compiled.Version` ‚Äî and that stays TRUE after this
  story.** Building the gate now would be speculative: there is no enforcement-significant version skew to
  protect against (A-4 holds ‚Äî the CP serves ONE version fleet-wide).
- **REGISTER the trigger (the real S8 seam):** the FIRST time the artifact gains an ENFORCEMENT-significant
  shape change that an old agent would MIS-enforce (a new subject KIND on the wire, a new match dimension,
  or the EPIC-8 multi-version-served-fleet-wide reality that breaks A-4), the agent MUST fail-closed on a
  `Compiled.Version` it doesn't understand (refuse the artifact ‚Üí cold-start deny-all, not silent-ignore).
  That is the gate to build THEN, not now. **This story does not build it; it records exactly when it
  becomes mandatory** (so it can't drift into a silent mis-enforcement ‚Äî the hash.go:26 warning made real).
- **RECOMMEND** defer-with-named-trigger. Ship v3 as observability; do NOT build the S8 gate; register
  its trigger + the A-4 dependency.

### D6 ‚Äî RBAC

- **Reuse `PermPolicyManage`.** Per-user and temporary grants are policy mutation at the SAME owner/admin
  grain as rule/group/mode management ‚Äî a grant IS a policy rule. Minting a new perm would fragment the
  policy-governance grain for no capability boundary. (The per-feature naming discipline, `rbac.go:33`,
  says mint-new only when the capability is genuinely distinct ‚Äî here it isn't.)
- **RECOMMEND** `PermPolicyManage` reuse. *Registered alternative (S7.5.4b):* if a "grant approver" grain
  emerges (someone who may issue TEMPORARY access but not rewrite permanent policy), mint `PermGrantIssue`
  then ‚Äî not now.

### D7 ‚Äî Edition gate + audit + standing guards

- **Edition:** enterprise-gated via the EXISTING policy port (`NewPolicyPort`/`policy_wire_enterprise.go`)
  ‚Äî grants are part of the Zero-Trust policy feature, NOT a new named edition bool. Open build: the policy
  endpoints already 403; grant fields ride that gate.
- **Audit:** human grant create/extend/revoke = HUMAN actor (`writeAudit` + principal). Auto-expiry via
  the sweeper = SYSTEM actor (`auditSystem`, cause `grant_expired`) ‚Äî no human initiated the lapse. Extend
  records the old‚Üínew window in metadata (the audit trail must show a grant's window was moved, by whom).
- **Standing guards auto-arm:** querylint (`policy_rules` already tenant-scoped); the codegen drift guard
  (spec ‚Üí Go/TS); `test-editions` (both builds); `TestProtocolVersionConstantsAgree` + hash-projection
  reds catch the v3 bump; the canonical-hash twin goldens catch a `nodepolicy` field-order drift.
- **RECOMMEND** as stated.

## Downgrade-release (restated for the ledger, per the binding law)

**Grants add no persistent enforcement state; enforcement is the compiled artifact, which disappears on
downgrade ‚Üí the fleet falls back to mesh (default-allow) ‚Üí grants strand no one (access widens).** No
boot-time release is required. The grants-are-additive-only lock (¬ßD3) is what keeps this true ‚Äî a
narrowing grant WOULD need a release, so narrowing is out of scope. This is the law honored by REASONING,
not by a skip. (S7.5.5 MFA-enforce, per the ledger, must likewise state its downgrade-release at its own
commit-one ‚Äî and unlike grants it likely DOES petrify, so it likely needs a real `Release*`.)

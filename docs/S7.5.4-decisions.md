## Decide-items

### D1 — Per-user subject (and the "USER as a subject kind" framing, resolved)

- **Model: a `src_kind` discriminator on `policy_rules`** mirroring the existing `dst_kind` pattern.
  `src_kind ∈ {group, user}`; `src_group_id` XOR `src_user_id` (a CHECK enforces exactly one, like the
  `dst_*` columns). A per-user rule names a single org member as the source subject.
- **Compiler: one branch at `compiler.go:188`.** The `userGroups[d.UserID]` membership test gains a
  `|| (r.SrcKind == "user" && r.SrcUserID == d.UserID)` arm — a per-user rule resolves to THAT user's
  active device /32s, skipping the group indirection. Everything downstream (dedup, allow-set keying,
  the 4 triggers) is unchanged.
- **RESOLVES the "USER as a subject kind in `policyspec.Compiled`" framing (the load-bearing
  clarification):** USER becomes a subject kind at the RULE-SPEC level (`policy_rules.src_kind`), **NOT
  on the compiled wire artifact** — which stays IP-only (/32→CIDR) because the per-user subject is
  resolved to the user's device /32s CP-side, identically to how a group is. **Therefore the per-user
  subject bumps NO enforcement artifact version.** The only version bump in this story is D4's
  observability `device_id`. (This is why the artifact stayed IP-only through S7.1/S7.2/S7.5.2 — identity
  never reaches the wire as a kind; S7.5.4 keeps that invariant.)
- The membership FK cascade that protects group grants (`group_members (org_id,user_id) → memberships ON
  DELETE CASCADE`) must have a per-user analog: a `src_user_id` rule must CASCADE-drop (or be swept) when
  its user leaves the org, or it dangles as a grant to a departed member. Sub-decision below.
- **DISPOSITIONED: the `src_kind` discriminator + the FK cascade** (`src_user_id →
  memberships(org_id,user_id) ON DELETE CASCADE` — structural > procedural, the S7.1 discipline).
- **RIDER (how the cascade reaches the WIRE — the F1 committed-removal-must-push law):** the cascade
  DELETE removes the grant row correctly in the DB, but a departed user's /32 leaving the compiled
  artifact must **recompile + push ORG-WIDE**, or the DB is correct while the gateway serves a stale
  grant to a departed member (the S7.5.2 "committed-removal-must-push" class). A pure `ON DELETE CASCADE`
  fires no application code → NO recompile. **Therefore the member-removal path (which S7.2 already wired
  as the membership recompile trigger — `RemoveGroupMember`/deactivation → `pushOrg`) MUST also cover the
  `src_user_id`-grant deletion:** member removal recompiles the org regardless, and the compiler reads
  the CURRENT `policy_rules` (post-cascade) — so as long as the cascade commits in the SAME transaction
  that the recompile reads after, the departed user's per-user grant is gone from the artifact on that
  push. **Slice-1 test obligation (red):** removing an org member with a live `src_user_id` grant
  recompiles + pushes org-wide AND the compiled artifact no longer contains that user's /32
  (`TestPerUserGrantDropsOnMemberRemoval`) — the cascade-correct-but-gateway-stale trap, proven closed.

### D2 — Temporary grants: expiry model + WINDOW-EXTENSIBLE (the inherited obligation)

- **Model: a nullable `expires_at timestamptz` on `policy_rules`.** NULL = permanent (every rule today);
  set = a temporary grant that lapses at `expires_at`. A rule can be per-user, temporary, both, or
  neither — the two features are orthogonal columns, not a new entity.
- **Compiler: a pre-loop filter** `WHERE expires_at IS NULL OR expires_at > now()` on `s.Rules` (the
  correctness backstop — an expired rule stops compiling on the NEXT recompile REGARDLESS of the
  sweeper). This is belt; the sweeper is braces.
- **The 5th recompile trigger — a grant-expiry SWEEPER** (cloned from `SweepStaleHealthBlocks`,
  `health.go:509`): a `time.Ticker` (interval ~1 min — expiry PROMPTNESS matters more than the 5-min
  health cadence, tunable) that, when a temporary rule crosses `expires_at`, recompiles + pushes the
  affected org so the grant's /32s leave the gateway within the sweep interval, not on the reconcile
  fallback. Per-lapse audit: system actor, cause `grant_expired` (`auditSystem`, mirroring
  `report_stale`).
- **EXTEND-BEFORE-LAPSE (the obligation, LOCKED):** extending a grant is `UPDATE policy_rules SET
  expires_at = $new WHERE id = $1 …` — bumping the window IN PLACE, never delete+recreate (which would
  churn the /32 out and back, and lose the rule's identity/audit lineage). **The sharp call — the
  extend-vs-sweep RACE:** the sweeper may lapse a grant in the same instant an admin extends it. Guard:
  **extend REFUSES a rule whose `expires_at` has already passed** (`… AND expires_at > now()` in the
  UPDATE; 0 rows → a `409 grant_lapsed` "this grant already expired — create a new one"). A lapsed grant
  is terminal; only a live grant's window moves. This makes extend and the sweeper serialize on the row
  without a lock: whoever commits first wins, and the loser sees the honest terminal state. (Diverges
  from every existing expiry row — `auth_tokens`/`cli_*` are consume-once, never extended — so this race
  is genuinely new; stating it is the point.)
- **DISPOSITIONED:** `expires_at` on `policy_rules` + pre-loop filter + ~1-min sweeper + extend-refuses-
  lapsed. **LINGER model (not delete-on-sweep) — the principled reason:** an expired temporary grant is
  AUDIT-RELEVANT HISTORY ("Alice had prod access Tue–Fri") — exactly the who-accessed-what-when
  visibility this epic builds; delete-on-sweep would destroy that record for a simplicity gain, and the
  restart-gap it would avoid is ALREADY handled by the compiler filter (correctness never depends on the
  sweep). Linger IS the feature.

> **AMENDMENT (story-end review, Option A — linger REVERSED to delete-on-sweep).** The full multi-finder
> review fired the budget tripwire: 4 of 9 findings clustered in this exact subsystem — linger's kept
> expired row is the root of [1] (re-grant after lapse dead-ends: extend refuses `grant_lapsed`, re-create
> collides on the expiry-agnostic unique index) + [2]/[3] (edit-with-expiry collisions), and the
> in-memory-window sweeper linger forced produced [4] (error advances the window → skipped audit) + [5]
> (restart resets the window → downtime lapses unaudited). **What the review revealed:** linger's only
> remaining value was DISPLAY of expired grants — extend-after-lapse was already refused by design
> (window-extensible = extend BEFORE lapse, the locked constraint), so the lingering row bought a
> rules-list cosmetic at the cost of being the defect root. The audit history it was meant to preserve
> lives in `audit_logs` (`grant_created` + `grant_expired`), not in the row. **What replaces it
> (reduce-not-patch, not a marker-on-top patch):** the sweeper **DELETEs** expired grants (audited
> `grant_expired`, system actor `grant-expiry` + cause, SAME-TX, then org-wide push) — a **STATELESS**
> sweep (no `last` window), which closes [4]/[5] BY CONSTRUCTION (a failed/interrupted tick just leaves
> rows for the next tick to delete+audit). A re-grant after lapse is a NEW grant with its own
> `grant_created` (cleaner than reviving a two-lifetime row). **Accepted residual:** a re-grant racing the
> sweep-delete can transiently `409` on the unique index (~≤1 min) — retry semantics, not engineered
> around. Extend audit now records **old→new** window (D7). `grant_expired` is a new audited-mutation
> class → gets ONE wire execution in the box-walk (the audit-nil-metadata armed-guard convention), incl.
> a downtime-lapse leg and a Leg-3b-shaped non-hosting-gateway strip.
- **LEDGER (bounded-linger-retention follow, NOT a slice-2 fix):** lingering expired-grant rows are
  display+audit retention and need a BOUNDED lifetime eventually — an inert-for-N-days expired grant gets
  real-swept (hard-deleted) or rides the flow-log retention discipline (`accesslog` retention sweep) — so
  the table can't grow unbounded forever. Registered so it can't be forgotten; not built now.
- **RIDER — extend and sweep COMPOSE on the same row lock (the S7.5.1 composed-critical-section lesson,
  on the grant lifecycle):** both the extend `UPDATE … WHERE id=$1 AND expires_at > now()` and the
  sweeper's expire-read must serialize on the SAME `policy_rules` row so a grant lapsing mid-extend
  resolves DETERMINISTICALLY to extended-OR-`409 grant_lapsed`, never torn. Mechanism: the extend runs in
  a tx that `SELECT … FOR UPDATE`s the row (or relies on the `AND expires_at > now()` predicate lock);
  the sweeper's recompile reads a consistent snapshot. Whoever commits first wins; the loser sees the
  honest terminal state. No lost-update, no half-extended grant.
- **RIDER — the lapse failure direction is SAFE (state it explicitly):** a lapsed-but-not-yet-swept grant
  keeps allowing for **≤ ~1 min** (until the next sweep tick), and **that is acceptable BECAUSE the
  failure direction is "access lingers briefly," never "access wrongly cut."** Grants are additive
  (§D3), so a late lapse over-permits for a bounded window — the safe direction — exactly as an early
  membership removal under-permits. The compiler filter (`expires_at > now()`) is the correctness
  backstop; the sweeper is promptness. We do NOT need the lapse to be instant because instant-cut is not
  a safety requirement here (contrast a deny/quarantine grant, where a late apply WOULD be unsafe — out
  of scope, §D3). Interval 1 min LOCKED (a promise like "expires at 5:00" leaving at 5:04 is a poor look;
  staleness was fuzzy at 5 min, expiry is a promise → tighter).

### D3 — Downgrade-release (the S7.5.3 ledger law — MANDATORY statement)

The law: any enterprise ENFORCEMENT flag must define what clears it on downgrade. For grants the honest
answer is **"nothing to clear" — but here is the REASONING the law requires, not a skip:**

- **Grants add NO persistent enforcement STATE on any row** (unlike `health_blocked`, a device-row flag
  that survived a downgrade and stranded the device). A grant is a `policy_rules` row that is a COMPILER
  INPUT. Enforcement lives ONLY in the compiled artifact.
- **On downgrade (policy edition OFF → `NewPolicyEdition()` nil port), the compiled artifact vanishes and
  agents fall back to the legacy MESH (default-ALLOW within the org).** So every grant becomes moot and
  access WIDENS, never narrows — no user is stranded off the network by a grant on downgrade. The grant
  ROWS persist harmlessly (re-compiled on re-upgrade if still unexpired; the sweeper resumes).
- **Therefore no boot-time `Release*` is needed** — the "release" is structural: disabling the feature
  removes the artifact that carried the enforcement, and the fallback is MORE open. This is the OPPOSITE
  of health_blocked's petrification, and the paper states it so a future reader (S7.5.5) doesn't assume
  every enforcement feature needs a boot-release. *(If a downgrade could ever STRAND access, a release
  would be mandatory — see the lock below that keeps grants additive so it can't.)*
- **LOCK — grants are ADDITIVE-ONLY (allow grants), never DENY/narrow.** A per-user or temporary grant
  only ADDS allow entries on top of the default-deny model; there is no "deny grant" / "restrict grant."
  This (a) keeps the downgrade safe (mesh reopens, nothing stranded), (b) keeps the default-deny model
  clean (a deny is the ABSENCE of an allow, not an explicit rule), and (c) is the scope fence — a
  temporary DENY (quarantine) is a DIFFERENT feature. Out of S7.5.4.
- **LEDGER (fence made inheritable, so the next story doesn't rediscover it):** a future **deny / narrow /
  quarantine grant DOES create persistent enforcement state** (it CUTS access that would otherwise flow,
  so on downgrade its absence must not silently re-open a quarantined device) — **and therefore DOES owe a
  downgrade-release**, like `health_blocked` (a device/grant-row flag cleared at open-build boot,
  audited). That story INHERITS the S7.5.3 downgrade-releases-enforcement obligation at its own commit-one
  — it does not get the by-construction free pass additive grants get here. The dividing line is exactly:
  **additive (grants access) = release-by-construction (mesh fallback widens); subtractive (cuts access) =
  owes an explicit release.**

### D4 — Flow-log identity stamping (the S7.5.1 deferred obligation — its named carrier)

"A per-user grant with no user in the flow log is half-blind" — you cannot audit "did user X's temporary
grant actually get used" if the event only has a src IP. This story makes USER-level events real:

- **Agent-STAMPED device identity (the honest path, NOT an `src_ip→device` DB lookup — that racy map is
  forbidden, `ingest.go:44`).** The compiled artifact carries the /32→device mapping so the agent can
  stamp it: **`AllowEntry` (or a parallel map on `Compiled`) gains `SrcDeviceID` — OBSERVABILITY, exactly
  like `RuleID`.** The agent decodes it, stamps `device_id` on the `flowlog.Event`, it rides `WireEvent`,
  and `ingest.enrich` writes it straight through to `access_events.src_device_id` — no DB IP lookup.
- **`src_user_id` joins CP-side** from `src_device_id → devices.user_id` at ingest (a clean FK join, not
  a wire field) — so the wire carries ONE new observability field (device_id), and user is resolved
  server-side.
- **This is a v3 `ProtocolVersion` bump — classified OBSERVABILITY:** `SrcDeviceID` stays OUT of
  `projectForHash` on BOTH sides (`policyspec` + `nodepolicy`), so existing pushed/applied canonical
  hashes are UNDISTURBED (no false staleness on every enforcing gateway — the exact trap v2's `rule_id`
  navigated). Guarded by `TestProtocolVersionConstantsAgree` (twin constants move together) + the hash-
  projection reds + the nodepolicy field-order golden.
- **DISPOSITIONED: agent-stamped `SrcDeviceID` (v3, OBSERVABILITY, out of hash) + CP-side `src_user_id`
  join, stamped for ALL flow events** — device attribution is valuable log-wide; a per-user-grant-only
  stamp would make a confusing PARTIAL log (some events attributed, some not, for no principled reason)
  and add a branch in the hot path. One field, universal. The flow-log UI surfacing of the new identity
  columns is a fast-follow, not commit-one.

### D5 — The S8 fail-closed-on-newer-version decide-item (DECIDE it, don't drift it)

The founder flagged the "newer-Version fail-closed decide-item." With v3 (device_id) landing, this is the
moment to decide — and the decision is **defer-with-a-named-trigger, reasoned:**

- **v3 is OBSERVABILITY, so an old agent ignoring `SrcDeviceID` is CORRECT and SAFE** — it simply doesn't
  stamp identity; enforcement is byte-identical (the field is out of the hash). So **v3 does NOT require a
  fail-closed gate** — additive-observability forward-compat (unknown JSON fields ignored) is exactly
  right here, as it was for v2.
- **The agent still does NOT fail-close on a newer `Compiled.Version` — and that stays TRUE after this
  story.** Building the gate now would be speculative: there is no enforcement-significant version skew to
  protect against (A-4 holds — the CP serves ONE version fleet-wide).
- **REGISTER the trigger (the real S8 seam):** the FIRST time the artifact gains an ENFORCEMENT-significant
  shape change that an old agent would MIS-enforce (a new subject KIND on the wire, a new match dimension,
  or the EPIC-8 multi-version-served-fleet-wide reality that breaks A-4), the agent MUST fail-closed on a
  `Compiled.Version` it doesn't understand (refuse the artifact → cold-start deny-all, not silent-ignore).
  That is the gate to build THEN, not now. **This story does not build it; it records exactly when it
  becomes mandatory** (so it can't drift into a silent mis-enforcement — the hash.go:26 warning made real).
- **DISPOSITIONED: defer-with-named-trigger.** Ship v3 as observability; do NOT build the S8 gate now
  (v3 is observability → old-agent-ignores is safe).
- **RIDER — write the sharpened trigger where EPIC 8's commit-one INHERITS it** (not just here): **EPIC 8
  (the multi-version / operator-reconciliation epic) must, at its commit-one, build the agent
  fail-closed-on-newer-`Compiled.Version` gate BEFORE it does anything that breaks A-4** (serves more than
  one artifact version fleet-wide) OR adds an enforcement-significant wire field. Until that gate exists,
  a newer enforcement-significant artifact reaching an old agent = SILENT mis-enforcement (the agent
  ignores the field it should honor). The trigger is recorded in this paper AND must be surfaced in EPIC
  8's planning so it can't drift — the `hash.go:26` warning ("Version becomes a divergence source at
  EPIC 8") made into an actionable inherited obligation.

## Downgrade-release (restated for the ledger, per the binding law)

**Grants add no persistent enforcement state; enforcement is the compiled artifact, which disappears on
downgrade → the fleet falls back to mesh (default-allow) → grants strand no one (access widens).** No
boot-time release is required. The grants-are-additive-only lock (§D3) is what keeps this true — a
narrowing grant WOULD need a release, so narrowing is out of scope. This is the law honored by REASONING,
not by a skip. (S7.5.5 MFA-enforce, per the ledger, must likewise state its downgrade-release at its own
commit-one — and unlike grants it likely DOES petrify, so it likely needs a real `Release*`.)

## Box-walk RESULTS + walk-found amendment (founder-present session, 2026-07-16)

### D8 (walk-found) — extend must NOT push: `ExtendGrant` uses `withTx`, not `mutate`

Leg 4's zero-churn half was written to CATCH a spurious push, and it did. `ExtendGrant` went through
`s.mutate` (= `withTx` + **unconditional `pushOrg`**), so every window-bump recompiled the whole org and
re-applied a **byte-identical** ruleset on every gateway — the wire showed the /32 allow's nft handle
churn (A `426→430`, B `510→514`) + counter reset. That contradicts the `ExtendPolicyRule` comment's own
"never … cause a spurious push" claim: the in-place UPDATE avoided a delete+recreate, but the service
layer pushed anyway.

- **Fix:** `ExtendGrant` uses `s.withTx` (no push). SAFE because `expires_at` is **not** in the compiled
  enforcement artifact (the `CanonicalHash` projection excludes it), and this endpoint is **extend-ONLY**
  — `ExtendGrantRequest` is `expires_at`-only (`additionalProperties:false`), `ExtendPolicyRule` SETs
  only `expires_at`, and no PATCH / `UpdatePolicyRule` path exists. So no artifact-affecting field can
  flow through it → dropping the push cannot hide an edit that SHOULD push (the inverse defect the
  mini-review specifically cleared). Lapse stays enforced by the compiler's `expires_at > now()` filter +
  the sweeper's delete-push.
- **RED first:** `TestExtendGrantWindow` now asserts the extend fires NO push (proven to FAIL pre-fix on
  an isolated DB; GREEN post-fix; the create/member/mode/rule push-pins in `TestMutationsFirePush` stay
  green). Mini-review (focused, not fan-out): no issues — inverse-defect ruled out, lapse enforcement
  intact, red valid, no other caller relies on `mutate`'s push.
- **Re-earned on the wire (post-fix):** handles IDENTICAL across extend on A (`720`) + B (`804`), node
  logs silent, window moved, `grant_extended` audit `old≠new`. The comment's intent is now TRUE on the
  wire and pinned by the red. Commit `3a6d7d3`.

### Leg-3 two-mechanism distinction (recorded — a strengthening, not a wart)

Downtime lapse surfaced that a lapsed grant is removed by TWO independent mechanisms with different
latencies: (a) the **compiler active-rules filter** (`expires_at > now()`) strips enforcement on the
NEXT real recompile/refetch — fast (≈3s after API restart, on the post-restart re-fetch); (b) the
**stateless sweeper** DELETEs the row + writes the `grant_expired` system audit + pushes — on its ≤60s
tick. So after a downtime lapse, enforcement is correct almost immediately while the audit/row-delete
"accounting" catches up on the next sweep (row deleted +62s post-restart, audit then present). The
stateless sweep (no window cursor) is what closes the downtime-audit gap: it picks up EVERY currently
-expired grant regardless of when it expired, so nothing is lost across an outage.

### Atomic-apply reliance (named, load-bearing)

D8's spurious push caused **no enforcement gap** because the node applies its whole ruleset in ONE atomic
`nft -f -` transaction (`egress_linux.go`) — any re-apply swaps old→new atomically, never exposing a
window with the allow missing. This is an S7.2 apply-design property we now KNOWINGLY rely on (a
byte-identical re-apply is invisible to traffic). If that apply ever becomes non-atomic (flush-then-add),
the "harmless re-apply" reasoning breaks — flagged so it's a known dependency, not an accident.

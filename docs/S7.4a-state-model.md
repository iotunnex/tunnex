# S7.4a — state-model (paper; budget-rule escalation, NO code until signed off)

Triggered by the final-round budget rule: the tight confirm found a NEW non-hygiene bug ([371]) in the
notices dimension. Rather than a 4th patch, this models the surface and proposes a REDUCTION.

## The four dimensions (loads × role × edition × notices)

- **loads** — every GET → `loadOne` → `Loaded<T>`; a failure renders retry, never absence. **SOLVED, pure.**
- **role × edition** — collapsed into the pure `accessView()` (loading / fatal / load_retry / upsell /
  role_retry / role_loading / member_gate / admin_body). Body renders only for `admin_body`. **SOLVED, pure.**
- **notices** — the ONLY un-reduced dimension, and the one still generating findings.

## The notices fault (root cause of [291]/[309]/[371])

The partial-swap warning is modelled as **TWO states for ONE fact**:
- `notice: string | null` (the amber copy)
- `staleRuleId: string | null` (which rule it refers to)

They must stay in lockstep but are written from different places, so they DESYNC — each desync is a finding:
- **[291]** notice rendered inside a load-branch → masked by a coincident load failure.
- **[309]** notice not cleared when its rule was resolved by a delete → stale warning persists.
- **[371]** notice cleared by an UNRELATED clean create → live-stale-rule warning vanishes (false reassurance).

The confirm's refuted item named it exactly: `notice` is fully DERIVABLE from `staleRuleId`
(`notice = staleRuleId ? swapPartialMessage(staleRuleId) : null`). One of the two states is redundant, and
redundant lockstep state is precisely where desync bugs breed.

## The REDUCTION — single source of truth: `staleRuleId`, notice DERIVED

Collapse the two states to one. `staleRuleId` is THE state; the notice is a pure projection.

```
state:   staleRuleId: string | null          // the rule the partial-swap left un-deleted, or null
derive:  const notice = staleRuleId ? swapPartialMessage(staleRuleId.slice(0,8)) : null   // pure, no state
set:     onDone(_, staleId) → if (staleId) setStaleRuleId(staleId)   // partial ONLY; clean create does NOTHING
clear:   useEffect([rules, staleRuleId]) → if staleNoticeCleared(staleRuleId, rules) setStaleRuleId(null)
render:  notice at top level (compose-not-compete, sectionRender) — unchanged
```

Why this ends the arc (each finding becomes STRUCTURALLY impossible, not handled-case-by-case):
- **[371]** a clean create never touches `staleRuleId` → the warning cannot be dropped while the rule lives.
- **[309]** clearing is ONLY the `staleNoticeCleared` useEffect (rule absent from list) → nothing else clears it.
- **[291]** notice still renders top-level (sectionRender) → a load failure replaces content, never the warning.
- No lockstep pair → no desync surface left. `swapPartialMessage(staleRuleId)` is already unit-tested;
  `staleNoticeCleared` already has its red. The only change is deleting the `notice` state + its writes.

Net diff: remove `notice` useState + every `setNotice`; add one derived `const`; `onDone` clean-path no
longer clears `staleRuleId`. Smaller than the current code, not larger.

## The problem

EPIC 8's site-to-site mesh is **hub-and-spoke**: one elected hub gateway carries transit between spokes
(S8.2 `siteLinkGraph`). The hub is a **single point of failure** — if the hub box dies, every spoke-to-spoke
path through it goes dark, and the CP's site-link health surfaces `site_hub_down` (S8.2 H5, honest) but
nothing FAILS OVER. S8.6 makes the hub role survivable.

Zero-touch + unlock-then-opt-in still bind: HA is founder-directed enterprise-adjacent, default OFF, and
the data-plane agent owns reconciliation (the CP never touches WireGuard).

## HA scope — the decide-items (RULED, with conditions folded)

### D1 — Hub-set election (CONFIRMED: CP-owned, ordered, generation-stamped, agents observe-never-vote)
`electSiteHub` (S8.3, CP, deterministic) is extended to elect an ORDERED `{primary, standby[]}` per site.
One-truth at its highest-stakes tier — a voting/gossip protocol among agents would be building Raft in a
weekend (how products die). Agents OBSERVE the pushed hub set and reconcile; they never vote.
- **Election inputs, stated EXHAUSTIVELY (condition):** a gateway is hub-CAPABLE iff it has a public
  endpoint present (a NAT'd spoke can't be a hub — it dials out) AND a fresh WG key. Among capable
  gateways, order by: (1) **admin priority override** (RULED: YES — an optional per-node `hub_priority`
  field; operators distrust magic, so the admin OUTRANKS the election, which only breaks ties); (2) node
  health (fresh handshake / not degraded); (3) a deterministic tiebreak (node-id) so the order is stable
  and re-computable. Absent an admin priority, it degrades to health-then-id (today's determinism).

### D2 CORRECTION (2026-07-20, Slice-3 build) — promotion = AllowedIPs reassignment, NOT route metric
The D2 condition below ("route-metric preference") verified the WRONG LAYER: it checked that Linux `ip route`
honors metric among duplicate prefixes (true) but never checked it against WireGuard's crypto-routing, where
**AllowedIPs owns peer selection within an interface and the OS metric never gets a vote**. With one `wg0`
there is ONE interface, so the metric would arbitrate a choice that doesn't exist (the REPORTED ≢ STORED
lesson's routing sibling — a ruling must cite the layer that actually DECIDES: the OS route picks the
interface, AllowedIPs picks the peer). RULED **(A)**:
- **Promotion = AllowedIPs REASSIGNMENT at compile**, not a metric flip. The D2 promise SURVIVES INTACT and
  wins on the merits: the standby tunnel is warm (handshaked + keepalive'd), so promotion still builds
  NOTHING — no handshake wait, no tunnel establishment; just a re-compile moving the remote subnets from the
  primary peer's AllowedIPs to the standby's, applied by the agent's existing uapi peer-update reconcile (no
  bounce). (B) per-hub interfaces REFUSED — buys literal metric fidelity with a multi-interface data plane
  the product doesn't have and the walk doesn't need (speculative-complexity class).
- **THE INVARIANT (replaces the metric reds):** a remote subnet lives in EXACTLY ONE peer's AllowedIPs per
  device, EVER — never both hubs (WG's undefined-which is a nondeterminism generator; the compiler makes
  the assignment single-valued BY CONSTRUCTION). Red: compile with primary+standby → subnets in the
  PRIMARY's AllowedIPs only, the standby's = its tunnel /32 exactly.
- **Standby peer shape:** endpoint present (so the handshake fires) · AllowedIPs = the standby hub's tunnel
  /32 ONLY · keepalive per S8.2b — warm + observable (the spoke's handshake with the standby lands in
  `node_peer_status`, Slice 1's substrate proving the warmth). Red: the standby peer's handshake age is
  visible in the table.
- **PROMOTION LATENCY (honest sentence):** the flip rides compile → push → reconcile (seconds), consistent
  with D3's CP-driven design — failover speed IS push-path speed, and CP-down = no failover is the named
  residue. The mechanism's dependency chain is stated (the metric version wouldn't have had it).
- **Reducer-identity-key concern WITHDRAWN** — no duplicate OS routes exist under (A); the red list carries
  no ghost of it.

### D2 (ORIGINAL condition — SUPERSEDED by the correction above; kept for the record)
The standby pre-establishes keepalive'd transit tunnels to every spoke (S8.2b keepalive), installs its
transit routes but at a WORSE preference, so promotion is a **preference flip, not a tunnel build** — the
design's core idea (failover becomes a preference problem, not a build problem).
- **Platform-capability VERIFIED (condition a):** the HA route-preference tier is GATEWAY-side (Linux,
  `apps/node`) — NOT the client helper. The darwin/windows S8.5 reducer is the CLIENT tier and is not on
  the HA path. Linux `ip route` already carries a metric on every site route (`wgctrl_linux.go:311`
  `siteRouteMetric = 8021`, `:356` `ip route replace ... metric`) and honors metric-based selection among
  DUPLICATE-prefix routes (lower wins). So the standby installs its per-spoke transit routes at a HIGHER
  metric than the primary's; both present, primary wins; on primary death its routes prune (stale tunnel)
  and the standby's survive. **No presence-flip fallback is needed — the one HA platform (Linux) expresses
  preference cleanly.** (Fallback stance, unused: were a tier unable to express metric, it would be
  presence-flip — standby routes absent until promotion, colder but honest.)
- **Cost STATED (condition b):** every spoke holds N warm keepalive'd tunnels (one per hub in the set).
  Trivial at N=2 (one primary + one standby); the keepalive traffic is the S8.2b cadence × N. Said, not
  hidden.

### D3 — Failover trigger: CP-driven off the existing staleness clock (CONFIRMED, residue named)
The CP's existing site-link staleness (`SiteLinkStale`/`site_hub_down`, S8.2 H5 — REUSED, no new health
vocabulary, the WF-1 discipline) crosses a threshold → CP re-elects the hub set (promote standby) → pushes
→ agents reconcile. The promotion is a desired-state change, not a new control path.
- **RESIDUE NAMED (condition, in the paper's own voice → docs):** the CP is the failover BRAIN, so
  **CP-down = no failover** — everyone keeps their last role (fail-static). A SIMULTANEOUS primary + CP
  failure has NO automatic answer in v1. This is the honest v1 trade, consistent with the product's whole
  CP-authoritative architecture; it is stated here and ships in the docs. An HA feature whose limits are
  unstated is the reassuring-green class at the feature tier.
- Per-hub freshness requires per-hub-PEER observation — which is exactly L1's storage question answering
  itself (the CP must see each hub's per-spoke handshake to know which hub is alive). See L1.

### D4 — Flap damping: asymmetric hysteresis (CONFIRMED, concrete numbers)
An oscillating primary (marginal link) must not ping-pong the role (each flip re-routes every spoke).
- **Concrete numbers (condition), derived from the S8.2 staleness clock cadence** (the reconcile/report
  loop; the exact tick is pinned in the build against `SiteLinkStale`'s clock): **promote the standby after
  N = 3 consecutive stale ticks** (~ the S8.2 site-link-down threshold), **fail back to the primary only
  after it is fresh for M = 5 ticks AND a hold window has elapsed** (fast-over, slow-back). The precise
  tick maps to the existing clock at build time, cited.
- **The flap RED is the design's proof:** an oscillating primary (stale/fresh/stale/fresh…) produces
  EXACTLY ONE failover, not a metronome.

Slice-2 build surfaced that "per-site ordered hub set + per-site generation" (my earlier D1/D5 wording) was
a DRIFT from the verified model, not a design intent: `electSiteHub` is ORG-WIDE (one transit hub for the
org's whole site mesh, `service.go:425`) — the wording imported "per-site" from D5's phrasing without
re-checking the substrate (the one-truth law applies to rulings too). RULED **(A) org-level**:
`electSiteHubSet` = the org's ORDERED hub-capable gateways; `members[0]` = the transit hub (today's
election, now ordered); the rest = failover candidates IN ORDER. **ONE generation per ORG** (D5 fencing
re-keyed: per-ORG monotonic, CP-persisted, survives restart). Schema keyed by **org_id** — named
`org_hub_set` so it can't be misread as a site attribute (this is the transit-hub set, not per-site).
- **CONSEQUENCE (stated):** with an org-level set the standby need NOT share the primary's site — an org can
  fail over from an AWS-site hub to an Azure-site hub. The walk's same-site standby is ONE case, not the
  definition; the capability gate (public endpoint + fresh key) is the ONLY membership criterion. Red: a
  capable gateway from a DIFFERENT site enters the set in correct order.
- **(B) NOT foreclosed (reserved evolution):** if a future customer needs independent per-site failover
  (multi-hub topologies, the S8.2 "reserved" note), a per-site election layers UNDER the org transit
  election — the org set's schema does not block it. Recorded, not built.

### D5 — Split-brain: the fence is CP-SIDE (CORRECTED 2026-07-20, Slice-4 build — item-1 survey)
**The original wording — "an agent applies ONLY a strictly HIGHER generation" — imported fencing
vocabulary from architectures where nodes receive state from MULTIPLE authorities and must arbitrate
(Raft peers, gossip meshes, dual-master DBs). This product has NONE of those.** The Slice-4 apply-path
survey found NO monotonic (refuse-older) check on the agent, and the trace shows one isn't needed:
- Agents have EXACTLY ONE artifact source — the CP. No peer-to-peer push; a demoted primary can push to
  no one. The Watch is monotonic by construction (long-poll on `since`, the server only advances). So a
  stale artifact CANNOT reach an agent.
- The split-brain D5 feared decomposes into (a) stale-artifact-reaches-agent — PRECLUDED (single source +
  monotonic Watch); (b) demoted-primary-keeps-serving — REAL but INERT: the spokes moved their AllowedIPs
  on the CP's word, so the stale primary receives NOTHING (the fail-static residue, already named).
- An agent-side generation check would fence a scenario the architecture precludes — dead code whose only
  future is convincing a maintainer the scenario is live (speculative-complexity class, refused).

**RULED: the fence is CP-SIDE.** It is the ATOMIC monotonic generation bump in `UpsertOrgHubSet` (Slice 2,
`generation = CASE WHEN members IS DISTINCT FROM ... THEN +1`) — which makes two concurrent hub-set writes
converge to ONE monotonic result — PLUS single-source artifact delivery. Agents need NO generation
awareness because they have no second authority to arbitrate against. The `generation` stays in
`org_hub_set` as the PROMOTION EVENT's identity (audit, debugging, the UI's "hub set v7"); its fencing
role IS the upsert's atomicity.
- **DEFERRED agent-side fence — trigger NAMED:** IF the architecture ever gains a SECOND artifact path
  (agent-to-agent relay, CP federation, offline artifact bundles), the agent-side monotonic check becomes
  MANDATORY in that story's commit-one. Recorded so the deferred fence can't be forgotten if the
  precondition changes.
- **CP-side fence RED (explicit, so the fence can't rot):** the two-writer convergence fixture (concurrent
  reconciles, different member views → one monotonic generation-ordered result), cited at the
  `CASE ... IS DISTINCT FROM` statement.
- **Demoted-primary-inert RED (structural, not narrated):** post-promotion compile → the demoted primary's
  pubkey appears in ZERO spokes' subnet-carrying AllowedIPs (it may persist as a keepalive-only peer — the
  warm fail-back line). The walk's tcpdump (spokes stop sending) confirms on iron what the red pins.
- **Spoke-side partition (Slice 5, red):** a spoke that can reach NEITHER the CP NOR the primary uses its
  last-known config against whichever hubs it CAN reach — a reachable standby carries traffic even while
  the CP is unreachable (fail-static). RED the reachable-standby case (Slice 5).

**COMPILER CONSUMES THE PERSISTED ACTIVE ORDER (Slice-4 integration):** the failover controller maintains
`org_hub_set.members` as the ACTIVE order (configured minus stale under hysteresis). The compiler
(`siteLinkGraphFrom`) reads `org_hub_set.members` (the ONE truth), NOT `electSiteHubSet(topo)` directly —
so a promotion (a change to the persisted members) flows through the ordinary compile+push with no
failover-special path. `electSiteHubSet` is the CONFIGURED intent that `ReconcileHubSet` (bind/unbind/pin)
and the tick DERIVE from; the persisted set is what agents see. (Empty `org_hub_set` → fall back to
`electSiteHubSet` so a not-yet-reconciled org still compiles single-hub.)

## Slice 4 — failover trigger (DESIGN LOCKED, builds FRESH; scope-call ratified 2026-07-20)
Builds fresh: the story's single most defect-prone shape — a stateful control loop with hysteresis,
testable only by scripted clock, on the highest-stakes path (the S8.4-arc profile). Design locked here;
next sitting is pure execution.

**Locked design:**
- **The tick rides the existing CP-loop precedent** (`main.go:216` runs a `time.NewTicker` goroutine for
  the accesslog sweep). The failover tick is the same shape: a ticker goroutine calling ONE
  `nodes.Service` method per interval.
- **Reads, never measures:** member freshness comes from `node_peer_status.last_handshake_at` (Slice 1) /
  `SiteLinkStale` — no new clock, no new health vocabulary. The tick asks ONE question: is `members[0]`
  stale by the D4 numbers? N=3 stale → re-elect (promote the next capable member), bump generation, push.
- **Asymmetric hysteresis:** promote after N=3 stale ticks; fail back only after the demoted primary reads
  M=5 fresh + the hold window. FAIL-BACK IS A PROMOTION EVENT — a membership-order change, generation bump,
  push; one mechanism, two directions, the asymmetry living entirely in the thresholds. Pin-restoration
  (the pinned primary resumes `members[0]` when healthy) IS the fail-back.
- **Promotion is COMPILE-ONLY:** re-election changes `members[0]` → the Slice-3 compiler moves the subnets'
  AllowedIPs → push → agents reconcile. No new push path, no failover-special messages. The demoted primary
  (if reachable) keeps its hub posture (hub-symmetry); only spoke-side AllowedIPs move.
- **In-memory tick state** (consecutive-stale counts, hold timers), rebuilt from stored freshness on a CP
  restart — conservative (a restart mid-window restarts the count, delaying failover by ≤N ticks, NEVER a
  spurious one). Dormant-machinery's cousin: no persistence for state the substrate re-derives.
- **Every transition audited:** promotion + fail-back land as audit rows naming old-primary → new-primary +
  the triggering condition (the operator's "why did my hub change at 3am" trail).

## S8.6b — wg0→wg0 hub-transit DOCKER-USER accept (WF-4's third orientation) — commit-one

### D-transit-1 (LEAD, RULED B) — relax the route rules' orientation predicates; NOT a second marked set

The two orientations (`eth0→wg0` route, `wg0→wg0` transit) were never two policies. They are ONE policy:
*"Docker must not structurally drop traffic whose daddr/saddr is a Route — the ZT chain adjudicates it."*
The separate-rule model (a `iifname wg0 oifname wg0 daddr∈Routes` accept) **COLLIDES**: it re-derives to the
same `d:route`/`s:route` key as the existing route rules (the idempotent key + `dockerUserRuleRE` capture
daddr/saddr only, NOT iif/oif) → the map holds one entry per key, the sweep can't tell the orientations
apart → broken idempotence + orphans. Building a second marked-set engine (distinct comment) to preserve a
distinction that carries **no security content** is the wrong shape: the enforcement boundary was never the
DOCKER-USER match — it is the `ip tunnex` chain (D-transit-3). So narrowing Docker's match by iif bought
nothing and cost a collision.

**RULED (founder): (B) — relax the existing route rules.** Drop the ingress/egress predicates so the one
rule says what the policy means:
```
forward: oifname wg0  ip daddr <route>     # was iifname != wg0 → now covers eth0→wg0 (route) AND wg0→wg0 (transit)
return:  iifname wg0   ip saddr <route>     # was oifname != wg0 → covers wg0→eth0 AND the wg0→wg0 reply
```

Keys stay daddr/saddr-only (the policy is daddr/saddr-only); the map, sweep, regex remain ONE engine. The
WF-4-local rules (daddr=LocalSubnet, disjoint from Routes) are UNCHANGED. Same lesson as metric-vs-AllowedIPs:
cite the layer that actually decides — Docker's chain decides structural drop, the tunnex chain decides
permission, the accept's job is only the former.

**WHY THE PREDICATES WERE THERE (history, for the future reader — narrowing-was-incidental):** the WF-4 fixes
were minimal-scope responses to specific walk findings — S8.2c found eth0→wg0 (route), S8.5 found wg0→eth0
(local), each honest at its moment and each scoped its accept to exactly the direction its fixture exercised.
The `iifname != wg0` on the route forward was never a security predicate — it was "the direction this walk
proved," frozen into the rule. The third orientation (wg0→wg0) revealed the union was the true shape all
along. **A future "security tightening" PR MUST NOT re-add the iif/oif predicates** — doing so silently
re-breaks transit while the ZT chain (the real boundary) is untouched. This paragraph is the guard rail.

## Post-walk fold — failover-does-not-fire (Deck B, merge-gating S8.6)

### D1 (RULED) — spoke-observed liveness is the ONE truth for hub-member death
Liveness is testimony from the living: a member's OWN report proves it can reach the
CP, never that its data plane serves. A dead hub's peer handshakes freeze (no living
peer can complete a new handshake with a corpse) — the spokes' observations are the
only signal that MOVES when the subject dies (the S7.4b class). Already the model:
`freshness[id]` reads `latestByPubKey` over node_peer_status, keyed by the member's
pubkey — cross-observations by OTHER nodes. Quorum: with one spoke, that spoke IS the
verdict; with N, any-fresh = alive (over-report on death, damped by the N=3
hysteresis). No-spokes edge: hubs with zero living observers = NO verdict, no
demotion possible (honest; joins the CP-down residue family).

### ROOT (RULED) — the window, not a writer bug
`failoverStaleWindow` was 90s. WireGuard renews handshakes on the REKEY_AFTER_TIME
(120s)/REJECT_AFTER_TIME (180s) cycle, so a LIVING link's observed handshake age
sawtooths up to ~180s; the box-walk saw healthy steady-state ages 1:29–3:22 (to
202s). A 90s window marked living hubs dead every cycle (the #4 false-stale badge)
and the rekey kept resetting the consecutive-stale counter, so a real death fired
erratically. Fix: window → **240s**, DERIVED (REJECT_AFTER_TIME 180s + report/tick
slack, above the observed 202s ceiling), derivation cited in-code. Relationship to
apps/node `siteLinkStaleWindow` (180s): NOT one shared constant — deliberately
distinct across the api↔node module boundary, same protocol cadence, tuned in
OPPOSITE error directions (the agent's 180s is over-report-safe for a health BADGE;
the CP's 240s is under-report-safe for a DEMOTION, the costlier error) — paired
cross-reference comments so neither drifts silently.

### D2 (REFUTED by iron) — no writer clobber
node_peer_status PK is (reporter, peer): each observer's row is distinct by
construction; the EXISTS guard blocks self-rows; latestByPubKey takes MAX across
observers. Iron query showed distinct coexisting observer rows. No clobber.

### NULL / no-spokes (RULED) — three-state: no observer, no verdict

## The problem

EPIC 8's site-to-site mesh is **hub-and-spoke**: one elected hub gateway carries transit between spokes
(S8.2 `siteLinkGraph`). The hub is a **single point of failure** — if the hub box dies, every spoke-to-spoke
path through it goes dark, and the CP's site-link health surfaces `site_hub_down` (S8.2 H5, honest) but
nothing FAILS OVER. S8.6 makes the hub role survivable.

Zero-touch + unlock-then-opt-in still bind: HA is founder-directed enterprise-adjacent, default OFF, and
the data-plane agent owns reconciliation (the CP never touches WireGuard).

## HA scope — the decide-items (RULED, with conditions folded)

### D1 — Hub-set election (CONFIRMED: CP-owned, ordered, generation-stamped, agents observe-never-vote)
`electSiteHub` (S8.3, CP, deterministic) is extended to elect an ORDERED `{primary, standby[]}` per site.
One-truth at its highest-stakes tier — a voting/gossip protocol among agents would be building Raft in a
weekend (how products die). Agents OBSERVE the pushed hub set and reconcile; they never vote.
- **Election inputs, stated EXHAUSTIVELY (condition):** a gateway is hub-CAPABLE iff it has a public
  endpoint present (a NAT'd spoke can't be a hub — it dials out) AND a fresh WG key. Among capable
  gateways, order by: (1) **admin priority override** (RULED: YES — an optional per-node `hub_priority`
  field; operators distrust magic, so the admin OUTRANKS the election, which only breaks ties); (2) node
  health (fresh handshake / not degraded); (3) a deterministic tiebreak (node-id) so the order is stable
  and re-computable. Absent an admin priority, it degrades to health-then-id (today's determinism).

### D2 CORRECTION (2026-07-20, Slice-3 build) — promotion = AllowedIPs reassignment, NOT route metric
The D2 condition below ("route-metric preference") verified the WRONG LAYER: it checked that Linux `ip route`
honors metric among duplicate prefixes (true) but never checked it against WireGuard's crypto-routing, where
**AllowedIPs owns peer selection within an interface and the OS metric never gets a vote**. With one `wg0`
there is ONE interface, so the metric would arbitrate a choice that doesn't exist (the REPORTED ≢ STORED
lesson's routing sibling — a ruling must cite the layer that actually DECIDES: the OS route picks the
interface, AllowedIPs picks the peer). RULED **(A)**:
- **Promotion = AllowedIPs REASSIGNMENT at compile**, not a metric flip. The D2 promise SURVIVES INTACT and
  wins on the merits: the standby tunnel is warm (handshaked + keepalive'd), so promotion still builds
  NOTHING — no handshake wait, no tunnel establishment; just a re-compile moving the remote subnets from the
  primary peer's AllowedIPs to the standby's, applied by the agent's existing uapi peer-update reconcile (no
  bounce). (B) per-hub interfaces REFUSED — buys literal metric fidelity with a multi-interface data plane
  the product doesn't have and the walk doesn't need (speculative-complexity class).
- **THE INVARIANT (replaces the metric reds):** a remote subnet lives in EXACTLY ONE peer's AllowedIPs per
  device, EVER — never both hubs (WG's undefined-which is a nondeterminism generator; the compiler makes
  the assignment single-valued BY CONSTRUCTION). Red: compile with primary+standby → subnets in the
  PRIMARY's AllowedIPs only, the standby's = its tunnel /32 exactly.
- **Standby peer shape:** endpoint present (so the handshake fires) · AllowedIPs = the standby hub's tunnel
  /32 ONLY · keepalive per S8.2b — warm + observable (the spoke's handshake with the standby lands in
  `node_peer_status`, Slice 1's substrate proving the warmth). Red: the standby peer's handshake age is
  visible in the table.
- **PROMOTION LATENCY (honest sentence):** the flip rides compile → push → reconcile (seconds), consistent
  with D3's CP-driven design — failover speed IS push-path speed, and CP-down = no failover is the named
  residue. The mechanism's dependency chain is stated (the metric version wouldn't have had it).
- **Reducer-identity-key concern WITHDRAWN** — no duplicate OS routes exist under (A); the red list carries
  no ghost of it.

### D2 (ORIGINAL condition — SUPERSEDED by the correction above; kept for the record)
The standby pre-establishes keepalive'd transit tunnels to every spoke (S8.2b keepalive), installs its
transit routes but at a WORSE preference, so promotion is a **preference flip, not a tunnel build** — the
design's core idea (failover becomes a preference problem, not a build problem).
- **Platform-capability VERIFIED (condition a):** the HA route-preference tier is GATEWAY-side (Linux,
  `apps/node`) — NOT the client helper. The darwin/windows S8.5 reducer is the CLIENT tier and is not on
  the HA path. Linux `ip route` already carries a metric on every site route (`wgctrl_linux.go:311`
  `siteRouteMetric = 8021`, `:356` `ip route replace ... metric`) and honors metric-based selection among
  DUPLICATE-prefix routes (lower wins). So the standby installs its per-spoke transit routes at a HIGHER
  metric than the primary's; both present, primary wins; on primary death its routes prune (stale tunnel)
  and the standby's survive. **No presence-flip fallback is needed — the one HA platform (Linux) expresses
  preference cleanly.** (Fallback stance, unused: were a tier unable to express metric, it would be
  presence-flip — standby routes absent until promotion, colder but honest.)
- **Cost STATED (condition b):** every spoke holds N warm keepalive'd tunnels (one per hub in the set).
  Trivial at N=2 (one primary + one standby); the keepalive traffic is the S8.2b cadence × N. Said, not
  hidden.

### D3 — Failover trigger: CP-driven off the existing staleness clock (CONFIRMED, residue named)
The CP's existing site-link staleness (`SiteLinkStale`/`site_hub_down`, S8.2 H5 — REUSED, no new health
vocabulary, the WF-1 discipline) crosses a threshold → CP re-elects the hub set (promote standby) → pushes
→ agents reconcile. The promotion is a desired-state change, not a new control path.
- **RESIDUE NAMED (condition, in the paper's own voice → docs):** the CP is the failover BRAIN, so
  **CP-down = no failover** — everyone keeps their last role (fail-static). A SIMULTANEOUS primary + CP
  failure has NO automatic answer in v1. This is the honest v1 trade, consistent with the product's whole
  CP-authoritative architecture; it is stated here and ships in the docs. An HA feature whose limits are
  unstated is the reassuring-green class at the feature tier.
- Per-hub freshness requires per-hub-PEER observation — which is exactly L1's storage question answering
  itself (the CP must see each hub's per-spoke handshake to know which hub is alive). See L1.

### D4 — Flap damping: asymmetric hysteresis (CONFIRMED, concrete numbers)
An oscillating primary (marginal link) must not ping-pong the role (each flip re-routes every spoke).
- **Concrete numbers (condition), derived from the S8.2 staleness clock cadence** (the reconcile/report
  loop; the exact tick is pinned in the build against `SiteLinkStale`'s clock): **promote the standby after
  N = 3 consecutive stale ticks** (~ the S8.2 site-link-down threshold), **fail back to the primary only
  after it is fresh for M = 5 ticks AND a hold window has elapsed** (fast-over, slow-back). The precise
  tick maps to the existing clock at build time, cited.
- **The flap RED is the design's proof:** an oscillating primary (stale/fresh/stale/fresh…) produces
  EXACTLY ONE failover, not a metronome.

### D1/D5 KEYING CORRECTION (2026-07-19) — org-level, not per-site
Slice-2 build surfaced that "per-site ordered hub set + per-site generation" (my earlier D1/D5 wording) was
a DRIFT from the verified model, not a design intent: `electSiteHub` is ORG-WIDE (one transit hub for the
org's whole site mesh, `service.go:425`) — the wording imported "per-site" from D5's phrasing without
re-checking the substrate (the one-truth law applies to rulings too). RULED **(A) org-level**:
`electSiteHubSet` = the org's ORDERED hub-capable gateways; `members[0]` = the transit hub (today's
election, now ordered); the rest = failover candidates IN ORDER. **ONE generation per ORG** (D5 fencing
re-keyed: per-ORG monotonic, CP-persisted, survives restart). Schema keyed by **org_id** — named
`org_hub_set` so it can't be misread as a site attribute (this is the transit-hub set, not per-site).
- **CONSEQUENCE (stated):** with an org-level set the standby need NOT share the primary's site — an org can
  fail over from an AWS-site hub to an Azure-site hub. The walk's same-site standby is ONE case, not the
  definition; the capability gate (public endpoint + fresh key) is the ONLY membership criterion. Red: a
  capable gateway from a DIFFERENT site enters the set in correct order.
- **(B) NOT foreclosed (reserved evolution):** if a future customer needs independent per-site failover
  (multi-hub topologies, the S8.2 "reserved" note), a per-site election layers UNDER the org transit
  election — the org set's schema does not block it. Recorded, not built.

### D5 — Split-brain: generation as the fencing token (CONFIRMED, per-ORG monotonic CP-persisted — see keying correction above)
CP-single-election makes two-primaries structurally hard (a standby NEVER self-promotes). The hub set
carries a **generation**; an agent applies ONLY a strictly HIGHER generation, so a stale
primary-still-thinks-it's-hub is overridden the moment it re-reaches the CP.
- **Conditions:** the generation is **per-site, monotonic, CP-PERSISTED** (survives CP restart — a reset
  generation would un-fence every agent, re-opening split-brain on the next election). Stored in the DB,
  bumped on every hub-set change.
- **Spoke-side partition (condition, red):** a spoke that can reach NEITHER the CP NOR the primary uses its
  last-known preference order against whichever hubs it CAN reach — so a reachable standby carries its
  traffic even while the CP is unreachable (the fail-static spoke stance). RED the reachable-standby case.

### D6 — The proof: three-VM primary-kill walk (CONFIRMED, infra ask staged NOW)
HA cannot be proven on the 2-gateway topology. The walk needs a **THIRD gateway VM** (a second hub-capable
box, either cloud) so a real primary-kill → standby-promotion → spoke-continuity is observed on the wire:
kill the primary hub, watch a spoke's transit fail over to the standby with bounded interruption, then the
primary rejoins as standby (no split-brain — generation fencing holds). **INFRA ASK staged now (below) so
it's provisioned before batch day**, not discovered on it.

### L1 — node-peer status storage (RULED: YES, rides S8.6; table designed here)
D3's per-hub freshness IS node-peer handshake state, so HA earns the storage the S8.5 L1 metrics also
needed — designed ONCE, serving the failover clock AND the topology cards. REPORTED ≢ STORED gets its fix
where the law was minted (the `UpsertDeviceStatus` gateway-peer no-op finally gets its sibling that STORES).

**The table (designed now):**
- **`node_peer_status`** — keyed **`(node_id, public_key)`**: the reporting gateway node + the peer's WG
  pubkey (a remote gateway for a site-link, distinct from `device_status` which is device-keyed and no-ops
  on a non-device pubkey — the exact S8.5 gap). Columns: `last_handshake_at`, `rx_bytes`, `tx_bytes`,
  `updated_at`.
- **Write path:** the SAME agent report the CP already receives (`ReportStatus` / `UpsertDeviceStatus`,
  `nodes/service.go`). Today it maps a pubkey → the active DEVICE and no-ops an unknown (gateway) pubkey.
  S8.6 adds a SIBLING upsert: every reported peer that resolves to a NODE (a site-link peer) lands in
  `node_peer_status`. One report, two sinks (device_status for device peers, node_peer_status for gateway
  peers) — no new agent field, no new report.
- **Retention stance:** last-write-wins gauge (like `device_status`) — NOT a time series (that's S11.1's
  job). One row per `(node_id, public_key)`, upserted each report; cascade-deleted with the node. No
  history, no growth beyond the peer count.
- **Reads:** (1) D3's failover clock — per-hub per-spoke `last_handshake_at` freshness; (2) the L1 topology
  metrics — bytes + handshake-age on the site-link cards (render-floor: cumulative totals labeled "since
  last handshake", refresh-on-load, no rates/sparklines, absent-report → absent-number never zero-as-data,
  per the S8.5 L1 ruling).

## Slice 3 BUILT (2026-07-20) — warm standby + AllowedIPs preference, two-tier membership (3)
- **Membership RULED (3), two-tier** (`electSiteHubSet`): ANY pins → the set = the PINNED capable gateways (HA opt-in — the pin declares "carry the org's transit"; capability still gates, a pinned-NAT'd/keyless is excluded). This dissolved the `capable ⇒ hub-posture` collision — an endpoint-bearing LEAF (the walk's azure-gw) is no longer drafted into hub duty.
- **Compiler (`siteLinkGraphFrom`):** a MEMBER compiles HUB posture (peers with every non-self, subnet-advertising gateway NOT in its own site — **same-site exclusion** kills the spurious same-L2 primary↔standby link; primary AND standby both carry it → hub-symmetry, promotion changes nothing hub-side). A non-member LEAF compiles SPOKE posture: the remote subnets in the PRIMARY (members[0]) peer ONLY (the **single-AllowedIPs invariant**), each standby a **keepalive-only empty-AllowedIPs** peer.
- **Standby `/32` → EMPTY AllowedIPs** (all gateways share the pool-gateway `10.99.0.1`, so there is no distinct per-hub tunnel address): empty makes "no traffic ever rides the standby" STRUCTURAL (WG cannot crypto-route to a peer with no AllowedIPs) while keepalive keeps it warm + observable in `node_peer_status`.
- **The asymmetric pre-promotion link (paper line, for the walk's tcpdump):** a leaf's standby peer is empty-AllowedIPs; the standby's leaf peer carries the leaf's subnets (full transit posture). WG AllowedIPs is per-END local policy — the leaf DROPS anything the standby sends beyond keepalives until promotion flips its side. That drop is the warm-standby's SAFETY property, not a bug.
- **Version/hash — NO bump (argued both faces):** *for-bump* — standby peers are new topology an old agent can't fail over with; *for-no-bump (wins)* — a standby is just another peer; an old agent holds it as an idle empty-AllowedIPs keepalive peer and serves single-hub transit EXACTLY as today (degraded-not-broken). Structural: standby peers add NO routes (routes = remote subnets, peer-count-independent), so the CanonicalHash (routes+policy) is unchanged → no RequiredVersion bump.

## Re-entry survey (cited, not assumed) — what remains genuinely S8.5's

The registration warned S8.5 SHRINKS by what S8.2c absorbed + what the S8.4 coupling reserved. The survey confirms a LARGE shrink — most of the data-plane already exists:

- **(a) enterprise — do granted resource-CIDRs reach device `AllowedIPs` today?** NO. `allowedIPsFor(fullTunnel, poolCIDR)` (`devices/service.go:265`) returns `[poolCIDR]` (split) or `[0.0.0.0/0, ::/0]` (full). No resource/site CIDR path.
- **(b) open — device-reachable gateway-LAN routes today?** MISSING. Devices get only the pool.
- **Device peering:** a device peers with ONE node (`NodeID`, `devices/service.go:77` "the gateway the peer connects through"). So device→remote-LAN rides device → its-gateway → (S8.2 site link) → far-gateway → LAN.
- **Gateway data-plane already EXISTS (the shrink):**
  - **NAT:** the S3.7 masquerade `ip saddr <pool> oifname != "wg0" masquerade` (`egress_linux.go:289`) already SNATs the device pool out ANY non-tunnel iface — including the LAN iface. **Device→LAN return-path NAT is already there.**
  - **Forward:** mesh `iifname "wg0" oifname != "wg0" accept` (`egress_linux.go:360`) already forwards wg→LAN; enforcing gates it via grants (`allowMatch`).
  - **Site→site transit** (S8.2 Routes + WF-4 DOCKER-USER accept) already carries a device's remote-LAN packet across the hub when the LAN is on another gateway.
- **Disjointness validator already has THREE classes** (`subnetguard.go:18-20`: `site_subnet` / `pool` / `reserved`) — the registration's "third input class" is ALREADY SATISFIED if routed ranges are modeled as site subnets.
- **The S8.4 abstraction** (`docs/S8.4-decisions.md` coupling clause): a resolvable internal zone is *"a reachable range with an optional resolver,"* keyed to the range, never a device/node. **A declared LAN is that same range** — it plugs into the SAME `dns_forwarding` table + the SAME device-resolver handoff, no redo.

**CONCLUSION — what is genuinely S8.5's (the un-absorbed core):**
1. **Push declared LAN ranges into split-tunnel device `AllowedIPs`** (the one missing data-plane piece — devices currently never route a LAN).
2. **The device-side route handoff** = the S8.4b/crash-sweep trigger + the client resolver path going live.
3. **The model + edition-gating** (open = push to all; enterprise = grant-gated destination — one mechanism).
4. **Version/hash + device-config-blast-radius classification** (argued, not assumed).
5. The carried riders (L1 metrics, WF-1, S8.4b NRPT, crash-sweep with its ordering precondition, stale-add-rule-button).

## Decide-items (proposed dispositions, HELD for ruling)

### D1 — The core model: a routed range REUSES the site approved-subnet (the coupling's "sibling"), consumed by DEVICES
**Proposed:** a "routed subnet" is NOT a new table — it is a **site's APPROVED subnet** (S8.1), now compiled into **device `AllowedIPs`** (not only gateway Routes). One model, one disjointness class, one DNS abstraction. Edition-gates the DEPTH: **open/off** → every device gets every approved subnet in `AllowedIPs` (the Pritunl push-route, mesh-reachable); **enterprise/enforcing** → the range is a policy DESTINATION, device→range gated by a grant (the S8.2 "routed ≠ permitted" red already holds). **One mechanism, edition-gated depth** — exactly the registration's phrasing.
- **Why reuse, not a parallel `routed_subnets` table:** the site subnet is ALREADY the "reachable range fronted by a gateway," already in `subnetguard` (ClassSiteSubnet), already the DNS-forward range. A parallel table would add a 4th disjointness class + a second model of the same concept + divorce it from the S8.4 DNS coupling — the dormant-machinery/one-truth smell.
- **Alternative (noted, proposed-rejected):** a lightweight per-gateway `routed_subnets` table for a Pritunl-simple "just add a CIDR" flow without the register-site→bind→advertise→approve ceremony. REJECTED because it forks the model; the ceremony is the S8.1 all-editions-core price, and a "declare a LAN" is genuinely "advertise + approve a site subnet."
- **Open-edition site ceremony concern:** the site model is all-editions core (D11), so open orgs already have sites/gateways/subnets. "Declare a LAN to push" = advertise + approve a site subnet. If the ceremony proves too heavy for open-edition parity, a **thin UI affordance** ("Advertise a routed LAN") over the same tables is the answer — NOT a second model. HELD.
- **S8.4 coupling honored:** the range carries its optional resolver (S8.4 `dns_forwarding`); when a device gains a route to a range that declares a resolver, the client resolver handoff fires (D4 below) — "site today, S8.5 LAN tomorrow, same code."

### D2 — Disjointness: NO new class (routed ranges ARE site subnets)
**Proposed:** because D1 reuses site subnets, a routed range is already validated by `subnetguard.Check` as `ClassSiteSubnet` against pool + reserved + other site subnets. The registration's "third input class" is **already satisfied** (the validator has had 3 classes since S8.1). **No new validator input, no new typed refusal** — the existing `subnet_not_disjoint` covers it. If D1's alternative (separate table) were chosen instead, THEN a 4th class would be owed; reuse avoids it.

### D3 — NAT / return-path: SNAT (pool→gateway-LAN-addr), stated explicitly; existing S3.7 rule covers it
**Proposed:** device→routed-LAN traffic is **source-NAT'd** at the fronting gateway (device-pool source → the gateway's LAN address) so a LAN host replies to the gateway and needs **ZERO route back to the device pool** — the open-edition/Pritunl expectation (the LAN admin adds nothing). This is ALREADY the S3.7 masquerade `ip saddr <pool> oifname != "wg0" masquerade` (`egress_linux.go:289`), which SNATs the pool out any non-tunnel iface. **State it explicitly in the paper as the chosen return-path model.**
- **Alternative (noted, proposed-rejected for open):** documented static route on the LAN (no NAT, preserves the real device source IP). REJECTED as the DEFAULT — it demands the customer's LAN/router add a route to the pool (breaks the zero-touch/Pritunl expectation). RESERVED as a future per-gateway "no-NAT + documented route" toggle IF a customer needs source-IP preservation (trigger = first real need); silent-ambiguity forbidden.
- **Cross-gateway case:** when the LAN is on a DIFFERENT gateway than the device's peer, the packet rides the S8.2 site link (wg0→wg0, not masqueraded — spoke↔spoke stays un-NAT'd), and the FRONTING gateway SNATs to its LAN. Stated so the two-hop NAT is unambiguous.

### D4 — The device→site route handoff (the S8.4b / crash-sweep TRIGGER — the ordering precondition)
**Proposed:** split-tunnel device `AllowedIPs` gains the org's declared routed ranges → the device routes them via its gateway. **This is the named trigger** for two deferred S8.4 items:
- **The client resolver path goes LIVE** — `config.dns_forwards` (empty in S8.4) is now populated for a device that has a route to a range declaring a resolver (the S8.4 D3 handoff). The macOS `set_resolvers` install path activates; S8.4b Windows NRPT lands here (its own slice, own review, the kill-switch/revocation-untouched probe inherited).
- **THE ORDERING PRECONDITION (binding, from the S8.4 round-3 reduce-by-removal):** the **crash/owner-loss resolver sweep MUST land BEFORE the first `set_resolvers` install path activates** in this story. A resolver file must never be able to exist without its teardown mechanism present. **Slice ordering: the crash-sweep slice precedes the resolver-handoff slice.** The crash-sweep is built exercisable + red-able + walk-provable (a crash-mid-session leg), NOT dormant lifecycle code (the dormant-machinery law). See `docs/laws.md` shared-territory #4.
- **Which resolver does a device use for a range?** the range's declared resolver (S8.4), reachable via the device's route to that range. Emitted only when the device HAS the route (split-tunnel includes it) — "a device with a route to a range that declares a resolver."

### D5 — Version / hash classification (argued per the D2-checklist row, NOT assumed)
**Proposed:** two distinct artifacts, classified separately:
- **The GATEWAY `policyspec.Compiled` artifact:** a routed range = a site subnet → it already appears as a `Route` (out-of-hash plumbing, S8.2) and, under enforcing, as a grant (`AllowEntry`, in-hash). **S8.5 adds NO new gateway-artifact field** → **NO new `ProtocolVersion` bump for the gateway artifact.** The enforcement significance (device→range permitted) is already carried by the existing grant path; the routing is already the out-of-hash `Routes`.
- **The DEVICE config (WG peer `AllowedIPs`):** this is device-config (control-plane→client), NOT the gateway `Compiled` artifact — it does not touch `CanonicalHash`. BUT the **device-config blast radius is FLEET-WIDE**: declaring/removing a routed range changes EVERY device's `AllowedIPs` → every device must re-fetch + re-apply its config. Stated: a routed-range change triggers a **device-config push to all devices in the org** (the S3.7 push path), the same way a pool resize does. **Enforcement-adjacent but hash-neutral** — it changes what a device ROUTES, gated (enforcing) by the unchanged grant path.
- **Argue-don't-assume result:** classified **hash-neutral / no version bump**, because the enforcement-significant fact (permission) rides the existing grant, and the routing rides the existing out-of-hash `Routes` + the device `AllowedIPs` (never hashed). The NEW behavior is a device-config recompute+push, not a new wire field. **VERIFY at build:** a twin-golden that a routed-range declaration leaves the gateway `CanonicalHash` unchanged (no false-desync on an enforcing gateway), mirroring the S8.4 `DNSForwards` hash-blind test.

### D6 — L1 metrics rider (WF-1 + the S8.3-close metrics L1): site-link bytes + handshake age on the topology cards
**Proposed:** the S8.3-close "Metrics L1" + the S8.2c WF-1 "positive site-link health" land here as a rider. The agent already parses `wg show dump` (S3.6) — extend the report path with per-site-link tx/rx byte counters + last-handshake age; the Sites topology card renders them. **RENDER-FLOOR bound (the law):** cumulative-since-handshake TOTALS labeled EXACTLY as that — NO rate graphs, NO sampling implied (that is S11.1). **WF-1 positive health:** a site link with a fresh handshake + moving bytes reads POSITIVELY (not merely "not-degraded") — the honest inverse of `site_link_down`. HELD.

## Walk shape (defined BEFORE code)

Cross-cloud, split-tunnel device + declared LANs + ZT layering. Inherits the behind-hub leg (hub-forwarding verify) + the Windows NRPT leg (S8.4b).
1. **Pritunl-parity (the money leg):** a **split-tunnel** device gets a declared LAN range in its `AllowedIPs`, reaches a host on that LAN **via the gateway**, and **everything else routes DIRECT** (not through the tunnel) — the split-tunnel proof.
2. **Routed ≠ permitted (enforcing red):** same device, ZT enforcing, 0 grants → the range is in `AllowedIPs` (routed) but the gateway forward chain DROPS it; add a grant → it flows. Routing ≠ permission, on live iron.
3. **Cross-gateway routed range:** the declared LAN is on a DIFFERENT gateway than the device's peer → the packet rides the S8.2 site link + the fronting gateway's SNAT. Two-hop, un-NAT'd inner + SNAT at the front.
4. **Device→site DNS goes live (the D4 handoff):** the declared range carries a resolver → the split-tunnel device resolves that range's names (S8.4 client path, now non-empty) — macOS `/etc/resolver` + **the S8.4b Windows NRPT leg**.
5. **THE CRASH-SWEEP ORDERING LEG (the precondition proof):** device up with resolvers installed → **crash/force-quit the client** → the owned resolver files are swept (the crash-sweep that landed BEFORE the resolver path, per D4) — no stale `/etc/resolver` residue pointing at a dead tunnel. Both macOS + Windows.
6. **L1 metrics render:** the Sites topology card shows site-link tx/rx totals + handshake age (WF-1 positive health) — render-floor (totals, not rates).
7. **Blast-radius:** declare a new routed range → EVERY device re-fetches + gains the route within the push interval (fleet-wide device-config push).

## Slice 2 — the AllowedIPs-push FORK, RULED: Option A (volatile-routes channel + live apply)

**The fork (surfaced before code, condition-#3 as designed):** the client bakes `AllowedIPs` at device mint and NEVER re-fetches (`deviceconfig.ts` — the D2 "client owns creation, never re-fetch" invariant; same-mode+device-exists → `return existing.config`). `PushOrgNodes` pushes to gateway NODES, not client DEVICES. So there is NO dynamic AllowedIPs sync to clients — the D5 "fleet push" mechanism does not exist client-side.

**RULED: Option A** — the client config model is two-tier (never named before): **stable identity** (keys/endpoint/pool — baked at mint, never re-fetched, the D2 security invariant) + **volatile operational state** (revocation status — polled via the S6.4 RevocationMonitor). **Routed ranges are the second kind** (admin-declared, change over a device's lifetime, zero identity material) → they ride the exact precedent (a lightweight authenticated poll) the client already trusts for a more security-critical signal. Option A files routes where they belong; it adds NO new architecture. **B refused** (a route change → tunnel bounce per device + re-opens never-re-fetch for a whole-config pull when one volatile field changed — max blast radius, min cause). **C refused** (stranding the installed fleet fails the story's own reason to exist — Pritunl-parity for the installed fleet is the migration promise).

**PAPER SENTENCE (the invariant survives, stated):** identity is still NEVER re-fetched; routes were never identity. The channel carries RANGES ONLY — no keys, no endpoint, no pool, no policy. One field, one direction.

# S7.3 device posture вАФ TARGETED box-proof runbook (for Pawan to run + sign off)

**Why targeted, not the S7.2 8-proof scale:** S7.3 adds NO new gateway/enforcement path. "Block
untrusted" is exclusion-via-EXISTING-filters вАФ a pending device is dropped from `ListActivePeersForNode`
(no peer/tunnel) AND from the compiler input (no grants) by the same `status='active'` filter the box
already proved in S7.2/S3.x. So the box proof only has to confirm the STATE MACHINE on the wire + the
one genuinely-new security surface (the F1-part-3 reject-leg IP-reuse). Four legs + one wire-security
proof.

## Setup (same shape as the S7.2 box run)
- Enterprise stack up (`edition=enterprise`) вАФ device approval is enterprise-gated.
- One org; one gateway node enrolled (gateway = `10.99.0.1`).
- Two spokes A, B as WireGuard peers via `deploy/testing/s72-spokes.sh` (reuse it) вАФ a client device
  each. (Same-wg-key caveat: don't run the same key on a real client and a netns spoke simultaneously.)
- CSRF = header presence: every mutating curl carries `-H 'X-Tunnex-CSRF: 1'`; auth = the
  `tunnex_session` cookie (browser) or the bearer.
- `NFT()` helper = run nft/tcpdump inside the node netns:
  `sudo nsenter -t $(docker inspect -f '{{.State.Pid}}' $(docker compose ps -q node-agent)) -n <cmd>`.

## Leg 1 вАФ device_approval=ON: a new device enrolls PENDING and CANNOT connect (on the wire)
1. Turn approval on:
   `PUT /api/v1/organizations/$ORG/device-approval  {"mode":"on"}` вЖТ 200, note `grandfathered_count`.
2. Enroll a device for spoke A (via the dashboard/CLI/curl create-device). Response carries
   **`pending_approval: true`**; the device row is `status=pending`.
3. Bring spoke A's wg up with the issued config.
4. **PROVE it cannot connect:** on the gateway,
   `NFT wg show wg0` вЖТ **no peer** for A's pubkey (pending is excluded from `ListActivePeersForNode`);
   from spoke A, `ping -c3 10.99.0.1` (gateway) and `ping -c3 <spoke-B-ip>` вЖТ **100% loss** (no peer,
   no handshake). Record: pending device = no peer, no reachability.
5. Under `zero_trust_mode=enforcing` (optional, if already enforcing): `NFT nft list ruleset` вЖТ A's /32
   appears in NO allow rule (pending is excluded from the compiler input too вАФ the two-layer exclusion).

**PASS:** pending device has no wg peer, no ping, no allow rule. (This is the connect-gate the desktop
client enforces app-side too вАФ but the SERVER exclusion is the real boundary, proven here.)

## Leg 2 вАФ APPROVE вЖТ connects within <5s (timed)
1. `t0` = the approve HTTP 2xx: `POST /api/v1/organizations/$ORG/devices/$DEV/approve` вЖТ 204.
2. Poll `NFT wg show wg0` for A's pubkey to appear + a handshake; from spoke A `ping 10.99.0.1`.
3. **Assert `t_connected вИТ t0 < 5s`** (the push targets the gateway org-wide; the peer lands + handshakes).
   Record t0, t_connected, ќФ.
4. `GET /devices` вЖТ A is `status=active`, `approved_by` = the approving admin's id.

**PASS:** approve вЖТ peer + reachability < 5s; approved_by attributed.

## Leg 3 вАФ REJECT вЖТ teardown observable + IP freed for reuse (timed)
Enroll a SECOND pending device for spoke B (approval still on); note its assigned IP = `IP_B`.
1. `POST /api/v1/organizations/$ORG/devices/$DEV_B/reject` вЖТ 204. `GET /devices` вЖТ B is `revoked`,
   `assigned_ip` now null (freed вАФ the D1b sweep).
2. Enroll a THIRD device (spoke B re-enrolls) вЖТ it is assigned **`IP_B` reused** (the freed address is the
   lowest-free). Record: reject frees the IP; the next enrollment reuses it.
   *(Desktop-client mirror, optional: with the client pointed at spoke B, reject-while-awaiting fires the
   S6.4 teardown вАФ config cleared, a loud notification вАФ never a spurious "revoked-blip".)*

**PASS:** reject вЖТ revoked + IP freed; next enrollment reuses IP_B.

## Leg 3b вАФ F1-part-3 ON THE WIRE (the live proof of THIS branch's own security fix) вШЕ
This is the branch's headline security fix (Revoke/Reject вЖТ org-wide push) proven on the wire вАФ cheap:
one grep + one ping. **Precondition:** `zero_trust_mode=enforcing`, and a policy rule whose DESTINATION
group contains the device's owner, so the device's /32 lands as a group-dst grant on a NON-hosting
gateway. (Two gateways: G1 hosts the reused-IP device's owner as a SOURCE; G2 hosts a source device that
is GRANTED reachability to the dst-group вАФ so the dst device's /32 appears in G2's ruleset.)
1. Before reuse: on **G2** (which does NOT host the device), `NFT2 nft list ruleset | grep IP_B` вЖТ
   the old device's /32 is (or was) a group-dst grant here.
2. Reject/revoke the old IP_B device вЖТ within <5s, on **G2**: `NFT2 nft list ruleset | grep IP_B` вЖТ
   **GONE** (the org-wide push stripped the stale grant on the non-hosting gateway).
3. Enroll a NEW device that reuses `IP_B`, owned by a user who is **NOT** in that dst-group.
4. From the granted SOURCE spoke on G2, `ping -c3 IP_B` вЖТ **100% loss**: the new holder did NOT inherit
   the old device's group-dst reachability. `NFT2 ... | grep IP_B` вЖТ no grant for the new device.

**PASS:** the stale group-dst grant was stripped org-wide <5s before the IP could be reused, so a
reallocation never inherits reachability on an unreconciled gateway. (If Revoke/Reject had pushed
own-node only, step 2's grep would still show IP_B on G2 and step 4's ping would SUCCEED вАФ the
address-reuse privilege leak this branch fixed.)

## Leg 4 вАФ D4: flip ON grandfathers the existing fleet (no mass black-hole)
1. Start with `device_approval=off` and вЙ•2 ACTIVE, connected devices.
2. `PUT device-approval {"mode":"on"}` вЖТ 200, `grandfathered_count` = the active-device count.
3. **Immediately** (no wait): the existing devices STAY connected вАФ `NFT wg show wg0` still shows their
   peers; their pings still succeed. `GET /devices` вЖТ they are still `active` (not retro-pended).
4. A NEW enrollment after the flip is `pending` (Leg 1 behavior).

**PASS:** flip on grandfathers the live fleet (no outage); only new enrollments gate.

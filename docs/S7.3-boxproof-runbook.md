# S7.3 device posture — TARGETED box-proof runbook (for Pawan to run + sign off)

**Why targeted, not the S7.2 8-proof scale:** S7.3 adds NO new gateway/enforcement path. "Block
untrusted" is exclusion-via-EXISTING-filters — a pending device is dropped from `ListActivePeersForNode`
(no peer/tunnel) AND from the compiler input (no grants) by the same `status='active'` filter the box
already proved in S7.2/S3.x. So the box proof only has to confirm the STATE MACHINE on the wire + the
one genuinely-new security surface (the F1-part-3 reject-leg IP-reuse). Four legs + one wire-security
proof.

## Setup (same shape as the S7.2 box run)
- Enterprise stack up (`edition=enterprise`) — device approval is enterprise-gated.
- One org; one gateway node enrolled (gateway = `10.99.0.1`).
- Two spokes A, B as WireGuard peers via `deploy/testing/s72-spokes.sh` (reuse it) — a client device
  each. (Same-wg-key caveat: don't run the same key on a real client and a netns spoke simultaneously.)
- CSRF = header presence: every mutating curl carries `-H 'X-Tunnex-CSRF: 1'`; auth = the
  `tunnex_session` cookie (browser) or the bearer.
- `NFT()` helper = run nft/tcpdump inside the node netns:
  `sudo nsenter -t $(docker inspect -f '{{.State.Pid}}' $(docker compose ps -q node-agent)) -n <cmd>`.

## Leg 1 — device_approval=ON: a new device enrolls PENDING and CANNOT connect (on the wire)
1. Turn approval on:
   `PUT /api/v1/organizations/$ORG/device-approval  {"mode":"on"}` → 200, note `grandfathered_count`.
2. Enroll a device for spoke A (via the dashboard/CLI/curl create-device). Response carries
   **`pending_approval: true`**; the device row is `status=pending`.
3. Bring spoke A's wg up with the issued config.
4. **PROVE it cannot connect:** on the gateway,
   `NFT wg show wg0` → **no peer** for A's pubkey (pending is excluded from `ListActivePeersForNode`);
   from spoke A, `ping -c3 10.99.0.1` (gateway) and `ping -c3 <spoke-B-ip>` → **100% loss** (no peer,
   no handshake). Record: pending device = no peer, no reachability.
5. Under `zero_trust_mode=enforcing` (optional, if already enforcing): `NFT nft list ruleset` → A's /32
   appears in NO allow rule (pending is excluded from the compiler input too — the two-layer exclusion).

**PASS (server boundary):** pending device has no wg peer, no ping, no allow rule.

### Leg 1 (client half) — the commit-six catch, proven LIVE on the desktop
6. With the Tunnex desktop client pointed at this tenant, hit **Connect** on the pending device.
7. **PROVE the connect-gate:** the UI shows the stable **"Awaiting admin approval…"** state; the
   privilege helper is **NEVER invoked** (no macOS admin prompt, no `utun`/WFP interface created — check
   `ifconfig`/`Get-NetAdapter`); and across **≥60s** there is **NO "revoked" notification** (the pre-fix
   bug fired a spurious "revoked" in ≤30s — its absence is the proof). The ApprovalMonitor is polling
   calmly.

**PASS (client gate):** connect on a pending device = awaiting state, helper never armed, no
spurious-revoked across ≥60s.

## Leg 2 — APPROVE → connects within <5s (timed)
1. `t0` = the approve HTTP 2xx: `POST /api/v1/organizations/$ORG/devices/$DEV/approve` → 204.
2. Poll `NFT wg show wg0` for A's pubkey to appear + a handshake; from spoke A `ping 10.99.0.1`.
3. **Assert `t_connected − t0 < 5s`** (the push targets the gateway org-wide; the peer lands + handshakes).
   Record t0, t_connected, Δ.
4. `GET /devices` → A is `status=active`, `approved_by` = the approving admin's id.

4b. **Client (refinement-2 observed):** on approve, the desktop shows the **ApprovalMonitor
    notification** ("device approved — click Connect"), the device becomes **connectable**, and the
    tunnel does **NOT** auto-connect (no `utun`/handshake until the human clicks Connect — a background
    poll must never arm the kill-switch).

**PASS:** approve → peer + reachability < 5s; approved_by attributed; client notifies + connectable, no
auto-connect.

## Leg 3 — REJECT → teardown observable + IP freed for reuse (timed)
Enroll a SECOND pending device for spoke B (approval still on); note its assigned IP = `IP_B`.
1. `POST /api/v1/organizations/$ORG/devices/$DEV_B/reject` → 204. `GET /devices` → B is `revoked`,
   `assigned_ip` now null (freed — the D1b sweep).
2. Enroll a THIRD device (spoke B re-enrolls) → it is assigned **`IP_B` reused** (the freed address is the
   lowest-free). Record: reject frees the IP; the next enrollment reuses it.
   *(Desktop-client mirror, optional: with the client pointed at spoke B, reject-while-awaiting fires the
   S6.4 teardown — config cleared, a loud notification — never a spurious "revoked-blip".)*

**PASS:** reject → revoked + IP freed; next enrollment reuses IP_B.

## Leg 3b — F1-part-3 ON THE WIRE (the live proof of THIS branch's own security fix) ★
This is the branch's headline security fix (Revoke/Reject → org-wide push) proven on the wire — cheap:
one grep + one ping. **Precondition:** `zero_trust_mode=enforcing`, and a policy rule whose DESTINATION
group contains the device's owner, so the device's /32 lands as a group-dst grant on a NON-hosting
gateway. (Two gateways: G1 hosts the reused-IP device's owner as a SOURCE; G2 hosts a source device that
is GRANTED reachability to the dst-group — so the dst device's /32 appears in G2's ruleset.)
1. Before reuse: on **G2** (which does NOT host the device), `NFT2 nft list ruleset | grep IP_B` →
   the old device's /32 IS a group-dst grant here (record the rule line).
2. **TIMED:** `t0` = the reject/revoke HTTP **2xx**. Poll `NFT2 nft list ruleset | grep IP_B` until
   **empty**. **Assert `t_gone − t0 < 5s`** — record t0, t_gone, Δ. This is the org-wide push stripping
   the stale grant on the non-hosting gateway.
3. Enroll a NEW device that reuses `IP_B`, owned by a user who is **NOT** in that dst-group.
4. **DIRECTIONAL — probe from the SOURCE side (the attack direction):** from the spoke on G2 that holds
   the grant TO the old device's dst-group, `ping -c3 IP_B`. **BEFORE** any new grant references the
   reused IP: the ping **DROPS (100% loss)** and G2's **`tunnex_default_drop` counter TICKS** for those
   packets (`NFT2 nft list ruleset | grep default_drop` before/after — the delta accounts for the probe).
   `NFT2 ... | grep IP_B` → no grant for the new holder.

**PASS:** the stale group-dst grant was stripped on the non-hosting gateway `Δ<5s` after reject (timed),
and the reused IP took packets to `default_drop` from the source side — the new holder inherited NO
reachability. (Had Revoke/Reject pushed own-node only, step 2 never goes empty on G2 and step 4's ping
SUCCEEDS — the address-reuse privilege leak this branch fixed.)

## Leg 4 — D4: flip ON grandfathers the existing fleet (no mass black-hole)
1. Start with `device_approval=off` and ≥2 ACTIVE, connected devices.
2. **Start a CONTINUOUS ping** on a grandfathered device (`ping 10.99.0.1` running throughout).
3. `PUT device-approval {"mode":"on"}` → 200. **Assert** `grandfathered_count` == the active-device
   count (the enumeration, best-effort but present on this happy path).
4. **QUANTIFIED (the proof-8 pattern):** across the flip, the continuous ping logs **ZERO lost packets**
   — the live fleet is untouched (no peer drop, no re-pend). Record the loss count.
5. **`approved_by IS NULL` on grandfathered rows** (one psql): `SELECT count(*) FROM devices WHERE
   org_id='$ORG' AND status='active' AND approved_by IS NULL;` == the grandfathered fleet (NULL =
   grandfathered/auto, D4 — they were never human-approved).
6. **Negative:** a fresh enrollment AFTER the flip lands `pending` (Leg 1 behavior).

**PASS:** flip on → grandfathered_count matches fleet, zero lost packets across the flip, grandfathered
rows have `approved_by IS NULL`, new enrollment gates pending.

# S7.5.3 — Device posture checks v1 (commit-one, decision-first)

**Paper before code.** Extends the S7.3 *known-device* gate to *healthy-device*: OS version · disk
encryption · EDR-present; block-or-warn per org; enterprise-gated. Client-reported attestation —
its limits stated HONESTLY (§Threat model), not oversold.

## What already exists (seam map, so we plug in, not greenfield)

- **S7.3 is device APPROVAL (known-device), not posture.** `organizations.device_approval` (`0020`),
  `Approve`/`SetDeviceApproval` (`devices/service.go:515,592`), edition bool `NewDeviceApprovalEdition`
  (`device_approval_wire_{open,enterprise}.go`). A pending device is blocked by **desired-state
  EXCLUSION** (`ListActivePeersForNode` / `ListActiveDevicesForOrg` filter `status='active'`), **not a
  connect-time check.** There is NO server-side "connect/authorize" endpoint — connection is the client
  bringing up WireGuard against a config it already holds. **The only server leverage over a live
  connection is desired-state inclusion/exclusion + the `<5s` `nodepush` (`nodepush/hub.go`).**
- **No client→CP posture channel today.** The AGENT reports wg telemetry (`POST /agent/report` →
  `device_status`); the CLIENT only creates/lists/revokes devices + polls its own status
  (`httpdeviceapi.ts`). S7.5.3 must **create** a client→CP posture-fact channel.
- **Org toggles = columns on `organizations`** (`zero_trust_mode`, `device_approval`), read via
  `GetOrganizationByID`, written by an audited `SetOrg…`.
- **Grandfather precedent (S7.3):** flip-on default-off touches no existing rows; only NEW devices read
  the flag at create; `SetDeviceApproval` never mutates device rows; best-effort blast-radius count
  post-commit. "A flip must not mass-blackhole a fleet" (the S7.2 cold-start lesson).
- **Naming collision:** `device_posture` is already migration `0020`'s name + `device_posture_handlers.go`
  (all S7.3 approval). S7.5.3 uses **`device_health`** identifiers to stay distinct.

## Sizing verdict (standing check)

Control-plane + a new client-report channel + client-side fact collection + a few active-device reader
edits + the reuse of the exclude-then-push machinery. **No kernel / no enforcement-chain contact.**
S7.5.2-sized. **The one sprawl vector is EDR-present** (see D5) — real EDR detection pulls toward
fragile process/bundle enumeration or TPM/attestation. If the paper starts reaching for real
attestation, that's the scope signal to CALL, not absorb.

## Threat model — what client-reported posture CAN and CANNOT do (load-bearing, stated up front)

The desktop client (via the privileged helper) reads real OS state (FileVault/BitLocker/OS version),
but **the REPORT is client-controlled → spoofable by a compromised or malicious client.**

- **CAN:** deter the *honest-but-noncompliant* device — a user with FileVault off is told/blocked and
  turns it on. Raises the floor for the ordinary fleet; gives compliance a legible signal + audit trail.
- **CANNOT:** stop a *compromised/malicious* client that LIES (reports compliant when it isn't) or goes
  SILENT to evade. This is NOT hardware attestation (no TPM/DeviceCheck/Secure-Enclave quote in v1).

**Honesty line that MUST ship in docs + the config UI** (verbatim intent): *"Posture checks deter honest
non-compliance and give you an audit trail. They are client-reported, not hardware-attested — a
compromised device can misreport. Treat posture as defense-in-depth, not a guarantee."* Selling this as
attestation would be the dishonest failure mode; the paper forbids it. (A future S7.5.3b could add a
hardware-attested channel; explicitly out of v1.)

This threat model directly decides **absence/staleness** (D4): since a malicious device can lie anyway,
failing *closed* on absence buys little against the malicious case while breaking compat for
non-reporting devices — so absence does NOT block (fail-open + surfaced), and only a POSITIVE
non-compliant report gates. We block what we can see is bad; we don't pretend to see what we can't.

## Decide-items

### D1 — Report mechanism + seam (who reports, which channel, schema)

- **Reporter = the DESKTOP CLIENT, not the agent.** The agent owns the data plane (wg); it has no view
  of FileVault/BitLocker/OS of the *user's* machine. The client (via its privileged helper, which
  already has the OS APIs) collects and reports. CLI-only + mobile-WG devices have no such client → they
  never report (→ D4 absence).
- **Channel = a NEW client→CP endpoint** `POST /api/v1/organizations/{orgId}/devices/{deviceId}/health`
  (there's nothing to fold into — the agent path is node telemetry, not client posture). Body:
  `{os_version, disk_encrypted: bool, edr_present: bool|null, collected_at}`. Authed as the **device
  OWNER** (bearer/session, self-report — same ownership rule as `createDevice`), NOT an admin perm.
- **Schema = a lean `device_health` table** (device_id PK, os_version, disk_encrypted, edr_present,
  reported_at, + a derived `evaluated_state`), mirroring the `device_status` precedent
  (`0013`: "report-driven telemetry lands in a LEAN table, not on the churny devices row"). Posture facts
  churn on every report → a table, not `devices` columns.
- **Ownership split:** CLIENT collects + reports raw facts; CP owns config, evaluation, and the
  enforcement flag; the AGENT is untouched (it just serves whatever desired state CP computes).
- **RECOMMEND** as stated. *Open sub-decision:* store only the current snapshot (v1) vs a short history
  (audit). Recommend snapshot-only for v1 (history = S7.5.3b), keep it small.

### D2 — Evaluation semantics: block-vs-warn, WHERE and WHEN (the sharpest item)

There is no connect gate, so evaluation is **continuous, server-side, reusing exclude-then-push:**

- Client re-reports on an interval (reuse the `ApprovalMonitor` cadence class). On each report, CP
  evaluates the facts against the org's per-check config (D3).
- **block-mode + noncompliant → set the device's health-gate flag (D7) → the active-device readers
  exclude it → `PushOrgNodes` → its /32 leaves every gateway within `<5s`.** Back to compliant on a
  later report → clear the flag + push → re-included. This is exactly the S7.3-pending / S7.5.2-deprovision
  machinery; **no new enforcement path.**
- **warn-mode + noncompliant → NEVER gates.** It records an audit event + surfaces a warning
  (dashboard device row + client tray/banner), access continues.
- **The drift answer (explicit, per prior direction — this is THE semantic):** an already-connected
  device that drifts noncompliant under a block-mode check **is disconnected mid-session** — the next
  re-report (≤ the report interval) flips the gate and the `<5s` push pulls its /32. Bounded by the
  report interval, not instant; stated honestly as "posture drift is caught within one report cycle,"
  NOT "instantly." Warn-mode drift only raises the banner.
- **RECOMMEND** continuous-reeval + exclude-then-push + interval-bounded drift disconnect. *Open
  sub-decision:* the report interval (tighter = faster drift catch + more load). Recommend ~10 min
  (aligns S7.5.2 poll + ApprovalMonitor), tunable.

### D3 — Per-check config model (the unlock-then-opt-in convention, f2385e1)

Convention: **org-level opt-in, default-OFF, per-check configurable** — an org can require
disk-encryption WITHOUT requiring EDR.

- **Config shape = a `org_health_checks` table** (org_id, check_kind, mode, param), one row per
  configured check. `check_kind ∈ {os_version, disk_encryption, edr_present}`; `mode ∈ {warn, require}`;
  `param` (e.g. `{"min":"14.0"}` for os_version, null for the booleans). **No row = that check is OFF**
  (opt-in by absence → default-off by construction). This DIVERGES from the S7.3 single-column pattern
  (`device_approval text off|on`) because per-check + params don't fit one column; the divergence is the
  justification. *Alternative:* three columns on `organizations` (`health_os_mode`, `health_disk_mode`,
  `health_edr_mode` + a `health_os_min`). Columns match precedent but bloat the org row + are awkward for
  params.
- **RECOMMEND the table** (per-check + params is the real shape). Written by an audited
  `SetOrgHealthCheck(org, kind, mode, param)`; the convention (default-off, per-check) is enforced by the
  table being empty until an admin opts in.
- Config CRUD is admin (D8 perm). Reading config for evaluation is org-scoped (querylint auto-arms).

### D4 — Staleness + absence + grandfather (the compat question)

Directly decided by the threat model:

- **Absence (never reported): FAIL-OPEN + surfaced, never blocks.** A device with no `device_health` row
  (CLI-only, mobile-WG, old client version, or a desktop that hasn't reported yet) is treated as
  `posture_unknown` — access continues; the dashboard/audit shows "posture unknown." Rationale: a
  malicious device can report-compliant anyway, so fail-closed-on-absence breaks compat (blackholes CLI
  /mobile fleets) for ~no security gain against the real adversary. Honest + compat-preserving.
- **Staleness (reported once, then went quiet past a TTL): treat as absence** (→ posture_unknown, not
  blocked). TTL ~ a few report intervals. Only a FRESH positive non-compliant report gates.
- **Grandfather = 0%-loss BY CONSTRUCTION:** turning a check ON (a) is default-off, (b) never mutates
  device rows in a migration, (c) blocks only on a *positive fresh non-compliant report* — so a fleet on
  old/non-reporting clients keeps working on flip-on (they're posture_unknown, not blocked). Best-effort
  blast-radius post-commit: count devices whose LAST report (if any) would fail the newly-enabled check
  (mirrors S7.3's `GrandfatheredCount`). This is stricter-safe than S7.3: even reporting devices aren't
  retro-blocked by the config write itself — only by their next evaluation.
- **RECOMMEND** as stated. *Open sub-decision:* a per-org "strict / require-reporting" toggle that makes
  absence WARN-visible-but-still-not-block, or (v-next) block. Recommend NOT shipping a
  block-on-absence mode in v1 (it's the dishonest-security foot-gun); leave a named hook for S7.5.3b.

### D5 — Platform coverage matrix (stated HONESTLY — what actually ships per platform)

| Check | macOS (read path) | Windows (read path) | v1 verdict |
|---|---|---|---|
| **OS version** | `ProcessInfo.operatingSystemVersion` / `sw_vers` — reliable | `RtlGetVersion` — reliable | **SHIP both** |
| **Disk encryption** | FileVault: `fdesetup status` / IOKit — reliable (helper is privileged) | BitLocker: `Win32_EncryptableVolume` WMI / `manage-bde` — reliable (privileged) | **SHIP both** |
| **EDR present** | No standard API. Only fragile process/bundle-id enumeration of known agents — incomplete + brittle | Windows Security Center WMI (`AntiVirusProduct`) — registered AV, "EDR" vs "AV" is fuzzy, WSC-only | **THE SPRAWL VECTOR** |

- **EDR is the honesty + sizing problem.** Reliable OS-version + disk-encryption on both platforms is
  clean, bounded, and honest. EDR has no reliable cross-platform read; doing it "well" pulls toward agent
  fingerprinting / an allow-list of vendor bundle-ids = maintenance treadmill + false confidence.
- **RECOMMEND for v1: ship OS-version + disk-encryption on both platforms.** Ship **EDR-present as
  Windows-only, best-effort, EXPLICITLY LABELED** ("Windows Security Center-registered protection; not
  available on macOS; best-effort") — OR **defer EDR entirely to S7.5.3b.** *This is the decide-item I
  most want your call on:* honest-limited-EDR-now vs clean-defer-EDR. My lean: **defer EDR to 5.3b** —
  it keeps v1 honest and S7.5.2-sized, and avoids shipping a check that reads as security theater on
  macOS. (The scope-signal the sizing note warned about is exactly EDR.)

### D6 — Honest-attestation framing (docs + UI)

Covered by §Threat model. As a decide-item: the config UI and `docs/` MUST carry the honesty line;
the check labels say "client-reported"; there is NO copy implying attestation/tamper-proof. **RECOMMEND**
the verbatim honesty line ship in both the admin config surface and the self-host docs.

### D7 — Enforcement seam (how a block actually excludes)

- **A device orthogonal `health_gate` flag** (e.g. `devices.health_blocked boolean NOT NULL DEFAULT
  false`, or a `health_state` enum) that the active-device readers AND-in:
  `ListActivePeersForNode` + `ListActiveDevicesForOrg` gain `AND NOT health_blocked`. Posture is
  ORTHOGONAL to approval `status` (a device can be active+approved but posture-blocked), so a separate
  flag beats extending the `status` CHECK enum. Set/cleared by the evaluation (D2) + `PushOrgNodes`.
- *Alternative:* add `status='health_blocked'` to the enum (reuses the pending machinery) — rejected
  because it conflates two orthogonal axes (a device could be both pending-approval and unhealthy).
- **RECOMMEND the orthogonal boolean.** It touches exactly the two active-device reader queries (+ their
  twins) — the bounded enforcement edit. querylint auto-arms the new queries.

### D8 — Standing guards auto-arm

- **Edition gate:** a per-feature bool `NewDeviceHealthEdition()` (open=false / enterprise=true), mirroring
  `device_approval_wire_{open,enterprise}.go`; wired in `main.go`; each handler `if !s.deviceHealthEnabled
  { return editionRequired() }`. (Named per feature — the standing discipline.)
- **RBAC:** config CRUD = a **new `PermDeviceHealthManage`** (owner+admin) — named per feature, not a
  reuse of `PermDeviceApprove` (distinct capability). The **report endpoint is device-OWNER-authed**
  (self-report, member-allowed for their own device — like `createDevice`), no manage perm.
- **401-walk:** spec-driven auto-arm; add `puthealthcheck`/`reportdevicehealth` bodies to `walkBodies`.
- **querylint:** `device_health` + `org_health_checks` auto-armed as tenant tables → queries scope
  `org_id` (report endpoint scopes by device→org; config by org).
- **RECOMMEND** as stated.

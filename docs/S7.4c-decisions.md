## Grounding (what exists)
- **`e2e` CI job** (`.github/workflows/ci.yml`): `docker compose up -d --build --wait` (**OPEN edition** — no
  `TUNNEX_BUILD_TAGS`) + `make seed` (org + users ONLY — no SSO config / policy / node / device) + Playwright
  vs `nginx:8080`. **`continue-on-error: true`** (opportunistic / non-blocking by the S6.0b mandate).
- **The committed enterprise build path** (from `fix`/S3.7-era): `TUNNEX_BUILD_TAGS=enterprise` build-arg →
  `make up-enterprise`. So an enterprise stack is buildable from committed config — no Dockerfile hack.
- **S4.5 substitute** (`e2e/tests/settings.spec.ts:25`): asserts the "Tunnex Enterprise feature" GATE (open
  build → SSO 403). The REAL assertion (GET sso config payload carries no `client_secret`, fingerprint
  present) needs an ENTERPRISE build where the endpoint returns 200 + a seeded SSO config row.
- **S4.5b substitute** (`settings.spec.ts:68,86`): `page.route(...pool-cidr...)` MOCKS the 409 orphan list.
  The REAL assertion needs a live enrolled gateway + a device holding a pool IP, so a shrink STRANDS it and
  the server returns a REAL 409 orphan list the UI renders (un-mock).

## D-c1 — a SEPARATE `e2e-enterprise` CI job (not extend the open `e2e` job)
The open `e2e` stays as-is (open-edition smoke). Add a second job `e2e-enterprise` that brings the stack up
with `TUNNEX_BUILD_TAGS=enterprise` + an enterprise seed, and runs an enterprise-tagged Playwright suite.
Rationale: two editions = two stacks (the open one must keep proving the open gate at settings.spec.ts:25;
the enterprise one proves the payload). One job can't be both editions.

## D-c2 — blocking vs opportunistic: OPPORTUNISTIC (continue-on-error), but it RUNS
The e2e class is non-blocking by the S6.0b mandate (heavy/flaky infra must not block merges). `e2e-enterprise`
inherits `continue-on-error: true`. **"Un-deferrable" means it EXISTS and RUNS the real assertions every CI**
— the SATISFY is "the assertion executes against a live enterprise stack and is green", not "it red-blocks
merge". (A red here is a signal, triaged like the open e2e; the gates+client jobs remain the merge gates.)
*Open question for sign-off: is opportunistic acceptable for the SATISFY, or do you want these two specific
assertions promoted to a blocking check?*

## D-c3 — enterprise seed (`make seed-enterprise` or a flag on seed)
Extend the seed with the minimum the two assertions need, enterprise-tagged:
- **SSO config row** for the demo org (a sealed `client_secret_sealed` + provider) → S4.5's GET returns a real
  `ConfigView` to assert on (no secret, fingerprint present).
- **An enrolled gateway node + a device holding a pool IP** → S4.5b: a shrink strands it → real 409.
Decide-before-code: seed as a distinct `seed-enterprise` target (keeps the open seed pure) vs a
`TUNNEX_SEED_ENTERPRISE=1` branch. Lean: a distinct target.

## D-c4 — the node in CI = mem-backend + a join-token mint step (the hard wrinkle)
A real enrolled node without NET_ADMIN: `TUNNEX_WG_BACKEND=mem` (the CI-friendly backend). Enrollment still
needs a one-time **join token** (mint via the API/CLI) → the agent enrolls → then seed/API creates a device
that allocates a `10.99.0.x` IP so the shrink strands it. Decide-before-code: mint-token-in-CI mechanics
(a compose one-shot / a small mint step) + whether the device is created by the seed or a Playwright step.
(The two-gateway box already proved the mem-backend enroll flow — S7.3; this is the CI-headless version.)

## D-c5 — un-mock the two assertions
- **S4.5**: replace the gate-only check with, under the enterprise suite, log in → GET `/organizations/{id}/sso`
  → assert 200 + the payload has NO `client_secret`/secret material + the keyed fingerprint is present.
- **S4.5b**: remove the `page.route` mock; drive a real pool shrink that strands the seeded device → assert
  the UI renders the REAL 409 orphan list (names + reasons). The existing enterprise `View` unit test +
  mocked render become the belt; this is the SATISFY.

# DISPOSITION (signed off 2026-07-14, conditional on S7.4b merge — confirmed PR#20 sha 6aa0fad)

- **D-c1 APPROVED** — separate `e2e-enterprise` job; open `e2e` keeps proving its 403 gate.
- **D-c2 OVERRULED IN PART → TWO TIERS.** The SATISFY is split by assertion class:
  - **S4.5 secret-payload → ALSO a BLOCKING API-level integration test** in the gates job (enterprise build,
    NO browser). Rationale: a security assertion must gate; four epics of `continue-on-error` would make
    "un-deferrable" decorative. Home = an enterprise-tagged Go httptest that runs in `make test-editions`
    (asserting the GET sso payload has NO client_secret + `SecretFingerprint` present).
  - **The Playwright tier stays OPPORTUNISTIC** (the flake mandate applies to browsers, not security asserts).
  - **S4.5b 409-render → legitimately OPPORTUNISTIC** (browser render), recorded as such.
- **D-c3 APPROVED** — distinct `seed-enterprise`, composing ON TOP of `seed` (never forking base data).
- **D-c4 VERIFY-BEFORE-BUILD → VERIFIED, WRINKLE EVAPORATED (2026-07-14).** Both assertion paths read DB
  state, not wire state, so NO CI agent runs:
  - **S4.5b:** `devices.ResizePool` computes orphans from `q.ListActiveDeviceAllocations(ctx, orgID)` — a
    pure DB read of device rows' `assigned_ip` (service.go:358). `ipalloc.Orphans(newCIDR, allocated)` is a
    pure function. `devices.node_id` is NOT NULL FK to `nodes(id)`, but the query does NO enrollment check —
    a plain seeded `nodes` ROW (no agent, no mTLS, no join token) + a `devices` row with `assigned_ip` in the
    pool is sufficient. A shrink that excludes that IP returns the REAL 409 orphan list.
  - **S4.5:** `sso.ConfigService.View` returns `ConfigView` (no secret, `SecretFingerprint` only) from a
    `sso_configs` row (`client_secret_sealed`). Pure API+DB.
  - **CONSEQUENCE:** the mem-backend agent + join-token mint are NOT built. `seed-enterprise` = three DB
    inserts (sealed `sso_configs` row + a `nodes` gateway row + a `devices` row holding a pool IP).
- **Q4 APPROVED** — ONE PR (job + blocking httptest + seed-enterprise + Playwright un-mock).

# S6.7 (Story B) — the `kill -9` pcap GATE (Windows twin of the macOS proof)

The one proof that lifts the S6.9 guard. With the persistent arming now wired into the live path
(Phase B), a full tunnel is up (via the Story A dev bypass); we `taskkill /F` the helper mid-tunnel
and prove **zero cleartext egress while the WFP block holds** (death = enforcement), then that
recovery restores networking. **Nothing lifts the guard on a partial pass.**

Runs on the **Windows client** + the **VM gateway**. Admin PowerShell on Windows. Install the
branch `.exe` first.

## 0. Enable the dev bypass (both flags) + connect full tunnel
- VM: `TUNNEX_ALLOW_WINDOWS_FULLTUNNEL=1 docker compose up -d api`
- Windows (admin), then restart the service:
  `setx TUNNEX_DANGEROUS_WINDOWS_FULLTUNNEL 1 /M ; sc.exe stop tunnex-helper ; sc.exe start tunnex-helper`
- App → tick **Route all traffic (full tunnel)** → **Connect**. Confirm it's the persistent path:
  ```
  curl.exe -s ifconfig.me ; echo          # = the GATEWAY's public IP (egress works)
  netsh wfp show state file=- | findstr /i tunnex   # Tunnex provider present (block armed)
  ```
  The app shows the amber **DEV PREVIEW** banner (persistent kill-switch + the --wfp-clean hint).

## 1. Capture on the physical NIC (optional but ideal — built-in pktmon)
```
pktmon start --capture --pkt-size 0 -f C:\tunnex-kill9.etl
```

## 2. kill -9 the helper mid-tunnel (death = enforcement)
```
taskkill /F /IM tunnex-helper.exe
```
The tunnel's wireguard-go dies with it. The WFP block is kernel-resident — it MUST hold. (If the
SCM auto-restarts the service and networking returns immediately, disable the restart for the test:
`sc.exe failure tunnex-helper reset= 0 actions= ""` then re-kill — we need the dead window to
observe the block, not an instant recovery.)

## 3. Prove NO cleartext egress while the block holds (THE gate)
Immediately, with the helper dead:
```
curl.exe -s --max-time 5 ifconfig.me ; echo    # MUST be empty / fail
ping -n 3 8.8.8.8                               # MUST fail (General failure / timeout)
```
- **PASS = both BLOCKED.** No traffic reaches the internet in cleartext after the tunnel owner
  died — the WFP kill-switch survived process death. (Contrast: pre-S6.7 the dynamic session
  auto-deleted here and this leaked.)

## 4. pcap confirms zero cleartext (if captured)
```
pktmon stop
pktmon etl2txt C:\tunnex-kill9.etl -o C:\tunnex-kill9.txt
```
- Inspect `C:\tunnex-kill9.txt` for OUTBOUND packets on the physical adapter to a public IP during
  the dead window (exclude loopback + the WG endpoint). **PASS = none** — the wire is silent.

## 5. Recovery restores networking (bounded fail-closed)
Any of the proven recovery paths brings it back:
```
"C:\Program Files\Tunnex\resources\helper\tunnex-helper.exe" --wfp-clean   # escape hatch
# OR: sc.exe start tunnex-helper   (startup CleanStale releases the block)
# OR: reboot
curl.exe -s ifconfig.me ; echo    # your real IP → networking back
```
(If you disabled SCM failure actions in §2, restore them: `sc.exe failure tunnex-helper reset= 86400 actions= restart/5000`.)

## S8.5 COUPLING CLAUSE (acceptance precondition — every decide-item states its S8.5 behavior)

S8.5 pushes **open-edition routed subnets** (declared LANs) into split-tunnel devices' AllowedIPs — a known **sibling** of site subnets. **The clause:** S8.4 models a resolvable internal zone as *"a reachable range with an optional resolver,"* keyed to the **site/range**, never to a device or a node. So an S8.5 declared-LAN plugs into the SAME `dns_forwarding` table and the SAME device-resolver handoff without a redo — S8.4 builds the mechanism generically over ranges-with-resolvers; S8.5 adds more ranges. **A design that special-cases "site" in the DNS path fails this clause.** Concretely: (1) the forwarding table is keyed by domain→resolver-reachable-via-a-range, agnostic to whether the range is a site subnet or an S8.5 LAN; (2) the device-side resolver handoff (D3) is emitted whenever a device has a route to a range that declares a resolver — site today, S8.5 LAN tomorrow, same code.

## Decide-items (proposed dispositions, HELD for ruling)

### D1 — Resolution model: FORWARDING, not a name registry (one-truth)
**Proposed:** S8.4 does **zone forwarding**, exactly per the seam — a site declares *"domain `X` is resolved by my internal resolver at `<IP-in-my-subnet>`"* (`dns_forwarding[] = {domain, resolver_ip}`). Tunnex makes those zones resolvable **cross-site over the tunnel**; it does NOT host per-host records. Names live in the org's existing site DNS (the org already runs DNS for `nas.mumbai.acme.com`); S8.4 forwards the query there over the WG link.
- **Shape:** `nas.mumbai.acme.com` resolves because `mumbai.acme.com` is forwarded to mumbai-site's resolver. No synthetic `<host>.<site>.<org-zone>` scheme invented by tunnex — the org owns the zone; tunnex owns the FORWARDING TABLE (which domain → which site's resolver). **One-truth:** the CP owns sites/subnets/nodes and now the forwarding table; the resolver_ip must be VALIDATED to lie inside that site's approved subnet (no second registry, no free-floating IP — the `subnetguard`/site-subnet truth gates it).
- **Split-horizon:** the forwarded zones resolve to internal IPs ONLY inside the tunnel; a device/site keeps its own resolver for everything else. Public names untouched.
- **Cross-site:** rides the tunnel (works today on gateway Routes — see verify (b)).
- **S8.5:** identical — an S8.5 LAN can declare a `dns_forwarding` entry; the mechanism doesn't branch on site-vs-LAN.
- **Open question for ruling:** enterprise verified-domain integration (auto-derive the org-zone from the SSO domain claim) vs purely admin-entered domains. Proposed: **admin-entered domain + resolver_ip in v1** (open + enterprise), verified-domain auto-suggest as an enterprise nicety deferred — keeps v1 edition-symmetric and avoids coupling DNS to SSO.

### D2 — Where resolution lives + the failure mode
**Proposed:** **gateway-as-forwarder** (what the seam implies). The agent runs a tiny DNS forwarder; a query for a remote zone goes device/host → the answering gateway's forwarder → tunnel → the target site's gateway → that site's declared resolver_ip. The CP compiles the org's forwarding table into the desired-state artifact (a new out-of-hash section, D5); the agent programs its forwarder from it.
- **Resolver component:** the node image has none → add a **minimal Go DNS forwarder in the agent** (miekg/dns or stdlib) rather than shelling to dnsmasq/CoreDNS (fewer moving parts, no extra process, testable). Listens on a tunnel-side address (a well-known port on the gateway's overlay IP).
- **FAILURE MODE (four-word model, stated):** **DNS-down ≠ tunnel-down.** The forwarder is a SEPARATE best-effort section — a forwarder crash/miscompile must NEVER fail the reconcile, tear down peers/routes, or imply data-plane failure. **fail-static applies to resolver state:** the agent keeps last-good forwarding config; a bad DNS section is dropped, the rest of the desired state still applies (the DesiredState-atomic law's sibling — but DNS is convenience, so a DNS-only compile error should NOT fail the whole fetch; **decide-item sub-point for ruling:** does a malformed `dns_forwarding` entry fail the whole artifact (atomic) or just skip that entry (degraded)? Proposed: **skip-the-entry, degraded**, because DNS is not an enforcement boundary — reachability-by-IP must survive a DNS typo).
- **Health:** a resolver that can't reach its declared upstream is surfaced advisory (a `dns_forward_unreachable`-style signal, B2 pattern) — never green-blackhole, never a tunnel alarm.
- **S8.5:** the same agent forwarder answers for S8.5 LAN zones; no new resolver path.

### D3 — Client/device integration + the HA-posture sentence
**Proposed:** a device learns the internal resolver via **domain-SCOPED** resolver config, NOT a blanket `DNS =` replace. wg-quick `DNS =` sets the WHOLE system resolver → for split-tunnel that would tunnel ALL queries (wrong). So the **helper** does conditional/scoped resolver programming for the forwarded zones only (macOS `/etc/resolver/<domain>` or SCDynamicStore; Windows NRPT rule per domain; Linux systemd-resolved per-link domain). The CP hands the device the set of `{domain → tunnel-side resolver IP}` for zones it can reach.
- **What changes:** the helper gains domain-scoped resolver programming; the CP device-config emits the scoped-DNS set. **What does NOT change:** full-tunnel still gets its blanket `1.1.1.1`; split-tunnel keeps its own resolver for everything outside the forwarded zones.
- **HA-POSTURE (the sentence the plan owes):** **DNS is keyed to SITES, never to a node.** The resolver a device forwards to is the SITE's forwarder (reached via the site, whichever gateway currently backs it), and the resolver_ip is a SITE-subnet address. So **gateway replacement (S8.1 D6) does NOT orphan names** — the site identity + its `dns_forwarding` persist across a node swap; the new gateway forwards the same zone.
- **S8.5 coupling (bites here):** a split-tunnel device only benefits from a forwarded zone once it has a ROUTE to that zone's resolver — which is exactly S8.5's device-routes. **Proposed:** S8.4 emits the scoped-DNS set for any range the device can route to; today that's the pool only (verify (a)), so device-side S8.4 DNS is INERT until S8.5 lands the routes — S8.4 ships the mechanism + the cross-SITE proof (behind-hosts, which route today), and device→site resolution activates when S8.5 routes arrive. **No S8.5 build-ahead — just the composable handoff.**

### D4 — Edition split (argued)
**Proposed:** the FORWARDING mechanism (D1/D2) is **all-editions core**, like the site model (S8.1 D11). Rationale from the coupling clause: **S8.5's open-edition routed subnets want basic resolution too** — making DNS forwarding enterprise-only would leave S8.5's open LANs nameless, a guaranteed redo. Enterprise ADDS: (proposed, held) governance over DNS — verified-domain auto-derivation, and/or policy over which sites may resolve which zones (a DNS-scoped grant). v1 keeps the split minimal: **core = the forwarding table + agent forwarder + cross-site resolution; enterprise = the verified-domain nicety + any DNS-policy governance, both deferrable past v1.** Argue against: DNS as enterprise-only contradicts the S8.5 coupling and the "sites are core" precedent.

### D5 — Version/hash classification (D2-checklist row, explicit)
**Proposed:** the `dns_forwarding` artifact section is **degraded-not-broken PLUMBING — OUT of CanonicalHash, NO RequiredVersion bump** — the same class as `Routes`/`LocalSubnets` (S8.2/S8.2c). RequiredVersion analysis: an old agent that ignores the DNS section loses ONLY name resolution — never reachability (reach by IP still works), never enforcement (grants gate by IP; DNS resolves a name to an IP the grant still governs — DNS can NOT bypass the enforcement boundary). So it is convenience/observability, not enforcement-significant → `projectForHash` excludes it, twin goldens stay byte-identical, no version bump, a no-DNS org's artifact is unchanged. (Contrast: `Routes` is ALSO out-of-hash but triggered v5 because a route-carrying artifact an old agent can't interpret is a reachability gap it must REFUSE — DNS has no such refuse-worthy gap, so it does not even trigger the version like Routes did. **Confirm at ruling:** DNS out-of-hash AND no version trigger.)

### D6 — Walk shape (defined before code)
**Proposed:** cross-cloud pair (AWS Sydney ↔ Azure West US, the S8.2c topology) resolving each other **by name** from a **genuinely-separate FORWARDED behind-host** (fixture-fidelity topology law — NOT the gateway host, NOT a dummy-in-netns).
- **Leg 1 — cross-site name resolution:** from the Azure behind-host, `dig nas.aws-site-zone` (a name in aws-site's declared `dns_forwarding` zone) resolves via azure-gw → tunnel → aws-gw → aws-site's resolver → an AWS host IP; then reach it by that name. Zero gateway SSH (Zero-Touch Law).
- **Leg 2 — fail-static / DNS-down ≠ tunnel-down:** kill the site resolver (or break the forward) → name resolution fails, BUT reach-by-IP and the tunnel stay UP (the four-word model on the wire — a DNS failure never tears down the data plane).
- **Leg 3 — HA (D6/keyed-to-site):** replace the gateway node bound to a site → the zone still resolves (the forwarding config rode the SITE, not the node).
- **Leg 4 — split-horizon:** a public name still resolves via the host's own resolver (tunnex only captured the declared internal zones).
- **Device→site resolution is a NAMED SUBSTITUTE** (unit-pinned) with trigger = S8.5's device-routes (per verify (a)/(b) — inert until then).

### D7 — Config surface + audit + validation
**Proposed:** per-site `dns_forwarding` managed on the Sites page (an admin adds `{domain, resolver_ip}`), through the audited site path (`site.dns_forwarding_set` / `_removed`), `authorize(PermSiteManage)` FIRST, all-editions core. **Validation (one-truth):** the resolver_ip MUST be inside one of that site's APPROVED subnets (the CP's authoritative subnet truth — no free-floating resolver); a domain must be well-formed. Full-sweep: removing an entry withdraws the forward on the next reconcile (the S8.2c WF-5 pattern).

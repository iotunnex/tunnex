```bash

make seed && make seed-enterprise                        # demo org + owner + strandable device + sso
```

`AUTH`/`PSQL`/`ORG`/`BASE` = the S7.3/S7.4 STEP-0 blocks. Confirm `/api/v1/meta` → `edition: enterprise`.

## STEP 2 — 3-LEG SPINE

### Leg 0-b — OFF-DEFAULT WIRE PROOF (cheap, FIRST) → then arm
Prove the safety default on the wire BEFORE arming: with flow logging OFF (g2 booted without
`TUNNEX_FLOWLOG_GROUP` in STEP 1), the forward chain must be byte-identical to pre-S7.5.1 (NO log clauses).
```bash
# 1. Capture the OFF ruleset:
docker exec tunnex-node-agent-g2 nft list ruleset > walk-artifacts/leg0b-OFF.nft
grep -c "log prefix" walk-artifacts/leg0b-OFF.nft            # EXPECT: 0 (no log clauses)
# 2. Arm the group (restart g2 WITH the group) + capture the ARMED ruleset:
docker rm -f tunnex-node-agent-g2
docker run -d --name tunnex-node-agent-g2 --network tunnex_default --cap-add NET_ADMIN \
  -e TUNNEX_JOIN_TOKEN=$JOIN2 -e TUNNEX_NODE_NAME=g2 -e TUNNEX_API_URL=http://api:8080 \
  -e TUNNEX_AGENT_URL=https://api:8443 -e TUNNEX_WG_BACKEND=mem -e TUNNEX_FLOWLOG_GROUP=100 tunnex-node-agent
#   (mint a fresh JOIN2 if the first token was consumed; g2 re-enrolls under a new cert.)
docker exec tunnex-node-agent-g2 nft list ruleset > walk-artifacts/leg0b-ON.nft
# 3. The SOLE delta OFF→ON is the log clauses (the accept/drop verdicts unchanged):
diff walk-artifacts/leg0b-OFF.nft walk-artifacts/leg0b-ON.nft > walk-artifacts/leg0b-DIFF.txt
```
EXPECTED EVIDENCE: `leg0b-OFF.nft` has ZERO `log prefix`; the diff adds ONLY ` log prefix "tnx:…" group 100`
clauses to the accept lines + the deny tail — every `accept`/`drop` verdict byte-identical.
**PASS:** OFF renders no log clause (safety default proven on the wire); arming adds ONLY observation, never a
verdict change. **SAVE:** `leg0b-OFF.nft` · `leg0b-ON.nft` · `leg0b-DIFF.txt`.

### Leg 1 — rule → allow-flow → event-WITH-rule_id
STIMULUS: add an allow rule (owner → a resource the device can reach); drive ONE allowed flow from a spoke
device through the gateway.
```bash
# add a rule (Access UI, or:) POST .../policy/rules {src_group, dst_kind, dst_*}  -> capture RULE_ID
# drive a flow from a device to the granted dst (e.g. curl/nc from the device's namespace).
```
EXPECTED EVIDENCE:
- **nft ruleset** (`nft list ruleset` on g2): the allow rule renders `... counter log prefix "tnx:<RULE_ID> "
  group 100 accept` (the log clause is the SOLE delta vs enforcement).
- **query API**: `GET $BASE/api/v1/organizations/$ORG/access-events` → an `allow` event carrying
  `rule_id == RULE_ID` + grant enrichment (`dst_resource_id`/`dst_group_id`), `src_device_id`/`src_user_id`
  NULL (grant-level attribution by design).
- **JSONL** (`$TUNNEX_FLOWLOG_DIR/access-*.jsonl`): the same event verbatim, a monotonic `seq`.
**PASS:** the allowed flow-start produces exactly one allow event, kernel-stamped rule_id (no re-match).
**SAVE:** the allow-event JSON (query API) · the matching `nft` accept line · the JSONL line.

### Leg 2 — delete → deny-flow → deny-event
STIMULUS: delete the Leg-1 rule; drive the SAME flow again (now unauthorized).
EXPECTED EVIDENCE:
- **nft ruleset**: the allow line is gone; the flow now falls to the deny tail
  `ct state new log prefix "tnx:deny " group 100 counter ... drop`.
- **query API** (`?denies_only=true`): a `deny` event (rule_id empty = default-deny), same src/dst.
- **JSONL**: the deny line, seq continues monotonic.
**SAVE:** the deny-event JSON · the deny-tail `nft` line · the JSONL deny line.
**PASS:** the revoked grant's NEW flow denies + logs a deny event. (NOTE: a pre-existing ESTABLISHED flow
SURVIVES until natural close — the ledgered conntrack-kill S7.2-follow-up; the `terminated` event seam (6/n)
is ready for it. NOT proven here.)

### Leg 3 — CP-down → buffer → reconnect → drop-count-visible-as-GAP
STIMULUS: stop the CP's mTLS ingest (stop the api, or block :8443); drive many flows (agent buffers, bounded
16384); overflow the buffer; restart the CP.
EXPECTED EVIDENCE:
- while CP down: `docker logs tunnex-node-agent-g2` shows `flowlog_report_failed` (best-effort, enforcement
  unaffected — flows still accept/drop).
- after reconnect: `GET .../access-events` shows a **`gap`** event with `deny_count == N` ("N events dropped
  here"); `ScanSeqGaps` over the JSONL shows the seq hole.
**PASS:** the loss is LEGIBLE as an explicit gap marker (kernel-overrun + buffer-overflow + report-failure all
fold into the one drop-count), never a silent skip. **SAVE:** the `gap` event JSON (deny_count=N) · the `ScanSeqGaps` output · the `flowlog_report_failed` log lines.

## STEP 3 — S7.2 GATEWAY LEGS WITH LOGGING ON (enforcement unchanged by observation)
Re-prove the S7.2 forward-chain guarantees hold with `TUNNEX_FLOWLOG_GROUP` armed:
- **default-deny drops:** enforcing + zero grants → a device→device flow is DROPPED (pcap/curl-timeout);
  the deny tail logs it.
- **grant flows:** an explicit grant → the flow is ACCEPTED (curl succeeds) + logs an allow event.
- **rule-delete blocks:** delete the grant → a NEW flow is DROPPED again.
- **nflog-full never alters packet fate:** flood the deny-log (a port scan); confirm packets still drop
  correctly (verdict unchanged) while log lines are dropped/aggregated (the kernel `log` is best-effort;
  overrun folds into the gap). Enforcement verdict is INDEPENDENT of log emission.
**PASS:** every S7.2 verdict holds byte-for-byte with logging on; only observability degrades under flood.
**SAVE:** per-sub-leg — the drop/accept evidence (pcap or curl-timeout) · the deny/allow events · the overrun-under-flood drop-count.

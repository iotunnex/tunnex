# S7.2 ‚Äî Zero Trust ENFORCEMENT ‚Äî decisions (decision-first)

EPIC 7 story 2. S7.1 shipped the model + the pure, deterministic compiler
(`policyspec.Compiled`) but nothing enforces it. **S7.2 makes it real on the wire:** push the compiled
artifact to the agent, program it into the gateway's nftables forward chain as **default-deny**, and
prove a real packet is DROPPED ‚Äî the load-bearing gate this whole epic exists for.

## Grounding (from S7.1 + the code sweep)

- **The compiler is done + tested** (`internal/enterprise/policy.Compile ‚Üí policyspec.Compiled`):
  per-node `{Mode, Mesh, Allow[]{SrcIP, DstCIDR, Protocol, PortLow, PortHigh}}`. `off ‚Üí Mesh=true`
  (legacy blanket mesh); `enforcing ‚Üí Mesh=false` + the grant set (empty = deny-all). Deterministic.
- **Enforcement point:** the gateway's nftables `forward` chain (`apps/node/internal/egress/
  egress_linux.go` `ruleset()`), today `policy drop` + a **blanket `iifname wg0 oifname wg0 accept`**
  (device‚Üîdevice) + `wg0‚Üí!wg0 accept` (egress). S7.2 replaces the blanket allows with compiled
  per-source‚Üíper-destination accepts when enforcing.
- **Push vehicle:** `reconcile.DesiredState` (apps/node) built by `nodes.Service.DesiredState()`
  (apps/api) and served over the mTLS agent channel; long-poll `Watch` + `nodepush.Hub.Notify` give
  the <5s revocation path. apps/api and apps/node are **separate modules** ‚Äî the artifact rides as
  JSON; apps/node needs a mirror of the `Compiled` shape (like it already mirrors `Peer`).
- **S7.1 ledgered obligations (must land here):** (1) `AffectedNodeIDs` gets a **direct test**;
  (2) **org-membership removal is the 4th recompile+push trigger** (with device revoke/create +
  group-membership change), all measured vs the **<5s** spec; (3) honor the F1 convention.

## Proposed design (LOCKED unless flagged OPEN)

### 1. Artifact on the wire
Add `Policy *policyspec.Compiled` (omitempty) to the desired-state payload. apps/api's
`nodes.Service.DesiredState()`: in the ENTERPRISE build, call the policy port's
`Compile`+`BuildSnapshot` for the node's org and attach the node's `Compiled`; in the OPEN build (nil
port) attach nothing ‚Üí agent keeps legacy behavior. apps/node mirrors the `Compiled`/`AllowEntry`
JSON shape in `reconcile`. **Determinism preserved** ‚Üí reconcile stays a steady-state no-op (the
existing dirty-check compares the whole desired state).

### 2. Enforcement in the forward chain (`egress_linux.go`)
The agent programs the forward chain from `Compiled`:
- **`Mesh=true` (off / open build / no policy):** exactly today's ruleset ‚Äî blanket `wg0‚Üîwg0 accept`
  + egress. **Zero behavior change** when Zero Trust is off.
- **`Mesh=false` (enforcing):** `policy drop`; `ct state established,related accept`; then **one accept
  per `AllowEntry`** (`ip saddr <SrcIP> ip daddr <DstCIDR> [ip protocol tcp/udp th dport <lo>-<hi>]
  accept`); egress (`wg0‚Üí!wg0`) is ALSO default-deny ‚Äî permitted only where an AllowEntry's `DstCIDR`
  covers the destination (a `0.0.0.0/0` resource = an egress grant). v4 and v6 entries land in the
  `ip`/`ip6` tables by address family. The masquerade rule stays (NATs allowed egress).
- **Atomic swap:** the whole tunnex forward ruleset is replaced in one `nft -f` transaction, so there
  is no window where the chain is half-open (same discipline as the kill-switch).

### 3. Fail-closed on apply error (enforcing)
If the nftables apply FAILS while enforcing, the agent keeps the **last-good deny-bearing** ruleset
(or, on first arm, a bare `policy drop`) and surfaces the error ‚Äî it never falls back to the blanket
mesh. Under Zero Trust, an apply failure denies, it does not open.

### 4. The <5s invalidation push (D4 + F1)
Policy mutations `Notify` the affected nodes so the agent re-fetches + re-applies within <5s. Triggers
(the compiler's inputs): **rule/resource/group CRUD, group-membership change, device revoke/create,
and org-MEMBERSHIP REMOVAL** (F1's 4th trigger ‚Äî a removed member's device must lose reachability just
as fast as a revoked device). `Service.AffectedNodeIDs` supplies the target set; **it gets a direct
test here** (S7.1-ledgered).

### 5. The proof (the DoD gate ‚Äî SUBSTITUTES do not satisfy)
On a real gateway + two spoke devices, org in `enforcing`:
- **default-deny on the wire:** device A ‚Üí device B with NO rule ‚áí **packet DROPPED** (ping/curl times
  out; pcap on the gateway shows the forward drop, no delivery). This is the load-bearing check.
- **grant works:** add a `groupA‚ÜígroupB` rule ‚áí A reaches B within <5s (no reconnect).
- **revocation latency:** revoke B's device OR remove B from the org ‚áí A loses B within <5s.
- **egress default-deny:** enforcing with no `0.0.0.0/0` grant ‚áí device internet BLOCKED; add the
  egress resource ‚áí internet works.
- **mode round-trip:** `off` mesh reachable ‚Üí `enforcing` denies ‚Üí back to `off` restores, each within
  <5s, no leak window.

## LOCKED decisions (signed off) + recorded conditions

**1 ‚Äî nftables shape = RULE-PER-ALLOW.** One accept per `AllowEntry`, whole ruleset rebuilt each apply.
- (1a) **Deterministic order:** rules are emitted in the compiler's SORTED order (S7.1's sortedness
  test guarantees the input; the ruleset builder MUST preserve it) so `nft list ruleset` bytes are
  stable and steady-state reconcile is a no-op ‚Äî same class as the `wg syncconf` idempotence lesson.
- (1b) **Auditability is the argument:** `nft list ruleset` IS the debugging story ‚Äî a readable rule
  list beats an invisible set-membership bug.
- (1c) **Sets/verdict-maps DEFERRED** with a named trigger: a MEASURED apply/reconcile cost at real
  rule counts (e.g. **>500 rules/node**), not speculation.

**2 ‚Äî egress under enforcing = DEFAULT-DENY**, and the full-tunnel interaction is resolved as **(a)**:
- Under enforcing a device reaches the internet ONLY via an explicit `0.0.0.0/0` (or narrower)
  resource grant. Implicit internet under "zero trust" would be a lie.
- **(a) The mode-enable API surface ENUMERATES affected full-tunnel devices** (count + names) so the
  S7.4 warn-and-confirm has real data and the S7.2-era API caller is told exactly what breaks ‚Äî
  **server still obeys** (S4.7 server-is-truth). **NOT (b)** auto-create an egress grant: that
  recreates implicit-allow by another name and would outlive its rationale.
- **(2-red) Deliberate-red:** a full-tunnel device's egress is DROPPED under enforcing-with-no-egress-
  grant (on the wire).
- **(2-coherence) Distinct policy-denied state:** a device can now have egress denied by POLICY on a
  gateway that is egress-CAPABLE ‚Äî this MUST NOT be conflated with `gateway_no_egress` (the gateway
  can't egress at all). Name the policy-denied state distinctly (e.g. `egress_policy_denied`) in the
  status/error surfaces; the two refusal paths stay separate.

**3 ‚Äî proof transport = pcap + `nft` drop-counter + app-level timeout** (meets the kill-switch bar),
with these obligations PINNED:
- (3a) **positive AND negative pair:** an allowed flow PASSES (pcap shows delivery) AND a non-granted
  flow is DROPPED (drop-counter increments, pcap shows zero delivery).
- (3b) **F1-class proof:** org-member removal ‚Üí their device-to-device reachability dies on the wire
  within the **<5s** spec, **timed**.
- (3c) **blanket-rule-deletion proof:** under enforcing, `nft list ruleset` contains **NO `wg0‚Üíwg0`
  blanket accept** (S7.1's structural guard, now proven on the box).
- (3d) **S3.7-pattern deliberate-red:** manually FLUSH the compiled chain ‚Üí reconcile RESTORES it
  within the interval (desired-state, never assumed).

**4 ‚Äî full ruleset REBUILD per apply**, atomic:
- (4a) **Atomic REPLACE primitive:** a single `nft -f` transaction whose file re-declares the whole
  `tunnex` table (`flush table ‚Ä¶ ` + full re-add IN THE SAME FILE = one kernel transaction) ‚Äî **never
  flush-then-separately-populate** (that has an empty-chain window = fail-open or total-drop). State
  the primitive in the code comment.
- (4b) **Fail-closed = previous ruleset STAYS on a failed apply** (the atomic swap gives this for
  free ‚Äî VERIFY, don't assume); the agent reports the apply failure up the STATUS channel and the
  control plane SURFACES it. A gateway silently running stale policy is a policy violation in slow
  motion ‚Äî **staleness must be visible**.
- (4c) **Idempotence:** same `Compiled` ‚Üí same ruleset bytes ‚Üí a steady-state no-op reconcile test
  (the reconcile-idempotence guard).

## Demonstrated-reds + build-time amendments (recorded as they happen)

- **DEMONSTRATED-RED (chunk-1, the hash-discipline pin earning its keep):** the first applied-status
  implementation hashed the RULESET TEXT ‚Äî which carries the node-local masquerade subnet line the
  control plane can never reproduce ‚Üí pushed-vs-applied would false-positive staleness PERMANENTLY.
  Fixed: hash = SHA-256 over the canonical `Compiled` JSON (`policyspec.CanonicalHash` ‚â°
  `nodepolicy.CanonicalHash`), pinned by **twin golden tests** (identical fixtures + expected hex in
  both modules ‚Äî the cross-module drift guard) + a decode-then-rehash wire round-trip test.
  ‚Üí Armed-Guards inventory candidate at story end.
- **STALENESS ALARM CONDITION (locked): N = 90 seconds.** `stale` = pushed hash ‚âÝ applied hash
  persisting **> 90s** (= 3 report intervals: the report loop runs every 30s, and a NORMAL apply
  window is push ‚Üí fetch ‚Üí kick-apply ‚âà ‚â§2s + up to one 30s report cycle before the new applied hash
  lands ‚Äî so a mismatch older than 90s cannot be a normal window). A transient mismatch inside 90s is
  NOT an alarm. **`policy_error` non-empty is surfaced IMMEDIATELY, regardless of hash state** ‚Äî an
  apply failure is never smoothed by the persistence window.
- **Both independence directions tested (chunk-1/2):** policy delivery does not wait on the WG
  converge (proven chunk-1), AND a policy/nftables apply failure does not block or abort the WG peer
  converge ‚Äî revocation still lands on the interface while the egress leg errors (proven by test;
  structural: `OnPolicy` has no error path into `runOnce`, and the egress apply runs in its own
  goroutine). A failure in one data-plane leg must never freeze the other.

## Recorded answer ‚Äî per-rule nftables counters (flow-log seam)

**Q (S7.2 review): can compiled rules carry per-rule nftables COUNTERS cheaply now, even if reporting
waits?** **A: yes, near-free.** `counter` is a native nftables statement ‚Äî appending it to each
`accept` line (`‚Ä¶ accept` ‚Üí `‚Ä¶ counter accept`) adds one kernel-maintained packet/byte counter per
rule, zero control-plane cost and negligible data-plane cost. Emitting it day-one costs one token in
`renderAllow`; RETROFITTING later means re-rendering + a version bump. The hard part is only the
REPORTING side (`nft -j list ruleset` ‚Üí per-rule counts, correlate to policy_rule ids, expose as
access/flow logs) ‚Äî that stays deferred. **Ledgered as a product candidate** (access/flow logs:
"rule X matched N times, last hit T"). **Recommendation: emit `counter` from day one** to reserve the
seam ‚Äî pending Pawan's go (not built speculatively). NB: `nft` counters reset on the atomic ruleset
replace, so per-rule cumulative counts need server-side accumulation across reports (a reporting-side
concern, not a blocker to emitting).

## Box-proof findings (folded after the on-the-wire run)

The 8-proof box run PASSED (default-deny drop, grants, no-blanket, resource+egress scoping,
revoke/offboard <5s + /32-absence, mode transitions both ways, fail-closed + kernel-rollback,
flush‚Üíreconcile). Three findings surfaced in AUXILIARY paths (the enforcement core was correct) and
were dispositioned:

- **#1 ‚Äî F1 push-target was on the wrong (unexposed) method ‚Üí FOLDED.** My cascade-FK + `PushOrgNodes`
  sat on `MembershipService.RemoveMember`, which has NO HTTP endpoint. The reachable offboarding is
  `DeactivateMember`, which pushed only `PushUserNodes` (the ex-member's OWN device-nodes) ‚Äî so on a
  MULTI-gateway org a node hosting another user's device that referenced the ex-member as a policy
  group-destination wouldn't be pushed (its /32 drains at the interval, not <5s). Fix: `Deactivate`/
  `ReactivateMember` now `PushOrgNodes` (org-wide); `RemoveMember` marked INTERNAL-ONLY with a note so
  a future endpoint-adder inherits the push. Guard: `TestDeactivatePushesOrgWideNotJustUserNodes` (TWO
  nodes, target's device on node1 only, asserts PushOrgNodes + node2 in the push set ‚Äî the test does
  NOT share the single-gateway blind spot). **F1-PART-2 LESSON: the model-layer /32-sweep was proven
  in S7.1, but the push-layer TARGETING was not ‚Äî "proven at one layer" ‚âÝ "proven end-to-end".**
- **#2 ‚Äî established non-granted flows drain via conntrack ‚Üí DOCUMENTED + DEFERRED.** `ct established,
  related accept` (the return-path guard) keeps in-flight non-granted flows alive after a restriction
  takes effect. **This covers BOTH cases: enabling `enforcing` AND deleting an existing grant** ‚Äî in
  either, flows established under the prior policy drain until idle; only NEW flows are blocked
  immediately. **Revocation/offboarding is unaffected** (wg peer removal tears down the tunnel, killing
  all flows). Ledgered: targeted conntrack-kill on grant change, **trigger = first customer/compliance
  need for immediate flow termination** (pairs naturally with the flow-logs candidate ‚Äî the counters
  already emitted). See PLAN.md.
- **#3 ‚Äî stale window measured from applied-hash-age ‚Üí FOLDED.** `policy_hash_since` (when the applied
  hash was first seen) made `PolicyStale` fire during EVERY normal push's 2‚Äì3s apply window (applied
  briefly ‚âÝ pushed, but `since` was old ‚Üí read >90s ‚Üí false alarm). Fix: the AGENT stamps
  `failingSince` = the mismatch ONSET (when apply FIRST starts failing, cleared on the next success)
  and reports it; `PolicyStale(now)` measures `now ‚àí failingSince`. A healthy node has no
  `failingSince` ‚Üí never stale; the 90s window measures the real failure duration. Guards:
  `TestFailingSinceMismatchOnset` (onset stamped once, cleared on recovery) + `TestPolicyStaleFrom
  MismatchOnset` (healthy=never; FALSE at onset+60s, TRUE at onset+95s ‚Äî the proof-7 shape).

## Story-end multi-finder review folds (after box-proof green ‚Äî green was never merge authorization)

The story-end review found **6 CONFIRMED fail-open / availability bugs** the box proof didn't exercise
(restart, policy-DB error, port_low NULL). Dispositioned + folded (fail-closed is never wrong for a
security boundary):

- **#1 ‚Äî half-set port range widened to all-ports ‚Üí FOLDED (fail-closed).** A grant with `port_low`
  set but `port_high` NULL (or an inverted range) rendered to a bare `ip protocol tcp` = ALL ports.
  Fix: `renderAllow` REFUSES a malformed range ‚Äî `return "", false` (skip the entry, no match), never
  widen. Validation additionally requires ports both-set-or-neither. Guard:
  `TestRenderAllowHalfSetPortRangeFailsClosed`.
- **#2 ‚Äî restart re-armed blanket mesh under enforcing ‚Üí FOLDED (three states, not two).** Cold start
  with no policy yet received decoded as `nil = mesh` = fail-OPEN blanket. Fix: an explicit
  `policyReceived` flag splits the states ‚Äî (a) policy received + mesh ‚Üí blanket; (b) received +
  enforcing ‚Üí grants; (c) **NEVER received ‚Üí deny-all regardless of mode** (fail-closed cold start,
  accepted including for off-mode orgs). This SPLITS the chunk-1 absent=Mesh decision: `nil` WITHIN a
  received policy still = mesh (unchanged); never-received ‚âÝ mesh. Guards:
  `TestNeverReceivedIsDenyAllNotMesh` (never-received = deny) + the absent=Mesh test asserts
  nil-in-received = mesh and does NOT assert never-received = mesh.
- **#3 ‚Äî policy DB error 500'd DesiredState, blocking peer revocation ‚Üí FOLDED (control-plane
  isolation).** A `CompiledForNode` error returned up `DesiredState`, failing the whole fetch ‚Äî so a
  policy-subsystem outage would freeze peer convergence / revocation. Fix: the policy error is caught;
  peers are STILL served, and the policy FAILS CLOSED (enforcing deny-all artifact, never nil = open
  mesh). Regression test = the control-plane TWIN of the node-side data-plane isolation test:
  `TestDesiredStatePolicyErrorServesPeersFailsClosed` (policy error ‚Üí peers still served + fail-closed
  policy, no error returned).
- **#4 ‚Äî verbose agent report truncated by the LimitReader ‚Üí FOLDED.** The status-channel body cap was
  4096 bytes; a verbose `policy_error` could be truncated mid-JSON and rejected, losing the alarm. Fix:
  agent bounds `policy_error` to 300 chars before send; server `LimitReader` raised to 16384.
- **#5/#7 ‚Äî staleness stored JSONB-only, never surfaced ‚Üí FOLDED (minimal API surface now, badge is
  S7.4).** The applied-policy status landed in the node `capabilities` JSONB but no API field read it,
  so a stale/desynced gateway was invisible. Fix (the API field is S7.2; the dashboard BADGE is
  S7.4): `nodes.Service.PolicyStatus(ctx, node, now) (stale, synced bool)` + two OpenAPI `Node` fields
  ‚Äî **`policy_stale`** (apply FAILING past the 90s window, from `failingSince`) and **`policy_synced`**
  (pushed-vs-applied hash compare ‚Äî restores the compare #3's fold dropped, so a NEVER-FETCHED gateway
  is catchable, not just a failing-apply one). `ListNodes` populates both. Guard:
  `TestPolicyStatusSyncedAndStale` (in-sync, silent-desync, failing-stale, open-build-never-stale).
- **#6 ‚Äî egress-arm failure conflated with policy staleness ‚Üí FOLDED.** A non-policy nftOS apply error
  (e.g. a mesh/off-mode ruleset failing) stamped `failingSince` and read as POLICY-stale. Fix:
  `applyAndTrack` only treats an ENFORCING policy's apply as policy-staleness; a non-policy apply
  failure clears the policy-staleness fields. Guard: `TestNonEnforcingApplyFailureIsNotPolicyStale`.

**Staleness model reconciled (supersedes the earlier "STALENESS ALARM CONDITION" note's surfacing
detail):** two INDEPENDENT signals, both surfaced on the `Node` API ‚Äî `policy_stale` = apply failing
> 90s (measured from `failingSince` ONSET, per #3-box); `policy_synced` = policy in force matches what
we'd push now (catches silent staleness / never-fetched). The dashboard warning BADGE that reads these
fields is S7.4.

**Perf follow-up ledger (#8/#9/#10 ‚Äî CORRECTNESS-NEUTRAL, deferred to an S7.2 perf pass):**
recompile-per-fetch (`CompiledForNode` recompiles on every DesiredState instead of caching by policy
version), no off-mode fast-path (off-mode orgs still walk the compile path to produce a mesh artifact),
redundant re-apply per fetch (identical `Compiled` re-renders + re-applies even when unchanged ‚Äî the
idempotence guard makes it a kernel no-op but it still burns a transaction). None change behavior; all
are throughput optimizations. Trigger = policy-fetch load becomes measurable. Recorded in PLAN.md.

## Second review pass (after the refold ‚Äî a security-critical refold gets re-reviewed)

The first refold's second review found 6 more defects (the first pass's fixes were correct but
incompletely executed, and the fail-closed posture had a wider blast radius than intended). Dispositioned
+ folded:

- **#3 (validation gap, completes fold #1) ‚Äî FOLDED.** `renderAllow` fails a half-set port range closed,
  but `validateResource` had NO both-or-neither check for tcp/udp ‚Äî so a `{tcp, port_low:8080}` grant
  returned 200 OK, stored, compiled, then got SILENTLY skipped at the gateway (a grant the API reported
  as created that never worked). Worse, the `renderAllow` comment CLAIMED the invariant existed
  (`validateResource requires both-or-neither`) ‚Äî a comment asserting a guarantee the code doesn't
  enforce. Fix: added the both-or-neither check to `validateResource`; rewrote the comment to state the
  renderer never TRUSTS upstream. Guard: `TestValidateResourcePortsBothOrNeither`.
- **#6 (protocol fail-open, symmetric to #1) ‚Äî FOLDED.** `renderAllow`'s switch matched only tcp/udp;
  `"any"` rendered all-protocol by ABSENCE of a case, and so did `""`/garbage ‚Äî a silent all-protocol
  widen. Fix: explicit `case "any"` (intended wide grant) + `default: return "", false` (fail closed).
  Guard: `TestRenderAllowUnknownProtocolFailsClosed`.
- **CONVENTION (recorded ‚Äî it has now bitten twice, port #1 + protocol #6):** the compiler/renderer
  **REFUSE any unknown or half-specified input; they never WIDEN on it.** `renderAllow` skips a malformed
  entry (‚Üí no match ‚Üí default-deny); `validateResource` rejects it at the API. Neither trusts the other
  as sufficient ‚Äî they are the two halves of one fail-closed invariant. **This is a review-checklist line
  for every new field added to `AllowEntry`:** does a malformed/unknown value fail closed at BOTH the API
  validation and the renderer?
- **#4 (false out-of-sync alarm) ‚Äî FOLDED.** `PolicyStatus` swallowed a transient `CompiledForNode`
  error into `pushed=""`, so a HEALTHY in-sync enforcing gateway reported `policy_synced=false` during
  any DB blip. A false alarm on a healthy system is the SAME class as the original #3 false-staleness ‚Äî
  it trains operators to ignore the signal, worse than no signal. Fix: on a compile error, report
  `synced=true` (unknown ‚âÝ desynced); a genuine desync still surfaces on the next successful compile.
  Guard: the #4 case in `TestPolicyStatusSyncedAndStale` + the batch's error case.
- **#5 (N+1 recompile on the ListNodes read path) ‚Äî FIXED (hoist, not ledgered).** `ListNodes` called
  `PolicyStatus` per node, each rebuilding the WHOLE org snapshot (‚âà6 queries + a compile). Materially
  different from the #8/#9/#10 steady-state agent-loop inefficiencies (user-invisible) ‚Äî this is a hot
  ADMIN read path. Fix: `nodes.Service.PolicyStatusForNodes` + `policy.Service.CompiledHashesForNodes`
  build the org snapshot ONCE and compute every node's hash from it (Compile already returns all nodes).
  Guard: `TestPolicyStatusForNodesBatchParityAndErrorNotDesync`.
- **#2 (fail-closed blast radius on off-mode orgs) ‚Äî SCOPED (earlier "regardless of mode" ruling was
  over-broad).** The #3 fold made `DesiredState` fail closed (deny-all) on ANY policy error, for EVERY
  org ‚Äî so a policy-subsystem DB blip blackholed OFF / open-build orgs that pre-S7.2 never touched that
  subsystem, imposing an availability dependency on a feature they don't use. Fix: scope by the org's
  mode (one cheap column already fetched for the pool CIDR) ‚Äî **fail closed if enforcing OR mode-unknown;
  serve mesh when OFF is confirmed.** The security property is preserved EXACTLY where it matters: an
  enforcing org never opens, and an unreadable org row still fails closed. Guard:
  `TestDesiredStatePolicyErrorScopedByMode` (off‚Üímesh, enforcing‚Üídeny-all, both serve peers). NB the NODE
  cold-start (#1) is NOT scopeable this way ‚Äî see the named AVAILABILITY LIMITATION in PLAN.md ‚Äî and is
  KEPT as accepted: the gateway cannot learn its mode without the CP, so deny-until-first-fetch is
  inherent; the alternative (mesh on cold start) would let a reboot-during-outage open an enforcing org.
